# 01. 메시지 보안 아키텍처 — 필터가 아니라 ChannelInterceptor

## 무엇을 / 왜

`web` 모듈을 공부했다면 보안 = "필터 체인"이라는 그림이 익숙할 것이다. 요청이 `SecurityFilterChain`을 위에서 아래로 통과하며 인증·인가가 일어난다. 그런데 메시징 보안에는 `Filter`가 없다. 대신 **`ChannelInterceptor`** 가 그 자리를 차지한다.

이유는 단순하다. WebSocket/STOMP 메시지는 `HttpServletRequest`로 흐르지 않는다. 클라이언트가 보낸 STOMP 프레임은 `Message<?>` 객체가 되어 `MessageChannel`(Spring Messaging의 채널 추상화)을 타고 흐른다. 서블릿 필터는 이 흐름에 손댈 수 없다. Spring Messaging이 채널 통과 지점에 제공하는 확장 훅이 바로 `ChannelInterceptor`이고, 이 모듈의 보안 로직은 전부 그 인터페이스 구현체로 작성된다.

이 장은 개별 인터셉터의 디테일에 들어가기 전에, **메시지 하나가 채널을 통과할 때 어떤 인터셉터들이 어떤 순서로 개입하는가**라는 전체 지도를 그린다.

## 핵심 타입 — ChannelInterceptor 확장 훅

Spring Messaging의 `ChannelInterceptor`는 메시지의 송수신 라이프사이클에 여러 콜백을 제공한다. 이 모듈이 사용하는 훅은 다음과 같다.

```
ChannelInterceptor (spring-messaging)
│
├── preSend(message, channel)              ── 메시지가 채널로 send 되기 직전
│       └ 인가 검증, CSRF 검증, 컨텍스트 적재가 여기서 일어난다
│
└── afterSendCompletion(...)               ── send 완료/실패 후 (정리)

ExecutorChannelInterceptor (preSend을 상속 + 핸들러 호출 경계 훅 추가)
│
├── beforeHandle(message, channel, handler) ── 핸들러(예: @MessageMapping)가 실제로
│       └ 호출되기 직전(보통 다른 스레드). 컨텍스트를 재적재
│
└── afterMessageHandled(...)               ── 핸들러 호출 후 (정리)
```

`preSend`는 "메시지가 채널에 올라타기 전"의 동기 지점이고, `beforeHandle`/`afterMessageHandled`는 "그 메시지를 실제로 처리하는 핸들러 호출 경계"다. 이 둘이 보통 **다른 스레드**라는 점이 02장에서 다룰 컨텍스트 전파 문제의 출발점이다.

이 모듈의 주요 인터셉터를 책임별로 묶으면 다음과 같다.

```
 분류            인터셉터                                        주 훅
 ───────────────────────────────────────────────────────────────────────────
 인증 전파       SecurityContextChannelInterceptor             preSend + beforeHandle
                SecurityContextPropagationChannelInterceptor   preSend + postReceive
 인가            AuthorizationChannelInterceptor                preSend
 CSRF            CsrfChannelInterceptor / XorCsrfChannelInterceptor  preSend
```

## 동작 흐름 — STOMP 메시지 한 건의 여정

브라우저가 이미 WebSocket으로 연결된 상태에서 `SEND /app/chat` STOMP 프레임을 보낸다고 하자. 이 메시지가 `clientInboundChannel`(클라이언트→서버 인바운드 채널)을 통과할 때 등록된 보안 인터셉터들이 순서대로 개입한다.

```
브라우저 ──STOMP SEND /app/chat──▶ WebSocket 트랜스포트
                                        │
                                        ▼  Message<?>로 변환
                            ┌───────────────────────────────────────┐
                            │   clientInboundChannel.send(message)   │
                            │                                        │
   (1) 인증 적재  preSend ─▶ │  SecurityContextChannelInterceptor     │
        헤더의 user(Authentication)를 꺼내 SecurityContextHolder에 set
                            │                                        │
   (2) CSRF      preSend ─▶ │  CsrfChannelInterceptor (CONNECT만)    │
        SEND이므로 통과(매처 불일치 시 그대로 반환)
                            │                                        │
   (3) 인가      preSend ─▶ │  AuthorizationChannelInterceptor       │
        "/app/chat 에 SEND할 권한이 있는가?" → 없으면 AccessDeniedException
                            │                                        │
                            └───────────────────────────────────────┘
                                        │ (인가 통과)
                                        ▼  (보통 다른 스레드로 디스패치)
                        beforeHandle ─▶ SecurityContextChannelInterceptor
                                        다시 컨텍스트 적재 (스레드가 바뀌었으므로)
                                        │
                                        ▼
                                @MessageMapping("/chat") 핸들러 실행
                                  @AuthenticationPrincipal 등 인자 해석
                                        │
                     afterMessageHandled ─▶ 컨텍스트 정리(cleanup)
```

핵심 관찰:

- **하나의 채널에 여러 인터셉터가 체인으로 등록**된다. HTTP 필터 체인과 형태는 비슷하지만, 실행 주체가 `MessageChannel`이다.
- 각 인터셉터의 `preSend`는 **변형된 메시지를 반환**할 수 있다(예: 인증 헤더를 덧붙인 새 메시지). 다음 인터셉터는 그 반환값을 받는다.
- 인가에 실패하면 `preSend`에서 예외를 던져 메시지가 채널에 올라타지 못하게 막는다 — 즉 "default deny"가 채널 진입 차단으로 구현된다(04장).

## 핵심 메서드 — preSend의 반환 계약

`ChannelInterceptor.preSend`의 시그니처는 `Message<?> preSend(Message<?> message, MessageChannel channel)`이다. 이 반환값 계약이 모듈 전체 설계를 관통한다.

- **그대로 통과**: 자기 책임 범위가 아니면 받은 메시지를 그대로 반환한다. 예) `CsrfChannelInterceptor`는 CONNECT가 아니면 `return message`로 즉시 빠진다.
- **메시지 변형**: `SecurityContextPropagationChannelInterceptor`는 `MessageBuilder.fromMessage(message).setHeader(...).build()`로 인증 헤더를 심은 **새 메시지**를 반환한다.
- **차단**: `AuthorizationChannelInterceptor`는 인가 실패 시 반환 대신 `AccessDeniedException`을 던진다. `null`을 반환하면 send가 취소된다는 Messaging 계약을 예외로 대체한 셈이다.

## 설계 포인트 / 확장점

- **단일 확장 지점 재사용** — 보안을 위한 새로운 인프라를 만들지 않고, Spring Messaging이 이미 가진 `ChannelInterceptor`에 전부 올라탔다. 덕분에 사용자는 `clientInboundChannel.interceptors(...)`에 인터셉터를 추가하는 익숙한 방식 그대로 보안을 켠다.
- **책임 분리** — 인증 전파/인가/CSRF가 각각 독립 인터셉터다. 필요 없는 책임은 등록하지 않으면 된다(예: CSRF 토큰을 안 쓰면 `CsrfChannelInterceptor`를 빼면 그만).
- **등록은 config 모듈이 담당** — 이 모듈은 인터셉터 클래스만 제공하고, 어떤 채널에 어떤 순서로 꽂을지는 `@EnableWebSocketSecurity` 등 상위 config DSL이 결정한다. 그래서 이 모듈 안에는 "체인 순서를 박아 둔 곳"이 없다.

## 정리

메시징 보안은 HTTP 필터 체인의 메시징 버전이다. 다만 주역이 `Filter`가 아니라 `ChannelInterceptor`이며, 무대는 `SecurityFilterChain`이 아니라 `MessageChannel`이다. 한 메시지는 `preSend`에서 인증 적재 → CSRF → 인가를 거쳐 채널에 올라타고, `beforeHandle`에서 컨텍스트가 재적재된 뒤 핸들러로 들어간다. 다음 장들은 이 흐름을 인터셉터별로 확대해서 본다.
