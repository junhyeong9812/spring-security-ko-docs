# 04. 메시지 인가 — AuthorizationChannelInterceptor와 매처 위임

## 무엇을 / 왜

인증 컨텍스트가 채워지고(02장) 매처로 메시지를 식별할 수 있게 됐으니(03장), 이제 본론인 인가다. 질문은 하나다 — **"이 `Authentication`이 이 `Message`를 채널에 올릴 자격이 있는가?"**

메시징 인가는 web 모듈의 인가와 동일한 `AuthorizationManager` 추상화를 재사용한다. 다만 인가의 진입점이 `AuthorizationFilter`가 아니라 `AuthorizationChannelInterceptor`이고, 인가 대상이 `HttpServletRequest`가 아니라 `Message<?>`라는 점만 다르다. 덕분에 `hasRole`, `permitAll`, `authenticated` 같은 친숙한 규칙이 그대로 통한다.

## 핵심 타입

```
access/intercept/
├── AuthorizationChannelInterceptor                 인가 진입점 (ChannelInterceptor)
│       preSend에서 AuthorizationManager.authorize() 호출, 실패 시 default deny
│
├── MessageMatcherDelegatingAuthorizationManager    매처→매니저 매핑 테이블 + Builder DSL
│       AuthorizationManager<Message<?>> 구현
│
└── MessageAuthorizationContext                     매칭된 Message + 추출 변수(03장)

access/expression/
├── MessageExpressionAuthorizationManager           SpEL 표현식 기반 인가 (7.1)
├── MessageAuthorizationContextSecurityExpressionHandler
├── DefaultMessageSecurityExpressionHandler
└── MessageSecurityExpressionRoot                    표현식 루트(hasRole 등 메서드 제공)
```

세 타입의 협력 관계:

```
AuthorizationChannelInterceptor
        │ has-a
        ▼
AuthorizationManager<Message<?>>
        │ (보통 구현체)
        ▼
MessageMatcherDelegatingAuthorizationManager
        │ 매처가 일치한 mapping의
        ▼
AuthorizationManager<MessageAuthorizationContext<?>>
   (hasRole/permitAll/authenticated/표현식 매니저 등)
```

## 동작 흐름 — 인가 결정의 전체 경로

`AuthorizationChannelInterceptor.preSend`가 출발점이다.

```
preSend(message, channel)
   │
   ├─ 1. authentication = SecurityContextHolder...getAuthentication()
   │        Supplier로 감싸 lazy 평가. null이면
   │        AuthenticationCredentialsNotFoundException
   │
   ├─ 2. result = preSendAuthorizationManager.authorize(authentication, message)
   │
   ├─ 3. eventPublisher.publishAuthorizationEvent(...)  (감사/이벤트 훅)
   │
   └─ 4. result == null || !result.isGranted()  → AccessDeniedException ("default deny")
            정상 → return message  (채널 진입 허용)
```

여기서 2번의 `authorize`로 위임받는 쪽이 `MessageMatcherDelegatingAuthorizationManager`다.

```
MessageMatcherDelegatingAuthorizationManager.authorize(authentication, message)
   │
   for (mapping : mappings) {          // 등록 순서대로
   │     matcher = mapping.getMessageMatcher()
   │     ctx = matcher.matcher(message)   // 03장: MatchResult → 변수 추출
   │     if (matched) {
   │         MessageAuthorizationContext authCtx =
   │             new MessageAuthorizationContext(message, 추출변수)
   │         return mapping.manager.authorize(authentication, authCtx)  // ★ 첫 매칭에서 결정
   │     }
   │ }
   └─ 어떤 매처도 안 맞으면 → null 반환 (abstain, 기권)
```

핵심은 **"첫 번째로 매칭된 규칙이 결정한다"**(first-match-wins)는 점이다. HTTP 인가의 규칙 순서 의미와 동일하다. 그리고 매칭되는 규칙이 하나도 없으면 `null`(기권)을 반환하는데, 이 `null`이 다시 `AuthorizationChannelInterceptor` 4번 단계에서 **deny로 귀결**된다. 즉 "정의하지 않은 메시지는 거부"가 두 클래스의 협력으로 구현된다.

`MessageAuthorizationContext`는 매칭된 메시지와 03장에서 추출한 경로 변수(`getVariables()`)를 함께 들고 다니는 단순 홀더다. 표현식 인가가 `#roomId` 같은 변수를 참조할 때 이 컨텍스트에서 값을 꺼낸다.

## 인가 규칙 작성 — Builder DSL

규칙 테이블은 `MessageMatcherDelegatingAuthorizationManager.builder()`로 만든다. 매처 선택 → 제약 지정을 체이닝한다.

```java
AuthorizationManager<Message<?>> manager =
    MessageMatcherDelegatingAuthorizationManager.builder()
        .nullDestMatcher().authenticated()                  // CONNECT 등은 인증만
        .simpSubscribeDestMatchers("/topic/admin/**").hasRole("ADMIN")
        .simpDestMatchers("/app/chat").authenticated()
        .anyMessage().denyAll()                             // 나머지 전부 거부
        .build();
```

매처 선택 메서드(`Builder` 측):

```
anyMessage()                 ANY_MESSAGE
nullDestMatcher()            destination 없는 프레임 (CONNECT/DISCONNECT...)
simpTypeMatchers(types...)   SimpMessageType 기준
simpDestMatchers(patterns)   목적지 패턴 (타입 무관)
simpMessageDestMatchers(...) SimpMessageType.MESSAGE + 목적지
simpSubscribeDestMatchers(.) SimpMessageType.SUBSCRIBE + 목적지
matchers(MessageMatcher...)  임의 매처 직접 지정
```

제약 지정 메서드(`Constraint` 측)는 web과 거의 동일하다 — `hasRole`, `hasAnyRole`, `hasAuthority`, `hasAnyAuthority`, `permitAll`, `denyAll`, `authenticated`, `fullyAuthenticated`, `rememberMe`, `anonymous`, 그리고 임의 매니저를 받는 `access(...)`. 이들은 내부적으로 core의 `AuthorityAuthorizationManager`, `AuthenticatedAuthorizationManager`, `SingleResultAuthorizationManager`로 위임한다. 즉 이 DSL은 **메시징 전용 인가 로직을 새로 만든 게 아니라**, 매처 선택과 core 인가 매니저를 연결하는 어댑터다.

빌더는 `ApplicationContextAware`라서, 컨텍스트에 `PathPatternMessageMatcher.Builder` 빈이 등록돼 있으면 `setApplicationContext`에서 그것을 받아 목적지 매처 생성에 사용한다(03장의 파서 교체가 여기로 연결된다).

## 표현식 기반 인가

`access(...)`에 임의의 `AuthorizationManager<MessageAuthorizationContext<?>>`를 넘길 수 있는데, 7.1에서 추가된 `MessageExpressionAuthorizationManager`가 SpEL 문자열을 그 매니저로 변환해 준다.

```java
.simpDestMatchers("/app/chat/{roomId}")
.access(MessageExpressionAuthorizationManager.withDefaults()
        .expression("hasRole('USER') and #roomId == authentication.name"))
```

평가 흐름:

```
authorize(authentication, MessageAuthorizationContext ctx)
   │
   ├─ expressionHandler.createEvaluationContext(authentication, ctx)
   │     · DefaultMessageSecurityExpressionHandler가
   │       MessageSecurityExpressionRoot를 루트로 세팅 (hasRole 등 제공)
   │     · MessageAuthorizationContextSecurityExpressionHandler가
   │       ctx.getVariables()의 경로 변수를 #roomId 처럼 변수로 주입
   │
   ├─ ExpressionUtils.evaluateAsBoolean(expression, ctx)
   │
   └─ new ExpressionAuthorizationDecision(granted, expression)  반환
```

여기서 03장의 경로 변수 추출이 빛을 발한다. `MessageAuthorizationContextSecurityExpressionHandler.createEvaluationContext`가 `variables`를 순회하며 SpEL 컨텍스트 변수로 등록하므로, 표현식에서 `#roomId`로 목적지 변수를 직접 참조할 수 있다. `MessageSecurityExpressionRoot`는 `SecurityExpressionRoot`를 상속해 `hasRole`, `isAuthenticated`, `permitAll` 등의 표현식 함수와 `message` 필드를 노출한다.

## 설계 포인트 / 확장점

- **default deny 이중 안전망** — 매처 미일치 시 매니저는 `null`(기권), 인터셉터는 `null`을 거부로 해석. "허용을 명시하지 않은 메시지는 차단"이 보장된다.
- **core 인가 추상화 완전 재사용** — `AuthorizationManager`/`AuthorizationResult`/`AuthorizationEventPublisher`를 그대로 쓴다. 메시징은 "인가 대상이 `Message`인 또 하나의 호출 지점"일 뿐이다.
- **이벤트 발행 훅** — `setAuthorizationEventPublisher(...)`로 인가 결정(허용/거부)을 이벤트로 발행해 감사 로깅·모니터링에 연결할 수 있다. 기본은 no-op.
- **SpEL 핸들러 교체** — `MessageExpressionAuthorizationManager.withSecurityExpressionHandler(...)`로 커스텀 표현식 핸들러를 주입해 표현식 함수를 확장할 수 있다.
- **`SecurityContextHolderStrategy` 주입** — 인터셉터가 인증을 어디서 읽을지 교체 가능.

## 정리

메시지 인가는 `AuthorizationChannelInterceptor.preSend`에서 시작해 `MessageMatcherDelegatingAuthorizationManager`로 위임된다. 후자는 등록된 매처를 순서대로 시도해 첫 매칭의 `AuthorizationManager`에게 결정을 맡기고(first-match-wins), 어디에도 안 맞으면 기권→거부로 떨어진다. 규칙은 빌더 DSL로 작성하며 내부적으로 core의 인가 매니저로 위임된다. 7.1의 `MessageExpressionAuthorizationManager`는 SpEL과 경로 변수를 엮어 더 세밀한 인가를 가능하게 한다.
