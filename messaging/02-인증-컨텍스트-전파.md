# 02. 인증 컨텍스트 전파 — 헤더와 SecurityContextHolder 사이

## 무엇을 / 왜

`SecurityContextHolder`는 기본적으로 **스레드 로컬**이다. HTTP에서는 한 요청을 한 스레드가 처음부터 끝까지 처리하니 문제가 없다. 그러나 메시징은 다르다.

- 핸드셰이크/CONNECT 시점에 인증된 사용자는 STOMP 세션의 `Principal`로 기억되고, 이후 메시지의 `simpUser` 헤더(`SimpMessageHeaderAccessor.USER_HEADER`)로 따라온다.
- 그런데 실제 `@MessageMapping` 핸들러는 **메시지를 처리하는 그 시점의 스레드**(보통 인바운드 채널의 별도 풀)에서 돌고, 그 스레드의 `SecurityContextHolder`는 비어 있다.

따라서 누군가 "메시지 헤더에 든 인증 정보를 → 핸들러 스레드의 `SecurityContextHolder`로 옮기고 → 처리가 끝나면 비워주는" 다리 역할을 해야 한다. 그게 `context` 패키지의 인터셉터들이다. 이 컨텍스트가 채워져 있어야 04장의 인가도, `@AuthenticationPrincipal` 해석도 동작한다.

## 핵심 타입

```
context/
├── SecurityContextChannelInterceptor             (가장 많이 쓰임)
│     헤더의 user → SecurityContextHolder.set, 처리 후 clear
│     ExecutorChannelInterceptor 구현 → preSend & beforeHandle 양쪽에서 적재
│
├── SecurityContextPropagationChannelInterceptor  (역방향)
│     현재 SecurityContext의 Authentication → 메시지 헤더에 심어 다른 스레드로 운반
│     (서버 내부에서 메시지를 다른 채널로 넘길 때 컨텍스트 유실 방지)
│
└── AuthenticationPrincipalArgumentResolver       (핸들러 인자 해석)
      @AuthenticationPrincipal 파라미터 ← Authentication.getPrincipal()
```

`handler/invocation/reactive` 패키지에는 같은 역할의 **리액티브 버전**(`AuthenticationPrincipalArgumentResolver`, `CurrentSecurityContextArgumentResolver`)이 따로 있다. 이쪽은 `ReactiveSecurityContextHolder`(Reactor Context 기반)에서 인증을 꺼내 `Mono`로 돌려준다.

## 동작 흐름 1 — SecurityContextChannelInterceptor의 적재와 정리

`SecurityContextChannelInterceptor`는 `ExecutorChannelInterceptor`를 구현한다. 즉 `preSend`(채널 진입)와 `beforeHandle`(핸들러 호출 직전) **두 지점 모두에서** 컨텍스트를 세팅한다. 두 지점이 다른 스레드일 수 있기 때문이다.

```
preSend(message) / beforeHandle(message)
        │
        ▼  setup(message)
  ┌──────────────────────────────────────────────────────────┐
  │ 1. 현재 SecurityContext를 originalContext 스택에 push       │
  │    (ThreadLocal<Stack<SecurityContext>>)                   │
  │ 2. message.getHeaders().get("simpUser") 로 user 추출       │
  │ 3. user가 Authentication이면 그대로, 아니면 anonymous       │
  │ 4. 새 빈 SecurityContext에 authentication 세팅             │
  │ 5. SecurityContextHolder.setContext(새 컨텍스트)           │
  └──────────────────────────────────────────────────────────┘
        │
        ▼  (핸들러 실행 / 다음 인터셉터)
        │
afterSendCompletion / afterMessageHandled
        │
        ▼  cleanup()
  ┌──────────────────────────────────────────────────────────┐
  │ 스택에서 직전 컨텍스트 pop                                  │
  │  - 비어있던 컨텍스트였으면 → clearContext (완전 정리)       │
  │  - 아니면 → 그 컨텍스트로 복원                              │
  └──────────────────────────────────────────────────────────┘
```

스택을 쓰는 이유는 **중첩 호출**(인터셉터가 재진입되거나 같은 스레드에서 메시지가 연쇄 처리되는 경우)에서 직전 상태를 정확히 되돌리기 위해서다. 단순 set/clear가 아니라 push/pop이라는 점이 포인트다.

`getAuthentication`의 처리도 주목할 만하다. 헤더 값이 `Authentication`이 아니면 던지지 않고 **익명 인증**(`AnonymousAuthenticationToken`, `ROLE_ANONYMOUS`)으로 대체한다. 그래서 인증되지 않은 연결도 컨텍스트는 "익명 사용자"로 채워져 인가 단계가 일관되게 동작한다.

```java
// SecurityContextChannelInterceptor.setup (발췌)
Object user = message.getHeaders().get(this.authenticationHeaderName); // 기본 "simpUser"
Authentication authentication = getAuthentication(user);  // 아니면 anonymous
SecurityContext context = this.securityContextHolderStrategy.createEmptyContext();
context.setAuthentication(authentication);
this.securityContextHolderStrategy.setContext(context);
```

## 동작 흐름 2 — Propagation 인터셉터의 역방향 운반

`SecurityContextPropagationChannelInterceptor`는 방향이 반대다. **이미 컨텍스트가 채워진 스레드에서** 메시지를 만들어 다른 채널/스레드로 넘길 때, 컨텍스트가 유실되지 않도록 인증 정보를 헤더에 실어 보낸다.

```
preSend  : 현재 SecurityContext의 Authentication(없으면 anonymous)을
           MessageBuilder로 헤더("simpUser")에 심어 새 메시지 반환
              │
              ▼  (메시지가 다른 스레드의 핸들러로 전달)
beforeHandle / postReceive : 헤더의 Authentication을 꺼내
           SecurityContextHolder에 적재 (setup, 위와 동일한 스택 방식)
              │
afterMessageHandled : cleanup
```

JavaDoc이 명시하듯 이 인터셉터는 **`SecurityContextChannelInterceptor`와 같은 채널에 함께 쓰면 안 된다**. 둘 다 같은 헤더/컨텍스트를 만지므로 충돌한다. 보통 서버 내부에서 인증 컨텍스트를 다른 실행 채널로 "전파"해야 하는 특수 상황에서만 쓴다.

## 동작 흐름 3 — @AuthenticationPrincipal 인자 해석

컨텍스트가 채워진 뒤 핸들러가 호출되면, `AuthenticationPrincipalArgumentResolver`가 `@AuthenticationPrincipal` 파라미터를 채운다.

```
@MessageMapping("/im")
void im(@AuthenticationPrincipal CustomUser user) { ... }
        │
        ▼  resolveArgument
  1. SecurityContextHolder.getContext().getAuthentication()  (null이면 null 반환)
  2. authentication.getPrincipal() 추출
  3. 애너테이션에 expression이 있으면 SpEL로 평가
       (rootObject = principal, #this = principal)
  4. 타입 불일치 시: errorOnInvalidType=true → ClassCastException
                     아니면 → null 반환
```

`findMethodAnnotation`은 `@AuthenticationPrincipal`을 직접 단 경우뿐 아니라, 그것을 **메타 애너테이션으로 가진 커스텀 애너테이션**(`@CurrentUser` 같은)도 `MergedAnnotations`로 찾아낸다. 리액티브 버전은 이 과정을 `ReactiveSecurityContextHolder.getContext()` 기반 `Mono` 체인으로 옮기고, 파라미터가 `Mono`/`Flux`면 `ReactiveAdapterRegistry`로 감싸 준다.

## 설계 포인트 / 확장점

- **`SecurityContextHolderStrategy` 주입** — 세 클래스 모두 `setSecurityContextHolderStrategy(...)`를 노출한다. 멀티테넌시나 비표준 컨텍스트 저장 전략을 쓰는 환경에서 교체 가능하다.
- **익명 인증 커스터마이즈** — `setAnonymousAuthentication(...)`으로 기본 `ROLE_ANONYMOUS` 토큰을 바꿀 수 있다.
- **헤더 이름 교체** — 생성자로 인증을 담은 헤더 이름을 지정한다. 기본은 STOMP 표준인 `simpUser`.
- **명령형 vs 리액티브 분리** — 같은 의미의 리졸버를 `context`(서블릿/명령형)와 `handler/invocation/reactive`(Reactor)로 나눠 두어, 두 실행 모델에 동일한 프로그래밍 모델(`@AuthenticationPrincipal`)을 제공한다.

## 정리

메시징에서 인증은 "메시지 헤더에 실려 다니는 데이터"이고, 핸들러가 `SecurityContextHolder`로 그것을 읽으려면 누군가 스레드 로컬에 옮겨 줘야 한다. `SecurityContextChannelInterceptor`가 `preSend`/`beforeHandle`에서 헤더→홀더로 적재하고 끝나면 비우며(스택 기반 push/pop), `@AuthenticationPrincipal` 리졸버가 그 컨텍스트에서 principal을 꺼내 핸들러 인자로 넘긴다. 이 다리가 놓여야 비로소 인가가 의미를 갖는다.
