# 01. 토큰 도메인과 검증

## 무엇을 / 왜

OAuth 2.0의 모든 흐름은 결국 "토큰"으로 수렴한다. 클라이언트는 access token으로 보호 자원에 접근하고, refresh token으로 만료된 access token을 재발급받고, device flow는 device/user code로 사용자를 끌어들인다. 이 챕터는 그 **토큰 값 객체 계층**과, 토큰의 유효성을 조립식으로 검사하는 **`OAuth2TokenValidator`** 를 다룬다.

핵심 설계 의도는 두 가지다. 첫째, 모든 토큰은 "값 + 발급시각 + 만료시각"이라는 **공통 골격**을 공유하므로 추상 베이스로 묶는다. 둘째, 토큰 종류마다 검증 규칙이 다르므로(만료, audience, issuer ...) 검증을 **합성 가능한 전략**으로 떼어 둔다.

## 핵심 타입

```
                 ┌──────────────────────────────┐
                 │  «interface» OAuth2Token     │  getTokenValue()
                 │   getIssuedAt() / getExpiresAt() (default null)
                 └──────────────┬───────────────┘
                                │ implements
                 ┌──────────────▼───────────────┐
                 │  AbstractOAuth2Token          │  불변, Serializable
                 │   tokenValue/issuedAt/expiresAt + equals/hashCode
                 └──────────────┬───────────────┘
            ┌──────────────┬────┴─────┬──────────────────┐
            ▼              ▼          ▼                  ▼
   OAuth2AccessToken  OAuth2Refresh  OidcIdToken     OAuth2DeviceCode /
   (+ TokenType,        Token        (05장,           OAuth2UserCode
     scopes)                          ClaimAccessor)
```

`OAuth2Token`은 최소 계약이다. `getTokenValue()`만 필수고, 발급/만료 시각은 `default` 메서드로 `null`을 허용한다 — 모든 토큰이 시각 정보를 갖지는 않기 때문이다.

```java
// OAuth2Token.java
public interface OAuth2Token {
    String getTokenValue();
    default @Nullable Instant getIssuedAt()  { return null; }
    default @Nullable Instant getExpiresAt() { return null; }
}
```

## AbstractOAuth2Token — 불변 + 자기검증

`AbstractOAuth2Token`은 공통 필드 3개를 `final`로 들고, 생성자에서 **불변식(invariant)** 을 강제한다.

```java
// AbstractOAuth2Token.java
protected AbstractOAuth2Token(String tokenValue, @Nullable Instant issuedAt, @Nullable Instant expiresAt) {
    Assert.hasText(tokenValue, "tokenValue cannot be empty");
    if (issuedAt != null && expiresAt != null) {
        Assert.isTrue(expiresAt.isAfter(issuedAt), "expiresAt must be after issuedAt");
    }
    ...
}
```

- 토큰 값은 비어 있을 수 없다.
- 둘 다 주어지면 만료는 발급 이후여야 한다 — "이미 만료된 채 발급된 토큰"이라는 모순 상태를 생성 시점에 차단한다.
- `equals`/`hashCode`는 세 필드 기반. 즉 토큰 객체는 **값 동등성**을 가진다.

이 불변성 덕분에 상위 모듈은 토큰을 캐시·세션에 안전하게 공유할 수 있다.

## OAuth2AccessToken — TokenType과 scope

access token은 베이스에 두 가지를 더한다: **토큰 타입**과 **scope 집합**.

```java
// OAuth2AccessToken.TokenType
public static final TokenType BEARER = new TokenType("Bearer");
public static final TokenType DPOP   = new TokenType("DPoP");   // since 6.5
```

`TokenType`은 enum이 아니라 **값 객체**다. 이유는 확장성 — `new TokenType("...")`로 표준 외 타입도 만들 수 있고, `equals`는 `equalsIgnoreCase`로 비교한다(HTTP에서 `Bearer`/`bearer`가 섞여 오기 때문). scope는 항상 `unmodifiableSet`으로 감싼다.

`OAuth2RefreshToken`은 scope/type 없이 베이스를 거의 그대로 쓰고, `OAuth2DeviceCode`/`OAuth2UserCode`(RFC 8628 device flow)는 device/user code 값을 토큰으로 모델링한 얇은 구현이다.

## 토큰 검증: OAuth2TokenValidator

토큰을 받았다고 끝이 아니다. "만료되지 않았나, audience가 맞나, issuer가 신뢰하는 곳인가"를 확인해야 한다. 이걸 **함수형 전략**으로 추상화한 것이 `OAuth2TokenValidator`다.

```java
@FunctionalInterface
public interface OAuth2TokenValidator<T extends OAuth2Token> {
    OAuth2TokenValidatorResult validate(T token);
}
```

반환 타입이 `boolean`이나 예외가 아니라 **`OAuth2TokenValidatorResult`** 인 점이 핵심이다. 검증은 "통과/실패"의 이분법이 아니라 **여러 에러를 누적**할 수 있어야 한다(예: 만료도 됐고 audience도 틀림). 결과는 에러 컬렉션을 감싼 불변 객체다.

```java
// OAuth2TokenValidatorResult.java
public static OAuth2TokenValidatorResult success();                 // NO_ERRORS 싱글턴
public static OAuth2TokenValidatorResult failure(OAuth2Error... e); // 비면 success로 축약
public boolean hasErrors();
public Collection<OAuth2Error> getErrors();
```

`success()`는 빈 결과 싱글턴(`NO_ERRORS`)을 재사용해 할당을 아낀다. `failure(...)`도 에러가 비어 있으면 같은 싱글턴으로 접는다.

### 합성: DelegatingOAuth2TokenValidator

개별 검증기(만료 검증기, audience 검증기 ...)를 **하나로 묶는** 컴포지트다.

```java
// DelegatingOAuth2TokenValidator.validate
Collection<OAuth2Error> errors = new ArrayList<>();
for (OAuth2TokenValidator<T> validator : this.tokenValidators) {
    errors.addAll(validator.validate(token).getErrors());   // 단락 평가 없이 전부 수집
}
return OAuth2TokenValidatorResult.failure(errors);
```

주목할 점: **단락 평가(short-circuit)하지 않는다.** 첫 실패에서 멈추지 않고 모든 검증기를 돌려 에러를 모은다. 덕분에 호출자는 "왜 거부됐는지"를 한 번에 받는다. 빈 errors면 `failure`가 내부에서 `success`로 접히므로 통과를 의미한다.

```
   OAuth2TokenValidator (조립 결과)
   ┌──────────────────────────────────────────┐
   │ DelegatingOAuth2TokenValidator            │
   │   ├─ JwtTimestampValidator   (만료/nbf)    │  ← oauth2-jose 쪽 구현
   │   ├─ JwtIssuerValidator      (iss)        │
   │   └─ JwtAudienceValidator    (aud)        │
   └──────────────────────────────────────────┘
        validate(token) ─▶ 모든 결과의 errors 합치기
                          ─▶ 빈 집합이면 success, 아니면 failure
```

`oauth2-core`에는 컴포지트와 결과 타입만 있고, 구체 검증기(JWT 타임스탬프 등)는 `oauth2-jose`/`oauth2-resource-server`가 제공한다. 이 모듈은 "검증을 어떻게 조립하느냐"의 **계약과 골격**만 책임진다.

## 설계 포인트 / 확장점

- **불변 값 객체 + 생성자 검증**: 토큰은 만들어지는 순간 유효한 상태임이 보장된다. 잘못된 시각 조합은 `IllegalArgumentException`으로 즉시 거부.
- **값 객체로서의 TokenType / Grant 타입 패턴**: enum 대신 `final class + public static 상수 + 공개 생성자`. 표준 상수는 제공하되 미래/벤더 확장을 막지 않는 Spring Security의 일관된 관용구다(`AuthorizationGrantType`, `ClientAuthenticationMethod`도 동일).
- **검증의 합성**: 새 규칙이 필요하면 `OAuth2TokenValidator` 하나를 구현해 `DelegatingOAuth2TokenValidator`에 추가하면 된다. 기존 코드 수정 없음(개방-폐쇄).
- **에러 누적**: `boolean` 대신 결과 객체를 반환해 다중 실패 사유를 보존.

## 정리

토큰 도메인은 `OAuth2Token` 계약 위에 `AbstractOAuth2Token`이 불변·자기검증 골격을 깔고, access/refresh/id/device 토큰이 각자의 추가 정보를 얹는 구조다. 검증은 `OAuth2TokenValidator`라는 함수형 전략과 `DelegatingOAuth2TokenValidator` 컴포지트로 조립되며, 결과는 실패 사유를 누적하는 `OAuth2TokenValidatorResult`로 표현된다. 상위 모듈은 이 골격에 구체 토큰과 구체 검증기를 끼워 넣는다.
