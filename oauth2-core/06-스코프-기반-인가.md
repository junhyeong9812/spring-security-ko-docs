# 06. 스코프 기반 인가

## 무엇을 / 왜

OAuth2에서 권한의 단위는 **scope**다. 토큰에 `scope: "read write"`가 있으면 그 토큰을 든 클라이언트는 read·write 작업을 할 수 있다. 리소스 서버는 보호 자원마다 "이 작업에는 어떤 scope가 필요한가"를 선언하고, 들어온 토큰의 scope와 대조해 허용/거부를 정한다.

이 챕터는 그 **scope → 권한 매핑과 인가 판단**을 표준화한 `authorization` 패키지를 다룬다. 핵심 질문은 "scope가 어떻게 Spring Security의 `GrantedAuthority`로 변환되어 기존 인가 인프라에 녹아드는가"다.

## 핵심 타입

```
   spring-security-core 인가 인프라
   AuthorizationManager<T>           AuthorityAuthorizationManager.hasAuthority("SCOPE_read")
   ReactiveAuthorizationManager<T>   AllAuthoritiesAuthorizationManager.hasAllAuthorities(...)
        ▲ 위에 OAuth2 의미를 입힘
        │
   ┌────┴─────────────────────────────────────────────┐
   │ OAuth2AuthorizationManagers   (서블릿, static)     │  hasScope/hasAnyScope/hasAllScopes
   │ OAuth2ReactiveAuthorizationManagers (리액티브)     │
   │ OAuth2AuthorizationManagerFactory (인스턴스, 7.1)  │  ← prefix 커스터마이즈 가능
   │   └ DefaultOAuth2AuthorizationManagerFactory       │
   └────────────────────────────────────────────────────┘
```

## 핵심 관용구: "SCOPE_" 접두사

리소스 서버가 토큰을 인증하면, 토큰의 각 scope는 `SCOPE_` 접두사가 붙은 `GrantedAuthority`가 된다. 즉 scope `read` → 권한 `SCOPE_read`. 이렇게 하면 OAuth2 scope가 Spring Security의 **일반 권한 메커니즘에 그대로 흡수**되어, 기존 `AuthorityAuthorizationManager`를 재사용할 수 있다.

`OAuth2AuthorizationManagers`는 이 매핑을 캡슐화한 정적 팩토리다.

```java
// OAuth2AuthorizationManagers.java
public static <T> AuthorizationManager<T> hasScope(String scope) {
    assertScope(scope);
    return AuthorityAuthorizationManager.hasAuthority("SCOPE_" + scope);
}

public static <T> AuthorizationManager<T> hasAnyScope(String... scopes) {
    String[] mappedScopes = new String[scopes.length];
    for (int i = 0; i < scopes.length; i++) {
        assertScope(scopes[i]);
        mappedScopes[i] = "SCOPE_" + scopes[i];
    }
    return AuthorityAuthorizationManager.hasAnyAuthority(mappedScopes);
}

public static <T> AuthorizationManager<T> hasAllScopes(String... scopes) {   // since 7.1
    ...
    return AllAuthoritiesAuthorizationManager.hasAllAuthorities(mappedScopes);
}
```

세 변형의 의미:
- `hasScope("read")` → `SCOPE_read` **하나** 요구.
- `hasAnyScope("read","write")` → 둘 중 **하나 이상**.
- `hasAllScopes("read","write")` → **모두**(7.1 신규, `AllAuthoritiesAuthorizationManager` 사용).

### 바보방지: assertScope

흔한 실수가 `hasScope("SCOPE_read")`처럼 접두사를 직접 붙이는 것이다. 그러면 `SCOPE_SCOPE_read`가 되어 절대 매칭 안 된다. 그래서 방어한다.

```java
private static void assertScope(String scope) {
    Assert.isTrue(!scope.startsWith("SCOPE_"),
        () -> scope + " should not start with SCOPE_ since SCOPE_ is automatically prepended ...");
}
```

접두사를 직접 다루고 싶으면 `AuthorityAuthorizationManager#hasAuthority`를 쓰라고 메시지가 안내한다.

## 동작 흐름 — 보호 엔드포인트 접근

```
요청 (+ Authorization: Bearer <token>)
   │
   ▼
BearerTokenAuthenticationFilter          (oauth2-resource-server)
   │  토큰 검증(01·02장) → JwtAuthenticationToken
   │  authorities = { SCOPE_read, SCOPE_profile, ... }   ← scope → SCOPE_* 매핑
   ▼
AuthorizationFilter
   │  AuthorizationManager.check(authentication, request)
   ▼
OAuth2AuthorizationManagers.hasScope("read")  ≡  hasAuthority("SCOPE_read")
   │  authentication.getAuthorities() 에 SCOPE_read 있나?
   ├─ 있음 → AuthorizationDecision(granted=true)  → 진행
   └─ 없음 → AuthorizationDeniedException
              → 403 (또는 insufficient_scope, 03장)
```

설정 쪽 사용 예:

```java
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/messages/**").access(OAuth2AuthorizationManagers.hasScope("messages.read"))
    .anyRequest().authenticated());
```

scope를 권한으로 환산해 두었기에, 인가 결정 자체는 `spring-security-core`의 기존 `AuthorizationManager`가 처리한다. 이 모듈은 "scope 어휘 → 권한 어휘" 번역만 책임진다.

## 리액티브 / 팩토리 변형

`OAuth2ReactiveAuthorizationManagers`는 동일한 매핑을 WebFlux용으로 제공한다 — 반환 타입이 `ReactiveAuthorizationManager<T>`이고 내부적으로 `AuthorityReactiveAuthorizationManager`를 쓸 뿐, `SCOPE_` 접두사 규칙과 `assertScope` 방어는 그대로다.

```java
// OAuth2ReactiveAuthorizationManagers.hasScope
return AuthorityReactiveAuthorizationManager.hasAuthority("SCOPE_" + scope);
```

### 접두사를 바꾸고 싶다면 — Factory (7.1)

정적 팩토리는 접두사가 `"SCOPE_"`로 고정이다. 일부 인가 서버는 다른 컨벤션(예: `scope:`)을 쓰거나, 접두사 없이 권한 매핑을 하기도 한다. 이를 위해 7.1에서 **인스턴스 기반 팩토리**가 추가됐다.

```java
// DefaultOAuth2AuthorizationManagerFactory.java
private String scopePrefix = "SCOPE_";   // setScopePrefix(...)로 변경 가능, 빈 문자열 허용

@Override
public AuthorizationManager<T> hasScope(String scope) {
    assertScope(scope);
    return this.authorizationManagerFactory.hasAuthority(this.scopePrefix + scope);
}
```

- `OAuth2AuthorizationManagerFactory<T>` 인터페이스가 `hasScope`/`hasAnyScope`/`hasAllScopes`를 default로 정의(기본은 정적 팩토리에 위임).
- `DefaultOAuth2AuthorizationManagerFactory`는 `setScopePrefix`로 접두사를 바꾸고, 실제 권한 검사는 주입받은 `AuthorizationManagerFactory<T>`(코어의 `DefaultAuthorizationManagerFactory`)에 위임한다.
- `assertScope`도 고정 `"SCOPE_"`가 아니라 설정된 `scopePrefix` 기준으로 검사해, 커스텀 접두사와 일관된다.

즉 정적 유틸(빠른 사용)과 인스턴스 팩토리(설정 가능)의 두 진입점을 제공한다.

## 설계 포인트 / 확장점

- **scope를 권한으로 흡수**: OAuth2 고유 개념(scope)을 Spring Security 공통 개념(`GrantedAuthority`)으로 번역해, 인가 결정 인프라를 통째로 재사용한다. 새 인가 엔진을 만들지 않는다.
- **세 가지 매처 + 단언 방어**: `hasScope`/`hasAnyScope`/`hasAllScopes`로 OR·AND 조합을 명시적으로 표현하고, 접두사 이중 부착을 컴파일 후 즉시 실패로 막는다.
- **서블릿·리액티브 대칭**: 동일 규칙을 `AuthorizationManager`와 `ReactiveAuthorizationManager` 양쪽에 제공.
- **접두사 커스터마이즈(7.1)**: 정적 팩토리의 하드코딩 한계를 인스턴스 팩토리로 푼다. `scopePrefix` 변경·위임 팩토리 주입이 확장점.

## 정리

스코프 기반 인가의 핵심은 단 하나의 변환이다 — OAuth2 scope `X`를 권한 `SCOPE_X`로 매핑해 Spring Security의 기존 권한 인가 인프라에 흘려보낸다. `OAuth2AuthorizationManagers`(서블릿)와 `OAuth2ReactiveAuthorizationManagers`(리액티브)는 `hasScope`/`hasAnyScope`/`hasAllScopes`로 이 매핑을 캡슐화하고, 접두사 이중 부착을 `assertScope`로 막는다. 7.1의 `OAuth2AuthorizationManagerFactory`/`DefaultOAuth2AuthorizationManagerFactory`는 고정 접두사 한계를 풀어, 인가 서버 컨벤션에 맞춰 prefix를 바꿀 수 있는 인스턴스 기반 확장점을 제공한다.
