# 02. 클레임 접근과 타입 변환

## 무엇을 / 왜

OAuth2/OIDC에서 토큰·UserInfo·인트로스펙션 응답은 모두 **JSON**으로 온다. 자바로 역직렬화하면 `Map<String, Object>`가 되는데, 같은 클레임이라도 제공자마다 타입이 제각각이다. `exp`가 `Long`(epoch)일 수도, `Instant`일 수도, 숫자 문자열일 수도 있다. `aud`는 문자열 하나일 수도, 배열일 수도 있다.

이 챕터는 이런 **느슨한 Map을 안전한 자바 타입으로 꺼내는 규약**을 다룬다. 두 축이다:
- `ClaimAccessor` — "Map에서 클레임을 타입 안전하게 읽는 인터페이스"
- `converter` 패키지(`ClaimConversionService`, `ClaimTypeConverter`) — "실제 타입 변환을 수행하는 엔진"

## 핵심 타입

```
   «interface» ClaimAccessor                getClaims(): Map<String,Object>
   │  getClaimAsString/Boolean/Instant/URL/Map/StringList(name)
   │      └── 내부적으로 ClaimConversionService 호출
   ├── OAuth2TokenIntrospectionClaimAccessor  (RFC 7662: active, sub, aud ...)
   └── StandardClaimAccessor → IdTokenClaimAccessor  (OIDC, 05장)

   converter 패키지
   ┌────────────────────────────────────────────────┐
   │ ClaimConversionService (GenericConversionService)│  공유 싱글턴
   │   ├─ ObjectToStringConverter                     │
   │   ├─ ObjectToBooleanConverter                    │
   │   ├─ ObjectToInstantConverter                    │
   │   ├─ ObjectToURLConverter                        │
   │   ├─ ObjectToListStringConverter                 │
   │   └─ ObjectToMapStringObjectConverter            │
   └────────────────────────────────────────────────┘
   ClaimTypeConverter  ── 클레임 이름별 Converter 맵을 적용해 Map 통째로 정규화
```

## ClaimAccessor — 타입 안전 게터의 집합

`ClaimAccessor`는 단 하나의 추상 메서드 `getClaims()`만 요구하고, 나머지는 전부 `default` 메서드다. 즉 "클레임 Map만 내놓으면, 안전한 타입 변환은 인터페이스가 해준다."

```java
// ClaimAccessor.java
Map<String, Object> getClaims();

default boolean hasClaim(String claim) { return getClaims().containsKey(claim); }

default <T> @Nullable T getClaim(String claim) {        // 변환 없이 캐스팅
    return !hasClaim(claim) ? null : (T) getClaims().get(claim);
}

default @Nullable Instant getClaimAsInstant(String claim) {
    ...
    Instant convertedValue = ClaimConversionService.getSharedInstance()
            .convert(claimValue, Instant.class);
    Assert.notNull(convertedValue, () -> "Unable to convert claim '" + claim + "' ... to Instant.");
    return convertedValue;
}
```

여기서 두 갈래를 구분하자:
- `getClaim(name)`은 **변환 없이 그대로 캐스팅** — 타입이 확실할 때 쓴다.
- `getClaimAsX(name)`은 **`ClaimConversionService`를 거쳐 변환** — 제공자 편차를 흡수한다. 변환 실패는 친절한 메시지의 `IllegalArgumentException`으로 즉시 드러난다.

`OidcIdToken`, `OidcUserInfo`, `OAuth2TokenIntrospectionClaimAccessor` 구현체는 이 인터페이스 덕에 `getSubject()`, `getExpiresAt()` 같은 의미 있는 게터를 **단 한 줄**로 정의한다.

```java
// IdTokenClaimAccessor (05장에서 자세히)
default @Nullable Instant getExpiresAt() { return getClaimAsInstant(IdTokenClaimNames.EXP); }
```

### 인트로스펙션 클레임 (RFC 7662)

`OAuth2TokenIntrospectionClaimAccessor`도 `ClaimAccessor`를 상속해 인트로스펙션 응답 전용 게터를 default로 제공한다.

```java
default boolean isActive() {
    return Boolean.TRUE.equals(getClaimAsBoolean(OAuth2TokenIntrospectionClaimNames.ACTIVE));
}
default @Nullable List<String> getScopes() { return getClaimAsStringList(...SCOPE); }
```

`active`는 인트로스펙션의 핵심 — 토큰이 살아있는지를 인가 서버가 알려주는 값이다. 리소스 서버(`oauth2-resource-server`)의 opaque token 검증이 이 접근자를 사용한다.

## ClaimConversionService — 변환 엔진

`getClaimAsX`가 실제로 부르는 곳이다. Spring의 `GenericConversionService`를 상속하고, OAuth2 클레임에 자주 필요한 변환기 6종을 등록한 **공유 싱글턴**이다.

```java
ClaimConversionService.getSharedInstance().convert(value, Instant.class);
```

등록된 변환기들이 흔한 편차를 흡수한다:
- `ObjectToInstantConverter` — `Long`(epoch seconds), `Instant`, `Date`, 숫자 문자열 → `Instant`
- `ObjectToListStringConverter` — 단일 값 또는 컬렉션 → `List<String>` (예: `aud`)
- `ObjectToMapStringObjectConverter`, `ObjectToURLConverter`, `ObjectToBooleanConverter`, `ObjectToStringConverter`

호출자는 어떤 형태로 왔든 항상 동일한 자바 타입을 받는다.

## ClaimTypeConverter — Map 통째 정규화

`ClaimAccessor`가 "읽을 때마다 변환"이라면, `ClaimTypeConverter`는 **한 번에 Map 전체를 표준 타입으로 정규화**하는 전처리기다. 클레임 이름별 변환기 맵을 받아 적용한다.

```java
// ClaimTypeConverter.convert
Map<String, Object> result = new HashMap<>(claims);
this.claimTypeConverters.forEach((claimName, typeConverter) -> {
    if (claims.containsKey(claimName)) {
        Object mappedClaim = typeConverter.convert(claims.get(claimName));
        if (mappedClaim != null) {
            result.put(claimName, mappedClaim);     // 변환 성공분만 덮어씀
        }
    }
});
return result;
```

동작 흐름:

```
   원본 claims (JSON 역직렬화 결과, 타입 뒤죽박죽)
        │
        ▼   claimName → Converter 맵 적용
   ┌──────────────────────────────────────┐
   │ "exp" → ObjectToInstantConverter      │
   │ "aud" → ObjectToListStringConverter   │
   │ "iat" → ObjectToInstantConverter      │
   └──────────────────────────────────────┘
        │  (변환 실패/없는 클레임은 원본 유지)
        ▼
   정규화된 claims (자바 타입 일관)
        │
        ▼  OidcIdToken / OAuth2User 등으로 래핑
```

`oauth2-client`의 ID 토큰 디코딩, UserInfo 매핑이 이 변환기로 클레임을 미리 다듬는다. `null` 변환 결과는 무시하고 원본을 유지하는 점이 안전장치다 — 변환 실패가 클레임 손실로 이어지지 않는다.

## 설계 포인트 / 확장점

- **default 메서드로 믹스인**: `ClaimAccessor`를 구현하면 수십 개의 타입 안전 게터를 공짜로 얻는다. 단일 추상 메서드만 채우면 된다.
- **공유 싱글턴 변환기**: `getSharedInstance()`로 무상태 변환 엔진을 재사용 — 인스턴스 폭증을 막는다.
- **두 가지 변환 시점**: 읽기 시점 변환(`getClaimAsX`) vs 일괄 전처리(`ClaimTypeConverter`). 후자는 클레임별 커스텀 변환기를 주입할 수 있는 확장점이다.
- **실패를 숨기지 않음**: `getClaimAsX`는 변환 불가 시 명확한 예외를 던지고, `ClaimTypeConverter`는 변환 불가 시 원본을 보존한다. 용도에 맞는 두 전략.

## 정리

클레임 처리는 "느슨한 JSON Map → 안전한 자바 타입"이라는 한 문제를 두 층위에서 푼다. `ClaimAccessor`는 읽을 때마다 `ClaimConversionService`로 변환해 의미 있는 게터를 default로 제공하고, `ClaimTypeConverter`는 클레임 Map 전체를 미리 정규화한다. 덕분에 `OidcIdToken`·`OAuth2User`·인트로스펙션 응답 같은 상위 타입은 변환 로직을 다시 짤 필요 없이 클레임 Map만 들고 있으면 된다.
