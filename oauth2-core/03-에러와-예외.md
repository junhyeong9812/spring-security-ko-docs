# 03. 에러와 예외

## 무엇을 / 왜

OAuth 2.0은 실패를 표준화한다. RFC 6749는 `invalid_request`, `invalid_grant`, `access_denied` 같은 **에러 코드**와, `error`/`error_description`/`error_uri` 세 필드로 된 **에러 응답 형식**을 못 박았다. 클라이언트와 리소스 서버는 이 코드를 보고 행동을 결정한다(재시도? 재인증? 거부?).

이 챕터는 그 에러를 모델링하는 `OAuth2Error`, 표준 코드 모음 `OAuth2ErrorCodes`, 그리고 이를 예외로 던지는 **두 갈래 예외 계층**을 다룬다. "왜 두 갈래인가"가 이 챕터의 핵심 질문이다.

## 핵심 타입

```
   OAuth2Error  (값 객체, Serializable)
     errorCode (필수) / description / uri
        ▲
        │ 담는다
        │
   ┌────┴───────────────────────────────────────────────┐
   │                                                     │
   OAuth2AuthenticationException        OAuth2AuthorizationException
   extends AuthenticationException      extends RuntimeException
   (Spring Security 인증 예외 계열)      (일반 런타임 예외)
        │                                     │
   인증 필터/프로바이더가 처리           클라이언트 측 흐름 실패
   → 401 + WWW-Authenticate              → 토큰 요청/교환 중 오류
```

## OAuth2Error — 에러 값 객체

에러 응답 한 건을 그대로 표현하는 불변 객체다. 최소 보장은 **에러 코드 하나**.

```java
// OAuth2Error.java
public OAuth2Error(String errorCode) { this(errorCode, null, null); }

public OAuth2Error(String errorCode, @Nullable String description, @Nullable String uri) {
    Assert.hasText(errorCode, "errorCode cannot be empty");
    ...
}

@Override
public String toString() {
    return "[" + getErrorCode() + "] " + (getDescription() != null ? getDescription() : "");
}
```

세 필드는 RFC 6749 §5.2의 에러 응답 본문(`error`, `error_description`, `error_uri`)과 1:1 대응한다. `description`/`uri`는 선택이라 `null` 허용. `toString()`이 `[invalid_token] The token expired` 형태로 로그·메시지에 바로 쓰이도록 만들어졌다.

### OAuth2ErrorCodes — 표준 코드 사전

`OAuth2Error`에 넣을 표준 문자열을 상수로 모았다. 직접 타이핑하다 오타 내지 말라는 것.

```java
// OAuth2ErrorCodes.java (발췌)
INVALID_REQUEST, UNAUTHORIZED_CLIENT, ACCESS_DENIED,
UNSUPPORTED_RESPONSE_TYPE, INVALID_SCOPE, INSUFFICIENT_SCOPE,  // RFC 6750
INVALID_TOKEN, SERVER_ERROR, TEMPORARILY_UNAVAILABLE,
INVALID_CLIENT, INVALID_GRANT, UNSUPPORTED_GRANT_TYPE,
UNSUPPORTED_TOKEN_TYPE, INVALID_REDIRECT_URI,
INVALID_DPOP_PROOF                                             // DPoP 확장
```

`insufficient_scope`/`invalid_token`은 리소스 서버가 `WWW-Authenticate` 헤더에 실어 보내는 Bearer 토큰 에러(RFC 6750)이고, `invalid_grant`/`invalid_client`는 토큰 엔드포인트가 클라이언트에게 돌려주는 에러다. 모두 한 사전에 모여 있다.

## 두 갈래 예외 — 왜 나뉘는가

같은 `OAuth2Error`를 담지만, 던져지는 **맥락이 다르다.**

### OAuth2AuthenticationException — 인증 실패

`AuthenticationException`을 상속한다. 즉 **Spring Security의 인증 파이프라인이 인지하는 예외**다. 인증 필터가 이걸 잡으면 `AuthenticationEntryPoint`가 동작해 401 응답을 만든다.

```java
// OAuth2AuthenticationException.java
public class OAuth2AuthenticationException extends AuthenticationException {
    private final OAuth2Error error;

    public OAuth2AuthenticationException(String errorCode) { this(new OAuth2Error(errorCode)); }
    public OAuth2AuthenticationException(OAuth2Error error) { this(error, error.getDescription()); }
    public OAuth2AuthenticationException(OAuth2Error error, @Nullable String message, @Nullable Throwable cause) {
        super(message);
        Assert.notNull(error, "error cannot be null");
        this.error = error;
        if (cause != null) { initCause(cause); }
    }
    public OAuth2Error getError() { return this.error; }
}
```

**언제**: 리소스 서버가 토큰을 거부할 때(만료·서명 불일치), OIDC 로그인이 ID 토큰 검증에 실패할 때, 클라이언트 인증이 깨질 때. JavaDoc이 시나리오를 나열한다 — 누락 파라미터, 잘못된 client_id, 잘못된 redirect_uri, 잘못된 scope, 거부된 grant 등.

### OAuth2AuthorizationException — 인가 흐름 오류

이쪽은 평범한 `RuntimeException`이다. 인증 파이프라인의 특수 처리를 받지 않는다.

```java
// OAuth2AuthorizationException.java
public class OAuth2AuthorizationException extends RuntimeException {
    private final OAuth2Error error;
    public OAuth2AuthorizationException(OAuth2Error error) { this(error, error.toString()); }
    ...
    public OAuth2Error getError() { return this.error; }
}
```

**언제**: 클라이언트가 인가 서버와 **프로토콜 대화를 진행하는 도중** 생기는 실패. 예를 들어 토큰 엔드포인트가 `invalid_grant`를 돌려줬거나, 토큰 응답을 파싱하지 못했거나, redirect로 `error=access_denied`가 돌아왔을 때. 이건 "현재 요청의 사용자 인증이 실패"한 게 아니라 "OAuth2 절차 자체가 실패"한 것이라, 인증 예외가 아니라 일반 런타임 예외로 표현한다.

```
  ┌─ 보호 자원 접근 시 토큰이 나쁨 ────────────────┐
  │  OAuth2AuthenticationException                │  → 인증 실패로 취급
  │  (extends AuthenticationException)            │  → EntryPoint → 401
  └───────────────────────────────────────────────┘

  ┌─ 클라이언트가 토큰을 받아오는 절차가 깨짐 ──────┐
  │  OAuth2AuthorizationException                 │  → 일반 런타임 오류
  │  (extends RuntimeException)                   │  → 흐름 중단/로깅/표시
  └───────────────────────────────────────────────┘
```

두 예외 모두 `getError()`로 동일한 `OAuth2Error`를 노출하므로, 잡는 쪽은 에러 코드/설명에 일관되게 접근할 수 있다. 차이는 "Spring Security 인증 메커니즘이 개입하느냐"뿐이다.

## 동작 흐름 — 리소스 서버 토큰 거부 예시

```
요청(+ 만료된 Bearer 토큰)
   │
   ▼
BearerTokenAuthenticationFilter        (oauth2-resource-server)
   │ authenticate(...)
   ▼
JwtAuthenticationProvider
   │ OAuth2TokenValidator.validate(jwt)  →  OAuth2TokenValidatorResult(errors)  (01장)
   │ errors.hasErrors() → true
   ▼
throw new OAuth2AuthenticationException(
        new OAuth2Error(OAuth2ErrorCodes.INVALID_TOKEN, "An error occurred ...", null))
   │
   ▼
AuthenticationEntryPoint
   → 401 + WWW-Authenticate: Bearer error="invalid_token", error_description="..."
```

`OAuth2Error`의 세 필드가 그대로 `WWW-Authenticate` 헤더 파라미터로 직렬화된다. 검증 결과(`OAuth2TokenValidatorResult`)의 에러가 예외의 `OAuth2Error`로, 다시 HTTP 응답으로 흐르는 일관된 경로다.

## 설계 포인트 / 확장점

- **에러를 값 객체로 1급화**: 예외에 문자열만 담지 않고 `OAuth2Error`를 담아, 코드·설명·URI를 구조적으로 보존한다. 직렬화(HTTP 응답)와 로깅 모두에 쓰인다.
- **상속 계층으로 의미 분리**: 같은 데이터(`OAuth2Error`)라도 `AuthenticationException` 계열이냐 `RuntimeException`이냐로 "인증 실패 vs 절차 실패"를 타입으로 구분. catch 지점과 처리기가 자연히 갈린다.
- **표준 코드 상수화**: 오타·비표준 코드를 막고, RFC ↔ 코드 매핑을 한 파일에서 관리.

## 정리

OAuth2의 실패는 `OAuth2Error`(코드+설명+URI) 값 객체로 모델링되고, 표준 코드는 `OAuth2ErrorCodes`에 박제된다. 이 에러는 두 예외로 던져지는데, 토큰/로그인 인증 실패는 `AuthenticationException`을 상속한 `OAuth2AuthenticationException`(인증 파이프라인이 처리, 401 유발)으로, 클라이언트의 토큰 획득 절차 실패는 일반 `RuntimeException`인 `OAuth2AuthorizationException`으로 표현된다. 둘 다 `getError()`로 같은 에러 객체를 노출해 처리 코드를 단순하게 유지한다.
