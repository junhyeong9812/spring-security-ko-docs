# 04. 프로토콜 상수와 엔드포인트 메시지

## 무엇을 / 왜

OAuth 2.0은 HTTP 위에서 정해진 단어로 대화한다. 인가 요청에는 `response_type=code&client_id=...&scope=...`가 실리고, 토큰 응답에는 `access_token`, `token_type`, `expires_in`이 담긴다. 이 챕터는 두 가지를 다룬다:

1. **프로토콜 상수** — `grant_type`, `client_id` 같은 표준 파라미터 이름과 grant/응답 타입 값. (오타 방지 + RFC 박제)
2. **엔드포인트 메시지** — 인가 요청/응답·토큰 응답을 표현하는 값 객체와, 이들을 HTTP/JSON으로 직렬화하는 변환기.

`oauth2-core`에는 엔드포인트(필터)가 없지만, 엔드포인트를 오가는 **메시지의 모양**은 여기서 정의한다.

## 핵심 타입 지도

```
[상수 — 값 객체형 enum 대용]            [상수 — 문자열 모음]
 AuthorizationGrantType                 OAuth2ParameterNames   (grant_type, client_id, code, scope, state,
   AUTHORIZATION_CODE / REFRESH_TOKEN     access_token, token_type, expires_in, refresh_token, error, ...)
   CLIENT_CREDENTIALS / JWT_BEARER      PkceParameterNames     (code_challenge, code_verifier, ...)
   DEVICE_CODE / TOKEN_EXCHANGE         OidcParameterNames     (id_token, nonce)
 ClientAuthenticationMethod
   CLIENT_SECRET_BASIC/POST/JWT ...     OAuth2AuthorizationResponseType.CODE
 AuthenticationMethod
   HEADER / FORM / QUERY                (Bearer 토큰을 어디에 실을지)

[엔드포인트 메시지 — 인가 코드 그랜트의 왕복]
 OAuth2AuthorizationRequest  ──(브라우저 redirect)──▶  인가 서버
 OAuth2AuthorizationResponse ◀─(redirect_uri 콜백)──   (code 또는 error)
 OAuth2AuthorizationExchange = Request + Response 묶음
 OAuth2AccessTokenResponse   ◀─(토큰 엔드포인트 JSON)── access/refresh token

[직렬화]
 http/converter/  : Spring MVC HttpMessageConverter
 web/reactive/function/ : WebClient BodyExtractor
 endpoint/Default*Converter : Map ↔ OAuth2AccessTokenResponse
```

## 상수: 값 객체 패턴의 일관성

`AuthorizationGrantType`, `ClientAuthenticationMethod`, `AuthenticationMethod`, `OAuth2AuthorizationResponseType`는 모두 같은 관용구를 따른다 — **enum이 아닌 `final class` + `public static` 표준 상수 + 공개 생성자**.

```java
// AuthorizationGrantType.java
public static final AuthorizationGrantType AUTHORIZATION_CODE = new AuthorizationGrantType("authorization_code");
public static final AuthorizationGrantType DEVICE_CODE       =
        new AuthorizationGrantType("urn:ietf:params:oauth:grant-type:device_code");
public AuthorizationGrantType(String value) { Assert.hasText(value, ...); this.value = value; }
```

왜 enum이 아닌가? OAuth2는 `urn:ietf:params:oauth:grant-type:jwt-bearer` 같은 **확장 grant**를 허용한다. enum이면 새 값을 못 만든다. 값 객체면 표준 상수의 편의와 확장 자유를 동시에 얻는다. `equals`/`hashCode`는 내부 문자열 기반이라 상수든 직접 생성이든 값이 같으면 동등하다.

`ClientAuthenticationMethod`는 한 발 더 나아가 `valueOf(String)` 팩토리를 제공한다 — 알려진 상수면 그 싱글턴을, 아니면 새 객체를 돌려준다.

```java
public static ClientAuthenticationMethod valueOf(String method) {
    for (ClientAuthenticationMethod m : methods()) { if (m.getValue().equals(method)) return m; }
    return new ClientAuthenticationMethod(method);   // 비표준도 수용
}
```

`AuthenticationMethod`(`HEADER`/`FORM`/`QUERY`)는 리소스 서버가 **Bearer 토큰을 요청 어디에서 읽을지**를 가리킨다.

### 파라미터 이름 사전

`OAuth2ParameterNames`는 와이어에 실리는 모든 표준 키를 상수로 모은다 — `grant_type`, `client_id`, `client_secret`, `code`, `redirect_uri`, `scope`, `state`, `access_token`, `token_type`, `expires_in`, `refresh_token`, `error`, `error_description`, `error_uri`, 그리고 device flow(`device_code`, `user_code`)·token exchange(`subject_token`, `audience` ...)까지. `PkceParameterNames`는 PKCE(`code_challenge`, `code_verifier`), `OidcParameterNames`는 OIDC(`id_token`, `nonce`)를 따로 담는다. 빌더가 쿼리스트링을 조립할 때 이 상수를 키로 쓴다(아래 참조).

## 엔드포인트 메시지: 인가 코드 그랜트의 왕복

가장 대표적인 흐름인 authorization code grant를 메시지 관점에서 따라가 보자.

```
사용자 브라우저        클라이언트(oauth2-client)              인가 서버
     │                       │                                  │
     │   보호 페이지 요청     │                                  │
     │──────────────────────▶│  OAuth2AuthorizationRequest 생성 │
     │                       │  .authorizationCode()...build()  │
     │  302 → getAuthorizationRequestUri()                      │
     │◀──────────────────────│                                  │
     │   GET /authorize?response_type=code&client_id=...&scope=…&state=…
     │─────────────────────────────────────────────────────────▶│
     │                       │       (사용자 로그인/동의)        │
     │  302 → redirect_uri?code=AUTH_CODE&state=…                │
     │◀─────────────────────────────────────────────────────────│
     │──────────────────────▶│  OAuth2AuthorizationResponse     │
     │                       │  .success(code) / .error(...)     │
     │                       │  (Request+Response = Exchange)    │
     │                       │  POST /token (code 교환) ────────▶│
     │                       │  ◀── JSON: OAuth2AccessTokenResponse
```

### OAuth2AuthorizationRequest — 나가는 요청

빌더로 만드는 불변 객체다. 핵심은 `build()`가 **인가 요청 URL을 직접 조립**한다는 점.

```java
// OAuth2AuthorizationRequest.AbstractBuilder.buildAuthorizationRequestUri
Map<String,Object> parameters = getParameters();           // response_type, client_id, scope, state, redirect_uri
...
MultiValueMap<String,String> queryParams = new LinkedMultiValueMap<>();
parameters.forEach((k,v) -> queryParams.add(encodeQueryParam(k), encodeQueryParam(String.valueOf(v))));
UriBuilder uriBuilder = this.uriBuilderFactory.uriString(this.authorizationUri).queryParams(queryParams);
return this.authorizationRequestUriFunction.apply(uriBuilder).toString();
```

- `getParameters()`가 `OAuth2ParameterNames` 상수를 키로 표준 파라미터를 채운다(`RESPONSE_TYPE`, `CLIENT_ID`, `SCOPE`는 공백 구분 문자열, `STATE`, `REDIRECT_URI`, 그리고 `additionalParameters`).
- 인코딩은 RFC 3986에 맞춰 빌더가 **직접** 처리한다(`UriBuilderFactory`의 인코딩은 `NONE`으로 꺼 둠) — 이미 인코딩된 파라미터의 이중 인코딩을 피하기 위해서다.
- 결과 `authorizationRequestUri`가 사용자에게 보낼 302 Location이 된다.

`from(request)`로 기존 요청을 복제해 일부만 바꾸는 빌더도 제공한다(예: PKCE 파라미터 추가).

### OAuth2AuthorizationResponse — 돌아오는 콜백

인가 서버가 `redirect_uri`로 돌려주는 결과다. 성공이면 `code`, 실패면 `OAuth2Error`를 담는다 — **둘 중 하나**.

```java
public static Builder success(String code);   // code + state
public static Builder error(String errorCode); // error/description/uri + state
public boolean statusOk();     // code 있음
public boolean statusError();  // error 있음
```

`state`는 양쪽에 다 들어가 CSRF 방어용으로 요청-응답을 묶는다. `OAuth2AuthorizationExchange`는 단순히 `Request + Response`를 한 객체로 묶어, 클라이언트가 코드 교환 단계에서 원래 요청 맥락(redirect_uri, scope 등)을 함께 들고 있게 한다.

### OAuth2AccessTokenResponse — 토큰 엔드포인트 응답

코드를 교환하면 토큰 엔드포인트가 JSON을 돌려준다. 이를 모델링한 값 객체다. 빌더의 `build()`에 RFC 준수 디테일이 숨어 있다.

```java
// OAuth2AccessTokenResponse.Builder
private Instant getExpiresAt() {
    if (this.expiresAt == null) {
        Instant issuedAt = getIssuedAt();
        this.expiresAt = (this.expiresIn > 0) ? issuedAt.plusSeconds(this.expiresIn)
                                               : issuedAt.plusSeconds(1);   // expires_in 누락 대비
    }
    return this.expiresAt;
}
```

`expires_in`은 RFC상 권장(필수 아님)이라 안 올 수 있다. 그러면 기본값 0이 되어 "이미 만료"가 될 위험이 있으므로, **발급시각 +1초**로 보정한다. `expiresIn`을 주면 `expiresAt`을 무효화하고(`expiresAt = null`) 재계산하는 등, 두 표현(절대 시각 vs 상대 초)을 일관되게 합친다.

## 직렬화: Map ↔ 메시지, 그리고 HTTP

JSON과 값 객체 사이를 잇는 변환기가 두 층으로 있다.

```
   토큰 엔드포인트 JSON
        │  (Spring MVC)                      │  (WebClient/리액티브)
        ▼                                    ▼
 OAuth2AccessTokenResponse              OAuth2BodyExtractors
 HttpMessageConverter                  .oauth2AccessTokenResponse()
        │ read: JSON → Map → 변환             │ → Mono<OAuth2AccessTokenResponse>
        ▼
 DefaultMapOAuth2AccessTokenResponseConverter   (Map<String,Object> → OAuth2AccessTokenResponse)
 DefaultOAuth2AccessTokenResponseMapConverter   (반대 방향, write 시)
```

`OAuth2AccessTokenResponseHttpMessageConverter`는 JSON 파싱을 범용 JSON 컨버터에 위임하고, 그 결과 `Map`을 `DefaultMapOAuth2AccessTokenResponseConverter`로 값 객체로 바꾼다(쓰기는 반대). 두 변환기는 **교체 가능한 확장점**이다.

```java
// OAuth2AccessTokenResponseHttpMessageConverter
public final void setAccessTokenResponseConverter(
        Converter<Map<String,Object>, OAuth2AccessTokenResponse> c) { ... }   // 비표준 응답 대응
```

같은 일을 리액티브 스택에서 하는 게 `OAuth2BodyExtractors.oauth2AccessTokenResponse()`다. `OAuth2ErrorHttpMessageConverter`는 에러 응답 JSON ↔ `OAuth2Error`를, `OAuth2DeviceAuthorizationResponseHttpMessageConverter`는 device flow 응답을 담당한다.

## 설계 포인트 / 확장점

- **상수 박제 + 값 객체 확장성**: 표준은 상수로, 확장은 공개 생성자/`valueOf`로. enum의 안전함과 개방성을 절충.
- **메시지 = 불변 빌더 객체**: 인가 요청/응답·토큰 응답은 모두 빌더로 만들고 한 번 만들면 못 바꾼다. URL 조립·시각 보정 같은 규칙을 빌더가 캡슐화한다.
- **직렬화 변환기 교체**: 제공자별 비표준 토큰 응답은 `setAccessTokenResponseConverter`로 커스텀 변환기를 끼워 대응한다(`oauth2-client`가 실제로 활용).
- **MVC와 리액티브 양쪽 지원**: 같은 도메인을 `HttpMessageConverter`(서블릿)와 `BodyExtractor`(WebClient) 두 진입점으로 직렬화.

## 정리

이 챕터의 상수들은 OAuth2/OIDC 와이어 포맷의 단어장이다 — grant·인증 방식·응답 타입은 확장 가능한 값 객체로, 파라미터 키는 문자열 사전으로 박제된다. 엔드포인트 메시지(`OAuth2AuthorizationRequest`/`Response`/`Exchange`, `OAuth2AccessTokenResponse`)는 인가 코드 그랜트 왕복의 각 단계를 불변 빌더 객체로 표현하고, URL 조립·`expires_in` 보정 같은 규칙을 안에 감춘다. JSON ↔ 객체 변환은 MVC용 `HttpMessageConverter`와 리액티브용 `BodyExtractor`로 제공되며, 내부 Map 변환기를 교체해 비표준 제공자에 대응할 수 있다.
