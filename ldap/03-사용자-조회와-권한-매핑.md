# 03. 사용자 조회와 권한 매핑

## 무엇을 / 왜

인증으로 "이 사람이 맞다"가 끝나면, 두 가지가 더 필요하다. (1) 디렉터리 엔트리를 애플리케이션이 쓰는 `UserDetails`로 바꾸는 일, (2) 사용자가 가진 권한(역할)을 채우는 일. 게다가 비밀번호 검증 없이 디렉터리를 단순 조회 백엔드로만 쓰고 싶을 때(모드 B)와, 사용자/그룹을 디렉터리에 직접 CRUD하고 싶을 때도 있다. 이 챕터는 `UserDetailsContextMapper`, `LdapAuthoritiesPopulator` 계열, 그리고 `LdapUserDetailsService`/`LdapUserDetailsManager`를 다룬다.

## 핵심 타입

```
UserDetailsContextMapper (interface)        엔트리 ↔ UserDetails
   └─ LdapUserDetailsMapper                 기본: LdapUserDetailsImpl 생성 (읽기 전용)
        └─ Person/InetOrgPerson ContextMapper  스키마별 풍부한 매핑

LdapAuthoritiesPopulator (interface)        권한 채우기
   ├─ DefaultLdapAuthoritiesPopulator       그룹 검색 (member=userDN)
   │     └─ NestedLdapAuthoritiesPopulator  중첩 그룹 재귀
   ├─ NullLdapAuthoritiesPopulator          권한 없음
   └─ UserDetailsServiceLdapAuthoritiesPopulator  권한을 다른 UserDetailsService 에서

UserDetailsService (core)
   └─ LdapUserDetailsService                모드 B: 검색+권한, 비번검증 없음

UserDetailsManager (core)
   └─ LdapUserDetailsManager                디렉터리에 사용자/그룹 CRUD + 비번변경
```

## 동작 흐름 ① — 권한 채우기 (DefaultLdapAuthoritiesPopulator)

권한의 출처는 보통 "그룹"이다. 그룹 엔트리(`groupOfNames` 등)의 `member` 속성에 사용자 DN이 들어 있으면 그 그룹을 역할로 본다. `DefaultLdapAuthoritiesPopulator.getGrantedAuthorities()` (`.../userdetails/DefaultLdapAuthoritiesPopulator.java`):

```
getGrantedAuthorities(userData, "ben")
  │ userDn = userData.getNameInNamespace()   (전체 DN)
  ▼
  getGroupMembershipRoles(userDn, "ben"):
     groupSearchBase 없으면 → 빈 집합 (그룹 검색 비활성)
     ldapTemplate.searchForMultipleAttributeValues(
         base   = groupSearchBase  (예: "ou=groups"),
         filter = "(member={0})"   ({0}=userDn, {1}=username 사용 가능),
         attrs  = ["cn"]  (groupRoleAttribute))
     각 매칭 그룹 record → authorityMapper:
         role = record["cn"][0]
         convertToUpperCase 면 대문자
         → SimpleGrantedAuthority("ROLE_" + role)
  │
  ├ + getAdditionalRoles(user, username)   (서브클래스 훅, 기본 빈 집합)
  └ + defaultRole (설정 시 모두에게)
```

기본값: 필터 `(member={0})`, 역할 속성 `cn`, 접두사 `ROLE_`, 대문자 변환 on, 단일 레벨 검색. 예컨대 `cn=developers` 그룹의 멤버면 `ROLE_DEVELOPERS` 권한을 받는다. `authorityMapper`(Function)를 통째로 교체하면 매핑 규칙을 완전히 바꿀 수 있다.

`groupSearchBase`가 `null`이면 그룹 검색 자체를 끈다 — 이 경우 권한은 사용자 엔트리의 속성(`LdapUserDetailsMapper.roleAttributes`)이나 다른 populator에서 와야 한다.

### 중첩 그룹 — NestedLdapAuthoritiesPopulator

그룹이 다른 그룹의 멤버인 경우(예: `j-developers` 안에 `java-developers`), `NestedLdapAuthoritiesPopulator`가 부모 그룹을 따라 올라가며 재귀 검색해 상위 역할까지 모은다. `DefaultLdapAuthoritiesPopulator`를 상속해 검색 한 번에 그치지 않고 발견된 그룹 DN으로 다시 검색을 반복한다.

## 동작 흐름 ② — UserDetails 매핑 (LdapUserDetailsMapper)

`LdapUserDetailsMapper.mapUserFromContext()` (`.../userdetails/LdapUserDetailsMapper.java`):

```
mapUserFromContext(ctx, "ben", authorities)
  │ essence = new LdapUserDetailsImpl.Essence()  (빌더)
  ├ dn       = ctx.getNameInNamespace()
  ├ password = ctx.getObjectAttribute("userPassword") (있으면)
  ├ username = "ben"
  ├ roleAttributes 설정 시: 엔트리의 해당 속성값들도 ROLE_* 로 추가
  ├ authorities (populator 가 준 것) 추가
  ├ PasswordPolicyResponseControl 있으면 만료/유예로그인 정보 반영
  └ essence.createUserDetails() → LdapUserDetailsImpl
```

`LdapUserDetailsImpl`은 `LdapUserDetails`(= `UserDetails` + `getDn()`)의 표준 구현이다. `mapUserToContext`는 이 기본 매퍼에서 `UnsupportedOperationException`을 던진다 — 읽기 전용이기 때문이다. 쓰기(생성/수정)가 필요하면 `Person`/`InetOrgPerson` 계열 ContextMapper를 쓴다(이들은 양방향 매핑을 지원하고 `cn`, `sn`, `mail` 같은 풍부한 속성을 다룬다).

## 동작 흐름 ③ — 모드 B: LdapUserDetailsService

비밀번호 검증 없이 디렉터리를 조회 백엔드로만 쓰는 경로 (`.../userdetails/LdapUserDetailsService.java`):

```
loadUserByUsername("ben")
  │
  ├ userData = userSearch.searchForUser("ben")           ← 검색만 (bind 안 함)
  └ return userDetailsMapper.mapUserFromContext(
              userData, "ben",
              authoritiesPopulator.getGrantedAuthorities(userData, "ben"))
```

`LdapAuthenticationProvider`와 같은 `LdapUserSearch`/`LdapAuthoritiesPopulator`/`UserDetailsContextMapper`를 재사용하지만, **비밀번호를 검증하지 않는다**. 예: 다른 수단으로 인증한 뒤 LDAP에서 사용자 정보·권한만 끌어올 때, 혹은 Remember-Me처럼 username만으로 사용자를 복원할 때 쓴다. 권한 populator를 안 주면 `NullLdapAuthoritiesPopulator`(권한 없음)가 들어간다.

## 동작 흐름 ④ — LdapUserDetailsManager (CRUD + 비번 변경)

`LdapUserDetailsManager`는 `UserDetailsManager`를 구현해 디렉터리에 사용자를 **직접 쓰는** 경우를 담당한다 (`.../userdetails/LdapUserDetailsManager.java`). 인증 provider 경로와는 별개다.

- `createUser/updateUser/deleteUser` — `LdapUsernameToDnMapper`로 username→DN을 만들고 엔트리를 bind/modify/unbind 한다. 동시에 그룹 멤버십(`uniquemember` 속성)을 추가/제거해 권한을 반영한다.
- `loadUserByUsername` — DN을 직접 조합해 엔트리를 읽고(검색 없이), 그룹 검색으로 권한을 채운 뒤 `InetOrgPersonContextMapper`(기본)로 매핑한다.
- `changePassword` — 두 전략. 기본은 사용자 자격증명으로 **재바인딩 후 속성 수정**(`oldPassword` 필요). `usePasswordModifyExtensionOperation`을 켜면 RFC 3062 **Password Modify 확장 연산**(BER 인코딩 요청)을 보낸다.

```
changePassword(old, new)
  │ username = SecurityContext 의 현재 인증에서
  ├ usePasswordModifyExtensionOperation == true
  │    → RFC3062 확장 연산 (PasswordModifyRequest BER 인코딩)
  └ false (기본)
       ├ old != null → 사용자로 reconnect 후 userPassword 속성 REPLACE
       └ old == null → 관리자 컨텍스트로 속성 REPLACE
```

## 설계 포인트 / 확장점

- **권한 출처의 유연성** — 그룹 검색(`Default`), 중첩 그룹(`Nested`), 다른 `UserDetailsService`(`UserDetailsServiceLdapAuthoritiesPopulator`), 없음(`Null`) 중 선택. `authorityMapper`로 매핑 규칙까지 교체 가능.
- **읽기/쓰기 매퍼 분리** — `LdapUserDetailsMapper`는 읽기 전용, `Person`/`InetOrgPerson` 매퍼는 양방향. CRUD에는 후자가 필요하다.
- **provider 경로 vs Manager 경로** — 전자는 인증과 조회를 합쳐 `AuthenticationProvider`로, 후자는 순수 디렉터리 CRUD로 `UserDetailsManager`로 동작한다. Manager는 "LDAP을 사용자 저장소로 운영"하는 시나리오용이다.
- **`getNameInNamespace()` 사용** — 권한 검색 시 부분 DN이 아니라 전체 DN을 써야 그룹의 `member` 값과 일치한다. populator/template이 일관되게 이를 사용한다.

## 정리

인증이 끝난 뒤 엔트리는 `UserDetailsContextMapper`로 `UserDetails`가 되고, 권한은 `LdapAuthoritiesPopulator`가 그룹 검색으로 채운다. 비밀번호 검증이 빠진 조회 전용 경로는 `LdapUserDetailsService`(모드 B)가, 디렉터리에 사용자를 직접 쓰는 경로는 `LdapUserDetailsManager`가 담당하며, 둘 다 같은 매퍼·populator 전략을 재사용한다.
