# 01. 개요와 핵심 추상화

## 무엇을 / 왜

LDAP 인증은 "비밀번호 검증", "사용자 찾기", "권한 채우기", "사용자 표현 만들기"라는 네 가지 일이 한 요청 안에서 일어난다. 이들을 한 클래스에 욱여넣으면 디렉터리 종류(OpenLDAP/AD)나 인증 방식(bind/compare)이 바뀔 때마다 전체를 갈아엎어야 한다. 이 모듈은 각 책임을 **독립된 전략 인터페이스**로 분리하고, `LdapAuthenticationProvider`가 이들을 조립(composition)하는 구조로 풀었다. 이 챕터에서는 그 네 전략과 데이터가 흐르는 타입(`DirContextOperations`)을 먼저 잡는다.

## 핵심 타입 — 4개의 전략 + 1개의 운반체

```
        LdapAuthenticationProvider (조립자, AuthenticationProvider)
                 │ 가진다(has-a)
   ┌─────────────┴───────────────┐
   ▼                             ▼
LdapAuthenticator            LdapAuthoritiesPopulator
"비번 검증 + 엔트리 획득"        "권한 채우기"
   │                             │
   │ (내부에서 필요시)            └─ DefaultLdapAuthoritiesPopulator (그룹 검색)
   ▼                                NestedLdapAuthoritiesPopulator (중첩 그룹)
LdapUserSearch
"엔트리 검색"
   └─ FilterBasedLdapUserSearch

       UserDetailsContextMapper
       "DirContextOperations → UserDetails"
       └─ LdapUserDetailsMapper (기본)

  ── 공통 운반체 ──
  DirContextOperations  : LDAP 엔트리(DN + 속성들)를 담은 Spring LDAP 객체.
                          authenticator가 만들어 populator/mapper로 흘려보낸다.
```

각 인터페이스의 책임을 한 줄씩 정리하면:

| 인터페이스 | 메서드 | 책임 | 소스 |
|---|---|---|---|
| `LdapAuthenticator` | `authenticate(Authentication) : DirContextOperations` | 비밀번호를 검증하고, 인증된 사용자의 디렉터리 엔트리를 돌려준다 | `ldap/src/main/java/org/springframework/security/ldap/authentication/LdapAuthenticator.java` |
| `LdapUserSearch` | `searchForUser(username) : DirContextOperations` | 로그인 이름으로 디렉터리에서 엔트리를 찾는다 | `.../ldap/search/LdapUserSearch.java` |
| `LdapAuthoritiesPopulator` | `getGrantedAuthorities(userData, username)` | 사용자 엔트리로부터 권한 집합을 만든다 | `.../ldap/userdetails/LdapAuthoritiesPopulator.java` |
| `UserDetailsContextMapper` | `mapUserFromContext(ctx, username, authorities) : UserDetails` | 엔트리+권한을 최종 `UserDetails`로 변환한다 | `.../ldap/userdetails/UserDetailsContextMapper.java` |

이 분리 덕분에, 예를 들어 "권한은 LDAP이 아니라 DB에서 가져오고 싶다"면 `LdapAuthoritiesPopulator`만 갈아끼우면 된다(`UserDetailsServiceLdapAuthoritiesPopulator`가 그 예). "bind 대신 비밀번호 비교"면 `LdapAuthenticator`만 바꾼다.

## 동작 흐름 — 인증 한 번에 일어나는 일

`LdapAuthenticationProvider`로 폼 로그인이 들어왔을 때(모드 A)의 큰 그림. 세부는 02챕터에서 다룬다.

```
authenticate(UsernamePasswordAuthenticationToken: "ben"/"pw")
  │
  │ ① 빈 username/password 거르기  (AbstractLdapAuthenticationProvider)
  ▼
  ② doAuthentication()  →  LdapAuthenticator.authenticate()
       └ bind 또는 compare 로 비번 검증, 성공 시 엔트리 반환
       반환: DirContextOperations userData  (DN + 속성)
  │
  ▼
  ③ loadUserAuthorities()  →  LdapAuthoritiesPopulator.getGrantedAuthorities(userData,"ben")
       └ groupSearchBase 아래에서 (member=<userDN>) 검색 → ROLE_* 집합
  │
  ▼
  ④ userDetailsContextMapper.mapUserFromContext(userData,"ben", 권한들)
       └ LdapUserDetailsImpl 생성 (DN, username, 비번(있으면), 권한)
  │
  ▼
  ⑤ createSuccessfulAuthentication()
       └ 권한에 PASSWORD_AUTHORITY(FactorGrantedAuthority) 추가
       반환: 인증된 UsernamePasswordAuthenticationToken
```

핵심은 `AbstractLdapAuthenticationProvider.authenticate()`가 ②③④⑤의 **순서를 고정한 템플릿 메서드**라는 점이다(`.../authentication/AbstractLdapAuthenticationProvider.java`):

```java
DirContextOperations userData = doAuthentication(userToken);          // ②
UserDetails user = this.userDetailsContextMapper.mapUserFromContext(  // ④
        userData, authentication.getName(),
        loadUserAuthorities(userData, authentication.getName(), password)); // ③
return createSuccessfulAuthentication(userToken, user);               // ⑤
```

`doAuthentication`과 `loadUserAuthorities`만 추상으로 남겨, 표준 LDAP과 Active Directory 두 서브클래스가 같은 골격을 공유한다.

## 핵심 메서드 — `createSuccessfulAuthentication`

성공 토큰을 만들 때 두 가지 결정이 들어간다(같은 파일):

```java
Object password = this.useAuthenticationRequestCredentials ? authentication.getCredentials()
        : user.getPassword();
...
authorities.add(FactorGrantedAuthority.fromAuthority(AUTHORITY)); // PASSWORD_AUTHORITY
```

- **비밀번호 출처** — `useAuthenticationRequestCredentials`(기본 true)면 토큰에 담긴 입력 비번을, false면 디렉터리에서 읽은 비번을 자격증명으로 보관한다. 디렉터리 비번을 못 읽는 경우가 많아 기본이 true다.
- **인증 요소 표시** — 결과 권한에 `PASSWORD_AUTHORITY` 팩터를 더해, 이 인증이 비밀번호 요소로 이뤄졌음을 남긴다(다중 인증 요소 정책에서 사용).

또한 `setAuthoritiesMapper`로 `GrantedAuthoritiesMapper`를 끼우면 populator가 만든 권한을 한 번 더 변환할 수 있다(기본은 변환 없는 `NullAuthoritiesMapper`).

## 설계 포인트 / 확장점

- **전략 패턴 + 합성** — 4개 인터페이스가 모두 교체 가능한 훅이다. 표준 설정(`config` 모듈)이 이들을 빈으로 조립한다.
- **템플릿 메서드** — `AbstractLdapAuthenticationProvider`가 흐름을, 서브클래스가 인증/권한 단계를 채운다.
- **`DirContextOperations`를 공통 통화로 사용** — authenticator가 만든 엔트리를 populator와 mapper가 재사용해, 권한 검색이나 속성 매핑을 위해 디렉터리를 다시 읽지 않아도 되게 한다.
- **두 모드의 분리** — 비번 검증이 필요 없으면 `LdapUserDetailsService`(모드 B)가 같은 `LdapUserSearch`/`LdapAuthoritiesPopulator`를 재사용한다.

## 정리

이 모듈의 뼈대는 "인증 = `LdapAuthenticator`, 검색 = `LdapUserSearch`, 권한 = `LdapAuthoritiesPopulator`, 표현 = `UserDetailsContextMapper`"라는 4분할이다. `LdapAuthenticationProvider`는 이들을 합성해 고정된 순서로 호출하는 조립자이며, `DirContextOperations`가 단계 사이를 흐르는 데이터다. 다음 챕터부터 각 전략을 깊이 본다.
