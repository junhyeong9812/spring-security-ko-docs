# 01. 인증 도메인 모델 — Authentication과 GrantedAuthority

## 무엇을 / 왜

Spring Security를 처음 들여다볼 때 가장 먼저 만나는 질문은 "**현재 사용자를 코드에서 어떻게 표현하는가**"이다. 폼 로그인이든 OAuth2든 JWT든, 인증이 끝나면 결국 하나의 객체가 만들어져 "이 사람은 누구이고, 무슨 권한을 갖는다"를 담는다. 그 객체가 바로 `Authentication`이다.

이 장은 `core` 모듈의 **가장 근본이 되는 어휘**를 다룬다. 이후 모든 장(컨텍스트 저장, 인증 처리, 인가 결정)은 전부 이 타입들 위에서 동작한다.

소스: `core/src/main/java/org/springframework/security/core/`, `.../authentication/`

## 핵심 타입

```
        Principal (java.security)
            ▲
            │ extends
       Authentication ───────────────┐ (인증 요청 토큰 ↔ 인증 결과 토큰)
            ▲                         │ 담는 것:
            │ implements              │  - getPrincipal()   누구
   AbstractAuthenticationToken       │  - getCredentials() 증명(비번 등)
   (+ CredentialsContainer)          │  - getAuthorities() 권한 목록
            ▲                         │  - getDetails()     부가정보(IP 등)
   ┌────────┼─────────────────┐      │  - isAuthenticated() 신뢰 여부
   │        │                 │      └─────────────────────
UsernamePassword  Anonymous  RememberMe   ... (각 인증 방식별 토큰)
AuthenticationToken  Token    Token

   GrantedAuthority ──── getAuthority():String  ("ROLE_ADMIN" 등)
        ▲
   SimpleGrantedAuthority / FactorGrantedAuthority
```

### Authentication — 요청이자 결과인 이중 인격

`Authentication`은 한 인터페이스가 **두 가지 역할**을 한다는 점이 핵심이다.

- **인증 요청(request)일 때:** principal=사용자가 입력한 username, credentials=입력한 password, authenticated=false.
- **인증 결과(result)일 때:** principal=`UserDetails`(풍부한 사용자 정보), credentials=보통 소거됨, authorities=부여된 권한, authenticated=true.

`Authentication.java`의 핵심 메서드 5개:

```java
Collection<? extends GrantedAuthority> getAuthorities(); // 부여된 권한 (인증 전엔 빈 컬렉션)
@Nullable Object getCredentials();   // 증명 (비밀번호 등)
@Nullable Object getDetails();       // 요청 부가정보 (IP, 세션ID 등)
@Nullable Object getPrincipal();     // 인증 주체 (username 또는 UserDetails)
boolean isAuthenticated();           // 이 토큰을 신뢰해도 되는가
```

`isAuthenticated()`의 의미가 특히 중요하다. 주석이 설명하듯, 이 값이 `true`면 보안 인터셉터가 이 토큰을 **다시 `AuthenticationManager`에 제출하지 않는다**(성능). 그래서 `setAuthenticated(true)`는 위험하며, 신뢰할 수 있는 `AuthenticationManager`만 호출해야 한다. 인터페이스 주석은 "구현은 불변(immutable)이거나, 생성 후 속성이 바뀌지 않음을 보장할 수 있을 때만 `true`를 반환하라"고 못 박는다.

`Authentication`은 `Principal`(JDK 표준)과 `Serializable`을 상속한다. `Serializable`인 이유는 세션에 저장되어 직렬화될 수 있어야 하기 때문이다.

### AbstractAuthenticationToken — 토큰들의 공통 기반

거의 모든 구체 토큰(`UsernamePasswordAuthenticationToken`, `AnonymousAuthenticationToken`, `RememberMeAuthenticationToken` 등)은 `AbstractAuthenticationToken`을 상속한다. 이 추상 클래스는 `Authentication` + `CredentialsContainer`를 구현하며, 공통 골격을 제공한다.

`AbstractAuthenticationToken.java`에서 눈여겨볼 두 가지:

**1) authorities의 불변 처리** — 생성자에서 들어온 권한 컬렉션을 방어적으로 복사해 `unmodifiableList`로 감싼다. 권한이 외부에서 변조되면 권한 상승 취약점이 되므로, 불변성이 보안의 일부다.

```java
this.authorities = Collections.unmodifiableList(new ArrayList<>(authorities));
```

**2) getName()의 다형적 해석** — principal의 타입에 따라 "이름"을 다르게 끄집어낸다. 인가 로직에서 `authentication.getName()`을 호출할 때 일관된 username이 나오도록 보장한다.

```java
if (this.getPrincipal() instanceof UserDetails userDetails) return userDetails.getUsername();
if (this.getPrincipal() instanceof AuthenticatedPrincipal ap)  return ap.getName();
if (this.getPrincipal() instanceof Principal principal)        return principal.getName();
return (getPrincipal() == null) ? "" : getPrincipal().toString();
```

`toString()`도 보안을 의식한다 — credentials는 항상 `[PROTECTED]`로 출력되어 로그에 비밀번호가 새지 않는다.

### CredentialsContainer — 자격 증명 소거 계약

인증이 끝나면 결과 토큰에 남은 비밀번호는 메모리상의 위험이다. `CredentialsContainer.eraseCredentials()`는 "내 안의 민감 정보를 지워라"는 계약이다. `AbstractAuthenticationToken`은 이를 구현해 credentials, principal, details가 또 `CredentialsContainer`이면 재귀적으로 소거한다.

```java
@Override public void eraseCredentials() {
    eraseSecret(getCredentials());
    eraseSecret(getPrincipal());
    eraseSecret(this.details);
}
```

이 메서드는 `ProviderManager`가 인증 성공 직후 자동으로 호출한다(03장 참고).

### GrantedAuthority — 권한의 최소 표현

`GrantedAuthority`는 `getAuthority()` 단 하나만 가진 인터페이스다. 문자열로 표현 가능한 권한은 그 문자열을 반환하고(예: `"ROLE_ADMIN"`), 문자열로 충분히 표현할 수 없는 권한만 `null`을 반환한다(이 경우 별도 `AuthorizationManager` 지원이 필요). 가장 흔한 구현은 `SimpleGrantedAuthority`(문자열 한 개를 감싸는 불변 객체)이고, `AuthorityUtils.createAuthorityList("ROLE_USER", ...)`로 손쉽게 만든다.

`"ROLE_"` 접두사 관례는 여기서 출발한다 — 인가 쪽(`AuthorityAuthorizationManager.hasRole`)이 `"ROLE_"`를 자동으로 붙여주므로, 권한 문자열은 이 관례를 따른다(05장).

## 동작 흐름 — 요청 토큰이 결과 토큰으로 바뀌는 순간

폼 로그인을 예로, `Authentication` 객체의 **생애주기**를 보자.

```
[1] 사용자가 username/password 제출
        │  (web 모듈의 필터가 토큰 생성)
        ▼
   UsernamePasswordAuthenticationToken.unauthenticated("alice", "pw1234")
   ├─ principal = "alice" (String)
   ├─ credentials = "pw1234"
   ├─ authorities = [] (빈 컬렉션)
   └─ authenticated = false        ← 아직 신뢰 못 함
        │
        ▼  AuthenticationManager.authenticate(token)   (03·04장)
        │
   [2] Provider가 검증 후 "결과 토큰"을 새로 생성
        ▼
   UsernamePasswordAuthenticationToken.authenticated(userDetails, creds, authorities)
   ├─ principal = UserDetails(alice, 이메일/권한 등 풍부)
   ├─ credentials = (이후 eraseCredentials로 소거)
   ├─ authorities = [ROLE_USER, FACTOR_PASSWORD]
   └─ authenticated = true         ← 이제 신뢰됨
        │
        ▼  SecurityContextHolder에 저장 (02장)
   이후 인가 단계에서 getAuthorities()로 접근 판단 (05장)
```

핵심은 **요청 토큰과 결과 토큰이 다른 인스턴스**라는 점이다. `UsernamePasswordAuthenticationToken`은 정적 팩토리 `unauthenticated(...)`(authenticated=false)와 `authenticated(...)`(authenticated=true)를 제공해, 두 상태를 명시적으로 구분한다. 결과 토큰의 권한은 한 번 채워지면 불변이다.

## 설계 포인트 / 확장점

- **인터페이스 우선 + 불변 토큰:** 애플리케이션 코드는 `Authentication` 인터페이스에 의존하고, 커스텀 정보는 principal(주로 `UserDetails`나 OAuth2의 principal) 안에 담는 것이 권장된다.
- **`toBuilder()` (7.0+):** `Authentication`에 빌더 API가 추가되어, 기존 토큰을 바탕으로 권한·details 등을 바꿔 **타입을 보존한 채** 새 토큰을 파생할 수 있다. `AbstractAuthenticationToken.AbstractAuthenticationBuilder`가 공통 빌더 골격을 제공하며, 기본 구현은 `SimpleAuthentication`을 만든다. `authorities(...)`로 권한을 채우면 자동으로 `authenticated=true`로 표시된다.
- **`Transient`:** `@Transient`가 붙은 `Authentication`은 세션에 저장되지 않아야 함을 표시한다(일회성 인증).
- **커스텀 토큰:** 새 인증 방식을 만들려면 `AbstractAuthenticationToken`을 상속하고 `getCredentials()`/`getPrincipal()`만 구현하면 된다. equals/hashCode/getName/erase는 상속받는다.

## 정리

`Authentication`은 "인증 요청"과 "인증 결과"를 같은 인터페이스로 표현하는 이중 인격의 도메인 객체로, principal·credentials·authorities·authenticated 네 축으로 사용자를 기술한다. `AbstractAuthenticationToken`이 불변 권한과 자격 증명 소거 같은 보안 골격을 제공하고, `GrantedAuthority`가 권한을 문자열 한 줄로 압축한다. 이 작은 타입 집합이 Spring Security 전체가 공유하는 **공통 언어**이며, 다음 장부터는 이 객체가 어디에 저장되고(02장) 어떻게 만들어지는지(03·04장)를 따라간다.
