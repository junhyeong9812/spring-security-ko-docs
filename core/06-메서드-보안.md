# 06. 메서드 보안 — AOP로 인가를 메서드에 거는 법

## 무엇을 / 왜

05장의 `AuthorizationManager`는 "무엇을 보호하는가(T)"에 대해 일반적이다. URL이 `T`면 URL 보안이고, **메서드 호출(`MethodInvocation`)이 `T`면 메서드 보안**이다. `@PreAuthorize("hasRole('ADMIN')")` 같은 애너테이션을 서비스 메서드에 붙이면, 그 메서드를 호출하기 직전(혹은 직후)에 인가 검사가 일어난다.

이 장은 그 메커니즘 — Spring AOP `Advisor`/`MethodInterceptor`로 인가 매니저를 메서드 호출에 끼워 넣는 `authorization/method` 패키지를 다룬다. "애너테이션 하나가 어떻게 실행을 가로채는가"가 핵심 질문이다.

소스: `core/src/main/java/org/springframework/security/authorization/method/`

## 핵심 타입

```
  @PreAuthorize / @PostAuthorize / @Secured / @RolesAllowed(JSR-250)
        │ (메서드/클래스에 부착)
        ▼
 ┌────────────────────────────────────────────────────────────┐
 │ AuthorizationAdvisor (Spring AOP Advisor + Ordered)          │
 │   getPointcut(): 어떤 메서드에 적용?  getAdvice(): 무엇을 실행? │
 └───────────────────────────┬────────────────────────────────┘
        구현 (= MethodInterceptor)│
   ┌───────────────────────────┼──────────────────────────────┐
   ▼                           ▼                              ▼
 AuthorizationManager       AuthorizationManager          PreFilter /
 BeforeMethodInterceptor    AfterMethodInterceptor        PostFilter
 (호출 전 검사)               (반환값 검사)                  Interceptor (인자/반환 컬렉션 필터링)
   │ 위임
   ▼
 ┌────────────────────────────────────────────────────┐
 │ AuthorizationManager<MethodInvocation> 구현들         │
 │   PreAuthorizeAuthorizationManager  (SpEL 평가)       │
 │   PostAuthorizeAuthorizationManager                  │
 │   SecuredAuthorizationManager       (@Secured)       │
 │   Jsr250AuthorizationManager        (@RolesAllowed)  │
 └────────────────────────────────────────────────────┘
```

### AuthorizationAdvisor — 포인트컷 + 어드바이스

Spring AOP에서 "어디에(pointcut) 무엇을(advice) 적용할지"를 묶은 것이 `Advisor`다. `AuthorizationAdvisor`는 거기에 `Ordered`를 더해, 여러 보안 어드바이저의 적용 순서를 고정한다.

각 인터셉터(예: `AuthorizationManagerBeforeMethodInterceptor`)는 `AuthorizationAdvisor`이면서 동시에 `MethodInterceptor`다 — 즉 `getAdvice()`가 `this`를 반환한다. 자기 자신이 포인트컷도 들고 있고, 가로채기 로직도 들고 있는 자족적 단위다.

### 적용 순서 — AuthorizationInterceptorsOrder

여러 애너테이션이 한 메서드에 동시에 걸릴 수 있으므로 순서가 중요하다. `AuthorizationInterceptorsOrder` enum이 이를 정의한다.

```
FIRST
PRE_FILTER      ← 메서드 호출 전, 입력 컬렉션 필터링 (@PreFilter)
PRE_AUTHORIZE   ← 메서드 호출 전, 접근 검사 (@PreAuthorize)
SECURED         ← @Secured
JSR250          ← @RolesAllowed / @PermitAll / @DenyAll
... (메서드 실행) ...
SECURE_RESULT   ← 반환 객체 자체를 인가 프록시로 감쌈 (@AuthorizeReturnObject)
POST_AUTHORIZE  ← 반환값 기준 접근 검사 (@PostAuthorize)
POST_FILTER     ← 반환 컬렉션 필터링 (@PostFilter)
LAST
```

`@PreFilter`(입력 정제) → `@PreAuthorize`(진입 차단) → **메서드 실행** → `@PostAuthorize`(결과 검사) → `@PostFilter`(결과 정제)라는 자연스러운 파이프라인이 순서로 강제된다.

## 동작 흐름 — @PreAuthorize 호출 가로채기

`userService.deleteUser(id)`에 `@PreAuthorize("hasRole('ADMIN')")`가 붙어 있다고 하자. 호출 시 일어나는 일.

```
caller → [AOP 프록시] userService.deleteUser(id)
   │
   ▼  AuthorizationManagerBeforeMethodInterceptor.invoke(mi)
   │      (pointcut이 @PreAuthorize 메서드와 매치하므로 인터셉터 발동)
   │
   ├─[1] result = authorizationManager.authorize(this::getAuthentication, mi)
   │        └ PreAuthorizeAuthorizationManager.authorize:
   │             attribute = registry.getAttribute(mi)   // @PreAuthorize의 SpEL 파싱·캐싱
   │             if (attribute == null) return null;     // 애너테이션 없으면 기권
   │             ctx = expressionHandler.createEvaluationContext(auth, mi)
   │             return ExpressionUtils.evaluate("hasRole('ADMIN')", ctx, ...)
   │                    → ExpressionAuthorizationDecision(granted)
   │
   ├─[2] if (result != null) eventPublisher.publishAuthorizationEvent(...)  // 감사 훅
   │
   ├─[3] if (result != null && !result.isGranted()):
   │        return handle(mi, result)    // 거부 처리 → 기본 ThrowingHandler → AccessDeniedException
   │
   └─[4] 허용 → proceed(mi) = mi.proceed()   // 실제 deleteUser(id) 실행
            (실행 중 AuthorizationDeniedException 나면 DeniedHandler로 대체값 반환 가능)
```

### getAuthentication() — 컨텍스트에서 인증 끌어오기

인터셉터는 현재 인증을 02장의 `SecurityContextHolder`에서 가져온다. 단, 정적 전역에 직접 묶이지 않고 `Supplier<SecurityContextHolderStrategy>`를 통해 전략을 들고 다닌다(테스트·격리 고려).

```java
private Authentication getAuthentication() {
    Authentication authentication =
        this.securityContextHolderStrategy.get().getContext().getAuthentication();
    if (authentication == null)
        throw new AuthenticationCredentialsNotFoundException(
            "An Authentication object was not found in the SecurityContext");
    return authentication;
}
```

이 메서드 참조(`this::getAuthentication`)가 05장의 `Supplier<Authentication>`로 매니저에 전달된다. 그래서 `permitAll` 같은 표현식은 이 supplier를 호출조차 안 해, 컨텍스트 조회 비용을 아낀다.

### SpEL 평가와 속성 캐싱

`@PreAuthorize`의 문자열은 SpEL 표현식이다. `PreAuthorizeAuthorizationManager`는 `PreAuthorizeExpressionAttributeRegistry`로 메서드별 표현식을 **한 번 파싱해 캐싱**하고(`getAttribute(mi)`), `MethodSecurityExpressionHandler`가 만든 평가 컨텍스트에서 평가한다. 이 컨텍스트에는 `hasRole`, `hasAuthority`, `authentication`, `principal` 같은 보안 함수/변수가 바인딩되어 있어, 표현식에서 바로 쓸 수 있다. 메서드 인자(`#id`)나 반환값(`returnObject`)도 이 컨텍스트로 참조 가능하다.

### before vs after

- **`AuthorizationManagerBeforeMethodInterceptor`** — 메서드 실행 **전**에 검사(`@PreAuthorize`, `@Secured`, JSR-250). 거부면 메서드를 아예 실행하지 않는다. order = `FIRST` 기본(애너테이션별로 세분).
- **`AuthorizationManagerAfterMethodInterceptor`** — 메서드 실행 **후** 반환값을 보고 검사(`@PostAuthorize`). `MethodInvocationResult`(invocation + 반환값)를 `T`로 받아, 반환된 객체의 소유자 검증 같은 판단을 한다.
- **`PreFilter`/`PostFilter` 인터셉터** — 컬렉션 인자/반환값을 요소별로 필터링한다. 인가가 boolean 결정이 아니라 "보여줄 수 있는 항목만 남기기"인 경우다.

### 거부 처리의 유연성

기본 거부 처리는 `ThrowingMethodAuthorizationDeniedHandler`로 `AuthorizationDeniedException`(= `AccessDeniedException`)을 던진다. 하지만 `@HandleAuthorizationDenied`나 매니저가 `MethodAuthorizationDeniedHandler`를 구현하면, **예외 대신 대체값**(예: null, 마스킹된 객체)을 반환할 수 있다. `PreAuthorizeAuthorizationManager`가 직접 `MethodAuthorizationDeniedHandler`를 구현하는 이유다. 인터셉터의 `proceed()`도 메서드 본문에서 던져진 `AuthorizationDeniedException`을 잡아 핸들러로 넘긴다.

## 설정과 프록시 생성

이 인터셉터들은 보통 직접 만들지 않는다. `config` 모듈의 `@EnableMethodSecurity`가 `AuthorizationManagerBeforeMethodInterceptor.preAuthorize()` 같은 정적 팩토리로 어드바이저 빈들을 등록하고, Spring AOP가 대상 빈을 프록시로 감싼다. `core`는 그 **재료(어드바이저·인터셉터·매니저)**를 제공하고, 조립은 `config`가 한다. `AuthorizationAdvisorProxyFactory`/`AuthorizationProxyFactory`는 반환 객체를 인가 프록시로 감싸는(`@AuthorizeReturnObject`) 등 프로그램적 프록시 생성을 지원한다.

## 설계 포인트 / 확장점

- **AOP Advisor 패턴:** 인가를 비즈니스 코드에서 분리해 횡단 관심사로 처리. 메서드는 보안을 의식하지 않고, 애너테이션 + 프록시가 가로챈다.
- **인가 매니저 재사용:** 메서드 보안은 05장의 `AuthorizationManager`를 `T = MethodInvocation`으로 특수화한 것일 뿐이다. URL 보안과 결정 엔진을 공유한다.
- **순서 보장:** `AuthorizationInterceptorsOrder`로 PreFilter→PreAuthorize→실행→PostAuthorize→PostFilter 파이프라인을 고정.
- **커스텀 표현식/핸들러:** `MethodSecurityExpressionHandler`로 표현식 함수를 확장하고, `MethodAuthorizationDeniedHandler`로 거부 동작(예외 vs 대체값)을 바꾼다.
- **리액티브 짝:** `*ReactiveMethodInterceptor` 계열이 `Mono`/`Flux` 반환 메서드에 대응한다.
- **이벤트:** `AuthorizationEventPublisher`로 인가 성공/거부를 이벤트화해 감사한다(기본은 `NoOpAuthorizationEventPublisher`).

## 정리

메서드 보안은 05장의 `AuthorizationManager`를 `MethodInvocation`에 적용한 것으로, Spring AOP `Advisor`/`MethodInterceptor`가 애너테이션이 붙은 메서드 호출을 가로채 인가를 검사한다. `@PreAuthorize`는 호출 전(`Before` 인터셉터)에 SpEL을 평가해 진입을 차단하고, `@PostAuthorize`는 반환값을 보고, `@PreFilter`/`@PostFilter`는 컬렉션을 정제한다. 인증은 `SecurityContextHolder`에서 지연 공급되고, 결정은 05장의 모델을 그대로 쓰며, 거부는 예외 또는 대체값으로 유연하게 처리된다. 이로써 `core` 모듈은 인증 도메인(01) → 컨텍스트(02) → 인증 처리(03·04) → 인가 결정(05) → 메서드 적용(06)까지, Spring Security 전체가 딛고 서는 토대를 완성한다.
