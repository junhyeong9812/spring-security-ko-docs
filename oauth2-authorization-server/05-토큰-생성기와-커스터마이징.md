# 05. 토큰 생성기와 커스터마이징 — `OAuth2TokenGenerator` 파이프라인

## 무엇을 / 왜

04장에서 모든 그랜트 Provider는 토큰을 직접 만들지 않고 `OAuth2TokenGenerator.generate(context)`에 위임했다. 이 챕터는 그 위임의 반대편 — "토큰 값과 클레임이 실제로 어떻게 만들어지는가" — 를 다룬다. 핵심은 **여러 생성기를 체인으로 묶어, 토큰 타입과 포맷에 맞는 생성기가 응답하도록** 한 설계와, 클레임을 가로채 바꾸는 `OAuth2TokenCustomizer` 훅이다.

## 핵심 타입

```
OAuth2TokenGenerator<T extends OAuth2Token>   (@FunctionalInterface)
   T generate(OAuth2TokenContext context)     ── 못 만들면 null 반환
        ▲
        │ 구현
   ┌────┴───────────────────────────────────────────┐
   │ DelegatingOAuth2TokenGenerator                  │  여러 생성기를 순회,
   │   tokenGenerators[] 를 순회하며 첫 non-null 채택  │  첫 non-null 채택
   ├─────────────────────────────────────────────────┤
   │ JwtGenerator              JWT access / OIDC id   │
   │ OAuth2AccessTokenGenerator 불투명(opaque) access  │
   │ OAuth2RefreshTokenGenerator 리프레시 토큰          │
   └─────────────────────────────────────────────────┘

   OAuth2TokenContext (입력)   ──▶ get(RegisteredClient/principal/issuer/
        DefaultOAuth2TokenContext     authorization/tokenType/grantType/...)

   OAuth2TokenCustomizer<C>    (클레임 후킹)
        JwtEncodingContext          JWT 헤더·클레임 빌더 접근
        OAuth2TokenClaimsContext    불투명 토큰 클레임 빌더 접근
```

### OAuth2TokenGenerator — "못 만들면 null"

`token/OAuth2TokenGenerator.java`는 한 메서드짜리 함수형 인터페이스다. 계약의 핵심은 **자기가 못 만드는 타입이면 `null`을 반환**한다는 규칙이다.

```java
@FunctionalInterface
public interface OAuth2TokenGenerator<T extends OAuth2Token> {
    @Nullable T generate(OAuth2TokenContext context);   // 지원 안 하면 null
}
```

이 "지원 안 하면 null" 규약이 체인을 가능하게 한다. `DelegatingOAuth2TokenGenerator`는 등록된 생성기를 순서대로 호출하다 처음으로 non-null을 주는 생성기의 결과를 채택한다.

```java
public @Nullable OAuth2Token generate(OAuth2TokenContext context) {
    for (OAuth2TokenGenerator<OAuth2Token> tokenGenerator : this.tokenGenerators) {
        OAuth2Token token = tokenGenerator.generate(context);
        if (token != null) return token;
    }
    return null;
}
```

기본 배선은 `new DelegatingOAuth2TokenGenerator(jwtGenerator, accessTokenGenerator, refreshTokenGenerator)`다(config 모듈의 `OAuth2ConfigurerUtils`). 따라서 한 번의 `generate(ACCESS_TOKEN)` 호출이 들어오면:

- 액세스 토큰 포맷이 `SELF_CONTAINED`(기본) → `JwtGenerator`가 JWT를 반환하고 끝.
- `REFERENCE`(불투명) → `JwtGenerator`는 `null`을 반환하고, `OAuth2AccessTokenGenerator`가 불투명 토큰을 만든다.

### JwtGenerator — 타입에 따라 자기 일이 아니면 빠진다

`token/JwtGenerator.java`. 가장 먼저 하는 일이 **"이건 내 일인가" 판별**이다.

```java
public @Nullable Jwt generate(OAuth2TokenContext context) {
    if (!OAuth2TokenType.ACCESS_TOKEN.equals(context.getTokenType()) &&
        !OidcParameterNames.ID_TOKEN.equals(context.getTokenType().getValue()))
        return null;                                   // access도 id도 아님 → 패스
    if (OAuth2TokenType.ACCESS_TOKEN.equals(context.getTokenType()) &&
        !OAuth2TokenFormat.SELF_CONTAINED.equals(
            context.getRegisteredClient().getTokenSettings().getAccessTokenFormat()))
        return null;                                   // 불투명 access 요청 → 패스
    ...
}
```

이 두 `return null`이 곧 체인 분기의 핵심이다. `JwtGenerator`는 "JWT 액세스 토큰" 또는 "ID 토큰"만 자기 일로 본다. 그 외에는 빠져서 다음 생성기에 기회를 넘긴다.

이후 표준 클레임을 채운다. issuer는 02장의 `AuthorizationServerContext`에서, subject는 principal에서, audience는 client_id에서 온다.

```java
String issuer = context.getAuthorizationServerContext().getIssuer();
JwtClaimsSet.Builder claimsBuilder = JwtClaimsSet.builder()
        .issuer(issuer)
        .subject(principal.getName())
        .audience(Collections.singletonList(registeredClient.getClientId()))
        .issuedAt(issuedAt).expiresAt(expiresAt).id(UUID.randomUUID().toString());
```

타입별 분기도 명확하다. 액세스 토큰이면 `nbf`와 `scope` 클레임을, ID 토큰이면 `azp`·`nonce`·`sid`·`auth_time` 같은 OIDC 클레임을 붙인다. 만료 시각도 다르다 — 액세스 토큰은 `TokenSettings.getAccessTokenTimeToLive()`, ID 토큰은 30분 고정(코드의 TODO로 표시됨).

마지막에 커스터마이저를 호출하고 인코딩한다.

```java
if (this.jwtCustomizer != null) {
    JwtEncodingContext jwtContext = JwtEncodingContext.with(jwsHeaderBuilder, claimsBuilder)
            .registeredClient(...).principal(principal)
            .authorizationServerContext(...).tokenType(...).authorizationGrantType(...)
            .build();
    this.jwtCustomizer.customize(jwtContext);     // 헤더/클레임 수정 기회
}
Jwt jwt = this.jwtEncoder.encode(JwtEncoderParameters.from(jwsHeader, claims));
```

서명은 직접 하지 않고 `oauth2-jose`의 `JwtEncoder`(Nimbus 기반)에 위임한다. 그 `JwtEncoder`가 쓰는 키가 곧 `/oauth2/jwks`로 공개되는 키다(06장).

### OAuth2TokenContext — 생성기의 단일 입력

`token/OAuth2TokenContext.java`는 `Context`(키-값 맵)를 상속한 인터페이스로, 생성기가 필요로 하는 모든 입력을 담는다: `RegisteredClient`, principal, `AuthorizationServerContext`, `OAuth2Authorization`, authorizedScopes, `OAuth2TokenType`, `AuthorizationGrantType`, authorizationGrant, 그리고 DPoP proof 같은 임의 키.

`default` 메서드로 타입 안전한 getter를 제공하고, `AbstractBuilder`로 채운다. 04장에서 본 `DefaultOAuth2TokenContext.builder()....build()`가 이 빌더의 구현이다. 생성기·커스터마이저가 같은 context를 공유하므로, **새 입력이 필요하면 `put(key, value)`로 context에 실어 보내면** 커스터마이저까지 그 값을 받을 수 있다(예: `OAuth2TokenContext.DPOP_PROOF_KEY`).

## 동작 흐름 — 액세스 토큰 한 장이 만들어지는 길

```
Provider (04장)
   │ DefaultOAuth2TokenContext.builder()
   │   .registeredClient/principal/authorizationServerContext/...
   │   .tokenType(ACCESS_TOKEN).build()
   ▼
DelegatingOAuth2TokenGenerator.generate(ctx)
   │
   ├─ JwtGenerator.generate(ctx)
   │     accessTokenFormat == SELF_CONTAINED ?
   │        예 → 클레임 구성 → jwtCustomizer.customize() → jwtEncoder.encode() → Jwt 반환 ─┐
   │        아니오 → null                                                                  │
   ├─ OAuth2AccessTokenGenerator.generate(ctx)   (위에서 null 일 때만 도달)               │
   │     불투명 토큰 값 + 클레임 → OAuth2AccessToken 반환                                  │
   ▼                                                                                       │
첫 non-null 채택 ◀───────────────────────────────────────────────────────────────────────┘
   │
Provider: OAuth2AuthenticationProviderUtils.accessToken(builder, generated, ctx)
   → 토큰 + 클레임 메타데이터를 OAuth2Authorization 에 적재
```

생성된 토큰이 `ClaimAccessor`를 구현하면(JWT가 그렇다) 그 클레임이 `OAuth2Authorization.Token`의 `CLAIMS_METADATA_NAME` 메타데이터로 저장된다(01장). 그래서 나중에 introspection이 토큰을 다시 디코딩하지 않고도 저장된 클레임을 그대로 응답에 쓸 수 있다(06장).

## 핵심 메서드 / 커스터마이징

- **`OAuth2TokenCustomizer<JwtEncodingContext>`**: JWT(액세스·ID)의 헤더·클레임을 바꾸는 가장 흔한 훅. `JwtGenerator.setJwtCustomizer(...)`로 주입되며, config의 `DefaultOAuth2TokenCustomizers`가 권한(authorities)을 클레임에 싣는 기본 커스터마이저를 제공한다. 사용자는 `OAuth2TokenCustomizer<JwtEncodingContext>` 빈을 정의하기만 하면 된다.
- **`OAuth2TokenCustomizer<OAuth2TokenClaimsContext>`**: 불투명(REFERENCE) 액세스 토큰의 클레임을 바꾸는 훅(`OAuth2AccessTokenGenerator` 쪽).
- **타입 분기는 context로**: 커스터마이저 안에서 `context.getTokenType()`·`context.getAuthorizationGrantType()`으로 "지금 만드는 게 액세스인지 ID인지, 어떤 그랜트인지"를 보고 조건부로 클레임을 넣는다.

## 설계 포인트 / 확장점

- **체인 + null 규약**: "지원 안 하면 null"이라는 단순 규약 하나로, 토큰 타입·포맷별 분기를 if-else 없이 생성기 목록의 순서로 표현했다. 새 토큰 종류를 추가하려면 `OAuth2TokenGenerator` 구현 하나를 체인에 더하면 된다.
- **생성기 전체 교체**: config의 `tokenGenerator(...)`로 `DelegatingOAuth2TokenGenerator` 자체를 갈아끼울 수 있다. 빈으로 `OAuth2TokenGenerator`를 정의하면 configurer가 집어간다.
- **서명 분리**: JWT 서명/키 관리는 이 모듈이 아니라 `oauth2-jose`의 `JwtEncoder`와 `JWKSource`가 맡는다. 이 모듈은 "어떤 클레임을 담을지"만 책임진다 — 관심사가 깔끔히 나뉜다.
- **ID 토큰 TTL은 아직 고정**: ID 토큰 만료가 30분 하드코딩(코드 TODO)이라는 점은 알아둘 만하다. 액세스/리프레시는 `TokenSettings`로 조정되지만 ID 토큰 TTL은 현재 설정 노출이 없다.

## 정리

토큰 생성은 `OAuth2TokenGenerator` 체인이 담당한다. `JwtGenerator`는 JWT 액세스·ID 토큰을, `OAuth2AccessTokenGenerator`는 불투명 토큰을, `OAuth2RefreshTokenGenerator`는 리프레시 토큰을 만들고, `DelegatingOAuth2TokenGenerator`가 "지원 안 하면 null" 규약으로 이들을 순회한다. 모든 입력은 `OAuth2TokenContext` 한 곳에 모이고, `OAuth2TokenCustomizer`가 마지막에 클레임을 손볼 기회를 준다. 실제 서명은 `oauth2-jose`에 위임된다 — 이 분리가 다음 장의 JWK 엔드포인트로 이어진다.
