# 02. 필터 체인과 요청 처리 골격 — 모든 엔드포인트가 공유하는 패턴

## 무엇을 / 왜

인가 서버에는 인가·토큰·검사·폐기·JWK·UserInfo 등 십수 개의 엔드포인트가 있다. 이들을 하나하나 외우는 건 무의미하다. 대신 **거의 모든 엔드포인트가 똑같은 골격을 공유**한다는 사실을 잡으면 전체가 한눈에 들어온다. 이 챕터는 그 공통 골격(Filter → Converter → AuthenticationManager → Provider → Handler)과, 모든 요청을 감싸는 `AuthorizationServerContext`, 그리고 토큰 발급의 전제인 클라이언트 인증 Filter를 다룬다.

## 핵심 타입

이 모듈은 새로운 인증 프레임워크를 만들지 않았다. **`spring-security-web`의 표준 인증 처리 패턴을 OAuth2 도메인에 그대로 적용**했다.

```
HTTP 요청
   │
   ▼
OncePerRequestFilter (엔드포인트 Filter)
   │  1) RequestMatcher.matches() 로 내 URI 인가 확인
   │  2) AuthenticationConverter.convert(request)  ── HTTP → *AuthenticationToken
   │  3) AuthenticationManager.authenticate(token)
   │        │
   │        ▼
   │     AuthenticationProvider (그랜트/엔드포인트별 비즈니스 로직)
   │        - RegisteredClientRepository / OAuth2AuthorizationService 읽고 씀
   │        - OAuth2TokenGenerator 호출
   │        ▼
   │     인증 결과 *AuthenticationToken
   │  4a) 성공 → AuthenticationSuccessHandler  (HTTP 응답 작성)
   │  4b) 실패 → AuthenticationFailureHandler  (OAuth2Error 응답)
```

이 패턴을 따르는 Filter들(모두 `web/` 또는 `oidc/web/`):

| Filter | 기본 URI | 위임 Provider |
|--------|----------|---------------|
| `OAuth2AuthorizationEndpointFilter` | `/oauth2/authorize` | AuthorizationCodeRequest / Consent |
| `OAuth2TokenEndpointFilter` | `/oauth2/token` | AuthorizationCode / Refresh / ClientCredentials / DeviceCode / TokenExchange |
| `OAuth2ClientAuthenticationFilter` | (토큰류 엔드포인트) | ClientSecret / JwtClientAssertion / PublicClient / X509 |
| `OAuth2TokenIntrospectionEndpointFilter` | `/oauth2/introspect` | TokenIntrospection |
| `OAuth2TokenRevocationEndpointFilter` | `/oauth2/revoke` | TokenRevocation |
| `OAuth2Device*` / `PushedAuthorizationRequest*` / `ClientRegistration*` | 각 설정값 | 해당 Provider |
| `NimbusJwkSetEndpointFilter` | `/oauth2/jwks` | (없음 — Provider 미사용, 03·06장) |
| `OAuth2AuthorizationServerMetadataEndpointFilter` | `/.well-known/...` | (없음) |

`NimbusJwkSetEndpointFilter`와 메타데이터 Filter는 인증할 게 없어 Provider를 쓰지 않고 직접 JSON을 쓴다. 나머지는 모두 위 골격을 따른다.

### Converter — HTTP를 도메인 인증으로

각 Filter는 생성자에서 `DelegatingAuthenticationConverter`로 여러 `AuthenticationConverter`를 묶는다. 예를 들어 토큰 엔드포인트는:

```java
// OAuth2TokenEndpointFilter 생성자
this.authenticationConverter = new DelegatingAuthenticationConverter(Arrays.asList(
        new OAuth2AuthorizationCodeAuthenticationConverter(),
        new OAuth2RefreshTokenAuthenticationConverter(),
        new OAuth2ClientCredentialsAuthenticationConverter(),
        new OAuth2DeviceCodeAuthenticationConverter(),
        new OAuth2TokenExchangeAuthenticationConverter()));
```

각 Converter는 `grant_type` 파라미터를 보고 자기 것이면 해당 `*AuthenticationToken`을, 아니면 `null`을 돌려준다. `DelegatingAuthenticationConverter`가 첫 번째 non-null을 채택한다. 즉 **"어떤 그랜트인지" 분기 로직이 Converter 목록에 데이터로 표현**되어 있고, 새 그랜트를 추가하려면 Converter 하나를 끼우면 된다.

### AuthorizationServerContext — "현재 요청"의 환경

`context/AuthorizationServerContext.java`, `context/AuthorizationServerContextHolder.java`.

토큰을 발급하려면 issuer URL이 필요한데, 멀티테넌트에서는 issuer가 요청마다 다를 수 있다. 그래서 `AuthorizationServerContextFilter`(config 모듈)가 **모든 인가 서버 요청의 가장 앞단**에서 issuer + settings를 묶은 `AuthorizationServerContext`를 만들어 `ThreadLocal`에 심고, 요청이 끝나면 비운다.

```java
// AuthorizationServerContextFilter (config 모듈)
try {
    String issuer = this.issuerResolver.resolve(request);     // 고정 issuer or 요청 호스트
    AuthorizationServerContext ctx =
        new DefaultAuthorizationServerContext(issuer, this.authorizationServerSettings);
    AuthorizationServerContextHolder.setContext(ctx);
    filterChain.doFilter(request, response);
}
finally {
    AuthorizationServerContextHolder.resetContext();
}
```

이 필터는 `SecurityContextHolderFilter` **뒤에** 추가된다(`OAuth2AuthorizationServerConfigurer.configure()`의 `addFilterAfter(...)`). 덕분에 모든 Provider는 `AuthorizationServerContextHolder.getContext().getIssuer()`로 현재 issuer를 ThreadLocal에서 꺼낼 수 있다 — 파라미터로 issuer를 줄줄이 넘길 필요가 없다. `JwtGenerator`(05장)와 `OAuth2AuthorizationCodeAuthenticationProvider`(04장)가 이 홀더를 직접 읽는다.

issuer 미설정 시 `IssuerResolver`는 요청 URL에서 엔드포인트 경로를 떼어내 issuer를 동적으로 만든다. 이것이 `multipleIssuersAllowed` 멀티테넌트의 동작 원리다.

## 동작 흐름 — 클라이언트 인증 Filter

토큰류 엔드포인트(`/oauth2/token`, `/oauth2/introspect`, `/oauth2/revoke` 등)는 **먼저 클라이언트를 인증**해야 한다. 그 일을 `OAuth2ClientAuthenticationFilter`가 한다. 이 필터는 위 골격을 따르되, 인증에 성공하면 **응답을 쓰지 않고 `SecurityContext`에 클라이언트 principal을 심은 뒤 체인을 계속 진행**한다는 점이 다르다.

```
POST /oauth2/token  (Authorization: Basic / 또는 body의 client credentials)
   │
   ▼ OAuth2ClientAuthenticationFilter
   │  convert(): Basic? PostBody? JWT assertion? PKCE public? X509?
   │      → OAuth2ClientAuthenticationToken (미인증)
   │  validateClientIdentifier(): client_id 문자가 %x20-7E 범위인지
   │  authenticationManager.authenticate(token)
   │      → ClientSecretAuthenticationProvider 등에서 비밀 검증
   │      → OAuth2ClientAuthenticationToken (인증됨, RegisteredClient 포함)
   │  onAuthenticationSuccess(): SecurityContextHolder 에 저장
   │  filterChain.doFilter(...)   ← 응답 안 쓰고 다음 필터로
   ▼
   OAuth2TokenEndpointFilter (이제 인증된 클라이언트 principal 사용 가능)
```

핵심 코드:

```java
// OAuth2ClientAuthenticationFilter.doFilterInternal
Authentication authenticationRequest = this.authenticationConverter.convert(request);
...
if (authenticationRequest != null) {
    validateClientIdentifier(authenticationRequest);
    Authentication authenticationResult = this.authenticationManager.authenticate(authenticationRequest);
    this.authenticationSuccessHandler.onAuthenticationSuccess(request, response, authenticationResult);
}
filterChain.doFilter(request, response);   // 항상 체인 진행
```

```java
private void onAuthenticationSuccess(...) {
    SecurityContext securityContext = SecurityContextHolder.createEmptyContext();
    securityContext.setAuthentication(authentication);   // 클라이언트를 principal로
    SecurityContextHolder.setContext(securityContext);
}
```

이렇게 클라이언트 인증이 `SecurityContext`에 자리잡으면, 뒤따르는 토큰 엔드포인트 Provider들은 `OAuth2AuthenticationProviderUtils.getAuthenticatedClientElseThrowInvalidClient(...)`로 그 principal을 꺼내 `RegisteredClient`를 얻는다(04장에서 반복적으로 등장).

실패 처리도 spec을 따른다. `invalid_client`이면 HTTP 401, 그 외에는 400을 반환하고, **에러 코드만** 노출해 정보 누출을 줄인다.

```java
OAuth2Error error = ((OAuth2AuthenticationException) exception).getError();
if (OAuth2ErrorCodes.INVALID_CLIENT.equals(error.getErrorCode()))
    httpResponse.setStatusCode(HttpStatus.UNAUTHORIZED);
else
    httpResponse.setStatusCode(HttpStatus.BAD_REQUEST);
OAuth2Error errorResponse = new OAuth2Error(error.getErrorCode());  // 코드만
```

지원하는 클라이언트 인증 방식(Converter ↔ Provider 짝):

| 방식 | Converter | Provider |
|------|-----------|----------|
| `client_secret_basic` | `ClientSecretBasicAuthenticationConverter` | `ClientSecretAuthenticationProvider` |
| `client_secret_post` | `ClientSecretPostAuthenticationConverter` | `ClientSecretAuthenticationProvider` |
| `private_key_jwt` / `client_secret_jwt` | `JwtClientAssertionAuthenticationConverter` | `JwtClientAssertionAuthenticationProvider` |
| `none` (퍼블릭 + PKCE) | `PublicClientAuthenticationConverter` | `PublicClientAuthenticationProvider` |
| `tls_client_auth` / `self_signed_tls_client_auth` | `X509ClientCertificateAuthenticationConverter` | `X509ClientCertificateAuthenticationProvider` |

## 전체 필터 배치

`OAuth2AuthorizationServerConfigurer`(config 모듈)가 `SecurityFilterChain`에 필터를 꽂는 순서를 단순화하면 다음과 같다.

```
SecurityContextHolderFilter
   └▶ AuthorizationServerContextFilter   (issuer/settings ThreadLocal 세팅)
        ...
   ┌── (각 *EndpointConfigurer 가 추가) ──┐
   │  NimbusJwkSetEndpointFilter           (AbstractPreAuthenticatedProcessingFilter 앞)
   │  OAuth2ClientAuthenticationFilter
   │  OAuth2AuthorizationEndpointFilter
   │  OAuth2TokenEndpointFilter
   │  OAuth2TokenIntrospection/Revocation EndpointFilter
   │  (옵션) Device / PAR / ClientRegistration / OIDC Filter
   └───────────────────────────────────────┘
```

configurer는 `getEndpointsMatcher()`로 이 엔드포인트 전체를 가리키는 matcher를 만들고, 그 경로들에 대해 **CSRF를 끈다**(`csrf.ignoringRequestMatchers(endpointsMatcher)`). 토큰 엔드포인트는 브라우저 폼이 아니라 클라이언트 자격증명으로 보호되기 때문이다. 또 `OAuth2TokenEndpointConfigurer` 등 토큰류 엔드포인트에는 인증 실패 시 401을 주도록 `HttpStatusEntryPoint`를 등록한다.

## 설계 포인트 / 확장점

- **균일한 골격**: 모든 엔드포인트가 `Filter(matcher+converter+manager+success/failure)` 형태라, 각 Filter에 `setAuthenticationConverter`/`setAuthenticationSuccessHandler`/`setAuthenticationFailureHandler` 세터가 있다. config 모듈의 `*EndpointConfigurer`가 이 세터를 통해 커스텀 Converter/Handler를 주입한다.
- **관심사 분리**: "HTTP 파싱"은 Converter, "비즈니스 규칙"은 Provider, "응답 직렬화"는 Handler로 갈라져 있다. 그래서 한 그랜트의 검증 로직만 바꾸고 싶으면 Provider만 건드리면 된다.
- **ThreadLocal 기반 문맥**: `AuthorizationServerContextHolder`와 `SecurityContextHolder` 두 ThreadLocal이 "현재 요청의 issuer"와 "현재 클라이언트/오너"를 운반한다. Provider 시그니처가 단순해지는 대가로, 비동기 경계를 넘을 때는 문맥 전파에 주의해야 한다.

## 정리

인가 서버의 엔드포인트는 전부 같은 리듬을 탄다: **matcher로 잡고 → Converter로 HTTP를 `Authentication`으로 만들고 → `AuthenticationManager`가 적절한 Provider에 위임하고 → Handler가 응답을 쓴다**. 그 앞단의 `AuthorizationServerContextFilter`가 issuer/settings를, `OAuth2ClientAuthenticationFilter`가 클라이언트 principal을 ThreadLocal에 깔아둔다. 이 골격을 머리에 넣으면 03~06장의 각 엔드포인트는 "이 Provider가 도메인 모델로 무엇을 하는가"만 보면 된다.
