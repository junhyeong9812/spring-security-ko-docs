# 01. 개념과 도메인 모델 — 인가 서버가 "기억하는 것"

## 무엇을 / 왜

인가 서버는 본질적으로 **상태를 가진 발급기**다. 토큰 하나를 발급하려면 "이 클라이언트가 누구이며 무엇을 할 수 있는지", "이 인가가 누구에게 어떤 scope로 주어졌는지", "이 코드가 이미 한 번 쓰였는지"를 알아야 한다. 이 챕터는 그 상태를 담는 세 핵심 모델 — `RegisteredClient`, `OAuth2Authorization`, `OAuth2AuthorizationConsent` — 와 그것을 저장하는 계약, 그리고 동작을 조정하는 `*Settings`를 다룬다.

엔드포인트나 그랜트 처리(03~04장)를 읽기 전에 이 도메인을 먼저 잡아야 한다. 모든 `AuthenticationProvider`가 결국 이 객체들을 읽고 쓰기 때문이다.

## 핵심 타입

```
   RegisteredClient ─────── "이 클라이언트는 누구인가" (등록 정보, 불변)
        │ withId() / from()
        │ clientId, clientAuthenticationMethods,
        │ authorizationGrantTypes, redirectUris, scopes,
        │ ClientSettings, TokenSettings
        ▼
   RegisteredClientRepository (계약)
        ├ InMemoryRegisteredClientRepository
        └ JdbcRegisteredClientRepository

   OAuth2Authorization ──── "이 발급은 어떤 상태인가" (집계 루트)
        │ withRegisteredClient() / from()
        │ registeredClientId, principalName, grantType,
        │ authorizedScopes, tokens(Map), attributes(Map)
        │   └ Token<T> : token + metadata(invalidated, claims)
        ▼
   OAuth2AuthorizationService (계약)
        ├ InMemoryOAuth2AuthorizationService
        └ JdbcOAuth2AuthorizationService

   OAuth2AuthorizationConsent ─ "리소스 오너가 허가한 권한"
        │ registeredClientId + principalName → Set<GrantedAuthority>
        ▼
   OAuth2AuthorizationConsentService (계약)
```

### RegisteredClient — 등록 클라이언트

`oauth2/oauth2-authorization-server/src/main/java/org/springframework/security/oauth2/server/authorization/client/RegisteredClient.java`.

빌더로만 만드는 불변 객체다. 단순한 데이터 묶음이 아니라 `build()` 단계에서 **규약을 강제**한다.

```java
public RegisteredClient build() {
    Assert.hasText(this.clientId, "clientId cannot be empty");
    Assert.notEmpty(this.authorizationGrantTypes, "authorizationGrantTypes cannot be empty");
    if (this.authorizationGrantTypes.contains(AuthorizationGrantType.AUTHORIZATION_CODE)) {
        Assert.notEmpty(this.redirectUris, "redirectUris cannot be empty");
    }
    ...
    if (this.clientSettings == null) {
        ClientSettings.Builder builder = ClientSettings.builder();
        if (isPublicClientType()) {                 // grant=code + method=NONE 단독
            builder.requireProofKey(true)           // PKCE 강제
                   .requireAuthorizationConsent(true);
        }
        this.clientSettings = builder.build();
    }
    validateScopes(); validateRedirectUris(); validatePostLogoutRedirectUris();
    return create();
}
```

여기서 두 가지가 중요하다. 첫째, **퍼블릭 클라이언트(인증 방식이 `NONE`뿐인 authorization_code 클라이언트)는 자동으로 PKCE와 consent를 강제**한다 — OAuth 2.1의 보안 기본값을 코드가 직접 박아둔 것이다. 둘째, `redirectUri`는 URI 파싱 후 **fragment(`#...`)가 있으면 거부**한다(`validateRedirectUri`). 등록 시점에 잘못된 값을 차단해 런타임 공격 표면을 줄인다.

`ClientSettings`(PKCE 요구·동의 요구·클라이언트 JWK URL 등)와 `TokenSettings`(토큰 수명·포맷·서명 알고리즘)를 클라이언트마다 따로 들고 있어, **클라이언트 단위로 정책을 다르게** 줄 수 있다.

### OAuth2Authorization — 발급 상태의 집계 루트

`.../OAuth2Authorization.java`. 이 모듈에서 가장 중심이 되는 객체다. 한 번의 인가가 만들어낸 모든 산출물(인가 코드, 액세스 토큰, 리프레시 토큰, ID 토큰)을 **한 묶음**으로 보관한다.

핵심 구조는 `Map<Class<? extends OAuth2Token>, Token<?>> tokens` 다. 토큰 종류별로 한 칸씩 차지하고, 각 `Token<T>`는 토큰 값 + **메타데이터**(invalidated 여부, claims)를 함께 들고 있다.

```java
public boolean isActive() {                 // Token<T> 의 메서드
    return !isInvalidated() && !isExpired() && !isBeforeUse();
}
```

`findByToken(...)` 으로 인가를 조회한 뒤 이 `isActive()` 하나로 "이 토큰을 받아줘도 되는가"를 판단한다. 토큰 검사·코드 교환·리프레시가 전부 이 메서드에 의존한다.

특히 주목할 것은 빌더의 `invalidate()` 다. 리프레시 토큰을 무효화하면 **연쇄적으로** 액세스 토큰과 인가 코드까지 무효화한다.

```java
public <T extends OAuth2Token> Builder invalidate(T token) {
    token(token, m -> m.put(Token.INVALIDATED_METADATA_NAME, true));
    if (OAuth2RefreshToken.class.isAssignableFrom(token.getClass())) {
        // 리프레시를 죽이면 access token + authorization code 도 함께 죽인다
        ...accessToken...  ...authorizationCode...
    }
    return this;
}
```

이 연쇄 무효화 덕분에 "인가 코드 재사용 공격이 탐지되면 이미 발급된 토큰까지 폐기"(04장)하는 보안 동작이 자연스럽게 구현된다.

`attributes`에는 직렬화 가능한 부가 상태가 들어간다. 대표적으로 인가 코드 흐름에서 `Principal.class.getName()`(리소스 오너 `Authentication`)과 `OAuth2AuthorizationRequest.class.getName()`(원래 인가 요청)이 저장되어, 나중에 코드→토큰 교환 시 다시 꺼내 쓴다.

### OAuth2AuthorizationConsent — 허가된 권한

`.../OAuth2AuthorizationConsent.java`. `(registeredClientId, principalName)` 키에 대해 리소스 오너가 허가한 `Set<GrantedAuthority>` 를 보관한다. 권한은 `SCOPE_` 접두사를 붙여 저장하고, `getScopes()` 는 그 접두사를 떼어 scope 집합으로 돌려준다.

```java
private static final String AUTHORITIES_SCOPE_PREFIX = "SCOPE_";
public Set<String> getScopes() { /* SCOPE_ 접두 권한 → scope 문자열 */ }
```

같은 클라이언트가 다음에 같은 scope를 다시 요청하면, 이미 동의가 있으므로 동의 화면을 건너뛴다(03장).

## 저장 계약 — 두 개의 간결한 인터페이스

```
OAuth2AuthorizationService          OAuth2AuthorizationConsentService
  void save(authorization)            void save(consent)
  void remove(authorization)          void remove(consent)
  findById(id)                        findById(clientId, principalName)
  findByToken(token, tokenType)
```

`findByToken(token, tokenType)` 의 `tokenType`은 **힌트**다. introspection처럼 토큰 종류를 모를 때는 `null`을 넘기고(`OAuth2TokenIntrospectionAuthenticationProvider` 참고), 코드 교환처럼 종류가 확실할 때는 `new OAuth2TokenType("code")`를 넘겨 검색을 좁힌다.

두 계약 모두 `InMemory*`(개발·테스트용, `ConcurrentHashMap`)와 `Jdbc*`(운영용, 테이블 기반) 구현을 제공한다. 운영에서는 인스턴스 재시작·수평 확장을 견디려면 `Jdbc*`를 써야 한다 — 인메모리 구현은 발급 상태가 JVM 메모리에만 있기 때문이다.

## *Settings — 동작을 조정하는 키-값 묶음

세 `Settings` 클래스는 모두 `AbstractSettings`(불변 `Map<String,Object>` 래퍼)를 상속한다. 기본값이 곧 정책이므로 표로 정리한다.

| 설정 | 기본값 | 의미 |
|------|--------|------|
| `AuthorizationServerSettings.authorizationEndpoint` | `/oauth2/authorize` | 인가 엔드포인트 |
| ` " .tokenEndpoint` | `/oauth2/token` | 토큰 엔드포인트 |
| ` " .jwkSetEndpoint` | `/oauth2/jwks` | 공개키 |
| ` " .tokenIntrospectionEndpoint` | `/oauth2/introspect` | 토큰 검사 |
| ` " .tokenRevocationEndpoint` | `/oauth2/revoke` | 토큰 폐기 |
| ` " .oidcUserInfoEndpoint` | `/userinfo` | OIDC UserInfo |
| ` " .issuer` | (없음) | issuer 식별자. 미설정 시 요청에서 동적 해석 |
| `ClientSettings.requireProofKey` | `true` | PKCE 강제 |
| ` " .requireAuthorizationConsent` | `false` | 동의 화면 요구 |
| `TokenSettings.accessTokenTimeToLive` | 5분 | 액세스 토큰 수명 |
| ` " .accessTokenFormat` | `SELF_CONTAINED`(JWT) | 액세스 토큰 포맷 |
| ` " .refreshTokenTimeToLive` | 60분 | 리프레시 토큰 수명 |
| ` " .reuseRefreshTokens` | `true` | 리프레시 시 토큰 재사용 여부 |
| ` " .idTokenSignatureAlgorithm` | `RS256` | ID 토큰 서명 알고리즘 |

`AuthorizationServerSettings.builder()`가 이 기본 URI들을 한 번에 박아넣는다. `accessTokenFormat`이 `SELF_CONTAINED`(기본)이면 JWT 액세스 토큰이 발급되고, `REFERENCE`이면 불투명 토큰이 발급된다 — 05장의 `JwtGenerator`/`OAuth2AccessTokenGenerator` 선택을 이 설정이 가른다.

`issuer`와 `multipleIssuersAllowed`는 동시에 설정할 수 없다(`build()`에서 검증). issuer를 고정하면 단일 테넌트, 비우면 요청 호스트에서 동적으로 issuer를 뽑는 멀티테넌트(02장 `AuthorizationServerContext`)가 된다.

## 설계 포인트 / 확장점

- **불변 + 빌더**: 세 도메인 객체 모두 protected 생성자 + 정적 `withXxx()`/`from()` 빌더로만 만든다. `from(existing)`은 기존 객체를 복제해 일부만 바꾼 새 객체를 만드는 데 쓰여, 동시성 문제 없이 상태 전이를 표현한다(예: `OAuth2Authorization.from(auth).invalidate(code).build()`).
- **저장소 교체**: `RegisteredClientRepository`·`OAuth2AuthorizationService`·`OAuth2AuthorizationConsentService` 빈을 직접 정의하면 Redis·MongoDB 등 임의 저장소로 갈아끼울 수 있다. config 모듈의 configurer가 이 빈들을 shared object로 집어간다.
- **정책의 위치**: "보안 기본값"이 설정 객체의 `builder()` 기본값과 `RegisteredClient.build()` 검증에 박혀 있다. 정책을 바꾸려면 `ClientSettings`/`TokenSettings`를 클라이언트별로 다르게 주면 된다.

## 정리

`RegisteredClient`는 "누가", `OAuth2Authorization`은 "무엇이 발급되어 어떤 상태인가", `OAuth2AuthorizationConsent`는 "오너가 무엇을 허락했는가"를 기억한다. 이 세 객체와 그 저장 계약, 그리고 `*Settings`가 인가 서버의 정적 모델이다. 다음 장부터는 이 모델을 읽고 쓰는 **요청 처리 흐름**으로 들어간다.
