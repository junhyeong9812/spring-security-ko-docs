# 03. 인가 엔드포인트와 동의 흐름 — `/oauth2/authorize`

## 무엇을 / 왜

Authorization Code Grant는 OAuth 2.1의 중심 흐름이다. 브라우저를 통해 리소스 오너에게 "이 클라이언트에게 권한을 줄래?"라고 묻고, 동의를 받으면 **인가 코드(authorization code)** 를 클라이언트의 redirect_uri로 돌려준다. 이 챕터는 그 일을 하는 `OAuth2AuthorizationEndpointFilter`와, 그 뒤의 두 `AuthenticationProvider`(인가 요청 검증 + 동의 제출 처리)를 따라간다. 실제 토큰은 여기서 나오지 않는다 — 코드만 나온다(토큰 교환은 04장).

## 핵심 타입

```
GET/POST /oauth2/authorize
   │
   ▼ OAuth2AuthorizationEndpointFilter
   │   (선행) OAuth2AuthorizationCodeRequestValidatingFilter  ── 사전검증 후 request attribute 로 전달
   │   convert():
   │     OAuth2AuthorizationCodeRequestAuthenticationConverter   (response_type=code 인 최초 요청)
   │     OAuth2AuthorizationConsentAuthenticationConverter       (동의 폼 제출)
   │
   ├─▶ OAuth2AuthorizationCodeRequestAuthenticationProvider
   │      → 동의 필요? → OAuth2AuthorizationConsentAuthenticationToken 반환 (코드 아님)
   │      → 동의 불필요 → OAuth2AuthorizationCodeRequestAuthenticationToken (코드 포함)
   │
   └─▶ OAuth2AuthorizationConsentAuthenticationProvider
          → 동의 저장 후 인가 코드 발급
```

요청 종류를 가르는 matcher가 중요하다. 같은 `POST /oauth2/authorize`라도 `response_type`·`redirect_uri`·`code_challenge` 등이 **없으면** 동의 제출로 간주한다(`createAuthorizationConsentMatcher`).

## 동작 흐름 1 — 인가 요청과 동의 게이트

`OAuth2AuthorizationCodeRequestAuthenticationProvider.authenticate()`가 이 흐름의 두뇌다. 단계별로:

```
1. (PAR이면) request_uri 해석 → 저장된 인가요청 복원
2. registeredClientRepository.findByClientId(clientId)   → 없으면 invalid_request
3. 미검증이면 4단계 검증을 순서대로:
     - grant_type 허용?           DEFAULT_AUTHORIZATION_GRANT_TYPE_VALIDATOR
     - redirect_uri / scope 유효?  authenticationValidator (교체 가능)
     - code_challenge (퍼블릭은 필수, PKCE) DEFAULT_CODE_CHALLENGE_VALIDATOR
     - prompt (OIDC)              DEFAULT_PROMPT_VALIDATOR
4. 리소스 오너가 인증됨?  아니면 → (prompt=none) login_required / 아니면 INVALID_REQUEST
5. OAuth2AuthorizationConsent 조회
6. 동의가 필요한가?  authorizationConsentRequired.test(...)
     ├ 필요   → 인가 저장(state 부여) + OAuth2AuthorizationConsentAuthenticationToken 반환
     └ 불필요 → 인가 코드 생성 + 인가 저장 + 코드 토큰 반환
```

"리소스 오너가 인증됐는가"를 보는 부분이 OAuth2와 일반 로그인을 잇는 접점이다.

```java
private static boolean isPrincipalAuthenticated(Authentication principal) {
    return principal != null
        && !AnonymousAuthenticationToken.class.isAssignableFrom(principal.getClass())
        && principal.isAuthenticated();
}
```

`/oauth2/authorize`는 일반 `SecurityFilterChain` 위에 있으므로, 사용자가 아직 로그인하지 않았다면 폼 로그인 필터가 먼저 가로채 로그인 페이지로 보낸다. 로그인 후 다시 이 엔드포인트로 돌아오면 principal이 인증된 상태가 되어 흐름이 이어진다. 즉 **이 모듈은 "사람을 어떻게 로그인시키는가"는 관여하지 않고, 이미 로그인된 principal을 전제로 인가만 처리**한다.

### 동의가 필요한지 판단하는 규칙

```java
private static boolean isAuthorizationConsentRequired(ctx) {
    if (!ctx.getRegisteredClient().getClientSettings().isRequireAuthorizationConsent())
        return false;                                   // 클라이언트가 동의 불요로 설정
    OAuth2AuthorizationRequest req = ctx.getAuthorizationRequest();
    if (req.getScopes().contains(OidcScopes.OPENID) && req.getScopes().size() == 1)
        return false;                                   // openid 단독은 동의 면제
    if (ctx.getAuthorizationConsent() != null
            && ctx.getAuthorizationConsent().getScopes().containsAll(req.getScopes()))
        return false;                                   // 이미 같은 scope 동의함
    return true;
}
```

세 가지 면제 조건이 명료하다. ① 클라이언트 설정상 동의 불요, ② `openid` scope만 요청(인증만 하므로 동의 불필요), ③ 과거에 같은(또는 더 넓은) scope를 이미 동의함. `setAuthorizationConsentRequired(Predicate)`로 이 규칙 전체를 교체할 수 있다.

동의가 필요하면 Provider는 코드를 만들지 않고 **state를 부여한 인가를 저장한 뒤** `OAuth2AuthorizationConsentAuthenticationToken`을 돌려준다. Filter는 이 토큰을 받으면 동의 화면을 띄운다.

```java
// OAuth2AuthorizationEndpointFilter.doFilterInternal
if (authenticationResult instanceof OAuth2AuthorizationConsentAuthenticationToken consentToken) {
    sendAuthorizationConsent(request, response, (OAuth2AuthorizationCodeRequestAuthenticationToken) authentication, consentToken);
    return;
}
```

`sendAuthorizationConsent`는 커스텀 consent 페이지 URI가 설정돼 있으면 그쪽으로 리다이렉트(scope·client_id·state를 쿼리로 전달)하고, 없으면 `DefaultConsentPage`로 기본 동의 HTML을 직접 렌더한다.

## 동작 흐름 2 — 동의 제출과 코드 발급

사용자가 동의 폼에서 scope를 골라 다시 `POST /oauth2/authorize`(response_type 없음)를 보내면 `OAuth2AuthorizationConsentAuthenticationProvider`가 처리한다. 이 Provider는 선택된 scope로 `OAuth2AuthorizationConsent`를 저장(`SCOPE_` 권한으로)하고, 인가 코드를 생성해 인가에 붙인 뒤 코드 토큰을 돌려준다. 이후는 흐름 1의 "동의 불필요" 분기와 합류한다.

코드가 준비되면 Filter의 success handler(`sendAuthorizationResponse`)가 redirect_uri에 `code`와 `state`를 붙여 브라우저를 클라이언트로 돌려보낸다.

```java
UriComponentsBuilder uriBuilder = UriComponentsBuilder.fromUriString(redirectUriForResponse)
        .queryParam(OAuth2ParameterNames.CODE, authorizationCode.getTokenValue());
if (StringUtils.hasText(state))
    uriBuilder.queryParam(OAuth2ParameterNames.STATE, UriUtils.encode(state, UTF_8));
String redirectUri = uriBuilder.build(true).toUriString();
this.redirectStrategy.sendRedirect(request, response, redirectUri);
```

전체 시퀀스를 한 장으로:

```
브라우저            인가서버                          저장소
  │  GET /authorize?response_type=code&client_id&scope&code_challenge
  │ ───────────────▶ │
  │                  │ 미인증? → (폼로그인 체인이 가로채 로그인)
  │ ◀─ 로그인 후 재요청 ─ │
  │                  │ Provider: 검증 → 동의 필요 판단
  │                  │  save(authorization, state)  ───────▶ │
  │ ◀── 동의 화면(HTML) ─ │
  │  POST /authorize (선택한 scope, state)
  │ ───────────────▶ │
  │                  │ ConsentProvider: save(consent) ─────▶ │
  │                  │  코드 생성 + save(authorization+code)▶ │
  │ ◀ 302 redirect_uri?code=...&state=... ─ │
```

## 핵심 메서드 / 안전장치

- **에러 시 redirect 여부 결정**: `createException(...)`은 에러 코드와 파라미터를 보고 redirect_uri를 줄지 말지 정한다. `client_id`·`state`·`request_uri` 관련 `invalid_request`는 **redirect를 막는다**(`redirectUri = null`) — 신뢰할 수 없는 redirect_uri로 에러를 흘리는 오픈 리다이렉트를 방지한다. 그 외 에러는 redirect_uri에 `error`·`error_description`을 붙여 클라이언트로 돌려준다(`sendErrorResponse`).
- **사전 검증 필터**: `OAuth2AuthorizationCodeRequestValidatingFilter`(엔드포인트 Filter의 내부 클래스)는 인가 엔드포인트 Filter보다 **앞서** 돌며, registeredClient 존재와 커스텀 validator를 미리 적용하고, 검증된 토큰을 request attribute로 넘긴다. PAR(request_uri) 요청은 이 사전 검증을 건너뛴다.
- **OIDC on/off에 따른 validator 주입**: `OAuth2AuthorizationServerConfigurer.init()`은 OIDC가 꺼져 있으면 "openid scope 요청을 거부"하는 validator를, 켜져 있으면 세션을 `SessionRegistry`에 등록하는 `SessionAuthenticationStrategy`를 주입한다. 즉 같은 인가 엔드포인트가 OIDC 활성 여부에 따라 다르게 행동한다.

## 설계 포인트 / 확장점

- **검증 파이프라인이 교체 가능**: `setAuthenticationValidator(Consumer<...Context>)`로 redirect_uri/scope 검증을 갈아끼울 수 있고, 기본 validator들은 `andThen`으로 합성된다(`getAuthenticationValidatorComposite`). 커스텀 검증을 끼우되 기본 PKCE·prompt 검증은 유지하는 식이 가능하다.
- **동의 UI 분리**: 동의 화면은 `setConsentPage(uri)`로 SPA·커스텀 페이지에 위임할 수 있다. 이 모듈은 "동의가 필요하다"는 신호와 필요한 파라미터(scope·state·client_id)만 제공한다.
- **코드 생성기 교체**: `setAuthorizationCodeGenerator(...)`로 코드 토큰 생성을 바꿀 수 있다(기본 `OAuth2AuthorizationCodeGenerator`, 05장 토큰 생성 추상화와 동일 계약).

## 정리

`/oauth2/authorize`는 두 단계다. 1단계 Provider가 요청을 검증하고 "동의가 필요한가"를 판단해, 필요하면 동의 토큰을, 아니면 인가 코드를 만든다. 동의가 필요하면 화면을 띄우고, 사용자가 제출하면 2단계 Provider가 동의를 저장하고 코드를 발급한다. 핵심은 **이 엔드포인트가 코드까지만 책임지고, 발급된 모든 상태를 `OAuth2Authorization`에 저장**한다는 것이다. 그 코드를 토큰으로 바꾸는 일은 다음 장의 토큰 엔드포인트가 맡는다.
