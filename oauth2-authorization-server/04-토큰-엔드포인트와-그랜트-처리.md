# 04. 토큰 엔드포인트와 그랜트 처리 — `/oauth2/token`

## 무엇을 / 왜

토큰 엔드포인트는 인가 서버의 심장이다. 클라이언트가 인가 코드·리프레시 토큰·클라이언트 자격증명 등을 들고 와서 **액세스 토큰(+리프레시 토큰, +ID 토큰)** 으로 바꿔 가는 곳이다. 이 챕터는 `OAuth2TokenEndpointFilter`가 그랜트를 어떻게 분기하는지, 그리고 가장 대표적인 `OAuth2AuthorizationCodeAuthenticationProvider`가 코드 한 장을 토큰 묶음으로 바꾸는 전 과정을 따라간다.

## 핵심 타입

```
POST /oauth2/token   (grant_type=...,  + 클라이언트는 이미 02장에서 인증됨)
   │
   ▼ OAuth2TokenEndpointFilter
   │   convert(): grant_type 으로 분기
   │     authorization_code   → OAuth2AuthorizationCodeAuthenticationToken
   │     refresh_token        → OAuth2RefreshTokenAuthenticationToken
   │     client_credentials   → OAuth2ClientCredentialsAuthenticationToken
   │     urn:...:device_code  → OAuth2DeviceCodeAuthenticationToken
   │     urn:...:token-exchange→ OAuth2TokenExchangeAuthenticationToken
   │   authenticationManager.authenticate(token)
   │        │
   │        ▼  (grant 별 Provider)
   │   OAuth2AccessTokenAuthenticationToken  (accessToken + refreshToken + extras)
   │   success → OAuth2AccessTokenResponseAuthenticationSuccessHandler (JSON 응답)
```

`OAuth2TokenEndpointFilter`는 가장 먼저 `grant_type` 파라미터가 정확히 1개인지 확인하고, Converter가 `null`을 주면 `unsupported_grant_type`을 던진다.

```java
String[] grantTypes = request.getParameterValues(OAuth2ParameterNames.GRANT_TYPE);
if (grantTypes == null || grantTypes.length != 1)
    throw createException(INVALID_REQUEST, GRANT_TYPE);
Authentication grantAuthn = this.authenticationConverter.convert(request);
if (grantAuthn == null)
    throw createException(UNSUPPORTED_GRANT_TYPE, GRANT_TYPE);
OAuth2AccessTokenAuthenticationToken result =
    (OAuth2AccessTokenAuthenticationToken) this.authenticationManager.authenticate(grantAuthn);
this.authenticationSuccessHandler.onAuthenticationSuccess(request, response, result);
```

그랜트별 Provider 일람:

| grant_type | Provider | 발급물 |
|------------|----------|--------|
| `authorization_code` | `OAuth2AuthorizationCodeAuthenticationProvider` | access(+refresh+id) |
| `refresh_token` | `OAuth2RefreshTokenAuthenticationProvider` | 새 access(+선택적 새 refresh) |
| `client_credentials` | `OAuth2ClientCredentialsAuthenticationProvider` | access (refresh 없음) |
| `urn:...:device_code` | `OAuth2DeviceCodeAuthenticationProvider` | access(+refresh) (폴링) |
| `urn:...:token-exchange` | `OAuth2TokenExchangeAuthenticationProvider` | 교환된 access (RFC 8693) |

## 동작 흐름 — 인가 코드를 토큰으로 (`OAuth2AuthorizationCodeAuthenticationProvider`)

이 Provider의 `authenticate()`는 검증과 발급을 한 호흡에 한다. 순서대로 보면:

```
1. 인증된 클라이언트 principal 꺼냄 → RegisteredClient
2. authorizationService.findByToken(code, CODE 타입)  → 없으면 invalid_grant
3. 저장된 인가에서 코드 토큰·원래 인가요청 복원
4. 검증:
     - 코드를 발급받은 client_id == 지금 요청한 client_id?   아니면 코드 무효화 + invalid_grant
     - redirect_uri 일치?                                   아니면 invalid_grant
     - 코드가 active 한가? (만료/무효화/재사용)
         └ 이미 무효화된 코드를 또 쓰면 → 이미 발급된 토큰까지 무효화 + invalid_grant
5. DPoP proof 검증 (있으면)
6. 토큰 생성:
     - access token  : tokenGenerator.generate(ACCESS_TOKEN context)
     - refresh token : 클라이언트가 refresh_token 그랜트 허용 시
     - id token      : 인가요청 scope 에 openid 포함 시
7. 코드 무효화(1회용) → authorizationService.save(authorization)
8. OAuth2AccessTokenAuthenticationToken 반환
```

### 재사용·교차 클라이언트 공격 방어

이 Provider에서 가장 교육적인 부분은 **방어 로직**이다. 코드는 1회용이므로, 재사용이 감지되면 그 코드로 이미 발급된 토큰까지 죽인다.

```java
// 다른 클라이언트가 코드를 쓰려 하면 코드 자체를 무효화
if (!registeredClient.getClientId().equals(authorizationRequest.getClientId())) {
    if (!authorizationCode.isInvalidated()) {
        authorization = OAuth2Authorization.from(authorization)
            .invalidate(authorizationCode.getToken()).build();
        this.authorizationService.save(authorization);
    }
    throw new OAuth2AuthenticationException(OAuth2ErrorCodes.INVALID_GRANT);
}
```

```java
// 이미 무효화된(=한 번 쓴) 코드를 또 들고 오면 발급 토큰을 폭파
if (!authorizationCode.isActive()) {
    if (authorizationCode.isInvalidated()) {
        OAuth2Authorization.Token<?> token = (authorization.getRefreshToken() != null)
            ? authorization.getRefreshToken() : authorization.getAccessToken();
        if (token != null) {
            authorization = OAuth2Authorization.from(authorization)
                .invalidate(token.getToken()).build();   // 연쇄 무효화 (01장)
            this.authorizationService.save(authorization);
        }
    }
    throw new OAuth2AuthenticationException(OAuth2ErrorCodes.INVALID_GRANT);
}
```

여기서 01장의 `OAuth2Authorization.Builder.invalidate()` 연쇄 무효화가 빛을 발한다. 리프레시 토큰 하나를 무효화하면 액세스 토큰과 코드까지 함께 죽으므로, "코드 도난 후 재사용" 시나리오에서 공격자가 이미 받았을 수 있는 모든 토큰이 한 번에 폐기된다.

### 토큰 생성 위임

실제 토큰 값은 이 Provider가 직접 만들지 않는다. `OAuth2TokenContext`를 채워 `OAuth2TokenGenerator`에 위임한다(05장 주제).

```java
DefaultOAuth2TokenContext.Builder tokenContextBuilder = DefaultOAuth2TokenContext.builder()
        .registeredClient(registeredClient)
        .principal(principal)
        .authorizationServerContext(AuthorizationServerContextHolder.getContext())  // 02장 ThreadLocal
        .authorization(authorization)
        .authorizedScopes(authorization.getAuthorizedScopes())
        .authorizationGrantType(AuthorizationGrantType.AUTHORIZATION_CODE)
        .authorizationGrant(authorizationCodeAuthentication);

OAuth2TokenContext tokenContext = tokenContextBuilder.tokenType(OAuth2TokenType.ACCESS_TOKEN).build();
OAuth2Token generatedAccessToken = this.tokenGenerator.generate(tokenContext);
if (generatedAccessToken == null)
    throw new OAuth2AuthenticationException(new OAuth2Error(SERVER_ERROR, "...failed to generate...", ERROR_URI));
```

토큰 타입(`ACCESS_TOKEN` → `REFRESH_TOKEN` → ID_TOKEN)을 바꿔가며 같은 context 빌더를 재사용한다. ID 토큰은 인가요청 scope에 `openid`가 있을 때만 만들고, 이때 `SessionRegistry`에서 세션을 찾아 `sid`·`auth_time` 클레임의 재료로 넘긴다(OIDC 세션 추적).

### refresh token 정책

리프레시 토큰은 무조건 주지 않는다. **클라이언트가 `refresh_token` 그랜트를 등록**했을 때만 발급한다. 주석대로 퍼블릭 클라이언트에는 리프레시를 주지 않는 것이 기본 방향이다.

```java
if (registeredClient.getAuthorizationGrantTypes().contains(AuthorizationGrantType.REFRESH_TOKEN)) {
    tokenContext = tokenContextBuilder.tokenType(OAuth2TokenType.REFRESH_TOKEN).build();
    OAuth2Token generatedRefreshToken = this.tokenGenerator.generate(tokenContext);
    ...
}
```

## 응답 작성

성공하면 `OAuth2AccessTokenResponseAuthenticationSuccessHandler`가 `OAuth2AccessTokenAuthenticationToken`을 표준 토큰 응답 JSON으로 직렬화한다.

```json
{ "access_token":"...", "token_type":"Bearer", "expires_in":300,
  "refresh_token":"...", "scope":"...", "id_token":"..." }
```

실패하면 `OAuth2ErrorAuthenticationFailureHandler`가 `{"error":"invalid_grant", ...}` 형태의 에러 응답을 쓰고, 토큰 엔드포인트 Filter는 `SecurityContextHolder.clearContext()`로 컨텍스트를 비운다.

## 다른 그랜트 한눈에

- **client_credentials**: 리소스 오너가 없는 머신 간 통신. principal이 곧 클라이언트이고 액세스 토큰만 발급(리프레시 없음). `OAuth2ClientCredentialsAuthenticationValidator`가 요청 scope가 등록 scope의 부분집합인지 검증한다.
- **refresh_token**: 저장된 인가를 리프레시 토큰으로 찾아 액세스 토큰을 재발급. `TokenSettings.isReuseRefreshTokens()`가 `false`면 리프레시 토큰도 새로 발급하고 기존 것을 무효화(회전).
- **device_code**: 입력장치가 빈약한 기기용. 클라이언트가 device_code로 폴링하면 사용자 승인 전에는 `authorization_pending`을, 승인 후에는 토큰을 돌려준다.
- **token-exchange(RFC 8693)**: 한 토큰을 다른 주체/대상의 토큰으로 교환. `OAuth2TokenExchangeCompositeAuthenticationToken`으로 subject/actor를 묶어 처리한다.

이들 모두 "검증 → `OAuth2TokenContext` 구성 → `OAuth2TokenGenerator.generate()` → `OAuth2Authorization` 저장 → `OAuth2AccessTokenAuthenticationToken` 반환"이라는 04장의 골격을 공유한다.

## 설계 포인트 / 확장점

- **Converter 추가로 그랜트 확장**: `OAuth2TokenEndpointFilter.setAuthenticationConverter(...)`와 config의 `tokenEndpoint(...)`로 커스텀 그랜트 Converter/Provider를 끼울 수 있다.
- **생성 위임의 일관성**: 모든 Provider가 토큰을 직접 만들지 않고 `OAuth2TokenGenerator`에 위임하므로, JWT/불투명/커스텀 클레임 정책을 한 곳(05장)에서 바꾸면 전 그랜트에 반영된다.
- **상태 전이는 항상 `from().build()`**: 무효화·토큰 추가 모두 기존 인가를 복제해 새 인가로 저장한다. 동시 요청이 같은 인가를 건드려도 불변 객체라 경합 상태가 단순해진다(최종 저장 승자 규칙).

## 정리

토큰 엔드포인트는 `grant_type`으로 적절한 Provider를 고르고, 그 Provider가 도메인 모델(`OAuth2Authorization`·`RegisteredClient`)을 검증한 뒤 `OAuth2TokenGenerator`에 토큰 생성을 위임한다. 인가 코드 흐름의 재사용·교차클라이언트 방어, 연쇄 무효화, OIDC ID 토큰 조건부 발급이 이 한 Provider 안에 응축돼 있다. 다음 장은 "토큰을 어떻게 만드는가"인 생성 파이프라인이다.
