# 03. 인증 프로바이더와 토큰

## 무엇을 / 왜

필터가 티켓(또는 ID/PW)을 `Authentication` 객체로 감싸 `AuthenticationManager` 에 넘기면, 실제 검증은 `AuthenticationProvider` 가 한다. 이 모듈은 01장에서 본 두 경로에 맞춰 **두 개의 프로바이더**와, 각 프로바이더의 입력/출력이 되는 **토큰 타입**들을 제공한다. 이 장은 그 검증 로직과 토큰의 의미를 본다.

소스: `kerberos/src/main/java/org/springframework/security/kerberos/authentication/`.

## 핵심 타입

```
[A] 경로                                  [B] 경로
─────────────────────────────────────   ──────────────────────────────────────
KerberosServiceRequestToken (입력=티켓)   UsernamePasswordAuthenticationToken (입력)
        │                                         │
        ▼                                         ▼
KerberosServiceAuthenticationProvider      KerberosAuthenticationProvider
  · ticketValidator.validateTicket()         · kerberosClient.login()
  · userDetailsService.load...()             · userDetailsService.load...()
        │                                         │
        ▼                                         ▼
KerberosServiceRequestToken (인증됨)        KerberosUsernamePasswordAuthenticationToken
  implements KerberosAuthentication           implements KerberosAuthentication
        └──────────────┬──────────────────────────┘
                       ▼
        KerberosAuthentication.getJaasSubjectHolder()
                → JaasSubjectHolder (Subject + 발급 티켓)
```

두 출력 토큰 모두 `KerberosAuthentication` 인터페이스를 구현한다. 이 인터페이스는 `getJaasSubjectHolder()` 하나만 노출하며, 04장의 멀티티어(위임) 기능이 이걸 통해 사용자의 Subject 에 접근한다.

## 동작 흐름 [A] — KerberosServiceAuthenticationProvider

```
authenticate(Authentication)
  ├ (KerberosServiceRequestToken) 캐스팅 → byte[] token 추출
  ├ ticketValidator == null? → IllegalStateException
  ├ userDetailsService == null? → IllegalStateException
  ├ KerberosTicketValidation v = ticketValidator.validateTicket(token)   ★검증
  ├ UserDetails u = userDetailsService.loadUserByUsername(v.username())   ★권한
  ├ userDetailsChecker.check(u)   (계정 만료/잠금 확인)
  ├ additionalAuthenticationChecks(u, auth)   (서브클래스 훅)
  └ return new KerberosServiceRequestToken(u, v, u.getAuthorities(), token)  (인증됨)
```

핵심 코드:

```java
KerberosServiceRequestToken auth = (KerberosServiceRequestToken) authentication;
byte[] token = auth.getToken();
KerberosTicketValidation ticketValidation = this.ticketValidator.validateTicket(token);
UserDetails userDetails = this.userDetailsService.loadUserByUsername(ticketValidation.username());
this.userDetailsChecker.check(userDetails);
additionalAuthenticationChecks(userDetails, auth);
KerberosServiceRequestToken responseAuth =
        new KerberosServiceRequestToken(userDetails, ticketValidation, userDetails.getAuthorities(), token);
```

주목할 점:

- **검증 ↔ 권한 분리(01장 원칙의 코드화)**: Kerberos 가 주는 건 `username` 뿐이고, 권한·계정 상태는 `userDetailsService` + `userDetailsChecker`(기본 `AccountStatusUserDetailsChecker`)가 책임진다.
- **`supports()`** 는 `KerberosServiceRequestToken` 타입만 받는다. 그래서 `AuthenticationManager` 가 토큰 타입으로 [A]/[B] 프로바이더를 자동 라우팅한다.
- **확장 훅**: `additionalAuthenticationChecks(...)` 는 빈 메서드로, 서브클래스가 추가 검증(예: 특정 도메인만 허용)을 끼울 자리다.

## 동작 흐름 [B] — KerberosAuthenticationProvider

폼 로그인의 `UsernamePasswordAuthenticationToken` 을 받아, ID/PW 로 **KDC 에 직접 로그인**해 검증한다.

```
authenticate(UsernamePasswordAuthenticationToken)
  ├ kerberosClient / userDetailsService null 체크
  ├ JaasSubjectHolder h = kerberosClient.login(name, password)   ★KDC 로그인
  ├ String username = h.getUsername()
  ├ UserDetails u = userDetailsService.loadUserByUsername(username)
  └ return new KerberosUsernamePasswordAuthenticationToken(u, credentials, u.getAuthorities(), h)
```

```java
JaasSubjectHolder subjectHolder = this.kerberosClient.login(auth.getName(), credentials.toString());
String username = subjectHolder.getUsername();
UserDetails userDetails = this.userDetailsService.loadUserByUsername(username);
KerberosUsernamePasswordAuthenticationToken output =
        new KerberosUsernamePasswordAuthenticationToken(userDetails, credentials, userDetails.getAuthorities(), subjectHolder);
```

[A]와 구조가 동일하다. 차이는 검증 협력자가 `KerberosTicketValidator`(티켓 검증) 대신 `KerberosClient`(KDC 로그인)이고, `supports()` 가 표준 `UsernamePasswordAuthenticationToken` 을 받는다는 점이다. 결과 토큰은 KDC 로그인으로 얻은 `JaasSubjectHolder` 를 품어, 이후 다른 서비스에 위임 인증할 자격을 보존한다.

## 토큰 타입 자세히

### `KerberosServiceRequestToken`

[A] 경로의 입력이자 출력인 양면 토큰이다. `AbstractAuthenticationToken` 을 확장하고 `KerberosAuthentication` 을 구현한다.

- **미인증 생성자** `(byte[] token)`: 필터가 만든다. 권한 없음(`NO_AUTHORITIES`), `ticketValidation=null`.
- **인증 생성자** `(principal, ticketValidation, authorities, token)`: 프로바이더가 만든다. `ticketValidation.subject()` 와 `username` 으로 `JaasSubjectHolder` 를 즉시 구성하고 `setAuthenticated(true)`.
- **`equals`/`hashCode` 는 토큰 바이트 기반**: "같은 티켓이면 같은 토큰"으로 본다.
- **재인증 불가**: 클래스 주석대로 한 번 쓴 Kerberos 티켓을 다시 검증하면 KDC 가 reply 에러를 낸다.
- **암복호화 메서드** `encrypt/decrypt`: `ticketValidation.getGssContext()` 가 살아 있으면(`holdOnToGSSContext=true`, 04장) `Subject.doAs` 안에서 `GSSContext.wrap/unwrap` 으로 메시지 기밀성을 제공한다. `responseToken()`/`getEncodedResponseToken()` 은 상호 인증용 응답 토큰을 base64 로 노출한다.

### `KerberosTicketValidation`

`KerberosTicketValidator.validateTicket()` 의 결과를 담는 불변 값 객체다. 보관하는 것: `username`, `subject`(검증된 서비스 주체로 구성된 JAAS Subject), `responseToken`(상호 인증용), `gssContext`(보유 시 암복호화 가능), `delegationCredential`(위임 자격, 있을 때).

### `KerberosUsernamePasswordAuthenticationToken`

[B] 경로의 출력. `UsernamePasswordAuthenticationToken` 을 확장하면서 `JaasSubjectHolder` 를 추가로 품는다. 이 Subject 의 private credential 에는 사용자의 Kerberos 티켓이 들어 있어, `KerberosMultiTier.authenticateService()` 로 **다른 서비스에 대한 새 티켓**을 발급할 수 있다(위임 시나리오, 04장).

### `JaasSubjectHolder`

인증된 사용자의 `Subject` 와, 추가로 발급한 서비스별 토큰 맵(`savedTokens`)을 보관하는 직렬화 가능 홀더다. `addToken(service, token)`/`getToken(service)` 로 멀티티어 위임에서 발급한 티켓을 캐싱한다.

## 설계 포인트 / 확장점

- **토큰 타입 = 라우팅 키**: 두 프로바이더가 `supports()` 로 입력 토큰 타입을 구분하므로, 하나의 `AuthenticationManager` 에 둘 다 등록해 SPNEGO 와 폼-Kerberos 를 동시에 지원할 수 있다.
- **`InitializingBean`**: `KerberosServiceAuthenticationProvider` 는 `afterPropertiesSet()` 에서 `ticketValidator`/`userDetailsService` 필수 주입을 검증한다.
- **교체 가능한 검증기**: `setTicketValidator`/`setKerberosClient` 로 SUN 구현 대신 다른 JRE 구현이나 모킹을 끼울 수 있다.

## 정리

두 프로바이더는 같은 골격("Kerberos 로 username 검증 → UserDetailsService 로 권한 채움 → 인증 토큰 반환")을 공유하며, 입력 토큰 타입으로 갈린다. 출력 토큰들은 `KerberosAuthentication` 을 구현해 `JaasSubjectHolder` 를 노출함으로써, 단순 인증을 넘어 위임(멀티티어)과 메시지 암복호화까지 가능하게 열어둔다. 실제 "검증" 알맹이는 `KerberosTicketValidator`/`KerberosClient` 구현에 있고, 그것이 다음 장의 주제다.
