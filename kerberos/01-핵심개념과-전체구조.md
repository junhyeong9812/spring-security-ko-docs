# 01. 핵심 개념과 전체 구조

## 무엇을 / 왜

이 모듈을 읽기 전에 알아야 할 세 가지 배경 개념이 있다. 이 장은 그 개념과, 그것들이 코드에서 어떤 타입으로 나타나는지, 그리고 모듈 전체가 어떻게 두 갈래의 인증 경로로 갈라지는지를 지도처럼 보여준다. 세부 동작은 02~05 장에서 다룬다.

### 배경 1 — Kerberos

Kerberos 는 신뢰받는 제3자인 **KDC(Key Distribution Center, Active Directory 의 도메인 컨트롤러)** 가 발급하는 암호화된 **티켓**으로 신원을 증명하는 인증 프로토콜이다. 핵심은 "사용자가 KDC 에 한 번 로그인해 TGT 를 받으면, 그 뒤로는 각 서비스에 대한 **서비스 티켓**을 비밀번호 재입력 없이 받아 제출한다"는 SSO 모델이다. 서비스(우리 웹앱)는 자신의 비밀키가 담긴 **keytab** 파일로 그 티켓을 복호화·검증한다.

### 배경 2 — SPNEGO (Negotiate)

브라우저가 Kerberos 서비스 티켓을 HTTP 로 실어 나르는 방법이 SPNEGO 다. 흐름은 단순하다.

```
브라우저 ──▶ GET /secure            (티켓 없음)
서버    ──▶ 401, WWW-Authenticate: Negotiate
브라우저 ──▶ GET /secure
            Authorization: Negotiate YII...(base64 서비스 티켓)
서버    ──▶ 200 (검증 성공, SecurityContext 채움)
```

이 모듈에서 401 응답을 만드는 쪽이 `SpnegoEntryPoint`, `Negotiate` 헤더를 파싱해 검증으로 넘기는 쪽이 `SpnegoAuthenticationProcessingFilter` 다.

### 배경 3 — JAAS / GSS-API

JRE 는 Kerberos 를 두 계층 API 로 노출한다. **JAAS**(`LoginContext`, `Krb5LoginModule`, `Subject`)는 자격증명을 로딩/보관하고, **GSS-API**(`GSSContext`, `GSSCredential`)는 그 자격증명으로 실제 토큰을 받아들이거나 만든다. 이 모듈의 `sun` 패키지는 이 저수준 호출을 캡슐화한다. 핵심 관용구는 `Subject.doAs(subject, action)` — "이 Subject(=keytab 으로 로그인한 서비스 신원)의 권한으로 GSS 작업을 실행하라"는 뜻이다.

## 핵심 타입 — 두 개의 추상 인터페이스

모듈 전체는 사실상 두 개의 작은 인터페이스를 중심으로 돈다. 나머지는 모두 이 둘의 구현이거나, 이 둘을 Spring Security 필터/프로바이더에 연결하는 접착제다.

```
            ┌─────────────────────────────────────────────┐
  [A] 경로  │ KerberosTicketValidator                      │
            │   validateTicket(byte[] token)               │
            │     → KerberosTicketValidation               │
            │   구현: SunJaasKerberosTicketValidator        │
            └─────────────────────────────────────────────┘

            ┌─────────────────────────────────────────────┐
  [B] 경로  │ KerberosClient                                │
            │   login(username, password)                  │
            │     → JaasSubjectHolder                      │
            │   구현: SunJaasKerberosClient                 │
            └─────────────────────────────────────────────┘
```

- **`KerberosTicketValidator`** (`kerberos/src/main/java/.../authentication/KerberosTicketValidator.java`): "받은 티켓이 진짜인가? 누구인가?"를 판단한다. 구현이 SUN/IBM JRE 별로 다를 수 있으므로 인터페이스로 분리되어 있다.
- **`KerberosClient`** (`.../authentication/KerberosClient.java`): "이 ID/PW 로 KDC 에 로그인되는가?"를 판단한다. 폼 로그인을 Kerberos 로 검증하는 [B] 경로에서 쓴다.

## 동작 흐름 — 두 경로의 큰 그림

### [A] SPNEGO 서버 검증 (브라우저 SSO, 가장 흔함)

```
브라우저
  │  Authorization: Negotiate <티켓>
  ▼
SpnegoAuthenticationProcessingFilter        (kerberos-web)
  │  헤더 base64 디코드 → byte[] 티켓
  │  new KerberosServiceRequestToken(티켓)   (미인증)
  ▼
AuthenticationManager.authenticate(...)
  ▼
KerberosServiceAuthenticationProvider        (kerberos-core)
  │  ticketValidator.validateTicket(token)    ── 검증 → username
  │  userDetailsService.loadUserByUsername()  ── 권한 로딩
  ▼
KerberosServiceRequestToken (인증됨)  →  SecurityContext
```

### [B] 폼 ID/PW 를 KDC 로 검증

```
폼 로그인 필터 (web 모듈 표준)
  │  UsernamePasswordAuthenticationToken
  ▼
KerberosAuthenticationProvider               (kerberos-core)
  │  kerberosClient.login(user, password)     ── KDC 로그인 → JaasSubjectHolder
  │  userDetailsService.loadUserByUsername()
  ▼
KerberosUsernamePasswordAuthenticationToken (인증됨) → SecurityContext
```

두 경로 모두 공통 패턴을 따른다. **(1) 모듈 고유 방식으로 신원을 검증해 username 을 얻고 → (2) 표준 `UserDetailsService` 로 권한을 채워 → (3) 인증된 `Authentication` 을 만든다.** 즉 "신원 증명"만 Kerberos 가 담당하고, "권한 부여"는 Spring Security 의 일반 메커니즘에 위임한다. 이것이 이 모듈의 가장 중요한 설계 결정이다.

## 설계 포인트

- **검증과 권한의 분리**: Kerberos 는 username 만 돌려준다. 권한(roles)은 `UserDetailsService` 가 책임진다. 그래서 두 프로바이더 모두 `ticketValidator`/`kerberosClient` 와 `userDetailsService` 두 협력자를 요구한다.
- **JRE 종속 격리**: `sun` 패키지가 SUN JRE 전용 `com.sun.security.*` 호출을 모은다. IBM JRE 등 다른 구현을 쓰려면 인터페이스만 갈아끼우면 된다.
- **서버/클라이언트 양방향**: 같은 모듈이 "우리가 보호받는 서비스"(core/web)와 "우리가 호출하는 클라이언트"(client) 두 역할을 모두 지원한다.

## 정리

Kerberos 모듈은 결국 두 개의 검증 인터페이스(`KerberosTicketValidator`, `KerberosClient`)와, 그것을 Spring Security 필터/프로바이더에 잇는 접착제로 이루어진다. SPNEGO 헤더로 들어온 티켓을 검증하는 [A] 경로가 핵심이고, 폼 ID/PW 를 KDC 로 검증하는 [B] 경로가 보조다. 어느 쪽이든 "신원은 Kerberos, 권한은 UserDetailsService" 라는 분업을 기억하면 나머지 코드가 쉽게 읽힌다.
