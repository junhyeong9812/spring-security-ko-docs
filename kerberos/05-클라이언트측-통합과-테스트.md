# 05. 클라이언트 측 통합과 테스트 지원

## 무엇을 / 왜

지금까지는 "우리 웹앱이 **보호받는 서비스**(acceptor)" 였다. 그런데 우리 앱이 거꾸로 **다른 Kerberos 보호 서비스를 호출하는 클라이언트(initiator)** 가 되는 경우도 흔하다. 예를 들어 SPNEGO 로 보호된 REST API 를 부르거나, GSSAPI 로 Active Directory(LDAP)를 조회해야 할 때다. `kerberos-client` 모듈이 이를 담당한다. 더불어 이 장은 KDC 없이 통합 테스트를 돌릴 수 있게 해주는 `kerberos-test` 모듈도 짧게 다룬다.

소스: `kerberos/src/main/java/org/springframework/security/kerberos/client/`, `.../test/`.

## 핵심 타입

```
 KerberosRestTemplate  (extends RestTemplate)
   doExecute(): LoginContext 로그인 → Subject.doAs(SPNEGO REST 호출)

 KerberosLdapContextSource (extends DefaultSpringSecurityContextSource)
   getDirContextInstance(): GSSAPI 로 LDAP(AD) 바인딩

 SunJaasKrb5LoginConfig  (extends Configuration, InitializingBean)
   위 둘에 주입되는 JAAS Krb5LoginModule 설정 빈

 MiniKdc / KerberosSecurityTestcase  (kerberos-test)
   내장 KDC 로 통합 테스트
```

## 동작 흐름 — KerberosRestTemplate

`RestTemplate` 을 확장해, 매 호출을 **JAAS 로그인 → `Subject.doAs` 안에서 실행**으로 감싼다. SPNEGO 협상 자체는 Apache HttpClient 5 의 `SPNegoSchemeFactory` 가 처리한다.

```
doExecute(url, ...)                       (RestTemplate 의 훅 오버라이드)
  ├ LoginContext lc = buildLoginContext()  ── keytab/캐시/비번 중 택1로 구성
  ├ lc.login()                              ── Kerberos 자격 획득
  ├ Subject serviceSubject = lc.getSubject()
  └ Subject.doAs(serviceSubject, () ->
        super.doExecute(url, ...))          ── 이 권한으로 실제 HTTP 호출
                                               (HttpClient 가 SPNEGO 헤더 부착)
```

자격증명 소스는 생성자 인자 조합으로 결정되며, `ClientLoginConfig.getAppConfigurationEntry()` 가 그에 맞춰 `Krb5LoginModule` 옵션을 만든다.

- keytab + principal 없음 → `useTicketCache=true` (OS 의 티켓 캐시 사용)
- keytab + principal 있음 → `useKeyTab=true, keyTab, principal, storeKey=true`
- password 제공 → `doNotPrompt=false` + `CallbackHandlerImpl` 가 비번 주입
- 항상 `isInitiator=true` (우리가 호출 개시자이므로)

HttpClient 쪽은 `buildHttpClient()` 가 SPNEGO/Kerberos 를 선호 스킴으로 등록하고, 실제 자격은 JAAS Subject 에서 오므로 `NullCredentials`(빈 자격)를 등록한다.

```java
Lookup<AuthSchemeFactory> authSchemeRegistry = RegistryBuilder.<AuthSchemeFactory>create()
    .register(StandardAuthScheme.SPNEGO, new SPNegoSchemeFactory(KerberosConfig.custom()
        .setStripPort(ENABLE).setUseCanonicalHostname(DISABLE).build(), SystemDefaultDnsResolver.INSTANCE))
    .build();
```

## 동작 흐름 — KerberosLdapContextSource

Active Directory 를 사용자/권한 저장소로 쓸 때, LDAP 바인딩 자체를 **Kerberos(GSSAPI)** 로 하는 컨텍스트 소스다. `spring-security-ldap` 의 `DefaultSpringSecurityContextSource` 를 확장한다.

```
getDirContextInstance(env)
  ├ env.put(Context.SECURITY_AUTHENTICATION, "GSSAPI")
  ├ Subject serviceSubject = login()         ── loginConfig 로 LoginContext 로그인
  └ Subject.doAs(serviceSubject, () ->
        super.getDirContextInstance(env))     ── 이 신원으로 LDAP 바인딩
```

`PrivilegedAction` 안에서 발생한 `NamingException` 을 배열에 담아 밖으로 다시 던지는(suppressedException) 패턴이 쓰인다. `Subject.doAs` 의 `run()` 이 검사 예외를 못 던지기 때문이다. 필수 협력자인 `loginConfig`(보통 `SunJaasKrb5LoginConfig`)는 `afterPropertiesSet()` 에서 주입 검증한다.

## 공유 JAAS 설정 — SunJaasKrb5LoginConfig

위 두 클라이언트에 주입되는 JAAS `Configuration` 빈이다. 04장의 `SunJaasKerberosTicketValidator` 내부 `LoginConfig` 와 목적은 같지만(코드로 `Krb5LoginModule` 옵션 구성), 이쪽은 **재사용 가능한 독립 빈**이라 `KerberosLdapContextSource` 등에 주입해 쓴다.

```java
options.put("principal", servicePrincipal);
if (keyTabLocation != null) { options.put("useKeyTab", "true"); options.put("keyTab", ...); options.put("storeKey", "true"); }
options.put("doNotPrompt", "true");
if (useTicketCache) { options.put("useTicketCache", "true"); options.put("renewTGT", "true"); }
options.put("isInitiator", isInitiator.toString());
```

`afterPropertiesSet()` 에서 `servicePrincipal` 필수 확인, `useTicketCache=false` 면 keytab 필수 확인, keytab 이 classpath 면 보안 경고를 남긴다.

## 테스트 지원 — MiniKdc / KerberosSecurityTestcase

실제 도메인 컨트롤러 없이 Kerberos 통합 테스트를 돌리기 위한 도구다.

- **`MiniKdc`**: Apache Directory 기반의 **내장 KDC** 다. 테스트용 realm·주체를 만들고 keytab 을 생성해, 진짜 KDC처럼 티켓을 발급한다.
- **`KerberosSecurityTestcase`**: JUnit 5 베이스 클래스. `@BeforeEach` 에서 `MiniKdc` 를 띄우고(`startMiniKdc`) `@AfterEach` 에서 정리한다(`stopMiniKdc`). `getTestDir()`/`createMiniKdcConf()` 를 오버라이드해 작업 디렉터리·KDC 설정을 바꿀 수 있다.

```java
@BeforeEach public void startMiniKdc() throws Exception {
    createTestDir(); createMiniKdcConf();
    this.kdc = new MiniKdc(this.conf, this.workDir);
    this.kdc.start();
}
```

이 베이스를 상속하면, 검증기/클라이언트가 실제 GSS 핸드셰이크를 수행하는 종단 테스트를 외부 인프라 없이 작성할 수 있다.

## 설계 포인트 / 확장점

- **`Subject.doAs` 일관성**: 클라이언트 쪽(REST·LDAP)도 서버 쪽 검증기와 똑같이 "로그인해서 Subject 얻기 → `doAs` 로 실행" 관용구를 따른다. 모듈 전체가 이 한 패턴으로 묶인다.
- **표준 타입 확장**: `KerberosRestTemplate`(=`RestTemplate`), `KerberosLdapContextSource`(=`LdapContextSource`)는 친숙한 Spring 타입을 그대로 확장해, Kerberos 인지 의식하지 않고 기존 코드에 끼워 넣을 수 있다.
- **설정 빈 재사용**: `SunJaasKrb5LoginConfig` 를 빈으로 분리해 여러 클라이언트가 같은 JAAS 설정을 공유한다.

## 정리

`kerberos-client` 는 모듈의 "우리가 클라이언트가 되는" 절반을 채운다. `KerberosRestTemplate` 은 REST 호출을, `KerberosLdapContextSource` 는 AD 조회를 각각 SPNEGO/GSSAPI 로 인증하며, 둘 다 `SunJaasKrb5LoginConfig` 같은 JAAS 설정으로 자격을 얻어 `Subject.doAs` 안에서 실행한다. `kerberos-test` 의 `MiniKdc` 는 이 모든 흐름을 실제 KDC 없이 검증하게 해준다. 이로써 서버 검증(02~04장)과 클라이언트 발신(이 장)을 합쳐 Kerberos 통합의 양방향이 완성된다.
