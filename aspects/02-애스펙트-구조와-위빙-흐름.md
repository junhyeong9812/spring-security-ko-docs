# 02. 애스펙트 구조와 위빙 흐름

## 무엇을 / 왜

이 챕터는 `org.springframework.security.authorization.method.aspectj` 패키지의 실제 `.aj` 파일들을 읽으며 **"메서드가 호출되면 애스펙트 안에서 무슨 일이 일어나는지"** 를 단계별로 따라간다. 핵심은 세 가지다.

1. 공통 골격 `AbstractMethodInterceptorAspect` 의 `around` advice
2. 각 애너테이션을 매칭하는 5개 구체 애스펙트의 pointcut
3. AspectJ join point 를 AOP Alliance `MethodInvocation` 으로 바꾸는 `JoinPointMethodInvocation` 어댑터

## 핵심 타입

```
            AbstractMethodInterceptorAspect  (abstract aspect)
            ├─ pointcut executionOfAnnotatedMethod()   ← 추상, 하위가 정의
            ├─ Object around() : <pointcut> { ... }     ← 공통 advice
            └─ MethodInterceptor securityInterceptor    ← 주입되는 인가 엔진
                     ▲ extends
        ┌────────────┼─────────────┬──────────────┬──────────────┐
   PreAuthorize   PostAuthorize  PreFilter     PostFilter      Secured
     Aspect         Aspect        Aspect         Aspect         Aspect
   @PreAuthorize  @PostAuthorize @PreFilter    @PostFilter    @Secured

   각 구체 애스펙트 = "어떤 메서드를 가로챌지(pointcut)"만 다르게 정의.
   가로챈 뒤의 처리는 전부 부모의 around advice 가 담당.

            JoinPointMethodInvocation  (class, implements MethodInvocation)
            └─ AspectJ thisJoinPoint 를 감싸 getMethod()/getArguments()/proceed() 제공
```

## 공통 골격: AbstractMethodInterceptorAspect

이 추상 애스펙트가 모든 애스펙트의 심장이다. `executionOfAnnotatedMethod()` 라는 **추상 pointcut** 을 선언하고, 그 pointcut 에 걸리는 모든 실행을 `around` advice 로 감싼다.

```java
// AbstractMethodInterceptorAspect.aj
protected abstract pointcut executionOfAnnotatedMethod();

private MethodInterceptor securityInterceptor;

Object around(): executionOfAnnotatedMethod() {
    if (this.securityInterceptor == null) {
        return proceed();                       // (1) 인터셉터 미설정 → 그냥 통과
    }
    MethodInvocation invocation =
        new JoinPointMethodInvocation(thisJoinPoint, () -> proceed());  // (2) 어댑팅
    try {
        return this.securityInterceptor.invoke(invocation);            // (3) 인가 엔진 호출
    } catch (Throwable t) {
        throwUnchecked(t);                       // (4) checked 예외도 그대로 던지기
        throw new IllegalStateException("Code unexpectedly reached", t);
    }
}
```

한 줄씩 의미를 보면:

- **(1)** `securityInterceptor` 가 주입되지 않았으면 보안을 적용하지 않고 원래 메서드를 그대로 실행한다. 위빙은 됐지만 설정이 안 된 상황에 대한 안전장치다.
- **(2)** AspectJ 가 주는 `thisJoinPoint`(현재 가로챈 호출 정보)와 "원래 메서드를 실행하는 람다 `() -> proceed()`" 를 묶어 `JoinPointMethodInvocation` 으로 감싼다. 이게 어댑터의 핵심이다.
- **(3)** core 의 `MethodInterceptor`(예: `AuthorizationManagerBeforeMethodInterceptor`)에게 그 `MethodInvocation` 을 넘긴다. 인터셉터는 인가 검사 → 통과하면 `invocation.proceed()` 호출 → 그 호출이 다시 (2)의 람다를 통해 AspectJ `proceed()` 로 이어진다.
- **(4)** AOP advice 의 반환 타입은 `Object` 라 checked 예외를 직접 던질 수 없다. `throwAny` 제네릭 트릭(`throw (E) ex`)으로 원래 예외를 **있는 그대로** 다시 던져서, 인가 실패(`AccessDeniedException`)나 비즈니스 예외가 호출자에게 그대로 전달되게 한다.

`setSecurityInterceptor(...)` setter 로 인터셉터가 주입되는데, 이 주입은 config 모듈이 담당한다(03 챕터).

## 구체 애스펙트: pointcut 만 다르다

5개 구체 애스펙트는 모두 부모를 `extends` 하고 `executionOfAnnotatedMethod()` pointcut 하나만 채운다. 즉 **"무엇을 가로챌지"** 만 선언하고 처리는 부모에게 맡긴다.

```java
// PreAuthorizeAspect.aj — @PreAuthorize 붙은 모든 메서드 실행
protected pointcut executionOfAnnotatedMethod()
    : execution(* *(..)) && @annotation(PreAuthorize);
```

`PostAuthorizeAspect`, `PreFilterAspect`, `PostFilterAspect` 도 애너테이션 이름만 바꾼 동일한 형태다. `execution(* *(..))` 은 "임의 반환·임의 인자의 모든 메서드 실행", `@annotation(...)` 은 "그 메서드에 해당 애너테이션이 있을 것" 을 뜻한다.

`SecuredAspect` 만 조금 더 넓다. 메서드 단위뿐 아니라 **타입 단위 `@Secured`** 도 잡는다.

```java
// SecuredAspect.aj
private pointcut executionOfAnyPublicMethodInAtSecuredType()
    : execution(public * ((@Secured *)+).*(..)) && @this(Secured);   // @Secured 타입(과 그 하위)의 모든 public 메서드
private pointcut executionOfSecuredMethod()
    : execution(* *(..)) && @annotation(Secured);                     // @Secured 메서드

protected pointcut executionOfAnnotatedMethod()
    : executionOfAnyPublicMethodInAtSecuredType() || executionOfSecuredMethod();
```

### 한 가지 주의: 애너테이션은 구현 클래스에

모든 구체 애스펙트의 클래스 주석은 같은 경고를 반복한다.

> "you **must** annotate the implementation class (and/or methods within that class), **not** the interface ... AspectJ follows Java's rule that annotations on interfaces are **not** inherited. This will vary from Spring AOP."

AspectJ 는 자바 규칙대로 **인터페이스에 붙은 애너테이션을 상속하지 않는다**. 따라서 인터페이스에 `@PreAuthorize` 를 달고 구현체에는 안 달면 위빙되지 않는다. 반드시 **구현 클래스(또는 그 메서드)** 에 애너테이션을 붙여야 한다. 이는 인터페이스 애너테이션도 잡아내던 Spring AOP(프록시) 와의 동작 차이다.

## 어댑터: JoinPointMethodInvocation

core 의 인터셉터는 AspectJ 를 모른다. AOP Alliance `MethodInvocation` 만 안다. 이 간극을 메우는 게 `JoinPointMethodInvocation` 이다. AspectJ `JoinPoint` 를 받아 `MethodInvocation` 의 각 메서드를 구현한다.

```java
// JoinPointMethodInvocation.aj
JoinPointMethodInvocation(JoinPoint jp, Supplier<Object> proceed) {
    if (jp.getTarget() != null) {
        this.target = jp.getTarget();
    } else {
        // SEC-1295: ITD(인터타입 선언) 사용 시 target 이 null 일 수 있음
        this.target = jp.getSignature().getDeclaringType();
    }
    String targetMethodName = jp.getStaticPart().getSignature().getName();
    Class<?>[] types = ((CodeSignature) jp.getStaticPart().getSignature()).getParameterTypes();
    Class<?> declaringType = jp.getStaticPart().getSignature().getDeclaringType();
    this.method = findMethod(targetMethodName, declaringType, types);  // 리플렉션으로 Method 복원
    Assert.notNull(this.method, ...);
    this.proceed = proceed;
}
```

핵심 변환은:

- `getMethod()` — join point 의 시그니처(이름+파라미터 타입)로 `declaringType.getMethod(...)`(없으면 `getDeclaredMethod`)를 리플렉션 조회해 `java.lang.reflect.Method` 를 복원한다. 인가 인터셉터가 메서드의 애너테이션/SpEL 표현식을 읽을 때 이 `Method` 가 필요하다.
- `getArguments()` — `jp.getArgs()` 를 그대로 노출. `@PreFilter`/`@PreAuthorize` 의 SpEL 이 인자를 참조(`#arg`)할 수 있게 한다.
- `getThis()` — 대상 객체. `new` 로 만든 객체나 ITD 처럼 target 이 null 인 경우는 선언 타입으로 대체한다.
- `proceed()` — 생성자로 받은 `Supplier`(즉 `() -> proceed()` 람다)를 실행한다. 인터셉터가 인가 통과 후 이 메서드를 부르면 결국 **AspectJ 의 원래 메서드 본문**이 실행된다.

## 동작 흐름: @PreAuthorize 메서드 한 번 호출

```
호출자 ──▶ securedBean.transfer(amount)
                │
                │  [위빙된 바이트코드 — PreAuthorizeAspect.around() 진입]
                ▼
   ┌─────────────────────────────────────────────────────────┐
   │ AbstractMethodInterceptorAspect.around()                 │
   │  securityInterceptor == null ?                            │
   │    ├─ yes ─▶ proceed()  (그냥 원본 실행)                  │
   │    └─ no                                                  │
   │         new JoinPointMethodInvocation(thisJoinPoint,      │
   │                                       () -> proceed())    │
   │         securityInterceptor.invoke(invocation) ───────────┼─┐
   └─────────────────────────────────────────────────────────┘ │
                                                                 ▼
              ┌────────────────────────────────────────────────────┐
              │ (core) AuthorizationManagerBeforeMethodInterceptor  │
              │  1. invocation.getMethod() 로 @PreAuthorize SpEL 획득 │
              │  2. AuthorizationManager.authorize(auth, invocation) │
              │  3. denied ? ─▶ throw AccessDeniedException ─────────┼─▶ 호출자에게 전파
              │  4. granted ─▶ invocation.proceed() ────────────────┼─┐
              └────────────────────────────────────────────────────┘ │
                                                                       ▼
                                              () -> proceed() 람다 실행
                                              = AspectJ 원래 transfer() 본문
                                                       │
                                                       ▼  반환값
                                              around 가 그대로 return
```

`@PostAuthorize`/`@PostFilter` 의 경우 인터셉터가 먼저 `invocation.proceed()` 로 본문을 실행한 뒤 **반환값을 보고** 인가/필터링한다는 점만 다르다. 그 순서 차이는 모두 core 의 인터셉터 안에 있고, 애스펙트 쪽 골격은 동일하다.

## 설계 포인트

- **Template Method 패턴**: 부모 애스펙트가 `around` 알고리즘을 고정하고, 변하는 부분(pointcut)만 추상 메서드로 빼 하위가 채운다. 새 애너테이션 지원을 추가하려면 pointcut 한 줄짜리 애스펙트만 만들면 된다.
- **Adapter 패턴**: `JoinPointMethodInvocation` 이 AspectJ 세계와 AOP Alliance 세계를 잇는다. 덕분에 인가 엔진(core 인터셉터)은 위빙 방식과 완전히 분리된다.
- **관심사 분리**: 이 모듈은 "어디에/언제 끼어들지"만 안다. "통과/거부를 어떻게 결정할지"는 core 가 안다. 그래서 인가 규칙이 진화해도 `aspects` 는 거의 그대로다.

## 정리

각 구체 애스펙트는 pointcut 한 줄로 "가로챌 메서드"만 정의하고, 공통 부모 `AbstractMethodInterceptorAspect` 의 `around` advice 가 join point 를 `JoinPointMethodInvocation` 으로 감싸 core 의 `MethodInterceptor` 에 위임한다. 인가 판단은 전부 core 가 하고, 이 모듈은 위빙 진입점과 어댑팅만 책임진다. 단, 인터페이스 애너테이션이 상속되지 않는다는 AspectJ 규칙은 PROXY 모드와 다른 주의점이다. 다음 챕터에서 이 애스펙트들이 config 에 의해 어떻게 빈으로 등록되고 인터셉터를 주입받는지 본다.
