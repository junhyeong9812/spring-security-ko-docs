# 03. 설정 연결과 레거시 애스펙트

## 무엇을 / 왜

02 챕터에서 애스펙트는 `securityInterceptor` 가 주입되어야 동작한다고 했다(없으면 그냥 `proceed()`). 그렇다면 **누가 그 애스펙트를 빈으로 만들고, 누가 인터셉터를 꽂아 주는가?** 답은 `aspects` 모듈 밖, `spring-security-config` 에 있다. 이 챕터는 그 연결 고리와, 같은 모듈에 남아 있는 deprecated `AnnotationSecurityAspect` 를 다룬다.

## 활성화 진입점: mode = ASPECTJ

사용자는 메서드 보안을 켤 때 모드를 고른다.

```java
@EnableMethodSecurity(mode = AdviceMode.ASPECTJ)   // 기본은 AdviceMode.PROXY
```

config 의 `MethodSecuritySelector` 안 `AutoProxyRegistrarSelector` 가 이 `mode` 를 보고 분기한다.

```java
// config: MethodSecuritySelector.AutoProxyRegistrarSelector
if (adviceMode == AdviceMode.ASPECTJ) {
    return new String[] { MethodSecurityAspectJAutoProxyRegistrar.class.getName() };
}
```

즉 `ASPECTJ` 모드일 때만 `MethodSecurityAspectJAutoProxyRegistrar` 가 import 되고, 이 레지스트라가 `aspects` 모듈의 애스펙트들을 무대에 올린다.

## 핵심 연결: aspectOf 로 애스펙트 빈 등록

AspectJ 애스펙트는 보통 스프링이 `new` 로 만들 수 없다. 애스펙트는 **싱글턴 인스턴스를 스스로 관리**하며, 정적 팩터리 메서드 `aspectOf()` 로 그 단일 인스턴스를 얻는다. 레지스트라는 바로 이 점을 이용한다.

```java
// config: MethodSecurityAspectJAutoProxyRegistrar
private void registerBeanDefinition(String beanName, String aspectClassName,
        String aspectBeanName, BeanDefinitionRegistry registry) {
    if (!registry.containsBeanDefinition(beanName)) {
        return;                                          // (1) 해당 인터셉터 빈이 없으면 skip
    }
    BeanDefinition interceptor = registry.getBeanDefinition(beanName);
    BeanDefinitionBuilder aspect =
        BeanDefinitionBuilder.rootBeanDefinition(aspectClassName);
    aspect.setFactoryMethod("aspectOf");                 // (2) new 대신 aspectOf() 로 인스턴스 획득
    aspect.setRole(BeanDefinition.ROLE_INFRASTRUCTURE);
    aspect.addPropertyValue("securityInterceptor", interceptor);  // (3) 인터셉터 주입
    registry.registerBeanDefinition(aspectBeanName, aspect.getBeanDefinition());
}
```

레지스트라는 5개 애스펙트 각각에 대해 이 작업을 반복한다.

```java
registerBeanDefinition("preFilterAuthorizationMethodInterceptor",  PreFilterAspect,    "preFilterAspect$0", ...);
registerBeanDefinition("postFilterAuthorizationMethodInterceptor", PostFilterAspect,   "postFilterAspect$0", ...);
registerBeanDefinition("preAuthorizeAuthorizationMethodInterceptor",  PreAuthorizeAspect,  "preAuthorizeAspect$0", ...);
registerBeanDefinition("postAuthorizeAuthorizationMethodInterceptor", PostAuthorizeAspect, "postAuthorizeAspect$0", ...);
registerBeanDefinition("securedAuthorizationMethodInterceptor",    SecuredAspect,      "securedAspect$0", ...);
```

세 단계의 의미:

- **(1)** 첫 인자(`preAuthorizeAuthorizationMethodInterceptor` 등)는 **core 가 PROXY/ASPECTJ 공통으로 만드는 `MethodInterceptor` 빈 이름**이다. 그 빈이 컨텍스트에 있을 때만 대응 애스펙트를 등록한다. (해당 애너테이션 지원이 꺼져 있으면 skip)
- **(2)** `setFactoryMethod("aspectOf")` 가 위빙의 진짜 연결점이다. ajc/LTW 가 위빙 과정에서 애스펙트 클래스에 `aspectOf()` 정적 메서드를 심어 두는데, 스프링은 이걸 팩터리로 써서 **위버가 관리하는 그 싱글턴 애스펙트 인스턴스**를 빈으로 가져온다. 그래야 바이트코드에 박힌 애스펙트와 스프링이 설정하는 애스펙트가 **같은 객체**가 된다.
- **(3)** 그 애스펙트 인스턴스에 core 의 인터셉터를 `securityInterceptor` 프로퍼티로 주입한다. 이 주입이 끝나야 02 챕터의 `around` advice 에서 `securityInterceptor != null` 분기를 타고 실제 인가가 일어난다.

### 전체 연결 그림

```
@EnableMethodSecurity(mode = ASPECTJ)
        │
        ▼
MethodSecuritySelector → MethodSecurityAspectJAutoProxyRegistrar
        │  for each (interceptor bean ↔ aspect):
        │     aspectOf() 로 위버의 애스펙트 싱글턴 획득
        │     + securityInterceptor = core 인터셉터 주입
        ▼
┌──────────────────────────────────────────────────────────┐
│  위빙된 클래스의 바이트코드                                 │
│    securedBean.transfer()                                  │
│        └─ PreAuthorizeAspect.aspectOf().around()          │
│               securityInterceptor(주입됨).invoke(...)      │
│                    │                                       │
│                    ▼                                       │
│        (core) AuthorizationManagerBeforeMethodInterceptor  │
└──────────────────────────────────────────────────────────┘
```

> 이름이 `...AspectJAutoProxyRegistrar` 라 헷갈릴 수 있는데, 실제로는 프록시를 만들지 않는다. 클래스 주석도 "AspectJAutoProxyRegistrar 가 `EnableAspectJAutoProxy` 만 지원해서 이 클래스가 필요했다" 고 밝힌다. 하는 일은 **애스펙트 빈 등록 + 인터셉터 주입**이다.

## 위빙은 별도 단계

여기서 한 가지가 빠져 있다. 레지스트라는 애스펙트를 **빈으로 등록하고 설정**할 뿐, **바이트코드 위빙 자체**를 하지 않는다. 위빙은 빌드/런타임의 책임이다.

- **CTW(컴파일 타임)**: AspectJ 컴파일러(ajc)가 애플리케이션 클래스를 컴파일하며 `.aj` 애스펙트를 짜 넣는다.
- **LTW(로드 타임)**: 클래스 로딩 시점에 `aspectjweaver` agent(또는 `spring-instrument`)가 위빙한다.

레지스트라가 만든 애스펙트 빈은 "위빙된 코드가 호출할 애스펙트 인스턴스에 인터셉터를 꽂는" 역할이고, 위빙으로 그 인스턴스의 `around` 가 실제 메서드에 연결된다. 둘이 맞물려야 메서드 보안이 완성된다.

## 레거시: AnnotationSecurityAspect (@Deprecated)

같은 모듈의 다른 패키지 `access/intercept/aspectj/aspect/` 에는 구버전 애스펙트가 하나 더 있다.

```java
// AnnotationSecurityAspect.aj  (@since 3.1)
@Deprecated   // Use aspects in authorization.method.aspectj instead
public aspect AnnotationSecurityAspect implements InitializingBean {

    private pointcut securedMethodExecution() :
        executionOfAnyPublicMethodInAtSecuredType()    // @Secured 타입
        || executionOfSecuredMethod()                  // @Secured 메서드
        || executionOfPrePostAnnotatedMethod();        // @Pre/Post* 메서드

    private AspectJMethodSecurityInterceptor securityInterceptor;

    Object around(): securedMethodExecution() {
        if (this.securityInterceptor == null) {
            return proceed();
        }
        AspectJCallback callback = () -> proceed();
        return this.securityInterceptor.invoke(thisJoinPoint, callback);
    }
}
```

현행 애스펙트와 비교하면 설계 차이가 뚜렷하다.

| 구분 | 현행 (`authorization.method.aspectj`) | 레거시 (`AnnotationSecurityAspect`) |
|---|---|---|
| 애스펙트 개수 | 애너테이션별 5개로 분리 | **단일 애스펙트**가 모든 애너테이션 처리 |
| 위임 대상 | AOP Alliance `MethodInterceptor` (현행 인가 엔진) | `AspectJMethodSecurityInterceptor` (구 `AccessDecisionManager`/메타데이터 소스 모델) |
| 어댑터 | `JoinPointMethodInvocation` (표준 `MethodInvocation`) | `AspectJCallback` (전용 콜백, core 의 인터셉터에 직접 thisJoinPoint 전달) |
| 상태 | 권장 | `@Deprecated` |

레거시는 `AuthorizationManager` 기반의 현대적 인가 모델 이전, `AccessDecisionManager`/`SecurityMetadataSource` 시대의 잔재다. 단일 애스펙트가 모든 보안 애너테이션을 한꺼번에 잡고, core 의 `AspectJMethodSecurityInterceptor`(역시 deprecated 계열)로 위임한다. 새 코드는 절대 쓰지 말고 현행 패키지를 써야 한다. 굳이 이 파일이 남은 이유는 하위 호환이다.

## 설계 포인트 / 확장점

- **모드 분리**: 같은 인가 엔진(core 인터셉터)을 PROXY/ASPECTJ 두 위빙 백엔드가 공유한다. 위빙 방식은 `mode` 한 줄로 바꿔 끼울 수 있다.
- **`aspectOf` 브리징**: 스프링 DI 와 AspectJ 위버라는 서로 다른 두 객체 수명주기를 잇는 표준 관용구. 위버 싱글턴을 스프링 빈으로 끌어와 설정한다.
- **조건부 등록**: 인터셉터 빈이 있을 때만 애스펙트를 등록(`containsBeanDefinition` 체크)하므로, 끈 기능에 대한 애스펙트는 만들어지지 않는다.

## 정리

`aspects` 모듈 자체에는 설정 코드가 없다. 활성화는 `@EnableMethodSecurity(mode = ASPECTJ)` 로 시작해 config 의 `MethodSecurityAspectJAutoProxyRegistrar` 가 각 애스펙트를 `aspectOf()` 팩터리로 빈 등록하고 core 의 `MethodInterceptor` 를 `securityInterceptor` 로 주입하는 흐름으로 이뤄진다. 실제 바이트코드 위빙은 CTW/LTW 가 별도로 수행한다. 한편 같은 모듈의 `AnnotationSecurityAspect` 는 구 인가 모델 시절의 단일 애스펙트로, 현재는 `@Deprecated` 이며 `authorization.method.aspectj` 의 분리형 애스펙트로 대체되었다.
