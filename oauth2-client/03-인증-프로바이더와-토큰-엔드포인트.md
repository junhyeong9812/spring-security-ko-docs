# 03. 인증 프로바이더와 Token Endpoint 클라이언트

## 무엇을 / 왜

02장에서 `OAuth2LoginAuthenticationFilter`는 `OAuth2LoginAuthenticationToken`을 `AuthenticationManager`에 위임하고 멈췄다. 그 뒤를 받는 게 이 장의 주역인 **AuthenticationProvider**다. Provider가 하는 일은 두 가지다.

1. 인가 코드를 **Token Endpoint에서 access/refresh token으로 교환** (`OAuth2AccessTokenResponseClient`).
2. (OAuth2 Login의 경우) 그 토큰으로 **UserInfo를 조회**해 `OAuth2User`를 만들고 최종 인증 결과를 조립.

OIDC(scope에 `openid`)인지 아닌지에 따라 두 Provider가 갈린다. 이 장은 일반 OAuth2 Login 경로와 토큰 교환 인프라를 다루고, OIDC 특화는 04장으로 넘긴다.

소스 위치: `oauth2/oauth2-client/src/main/java/org/springframework/security/oauth2/client/authentication/`, `.../endpoint/`

## 동작 흐름: 두 Provider의 협력 (openid 분기)

`AuthenticationManager`(보통 `ProviderManager`)에는 두 Provider가 등록되며, 둘 다 `supports(OAuth2LoginAuthenticationToken)`이 `true`다. 그래서 **요청 자체를 보고 자기 차례가 아니면 `null`을 반환**하는 방식으로 분기한다.

```
ProviderManager.authenticate(OAuth2LoginAuthenticationToken)
        │
        ├─▶ OidcAuthorizationCodeAuthenticationProvider.authenticate()
        │     scope에 "openid" 없으면 → return null (내 일 아님)
        │
        └─▶ OAuth2LoginAuthenticationProvider.authenticate()
              scope에 "openid" 있으면 → return null (OIDC가 처리)
```

`OAuth2LoginAuthenticationProvider`의 첫 줄이 정확히 이 가드다.

```java
if (loginAuthenticationToken.getAuthorizationExchange().getAuthorizationRequest()
        .getScopes().contains("openid")) {
    return null;  // OIDC면 OidcAuthorizationCodeAuthenticationProvider에게 양보
}
```

`ProviderManager`는 `null`을 "이 Provider는 처리 못 함"으로 보고 다음 Provider로 넘어가므로, 같은 토큰 타입을 두 Provider가 scope로 자연스럽게 나눠 갖는다.

### OAuth2LoginAuthenticationProvider — code→token→user

openid가 아니면 본격 처리에 들어간다.

```java
// 1) 내부 위임: 인가 코드를 토큰으로 교환
authorizationCodeAuthenticationToken =
    this.authorizationCodeAuthenticationProvider.authenticate(
        new OAuth2AuthorizationCodeAuthenticationToken(clientRegistration, authorizationExchange));
OAuth2AccessToken accessToken = authorizationCodeAuthenticationToken.getAccessToken();

// 2) 토큰으로 UserInfo 조회 → OAuth2User
OAuth2User oauth2User = this.userService.loadUser(
        new OAuth2UserRequest(clientRegistration, accessToken, additionalParameters));

// 3) 권한 매핑 + authorization_code factor 권한 추가
Collection<GrantedAuthority> mappedAuthorities =
        new LinkedHashSet<>(this.authoritiesMapper.mapAuthorities(authorities));
mappedAuthorities.add(FactorGrantedAuthority.fromAuthority(AUTHORITY)); // AUTHORIZATION_CODE

// 4) 최종 인증 결과(주체=oauth2User) 조립
return new OAuth2LoginAuthenticationToken(clientRegistration, authorizationExchange,
        oauth2User, mappedAuthorities, accessToken, refreshToken);
```

토큰 교환은 또다시 내부의 `OAuth2AuthorizationCodeAuthenticationProvider`에 위임된다는 점에 주목한다. 그 Provider는 `OAuth2AccessTokenResponseClient`를 호출해 실제 HTTP 토큰 요청을 보낼 뿐이다(아래). `FactorGrantedAuthority`로 "이 인증은 authorization_code 흐름을 거쳤다"는 사실을 권한에 새기는 것도 7.x의 multi-factor 표현이다.

### Token Endpoint 호출 — AbstractRestClientOAuth2AccessTokenResponseClient

모든 RestClient 기반 토큰 클라이언트의 공통 골격이다. `getTokenResponse`가 grant 요청을 받아 HTTP POST를 던지고 JSON 토큰 응답을 파싱한다.

```java
public OAuth2AccessTokenResponse getTokenResponse(T grantRequest) {
    OAuth2AccessTokenResponse accessTokenResponse = this.requestEntityConverter.convert(grantRequest)
            .retrieve()
            .body(OAuth2AccessTokenResponse.class);
    ...
}
```

내부에서 일어나는 일을 풀면:

```
grantRequest ─▶ validateClientAuthenticationMethod()  ── basic/post/none만 기본 지원
             ─▶ parametersConverter   → form 파라미터 (grant_type, code, redirect_uri ...)
             ─▶ headersConverter      → Authorization 헤더 (client_secret_basic 등)
             ─▶ RestClient POST tokenUri  body=form
             ─▶ OAuth2AccessTokenResponseHttpMessageConverter 로 JSON 파싱
             ─▶ OAuth2ErrorResponseErrorHandler 가 오류 응답을 OAuth2Error로 변환
```

- **메시지 컨버터**: 요청은 `FormHttpMessageConverter`(application/x-www-form-urlencoded), 응답은 `OAuth2AccessTokenResponseHttpMessageConverter`(JSON→`OAuth2AccessTokenResponse`).
- **클라이언트 인증 검증**: 기본적으로 `client_secret_basic`/`client_secret_post`/`none`만 허용. `private_key_jwt`/`client_secret_jwt`를 쓰려면 `NimbusJwtClientAuthenticationParametersConverter`를 `addParametersConverter`로 합성한다.
- **확장**: `setParametersConverter`/`addParametersConverter`, `setHeadersConverter`/`addHeadersConverter`, `setParametersCustomizer`로 요청을 자유롭게 변형한다. `set/add`의 차이는 "기본 컨버터를 대체하느냐, 기본 위에 합성하느냐"다.

grant 종류별 구현(`RestClientAuthorizationCodeTokenResponseClient` 등 5종)은 이 추상 클래스를 상속하고, 자기 grant에 맞는 `DefaultOAuth2TokenRequestParametersConverter`를 기본값으로 둔다. 6.4부터 권장 구현이 `RestClient` 기반으로 통일되었다(이전 `DefaultAuthorizationCodeTokenResponseClient`/`RestTemplate` 계열 대체).

## 핵심 메서드

| 메서드 | 하는 일 |
|---|---|
| `OAuth2LoginAuthenticationProvider.authenticate` | openid 가드 → code→token 위임 → UserInfo → 최종 토큰 조립 |
| `OAuth2AuthorizationCodeAuthenticationProvider.authenticate` | `OAuth2AccessTokenResponseClient` 호출로 토큰 교환만 수행 |
| `AbstractRestClientOAuth2AccessTokenResponseClient.getTokenResponse` | Token Endpoint에 form POST, JSON 응답 파싱, 오류 변환 |
| `...addParametersConverter / addHeadersConverter` | 토큰 요청을 합성 방식으로 커스터마이즈(JWT 클라이언트 인증 등) |

## 설계 포인트 / 확장점

- **null-반환 분기**: 같은 `OAuth2LoginAuthenticationToken`을 OIDC/비OIDC Provider가 scope 기준으로 나눠 갖는다. `ProviderManager`의 "null이면 다음 Provider" 규약을 활용한 깔끔한 분기.
- **위임의 위임**: 로그인 Provider는 토큰 교환을 `OAuth2AuthorizationCodeAuthenticationProvider`에, HTTP 호출은 `OAuth2AccessTokenResponseClient`에 위임한다. 각 책임이 독립적으로 교체 가능하다.
- **Converter 합성 모델**: 토큰 요청의 헤더/파라미터를 `set`(대체) 또는 `add`(합성)로 조립하는 일관된 확장 패턴. 비표준 인가 서버 대응의 주된 훅.
- **권한 매핑 훅**: `setAuthoritiesMapper(GrantedAuthoritiesMapper)`로 `OAuth2User`의 scope 권한을 조직 내부 ROLE로 매핑한다.

---

## 정리

필터가 넘긴 `OAuth2LoginAuthenticationToken`은 `AuthenticationManager` 안에서 scope의 `openid` 유무로 두 Provider 중 하나가 잡는다. 비OIDC 경로의 `OAuth2LoginAuthenticationProvider`는 토큰 교환(`OAuth2AccessTokenResponseClient`→Token Endpoint POST)과 UserInfo 조회를 엮어 `OAuth2User`를 주체로 하는 인증 결과를 만든다. 토큰 교환 인프라는 grant별 RestClient 구현으로 표준화되어 있고, Converter 합성으로 비표준 서버까지 대응한다.
