# 02. 인가 코드 그랜트 흐름과 서블릿 필터

## 무엇을 / 왜

OAuth2 Login의 핵심은 **Authorization Code Grant**(RFC 6749 §4.1)다. 사용자는 우리 앱이 아니라 인가 서버에서 인증하고, 인가 서버는 우리 앱으로 짧은 수명의 **코드**를 돌려보낸다. 우리 앱은 그 코드를 뒤에서(back-channel) 토큰으로 교환한다. 이 왕복은 두 개의 HTTP 요청으로 쪼개지므로, **두 개의 서블릿 필터**가 흐름을 나눠 맡는다.

```
OAuth2AuthorizationRequestRedirectFilter   ── (1) 인가 서버로 사용자 리다이렉트
OAuth2LoginAuthenticationFilter            ── (2) 돌아온 코드로 로그인 처리
```

이 장은 이 두 필터, 그리고 그 사이에 **state·PKCE·원래 요청 URL**을 안전하게 보관하는 `AuthorizationRequestRepository`를 다룬다. (로그인이 아니라 순수 토큰 획득만 하는 `OAuth2AuthorizationCodeGrantFilter`도 끝에서 짚는다.)

소스 위치: `oauth2/oauth2-client/src/main/java/org/springframework/security/oauth2/client/web/`

---

## 핵심 타입

```
OAuth2AuthorizationRequestRedirectFilter (OncePerRequestFilter)
   - 매칭 URI: /oauth2/authorization/{registrationId}
   - resolver: OAuth2AuthorizationRequestResolver
   - repo:     AuthorizationRequestRepository  ← state/PKCE 저장

OAuth2AuthorizationRequestResolver
   └─ DefaultOAuth2AuthorizationRequestResolver
        - registrationId 추출 → ClientRegistration 조회
        - state, PKCE(code_verifier/challenge), nonce 생성
        - authorizationRequestUri 조립

OAuth2LoginAuthenticationFilter (AbstractAuthenticationProcessingFilter)
   - 매칭 URI: /login/oauth2/code/*
   - authorizationRequestRepository.removeAuthorizationRequest(...)
   - AuthenticationManager.authenticate(OAuth2LoginAuthenticationToken)
   - 성공 시 OAuth2AuthorizedClientRepository.saveAuthorizedClient(...)

AuthorizationRequestRepository<OAuth2AuthorizationRequest>
   └─ HttpSessionOAuth2AuthorizationRequestRepository (기본, 세션 보관)
```

## 동작 흐름: 위임 로그인 전체 시퀀스

```
 사용자(브라우저)        우리 앱(필터 체인)                    인가 서버
     │                       │                                  │
 (a) │  GET /oauth2/authorization/google                        │
     │──────────────────────▶│ RedirectFilter                   │
     │                       │  resolver.resolve(req)           │
     │                       │   → ClientRegistration 조회       │
     │                       │   → state, PKCE, (openid면 nonce) │
     │                       │   → authReq를 Repository(세션)에 저장│
     │  302 Location: AS/authorize?client_id&scope&state&...      │
     │◀──────────────────────│                                  │
     │                                                          │
 (b) │  사용자가 인가 서버에서 로그인 + 동의                       │
     │─────────────────────────────────────────────────────────▶│
     │  302 redirect_uri?code=...&state=...                      │
     │◀─────────────────────────────────────────────────────────│
     │                       │                                  │
 (c) │  GET /login/oauth2/code/google?code=..&state=..           │
     │──────────────────────▶│ OAuth2LoginAuthenticationFilter  │
     │                       │  attemptAuthentication():        │
     │                       │   1. authResponse 파싱            │
     │                       │   2. Repository에서 authReq 제거   │
     │                       │   3. registrationId로 등록 재조회   │
     │                       │   4. OAuth2LoginAuthenticationToken│
     │                       │      → AuthenticationManager       │
     │                       │        (Provider가 code→token 교환,│
     │                       │         UserInfo 호출 — 03/04장)   │
     │                       │   5. OAuth2AuthorizedClient 저장    │
     │                       │   6. OAuth2AuthenticationToken 반환 │
     │  302 → 원래 요청 URL (성공 핸들러)                          │
     │◀──────────────────────│                                  │
```

### (a) RedirectFilter — 인가 요청 시작

`doFilterInternal`은 매 요청마다 resolver에게 "이 요청이 인가 시작 요청인가?"를 묻는다. 해당하면 리다이렉트하고 체인을 끊는다.

```java
OAuth2AuthorizationRequest authorizationRequest = this.authorizationRequestResolver.resolve(request);
if (authorizationRequest != null) {
    this.sendRedirectForAuthorization(request, response, authorizationRequest);
    return;
}
```

`sendRedirectForAuthorization`은 grant가 authorization_code일 때만 `authorizationRequestRepository.saveAuthorizationRequest(...)`로 **state/PKCE/nonce를 세션에 보관**한 뒤, `authorizationRequestUri`로 302 리다이렉트한다. 보관해 두는 이유는 (c)에서 응답의 state를 대조하고 PKCE `code_verifier`를 꺼내 쓰기 위해서다.

이 필터의 두 번째 역할은 **사후 처리**다. 체인을 그냥 통과시켰는데 하위에서 `ClientAuthorizationRequiredException`이 터지면(= "이 클라이언트는 아직 인가가 필요하다"), 그것을 잡아 같은 리다이렉트 흐름을 시작한다. `@RegisteredOAuth2AuthorizedClient`로 토큰을 요구했는데 없을 때 자동으로 로그인 흐름이 시작되는 메커니즘이 바로 이것이다.

### Resolver — state·PKCE 생성의 책임지

`DefaultOAuth2AuthorizationRequestResolver`는 URI 패턴 `/oauth2/authorization/{registrationId}`을 매칭해 registrationId를 뽑고, 등록을 찾아 인가 요청을 조립한다. 보안에 결정적인 두 값을 여기서 만든다.

- **state**: `Base64StringKeyGenerator`로 생성. CSRF(요청 위조) 방어. (c)에서 응답 state와 대조.
- **PKCE**(`requireProofKey`이거나 public client일 때): `code_verifier`(96바이트 랜덤)를 만들어 세션에 보관하고, 그 SHA-256 해시인 `code_challenge`만 인가 서버로 보낸다. 코드 가로채기 공격을 막는다. `OAuth2AuthorizationRequestCustomizers.withPkce()`가 적용된다.
- **nonce**(scope에 `openid` 포함 시): id_token 재생(replay) 방어용. (04장에서 검증.)

### (c) LoginFilter — 코드 회수와 인증 위임

`attemptAuthentication`이 흐름의 심장이다. 단계별로:

```java
// 1. 쿼리 파라미터가 정상 인가 응답인지 확인 (code 또는 error)
if (!OAuth2AuthorizationResponseUtils.isAuthorizationResponse(params)) { throw ... }
// 2. (a)에서 저장한 인가 요청을 꺼내며 제거 — 없으면 위조/만료
OAuth2AuthorizationRequest authorizationRequest =
        this.authorizationRequestRepository.removeAuthorizationRequest(request, response);
if (authorizationRequest == null) { throw "authorization_request_not_found"; }
// 3. 저장된 요청 안의 registrationId로 ClientRegistration 재조회
String registrationId = authorizationRequest.getAttribute(OAuth2ParameterNames.REGISTRATION_ID);
ClientRegistration clientRegistration = this.clientRegistrationRepository.findByRegistrationId(registrationId);
// 4. 요청+응답을 묶은 인증 토큰을 만들어 Manager에 위임
OAuth2LoginAuthenticationToken authenticationRequest =
        new OAuth2LoginAuthenticationToken(clientRegistration,
                new OAuth2AuthorizationExchange(authorizationRequest, authorizationResponse));
OAuth2LoginAuthenticationToken authenticationResult =
        (OAuth2LoginAuthenticationToken) getAuthenticationManager().authenticate(authenticationRequest);
```

여기서 필터는 **토큰 교환·UserInfo 호출을 직접 하지 않는다.** `OAuth2LoginAuthenticationToken`을 만들어 `AuthenticationManager`에 넘기고, 그 일은 `OAuth2LoginAuthenticationProvider`(03장)가 처리한다. 필터/프로바이더 분리는 Spring Security의 일관된 패턴이다.

인증이 끝나면 필터는 결과에서 토큰을 꺼내 `OAuth2AuthorizedClient`를 만들고 저장한 뒤, SecurityContext에 들어갈 최종 `OAuth2AuthenticationToken`을 반환한다.

```java
OAuth2AuthorizedClient authorizedClient = new OAuth2AuthorizedClient(
        authenticationResult.getClientRegistration(), oauth2Authentication.getName(),
        authenticationResult.getAccessToken(), authenticationResult.getRefreshToken());
this.authorizedClientRepository.saveAuthorizedClient(authorizedClient, oauth2Authentication, request, response);
return oauth2Authentication;
```

`OAuth2LoginAuthenticationFilter`는 `AbstractAuthenticationProcessingFilter`를 상속하므로, 성공/실패 핸들러·SecurityContext 저장·세션 전략 등 표준 로그인 인프라를 그대로 물려받는다.

## OAuth2AuthorizationCodeGrantFilter — 로그인 아닌 순수 토큰 획득

같은 패키지의 `OAuth2AuthorizationCodeGrantFilter`는 "로그인 인증"이 목적이 아니라, **이미 로그인한 사용자**가 어떤 클라이언트의 토큰을 추가로 얻고 싶을 때 쓰인다(예: 별도 OAuth2 API 연동). 인가 응답을 받아 `OAuth2AuthorizationCodeAuthenticationToken`으로 인증해 토큰만 얻고 `OAuth2AuthorizedClient`로 저장할 뿐, SecurityContext의 주체를 바꾸지 않는다. OAuth2 Login과 "토큰만 필요한 client" 시나리오를 구분하는 장치다.

## 핵심 메서드

| 메서드 | 하는 일 |
|---|---|
| `RedirectFilter.doFilterInternal` | resolve→리다이렉트, 또는 하위 `ClientAuthorizationRequiredException` 포착해 재리다이렉트 |
| `DefaultOAuth2AuthorizationRequestResolver.resolve` | registrationId 매칭, state/PKCE/nonce 생성, 인가 URI 조립 |
| `LoginFilter.attemptAuthentication` | 응답 검증 → 저장된 요청 회수 → Manager 위임 → AuthorizedClient 저장 |
| `HttpSessionOAuth2AuthorizationRequestRepository.save/remove` | 인가 요청(state/PKCE)을 세션에 보관/회수 |

## 설계 포인트 / 확장점

- **2-필터 분리**: 인가 코드 그랜트의 두 HTTP 왕복을 두 필터로 명확히 나눴다. 사이의 상태는 `AuthorizationRequestRepository`가 책임진다.
- **상태 보관 전략 교체**: 기본은 세션(`HttpSession...`)이지만, 무상태 환경(쿠키 기반)을 위해 `AuthorizationRequestRepository`를 구현해 교체할 수 있다. 두 필터·resolver 모두 `setAuthorizationRequestRepository`로 갈아끼운다.
- **요청 커스터마이징**: resolver의 `setAuthorizationRequestCustomizer(...)`로 인가 요청에 파라미터를 더하거나 PKCE를 명시적으로 켤 수 있다(`OAuth2AuthorizationRequestCustomizers.withPkce()`).
- **콜백/기본 URI 변경**: 기본 콜백은 `/login/oauth2/code/*`, 인가 시작은 `/oauth2/authorization`. 생성자 인자로 변경 가능.

---

## 정리

위임 로그인은 두 필터로 흐른다. `OAuth2AuthorizationRequestRedirectFilter`가 state/PKCE를 세션에 심고 사용자를 인가 서버로 보내고, 돌아온 코드는 `OAuth2LoginAuthenticationFilter`가 받아 — 저장해 둔 요청을 회수해 state를 대조한 뒤 — `OAuth2LoginAuthenticationToken`을 `AuthenticationManager`에 위임한다. 실제 토큰 교환과 사용자 조회는 다음 장의 프로바이더 몫이다.
