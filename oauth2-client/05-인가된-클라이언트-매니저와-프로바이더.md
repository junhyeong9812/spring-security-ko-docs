# 05. 인가된 클라이언트 매니저와 프로바이더

## 무엇을 / 왜

지금까지(02~04장)는 **로그인** 이야기였다. 이 장은 다른 축, 즉 **토큰을 획득·갱신·보관하고 리소스 서버 호출에 쓰는 인프라**다. "Google로 로그인" 없이도, 백그라운드 작업이 `client_credentials`로 토큰을 받아 외부 API를 호출하거나, 로그인한 사용자의 access token이 만료되면 refresh token으로 조용히 갱신해야 한다.

이 모든 것의 진입점이 `OAuth2AuthorizedClientManager`이고, 실제 grant별 로직은 `OAuth2AuthorizedClientProvider`들이 맡는다. 이 장은 Manager/Provider 구조, 표준 grant 5종, 그리고 토큰을 실제로 HTTP 호출에 주입하는 방법(`@RegisteredOAuth2AuthorizedClient`, RestClient 인터셉터)을 다룬다.

소스 위치: `oauth2/oauth2-client/src/main/java/org/springframework/security/oauth2/client/` (패키지 루트)

---

## 핵심 타입

```
OAuth2AuthorizedClientManager (FunctionalInterface)
   authorize(OAuth2AuthorizeRequest) → OAuth2AuthorizedClient | null
   ├─ DefaultOAuth2AuthorizedClientManager            (HttpServletRequest 컨텍스트)
   └─ AuthorizedClientServiceOAuth2AuthorizedClientManager  (요청 밖, Service 기반)

OAuth2AuthorizedClientProvider (FunctionalInterface)
   authorize(OAuth2AuthorizationContext) → OAuth2AuthorizedClient | null
   ├─ AuthorizationCodeOAuth2AuthorizedClientProvider   인가 필요 시 예외 발생
   ├─ RefreshTokenOAuth2AuthorizedClientProvider        만료 시 refresh
   ├─ ClientCredentialsOAuth2AuthorizedClientProvider
   ├─ JwtBearerOAuth2AuthorizedClientProvider
   ├─ TokenExchangeOAuth2AuthorizedClientProvider
   └─ DelegatingOAuth2AuthorizedClientProvider          (위 들을 순서대로 시도)

OAuth2AuthorizationContext     Provider 입력 (등록 또는 기존 authorizedClient + principal + attributes)
OAuth2AuthorizedClientProviderBuilder   Provider 조합 빌더
OAuth2AuthorizationSuccess/FailureHandler   성공 시 저장 / 실패(invalid_grant) 시 제거
```

---

## 동작 흐름: authorize() 한 번에 무슨 일이

`DefaultOAuth2AuthorizedClientManager.authorize`가 표준 시나리오다. "이 사용자의 이 클라이언트 토큰을 다오 — 없으면 받아오고, 만료면 갱신해서"라는 한 문장을 코드로 풀면:

```
authorize(OAuth2AuthorizeRequest)
   │
   1. 요청에 authorizedClient가 직접 들어있나?
   │     예 → 그것으로 context (재인가 경로)
   │     아니오 → Repository.loadAuthorizedClient(regId, principal, request)
   │              있으면 그것으로, 없으면 ClientRegistration 조회로 context (최초 인가)
   │
   2. context = withClientRegistration|withAuthorizedClient
   │             .principal(principal)
   │             .attributes(contextAttributesMapper.apply(req))  ← 예: 요청 scope
   │             .build()
   │
   3. authorizedClient = authorizedClientProvider.authorize(context)
   │     (DelegatingProvider가 grant별 Provider를 순서대로 시도)
   │     ├─ 예외(OAuth2AuthorizationException) → failureHandler.onAuthorizationFailure()
   │     │     RemoveAuthorizedClient...: invalid_grant이면 저장된 토큰 제거 후 재던짐
   │     └─ 성공·non-null → successHandler.onAuthorizationSuccess()
   │           기본: Repository.saveAuthorizedClient(...)
   │
   4. provider가 null 반환(재인가 불필요)이면 context의 기존 authorizedClient 반환
```

기본 Provider 조합은 생성 시점에 고정된다.

```java
private static final OAuth2AuthorizedClientProvider DEFAULT_AUTHORIZED_CLIENT_PROVIDER =
    OAuth2AuthorizedClientProviderBuilder.builder()
        .authorizationCode()
        .refreshToken()
        .clientCredentials()
        .build();
```

이 `DelegatingOAuth2AuthorizedClientProvider`는 등록된 Provider들을 순서대로 호출하며, **각자 "내 grant가 아니거나 처리할 게 없으면 null"** 을 반환한다. 첫 non-null을 채택한다.

### Provider의 null-반환 = "내 차례 아님"

`RefreshTokenOAuth2AuthorizedClientProvider`가 전형적이다. authorize의 첫머리에서 "재인가가 불필요한 조건"을 모두 null로 걸러낸다.

```java
public OAuth2AuthorizedClient authorize(OAuth2AuthorizationContext context) {
    OAuth2AuthorizedClient authorizedClient = context.getAuthorizedClient();
    if (authorizedClient == null                              // 아직 인가 안 됨
            || authorizedClient.getRefreshToken() == null     // refresh token 없음
            || !hasTokenExpired(authorizedClient.getAccessToken())) { // 아직 안 만료
        return null;  // → DelegatingProvider가 다음 Provider로
    }
    // 만료 + refresh token 있음 → refresh_token 그랜트로 갱신
    OAuth2RefreshTokenGrantRequest req = new OAuth2RefreshTokenGrantRequest(...);
    OAuth2AccessTokenResponse tokenResponse = this.accessTokenResponseClient.getTokenResponse(req);
    OAuth2AuthorizedClient refreshed = new OAuth2AuthorizedClient(..., tokenResponse.getAccessToken(), ...);
    // 갱신 이벤트 발행 (OIDC user 갱신 등 후속 처리용)
    this.applicationEventPublisher.publishEvent(new OAuth2AuthorizedClientRefreshedEvent(...));
    return refreshed;
}
```

`hasTokenExpired`는 `clockSkew`(기본 60초)를 빼서 미리 만료로 본다 — 호출 직전에 만료되는 경계 오류를 막는다. 만료된 토큰을 refresh로 갱신하면 새 `OAuth2AuthorizedClient`를 만들고 `OAuth2AuthorizedClientRefreshedEvent`를 발행한다. (OIDC라면 `OidcAuthorizedClientRefreshedEventListener`가 이 이벤트를 받아 `OidcUser`도 갱신한다.)

`AuthorizationCodeOAuth2AuthorizedClientProvider`는 다르다. 토큰이 없으면 스스로 갱신할 수 없으므로(사용자 리다이렉트가 필요) `ClientAuthorizationRequiredException`을 던진다. 02장의 `OAuth2AuthorizationRequestRedirectFilter`가 이 예외를 잡아 로그인 리다이렉트를 시작한다 — 두 축이 여기서 연결된다.

## 토큰을 실제 HTTP 호출에 주입하기

토큰을 받아도 결국 리소스 서버를 호출해야 의미가 있다. 세 가지 주입 경로가 있다.

### 1) @RegisteredOAuth2AuthorizedClient — 컨트롤러 인자

```java
@GetMapping("/articles")
public String articles(@RegisteredOAuth2AuthorizedClient("github") OAuth2AuthorizedClient client) {
    String token = client.getAccessToken().getTokenValue();
    ...
}
```

`OAuth2AuthorizedClientArgumentResolver`(HandlerMethodArgumentResolver)가 이 파라미터를 보고 내부적으로 `OAuth2AuthorizedClientManager.authorize(...)`를 호출해 토큰을 확보하거나 갱신해 주입한다. 토큰이 없으면 Manager가 `ClientAuthorizationRequiredException` 경로로 로그인을 유발한다.

### 2) RestClient 인터셉터 — OAuth2ClientHttpRequestInterceptor

```java
OAuth2ClientHttpRequestInterceptor interceptor =
        new OAuth2ClientHttpRequestInterceptor(authorizedClientManager);
RestClient restClient = RestClient.builder().requestInterceptor(interceptor).build();
```

매 요청 직전에 Manager로 토큰을 얻어 `Authorization: Bearer ...` 헤더를 붙인다. 더해, 리소스 서버가 401/403을 돌려주면 그것을 `OAuth2AuthorizationFailureHandler`로 전달할 수 있다 — `RemoveAuthorizedClientOAuth2AuthorizationFailureHandler`를 쓰면 무효해진 토큰을 저장소에서 제거해 다음 호출에 재발급을 유도한다. 어떤 registrationId를 쓸지는 `RequestAttributeClientRegistrationIdResolver` 등으로 요청 속성에서 해석한다.

### 3) WebClient ExchangeFilterFunction (리액티브 — 06장)

`web/reactive/function/client`의 `ServerOAuth2AuthorizedClientExchangeFilterFunction`이 같은 역할을 WebClient에서 한다.

## 핵심 메서드

| 메서드 | 하는 일 |
|---|---|
| `DefaultOAuth2AuthorizedClientManager.authorize` | 저장된 토큰 로드 → Provider 위임 → 성공 저장 / 실패 제거 |
| `OAuth2AuthorizedClientProviderBuilder.builder()...build()` | authorization_code/refresh/client_credentials 등 Provider 조합 |
| `RefreshTokenOAuth2AuthorizedClientProvider.authorize` | 만료+refresh 가능 시에만 갱신, 아니면 null |
| `AuthorizationCodeOAuth2AuthorizedClientProvider.authorize` | 토큰 없으면 `ClientAuthorizationRequiredException`(→리다이렉트) |
| `OAuth2ClientHttpRequestInterceptor.intercept` | Manager로 토큰 확보 후 Bearer 헤더 주입, 실패 핸들러 연동 |

## 설계 포인트 / 확장점

- **Manager(관리) vs Provider(전략) 분리**: Manager는 "로드→위임→저장/제거"라는 오케스트레이션만, Provider는 grant별 토큰 획득만 책임진다. 새 grant(jwt-bearer, token-exchange)는 Provider만 추가하면 된다.
- **Delegating + null 규약**: 여러 grant Provider를 순서대로 시도하고 "내 일 아니면 null". 03장 AuthenticationProvider와 동일한 패턴이 인가 측에도 적용된다.
- **요청 컨텍스트 유무로 Manager 2종**: `DefaultOAuth2AuthorizedClientManager`(웹)와 `AuthorizedClientServiceOAuth2AuthorizedClientManager`(스케줄러·백그라운드). 후자는 `OAuth2AuthorizedClientService`로 토큰을 보관한다.
- **성공/실패 핸들러 훅**: `setAuthorizationSuccessHandler`/`setAuthorizationFailureHandler`로 저장 정책을 바꾼다. 기본 실패 핸들러는 `invalid_grant`(만료된 refresh token 등)에서 토큰을 제거한다.
- **clockSkew/Clock**: 만료 판정의 경계를 테스트·운영 환경에 맞게 조정.

## 정리

`OAuth2AuthorizedClientManager`는 "토큰을 다오(없으면 받아오고 만료면 갱신해서)"라는 단일 진입점이다. 내부적으로 저장된 토큰을 로드해 `OAuth2AuthorizationContext`로 감싸고, `Delegating` Provider가 grant별 Provider(refresh/client_credentials/authorization_code/jwt-bearer/token-exchange)를 null-규약으로 순회한다. 성공하면 저장, `invalid_grant` 실패면 제거한다. 확보된 토큰은 `@RegisteredOAuth2AuthorizedClient`, RestClient 인터셉터, WebClient 필터를 통해 리소스 서버 호출의 Bearer 헤더로 흘러간다.
