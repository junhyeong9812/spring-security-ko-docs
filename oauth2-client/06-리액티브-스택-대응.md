# 06. 리액티브 스택(WebFlux) 대응

## 무엇을 / 왜

지금까지의 필터·프로바이더·서비스는 대부분 **서블릿(Web MVC, 블로킹)** 스택을 전제로 한다 — `HttpServletRequest`, `OncePerRequestFilter`, `RestClient`/`RestOperations`. 그러나 WebFlux 기반 리액티브 애플리케이션도 똑같이 OAuth2 Login과 토큰 관리가 필요하다.

이 모듈은 서블릿판 컴포넌트마다 **리액티브 미러**를 제공한다. 이름 규칙은 일관적이다: `Reactive`(논블로킹 도메인) 또는 `Server`(WebFlux `ServerWebExchange` 기반) 접두/접미가 붙고, 반환 타입이 `Mono`/`Flux`이며, HTTP 통신은 `RestClient` 대신 `WebClient`를 쓴다. 이 장은 그 대응 관계를 한눈에 정리해, 앞 장의 개념을 리액티브로 그대로 옮기는 지도를 제공한다.

소스 위치: `oauth2/oauth2-client/src/main/java/org/springframework/security/oauth2/client/` 전반의 `Reactive*` / `Server*`, 그리고 `web/server/`, `web/reactive/`.

## 무엇이 같고 무엇이 다른가

### 같은 것 — 개념과 흐름

01~05장에서 설명한 **개념적 흐름은 그대로다.** 인가 코드 그랜트의 두 단계(리다이렉트→코드 회수), Provider의 null/`Mono.empty()` 분기, Manager의 "로드→위임→저장", id_token 검증·nonce·sub 일치 같은 보안 검증은 리액티브판도 동일하게 수행한다. 예를 들어 `OAuth2LoginReactiveAuthenticationManager`도 scope의 `openid` 유무로 OIDC 매니저에 양보하고, `RefreshTokenReactiveOAuth2AuthorizedClientProvider`도 만료+refresh 가능 조건만 처리한다.

### 다른 것 — 실행 모델

- **반환 타입**: 모든 단계가 `Mono<...>`/`Flux<...>`를 반환해 논블로킹으로 연결된다. "처리 못 함"은 `null` 대신 `Mono.empty()`로 표현한다.
- **HTTP 클라이언트**: 블로킹 `RestClient`/`RestOperations` 대신 논블로킹 `WebClient`. Token Endpoint/UserInfo 호출이 이벤트 루프를 막지 않는다.
- **요청 추상화**: `HttpServletRequest`/`Response` 대신 `ServerWebExchange`. 세션도 `HttpSession` 대신 WebFlux `WebSession`(`WebSession...Repository`).
- **필터 모델**: 서블릿 `Filter` 대신 WebFlux `WebFilter`. 로그인은 일반 `AuthenticationWebFilter`에 OAuth2용 converter(`ServerOAuth2AuthorizationCodeAuthenticationTokenConverter`)를 끼워 구성한다.
- **보안 컨텍스트**: `SecurityContextHolder`(ThreadLocal) 대신 리액터 컨텍스트 기반 `ReactiveSecurityContextHolder`.

### WebClient 리소스 호출

서블릿의 `OAuth2ClientHttpRequestInterceptor`에 대응하는 것이 `ServerOAuth2AuthorizedClientExchangeFilterFunction`이다.

```java
ServerOAuth2AuthorizedClientExchangeFilterFunction oauth2 =
        new ServerOAuth2AuthorizedClientExchangeFilterFunction(authorizedClientManager);
WebClient webClient = WebClient.builder().filter(oauth2).build();
```

매 교환마다 `ReactiveOAuth2AuthorizedClientManager`로 토큰을 확보(필요 시 refresh)해 Bearer 헤더를 붙인다. 서블릿판과 동일하게, registrationId/principal 해석과 401/403 실패 처리를 지원한다.

## 핵심 포인트

| 축 | 서블릿 | 리액티브 |
|---|---|---|
| 비동기 표현 | 동기 반환, `null`=처리불가 | `Mono`/`Flux`, `Mono.empty()`=처리불가 |
| HTTP | RestClient / RestOperations | WebClient |
| 요청/세션 | HttpServletRequest / HttpSession | ServerWebExchange / WebSession |
| 필터 | Servlet Filter | WebFilter |
| 컨텍스트 | SecurityContextHolder | ReactiveSecurityContextHolder |

---

## 정리

리액티브 스택은 앞 장들의 **개념을 1:1로 미러링**하되, 실행 모델만 논블로킹으로 바꾼다. 컴포넌트 이름에서 `Reactive`/`Server`를 떼면 대응하는 서블릿 컴포넌트를, 동작 원리는 해당 장(02~05)을 그대로 참조하면 된다. 차이는 본질이 아니라 표면(`Mono`/`WebClient`/`ServerWebExchange`)에 있다. 따라서 OAuth2 Login·토큰 관리의 보안 검증과 흐름은 두 스택에서 동일하게 보장된다.
