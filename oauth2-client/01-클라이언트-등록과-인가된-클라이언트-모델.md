# 01. 클라이언트 등록과 인가된 클라이언트 모델

## 무엇을 / 왜

OAuth2 클라이언트의 모든 흐름은 결국 두 가지 데이터를 다룬다.

- **"우리는 어떤 클라이언트인가"** — 인가 서버에 등록된 `clientId`, `clientSecret`, 요청할 `scope`, 사용할 grant 타입, 그리고 authorization/token/userinfo/jwk 엔드포인트 URI. 이것이 `ClientRegistration`이다.
- **"이 사용자에게 어떤 토큰이 발급되었는가"** — 특정 등록(Client)을 특정 사용자(Principal)에게 발급된 access/refresh token과 묶은 것. 이것이 `OAuth2AuthorizedClient`다.

이 장은 모듈 전체가 읽고 쓰는 이 두 타입과, 그것을 보관하는 저장소 계층을 다룬다. 흐름(필터·프로바이더)을 보기 전에 데이터 토대를 먼저 잡는다.

소스 위치: `oauth2/oauth2-client/src/main/java/org/springframework/security/oauth2/client/registration/`, 그리고 동일 패키지 루트의 `OAuth2AuthorizedClient` / `OAuth2AuthorizedClientService`.

## 핵심 타입

```
ClientRegistration (불변, Builder로 생성)
   ├─ registrationId           "google", "github" 같은 논리 키
   ├─ clientId / clientSecret
   ├─ clientAuthenticationMethod  client_secret_basic / _post / none ...
   ├─ authorizationGrantType      authorization_code / client_credentials / ...
   ├─ redirectUri (템플릿 가능)   {baseUrl}/login/oauth2/code/{registrationId}
   ├─ scopes                      openid, profile, email ...
   ├─ clientSettings              requireProofKey (PKCE 강제 여부)
   └─ ProviderDetails
        ├─ authorizationUri / tokenUri / jwkSetUri / issuerUri
        └─ UserInfoEndpoint (uri, authenticationMethod, userNameAttributeName)

ClientRegistrationRepository
   └─ findByRegistrationId(id) → ClientRegistration | null

OAuth2AuthorizedClient (직렬화 가능)
   ├─ clientRegistration   ← 어떤 클라이언트
   ├─ principalName        ← 누구에게
   ├─ accessToken          ← 무슨 토큰 (필수)
   └─ refreshToken         ← 갱신용 (nullable)

OAuth2AuthorizedClientService   (HTTP 요청 밖에서도 사용)
OAuth2AuthorizedClientRepository (HttpServletRequest 컨텍스트 기반, 02장)
```

### ClientRegistration — 불변 + Builder + 검증

`ClientRegistration`은 `final` 클래스이며 생성자는 `private`다. 오직 `withRegistrationId(...).build()` 빌더로만 만든다. `build()`는 grant 타입에 따라 필수값을 검증한다.

```java
// ClientRegistration.Builder.build()
if (AuthorizationGrantType.CLIENT_CREDENTIALS.equals(this.authorizationGrantType)) {
    this.validateClientCredentialsGrantType();  // registrationId, clientId, tokenUri 필수
}
else if (AuthorizationGrantType.AUTHORIZATION_CODE.equals(this.authorizationGrantType)) {
    this.validateAuthorizationCodeGrantType();  // + redirectUri, authorizationUri 필수
}
```

빌더가 처리하는 두 가지 영리한 기본값에 주목한다.

- **clientAuthenticationMethod 추론**(`deduceClientAuthenticationMethod`): 명시하지 않으면, authorization_code인데 secret이 없으면 `NONE`(공개 클라이언트), 그 외엔 `CLIENT_SECRET_BASIC`으로 결정한다.
- **PKCE 강제**: `ClientSettings.requireProofKey`의 빌더 기본값은 `true`다. 단 grant가 authorization_code가 아니면 `validateAuthorizationGrantTypes()`에서 `false`로 되돌린다. 즉 PKCE는 authorization_code에만 의미가 있다.

`ProviderDetails`와 그 안의 `UserInfoEndpoint`는 내부 클래스로, 인가 서버 측 엔드포인트 정보를 묶는다. `userNameAttributeName`은 UserInfo 응답에서 "사용자 이름"으로 쓸 속성 키인데, 제공자마다 달라서(`sub`, `id`, `login`...) 등록 시 지정해야 한다(04장).

### ClientRegistrationRepository — registrationId로 찾는 단일 메서드

```java
public interface ClientRegistrationRepository {
    @Nullable ClientRegistration findByRegistrationId(String registrationId);
}
```

JavaDoc이 강조하듯, 등록 정보의 **원본 소유자는 인가 서버**이고 이 저장소는 그 복사본을 클라이언트 쪽에 두는 것뿐이다. 구현체:

- `InMemoryClientRegistrationRepository` — 설정에서 정적으로 등록(가장 흔함).
- `SupplierClientRegistrationRepository` — 지연 로딩(`Supplier`)으로 구성.
- 같은 패키지의 `ClientRegistrations`(클래스) — `issuerUri` 하나로 OIDC/OAuth2 **discovery 문서**(`.well-known/openid-configuration`)를 가져와 빌더를 자동 채워 주는 정적 헬퍼.

### OAuth2AuthorizedClient — "등록 + 사용자 + 토큰"의 결합

```java
public class OAuth2AuthorizedClient implements Serializable {
    private final ClientRegistration clientRegistration;
    private final String principalName;
    private final OAuth2AccessToken accessToken;       // 필수
    private final @Nullable OAuth2RefreshToken refreshToken;
}
```

이 객체 하나가 "이 사용자가 이 클라이언트에 권한을 부여했고, 그 결과 이 토큰을 받았다"는 사실을 표현한다. 리소스 호출 시 `getAccessToken()`을 꺼내 Bearer 헤더로 쓰고, 만료되면 `getRefreshToken()`으로 갱신한다(05장).

## 동작 흐름: 토큰은 어디에 저장되는가

발급된 `OAuth2AuthorizedClient`를 보관하는 추상화가 두 갈래로 나뉜다는 점이 핵심이다.

```
  HTTP 요청 컨텍스트가 있는가?
        │
   ┌────┴─────────────────────────────┐
   │ 있다 (필터/컨트롤러 안)            │ 없다 (스케줄러/백그라운드)
   ▼                                   ▼
 OAuth2AuthorizedClientRepository    OAuth2AuthorizedClientService
   save/load/remove(                   save/load/remove(
     ..., HttpServletRequest,            clientRegistrationId,
     HttpServletResponse)                principalName)
   │ 기본 구현                          │ 구현
   │ AuthenticatedPrincipal-           ├─ InMemoryOAuth2AuthorizedClientService
   │   OAuth2AuthorizedClientRepository├─ JdbcOAuth2AuthorizedClientService
   │   (인증된 사용자면 Service로 위임, │   (oauth2_authorized_client 테이블)
   │    익명이면 HttpSession에 보관)    │
   └───────────────────────────────────┘
```

`OAuth2AuthorizedClientService`는 HTTP 요청과 무관한 키(`clientRegistrationId`, `principalName`)만 받는다. 그래서 배치/스케줄러처럼 요청 바깥에서도 토큰을 꺼낼 수 있다.

반면 `OAuth2AuthorizedClientRepository`(02장에서 상세)는 `HttpServletRequest`/`Response`를 받는다. 기본 구현 `AuthenticatedPrincipalOAuth2AuthorizedClientRepository`는 영리하게 위임한다.

- 현재 사용자가 **인증된 상태**면 → 내부의 `OAuth2AuthorizedClientService`에 저장(서버 공유 저장소).
- **익명(아직 로그인 안 됨)**이면 → `HttpSessionOAuth2AuthorizedClientRepository`로 세션에 저장.

이 분기 덕분에 "로그인 진행 중(익명)"과 "로그인 완료 후"의 토큰을 자연스럽게 같은 인터페이스로 다룬다.

## 핵심 메서드

| 타입.메서드 | 하는 일 |
|---|---|
| `ClientRegistration.withRegistrationId(id)` | 빌더 시작. `build()`에서 grant별 필수값·PKCE·인증방식을 검증/추론 |
| `ClientRegistrationRepository.findByRegistrationId(id)` | 콜백 URL의 `{registrationId}`로 등록을 찾는 모든 필터의 출발점 |
| `OAuth2AuthorizedClientService.loadAuthorizedClient(regId, principalName)` | 요청 밖에서 토큰 조회 |
| `OAuth2AuthorizedClientService.saveAuthorizedClient(client, principal)` | 발급/갱신된 토큰 영속화 |
| `JdbcOAuth2AuthorizedClientService` | RDB에 토큰 저장(다중 인스턴스 환경에서 세션 대신 사용) |

---

## 설계 포인트 / 확장점

- **불변 + Builder + 검증 응집**: `ClientRegistration`은 만들어진 뒤 바뀌지 않는다. 모든 유효성 검사가 `build()` 한 곳에 모여 있어, 잘못된 등록은 부팅 시점에 빠르게 실패한다.
- **저장소 인터페이스 2분할**: "요청 컨텍스트 유무"라는 단 하나의 축으로 `Service`/`Repository`를 갈라, 백그라운드 토큰 사용과 웹 로그인 토큰 보관을 분리했다.
- **교체 지점**: 등록을 동적으로(DB·테넌트별) 관리하려면 `ClientRegistrationRepository`를 직접 구현한다. 토큰을 RDB/Redis에 두려면 `OAuth2AuthorizedClientService`를 구현(`JdbcOAuth2AuthorizedClientService`가 좋은 출발점)한다.

---

## 정리

`ClientRegistration`은 "우리가 누구이고 인가 서버의 어느 엔드포인트와 통신하는가"를, `OAuth2AuthorizedClient`는 "그 결과 어떤 사용자에게 어떤 토큰이 발급되었는가"를 표현하는 불변 데이터다. 이 둘을 `ClientRegistrationRepository`(registrationId→등록)와 `OAuth2AuthorizedClientService`/`Repository`(토큰 보관, 요청 컨텍스트 유무로 분기)가 보관한다. 이어지는 모든 장의 필터·프로바이더는 결국 이 두 타입을 읽고 쓰는 흐름이다.
