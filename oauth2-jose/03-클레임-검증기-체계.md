# 03. 클레임 검증기 체계

## 무엇을 / 왜

서명이 유효해도 토큰이 곧바로 신뢰되는 것은 아니다. 만료되었을 수도, 다른 발급자가 만든 것일 수도, 우리 서비스를 대상(`aud`)으로 하지 않을 수도 있다. 이런 **내용(클레임) 검증**을 2장의 디코더와 분리해 작은 부품으로 구현한 것이 `OAuth2TokenValidator<Jwt>` 패밀리다. 디코더의 4단계(`validateJwt`)는 이 검증기 하나에 위임하며, 그 검증기는 보통 여러 개를 합친 `DelegatingOAuth2TokenValidator`다.

이 장은 "작은 검증기들을 어떻게 만들고 조합하는가", 그리고 표준 묶음을 만드는 `JwtValidators`를 본다.

## 핵심 타입

```
   OAuth2TokenValidator<Jwt>  (oauth2-core, 함수형)
        OAuth2TokenValidatorResult validate(Jwt)
              │
   ┌──────────┼───────────────────────────────────────────┐
   │ 합성기                                                  │
   │  DelegatingOAuth2TokenValidator  ← 여러 검증기를 AND   │
   ├──────────────────────────────────────────────────────┤
   │ 단위 검증기                                            │
   │  JwtTimestampValidator   exp/nbf 시각 (+clock skew)    │
   │  JwtIssuerValidator      iss == 기대값                 │
   │  JwtAudienceValidator    aud 포함 여부                 │
   │  JwtIssuedAtValidator    iat 존재/유효                 │
   │  JwtTypeValidator        typ 헤더 화이트리스트          │
   │  X509CertificateThumbprintValidator  cnf.x5t#S256(mTLS)│
   │  JwtClaimValidator<T>    임의 클레임 + Predicate       │
   └──────────────────────────────────────────────────────┘
              │ 팩토리
   JwtValidators : createDefault / createDefaultWithIssuer /
                   createDefaultWithValidators / createAtJwtValidator
```

검증기는 성공/실패를 `OAuth2TokenValidatorResult`로 돌려준다. 실패는 `OAuth2Error`(코드 `invalid_token` 등)를 담는다. 예외를 던지지 않고 결과 객체로 표현하는 것이 이 체계의 일관된 약속이며, 최종적으로 디코더가 모아 `JwtValidationException`으로 던진다.

## JwtClaimValidator — 가장 일반적인 부품

`JwtClaimValidator<T>`는 "특정 클레임을 꺼내 Predicate로 검사"하는 범용 검증기다. 다른 여러 검증기가 내부적으로 이걸 쓴다.

```java
@Override
public OAuth2TokenValidatorResult validate(Jwt token) {
    T claimValue = token.getClaim(this.claim);
    if (claimValue != null && this.test.test(claimValue)) {
        return OAuth2TokenValidatorResult.success();
    }
    this.logger.debug(this.error.getDescription());
    return OAuth2TokenValidatorResult.failure(this.error);
}
```

클레임이 없거나(`null`) Predicate가 거짓이면 실패. 예를 들어 `JwtIssuerValidator`는 이 부품을 그대로 위임한다.

```java
public JwtIssuerValidator(String issuer) {
    Predicate<Object> testClaimValue = v -> v != null && issuer.equals(v.toString());
    this.validator = new JwtClaimValidator<>(JwtClaimNames.ISS, testClaimValue);
}
```

즉 `JwtIssuerValidator`/`JwtAudienceValidator` 등은 `JwtClaimValidator`에 적절한 클레임 이름과 Predicate를 끼운 얇은 래퍼다.

## JwtTimestampValidator — 시각과 시계 오차

가장 자주 작동하는 검증기. `exp`와 `nbf`를 본다.

```java
@Override
public OAuth2TokenValidatorResult validate(Jwt jwt) {
    Instant expiry = jwt.getExpiresAt();
    if (!this.allowEmptyExpiryClaim && ObjectUtils.isEmpty(expiry)) {
        return createOAuth2Error("exp is required");
    }
    if (expiry != null && Instant.now(this.clock).minus(this.clockSkew).isAfter(expiry)) {
        return createOAuth2Error(String.format("Jwt expired at %s", jwt.getExpiresAt()));
    }
    Instant notBefore = jwt.getNotBefore();
    if (!this.allowEmptyNotBeforeClaim && ObjectUtils.isEmpty(notBefore)) {
        return createOAuth2Error("nbf is required");
    }
    if (notBefore != null && Instant.now(this.clock).plus(this.clockSkew).isBefore(notBefore)) {
        return createOAuth2Error(String.format("Jwt used before %s", jwt.getNotBefore()));
    }
    return OAuth2TokenValidatorResult.success();
}
```

- **clock skew**(기본 60초): 발급 서버와 검증 서버의 시계가 다를 수 있으므로, `exp`/`nbf` 비교에 여유를 둔다. "현재 - skew"가 `exp`보다 뒤일 때만 만료로 본다.
- `exp`/`nbf`가 아예 없어도 기본은 통과(`allowEmpty*=true`). 7.0부터 `setAllowEmptyExpiryClaim(false)`로 "exp 필수"를 강제할 수 있다.
- `setClock(Clock)`으로 테스트에서 시간을 고정할 수 있다.

## 타입과 인증서 지문 검증

- `JwtTypeValidator`: `typ` 헤더가 화이트리스트(`validTypes`)에 있는지 본다. `JwtTypeValidator.jwt()`는 "`typ`이 `JWT`이거나 비어 있으면 통과"하는 기본 인스턴스다. `at+jwt` 토큰이라면 `new JwtTypeValidator("at+jwt", "application/at+jwt")`처럼 구성한다.
- `X509CertificateThumbprintValidator`(mTLS, RFC 8705): 토큰의 `cnf.x5t#S256` 확인 클레임이 있으면, 현재 요청의 클라이언트 인증서 SHA-256 지문과 일치하는지 검사한다. 확인 클레임이 없으면 통과(certificate-bound가 아닌 토큰 허용).

## 표준 묶음 — JwtValidators

대부분의 사용자는 단위 검증기를 직접 조립하지 않고 `JwtValidators` 팩토리를 쓴다.

```java
public static OAuth2TokenValidator<Jwt> createDefault() {
    return new DelegatingOAuth2TokenValidator<>(Arrays.asList(
        JwtTypeValidator.jwt(),
        new JwtTimestampValidator(),
        new X509CertificateThumbprintValidator(DEFAULT_X509_CERTIFICATE_SUPPLIER)));
}
```

기본 묶음은 **타입 + 만료 + 인증서지문** 세 가지다(발급자 검증은 issuer를 알아야 하므로 제외). `createDefaultWithIssuer(issuer)`는 여기에 `JwtIssuerValidator`를 더한다.

핵심 메서드 `createDefaultWithValidators(...)`는 "사용자 검증기 + 표준 검증기"를 합치되 **중복을 피한다**.

```java
public static OAuth2TokenValidator<Jwt> createDefaultWithValidators(List<OAuth2TokenValidator<Jwt>> validators) {
    List<OAuth2TokenValidator<Jwt>> tokenValidators = new ArrayList<>(validators);
    if (CollectionUtils.findValueOfType(tokenValidators, X509CertificateThumbprintValidator.class) == null) {
        tokenValidators.add(0, new X509CertificateThumbprintValidator(DEFAULT_X509_CERTIFICATE_SUPPLIER));
    }
    if (CollectionUtils.findValueOfType(tokenValidators, JwtTimestampValidator.class) == null) {
        tokenValidators.add(0, new JwtTimestampValidator());
    }
    if (CollectionUtils.findValueOfType(tokenValidators, JwtTypeValidator.class) == null) {
        tokenValidators.add(0, JwtTypeValidator.jwt());
    }
    return new DelegatingOAuth2TokenValidator<>(tokenValidators);
}
```

사용자가 이미 같은 종류의 검증기를 넣었으면 그것을 존중하고, 없을 때만 표준 검증기를 앞에 끼워 넣는다. "기본은 보장하되 사용자 의도는 덮어쓰지 않는다"는 정책이다.

## RFC 9068 access token 검증 — AtJwtBuilder

`createAtJwtValidator()`는 OAuth2 access token JWT 프로파일(RFC 9068)을 준수하는 검증기를 빌드한다. 생성자에서 필수 클레임 검증기를 미리 등록한다.

```java
private AtJwtBuilder() {
    JwtTimestampValidator timestamps = new JwtTimestampValidator();
    this.validators.put(JoseHeaderNames.TYP, new JwtTypeValidator(List.of("at+jwt", "application/at+jwt")));
    this.validators.put(JwtClaimNames.EXP, require(JwtClaimNames.EXP).and(timestamps));
    this.validators.put(JwtClaimNames.SUB, require(JwtClaimNames.SUB));
    this.validators.put(JwtClaimNames.IAT, require(JwtClaimNames.IAT).and(timestamps));
    this.validators.put(JwtClaimNames.JTI, require(JwtClaimNames.JTI));
    this.validators.put("client_id", require("client_id"));
}
```

`typ`은 `at+jwt`여야 하고, `exp`/`sub`/`iat`/`jti`/`client_id`가 모두 필수다. `issuer(...)`/`audience(...)`는 빌드 전에 반드시 지정해야 하며, `build()`는 필수 항목이 다 채워졌는지 `Assert`로 확인한다. 검증기들을 claim 이름으로 키잉한 `Map`에 담아 두어, `validators((v) -> v.put(...))`로 개별 교체/추가가 쉽다.

## 동작 흐름 — 디코더 안에서

```
 NimbusJwtDecoder.validateJwt(jwt)
        │
        ▼
 jwtValidator.validate(jwt)            ← 보통 DelegatingOAuth2TokenValidator
        │
        ├─▶ JwtTypeValidator        success
        ├─▶ JwtTimestampValidator   success
        ├─▶ JwtIssuerValidator      success
        ├─▶ X509Thumbprint...       success
        └─▶ (사용자 검증기)         ...
        │  결과 합산(하나라도 실패하면 errors 누적)
        ▼
 result.hasErrors() ?
    예 → throw JwtValidationException(설명, errors)
    아니오 → 검증된 Jwt 반환
```

`DelegatingOAuth2TokenValidator`가 모든 단위 검증기를 돌려 오류를 모으고, 하나라도 실패하면 디코더가 `JwtValidationException`을 던진다.

## 설계 포인트 / 확장점

- **조합(Composite) 패턴**: 작은 단일 책임 검증기 + `DelegatingOAuth2TokenValidator`. 새 규칙이 필요하면 `OAuth2TokenValidator<Jwt>`(또는 `JwtClaimValidator`)를 하나 만들어 합치기만 하면 된다.
- **결과 객체 vs 예외**: 단위 검증기는 `OAuth2TokenValidatorResult`로 결과를 반환(여러 오류 누적 가능)하고, 예외화는 디코더가 마지막에 한 번 한다.
- **교체 지점**: 디코더의 `setJwtValidator(...)`가 유일한 진입점. `JwtValidators.createDefaultWithValidators(...)`로 표준+커스텀을 안전하게 합쳐 넣는 것이 표준 패턴이다.
- **clock skew·필수성 토글**: 시계 오차(`JwtTimestampValidator(Duration)`), `exp`/`nbf` 필수 여부 등 운영 정책을 세터로 조정.

## 정리

내용 검증은 `OAuth2TokenValidator<Jwt>`라는 작은 부품과 이를 AND로 묶는 `DelegatingOAuth2TokenValidator`로 구성된다. `JwtClaimValidator`가 범용 토대이고, 그 위에 `JwtTimestampValidator`(만료/시계오차), `JwtIssuerValidator`, `JwtTypeValidator`, `X509CertificateThumbprintValidator` 등이 얹힌다. `JwtValidators`가 표준 묶음과 RFC 9068 빌더를 제공하며, 중복을 피해 사용자 검증기와 표준을 합친다. 디코더는 이 검증기 하나를 `setJwtValidator`로 받아 마지막 단계에서 실행한다.
