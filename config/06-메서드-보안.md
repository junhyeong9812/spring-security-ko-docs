# 06. 메서드 보안

## 무엇을 / 왜

지금까지는 URL/필터 단위의 웹 보안이었다. 하지만 `@PreAuthorize("hasRole('ADMIN')")`처럼 **서비스 메서드 호출 자체**를 보호하고 싶을 때가 있다. `@EnableMethodSecurity`가 그 스위치다. 이 장은 그 애너테이션이 어떻게 AOP 어드바이저(Advisor)를 등록해, 보안 애너테이션이 붙은 빈을 프록시로 감싸고 호출을 가로채는지를 본다.

웹 보안이 "서블릿 필터 체인"이라면, 메서드 보안은 "스프링 AOP 인터셉터 체인"이다. 둘 다 `config` 모듈이 빌더/셀렉터로 조립하지만 산출물이 다르다.

## 핵심 타입

```
   @EnableMethodSecurity(prePostEnabled=, securedEnabled=, jsr250Enabled=, mode=)
        │  @Import
        ▼
   MethodSecuritySelector (ImportSelector)
        │  속성에 따라 동적으로 @Configuration 선택
        ├─ prePostEnabled → PrePostMethodSecurityConfiguration
        ├─ securedEnabled → SecuredMethodSecurityConfiguration
        ├─ jsr250Enabled  → Jsr250MethodSecurityConfiguration
        ├─ (항상) AuthorizationProxyConfiguration
        └─ AutoProxyRegistrar (AOP 인프라 켜기)

   PrePostMethodSecurityConfiguration 가 등록하는 빈들:
        preFilter / preAuthorize / postAuthorize / postFilter
            각각  MethodInterceptor(=어드바이저) , 고유 Pointcut 보유
```

- 소스: `config/src/main/java/org/springframework/security/config/annotation/method/configuration/EnableMethodSecurity.java`
- 소스: `.../method/configuration/MethodSecuritySelector.java`
- 소스: `.../method/configuration/PrePostMethodSecurityConfiguration.java`
- 소스: `.../method/configuration/DeferringMethodInterceptor.java`

### @EnableMethodSecurity — 무엇을 켤지 고르는 플래그

```java
@Retention(RUNTIME) @Target(TYPE) @Documented
@Import(MethodSecuritySelector.class)
public @interface EnableMethodSecurity {
    boolean prePostEnabled() default true;   // @PreAuthorize/@PostAuthorize/@PreFilter/@PostFilter
    boolean securedEnabled() default false;  // @Secured
    boolean jsr250Enabled() default false;   // @RolesAllowed (JSR-250)
    boolean proxyTargetClass() default false;
    AdviceMode mode() default AdviceMode.PROXY;
    int offset() default 0;                  // 인터셉터 순서 오프셋
}
```

기본은 `prePostEnabled=true`만 켜진다. `@Secured`나 JSR-250을 쓰려면 명시적으로 켜야 한다. 이 플래그들이 `MethodSecuritySelector`가 어떤 설정 클래스를 import할지를 결정한다.

### MethodSecuritySelector — 조건부 설정 선택

```java
public String[] selectImports(AnnotationMetadata importMetadata) {
    ...
    List<String> imports = new ArrayList<>(Arrays.asList(this.autoProxy.selectImports(importMetadata)));
    if (annotation.prePostEnabled()) imports.add(PrePostMethodSecurityConfiguration.class.getName());
    if (annotation.securedEnabled()) imports.add(SecuredMethodSecurityConfiguration.class.getName());
    if (annotation.jsr250Enabled())  imports.add(Jsr250MethodSecurityConfiguration.class.getName());
    imports.add(AuthorizationProxyConfiguration.class.getName());
    ...
    return imports.toArray(new String[0]);
}
```

`ImportSelector`는 "런타임 속성에 따라 등록할 빈 구성을 동적으로 고르는" 스프링 메커니즘이다. 켜지 않은 기능의 설정 클래스는 아예 로드되지 않아, 불필요한 인터셉터가 빈으로 뜨지 않는다. `autoProxy.selectImports`는 `mode`(PROXY vs ASPECTJ)에 따라 AOP 인프라를 켠다.

## 동작 흐름: @PreAuthorize 가 호출을 막기까지

```
[기동]
  PrePostMethodSecurityConfiguration 가 4개의 MethodInterceptor 빈 등록
    preFilterAuthorizationMethodInterceptor      (Pointcut: @PreFilter)
    preAuthorizeAuthorizationMethodInterceptor   (Pointcut: @PreAuthorize)
    postAuthorizeAuthorizationMethodInterceptor  (Pointcut: @PostAuthorize)
    postFilterAuthorizationMethodInterceptor     (Pointcut: @PostFilter)
    │  각 인터셉터는 (Pointcut + AuthorizationManager) 를 들고 있음
    ▼
  AutoProxyRegistrar 가 켠 AOP 인프라가
  Pointcut 에 매칭되는 빈(=보안 애너테이션 붙은 메서드를 가진 빈)을 프록시로 래핑

[호출 시점]
  client → adminService.deleteUser(id)   (프록시)
    │
    ▼
  preAuthorizeAuthorizationMethodInterceptor.invoke(mi)
    │  PreAuthorizeAuthorizationManager.check(authentication, methodInvocation)
    │    @PreAuthorize("hasRole('ADMIN')") 표현식 평가
    │    ├ 허용 → mi.proceed() (실제 메서드 실행)
    │    └ 거부 → AccessDeniedException
    ▼
  (메서드 반환 후) postAuthorize / postFilter 인터셉터가 결과를 검사/필터
```

### 인터셉터 빈은 왜 DeferringMethodInterceptor 인가

```java
@Bean
@Role(BeanDefinition.ROLE_INFRASTRUCTURE)
static MethodInterceptor preAuthorizeAuthorizationMethodInterceptor(
        ObjectProvider<PrePostMethodSecurityConfiguration> _config) {
    return new DeferringMethodInterceptor<>(preAuthorizePointcut,
            () -> _config.getObject().preAuthorizeMethodInterceptor);
}
```

인터셉터 빈은 `static @Bean` 메서드로 만들어지고, 실제 인터셉터(`preAuthorizeMethodInterceptor`)는 `ObjectProvider`로 **지연 해석**된다. 이는 보안 인터셉터가 일반 빈보다 먼저 등록되어야 하는(다른 빈을 프록시하기 위해) AOP 인프라 빈인데, 그 인터셉터의 협력자(`MethodSecurityExpressionHandler`, `AuthorizationEventPublisher` 등)는 나중에 `@Autowired(required=false)`로 주입되기 때문이다. `DeferringMethodInterceptor`가 "껍데기 먼저 등록, 알맹이는 첫 호출 때 해석"으로 이 순서 문제를 푼다. `@Role(ROLE_INFRASTRUCTURE)`는 이 빈들을 일반 후처리 대상에서 제외해 조기 초기화 부작용을 막는다.

### 표현식 핸들러와 커스터마이징 주입

`PrePostMethodSecurityConfiguration`은 협력자들을 `@Autowired(required = false)`로 받아 네 인터셉터/매니저에 일괄 주입한다.

```java
@Autowired(required = false)
void setExpressionHandler(MethodSecurityExpressionHandler expressionHandler) {
    this.preFilterMethodInterceptor.setExpressionHandler(expressionHandler);
    this.preAuthorizeAuthorizationManager.setExpressionHandler(expressionHandler);
    this.postAuthorizeAuthorizationManager.setExpressionHandler(expressionHandler);
    this.postFilterMethodInterceptor.setExpressionHandler(expressionHandler);
}
```

즉 `MethodSecurityExpressionHandler`나 `AuthorizationEventPublisher` 빈을 정의하기만 하면 네 인터셉터 전부에 자동 반영된다. `setImportMetadata`는 `@EnableMethodSecurity(offset = ...)`을 읽어 인터셉터들의 실행 순서를 보정한다(여러 어드바이스가 같은 조인포인트에 걸릴 때).

## 핵심 메서드

- **`MethodSecuritySelector.selectImports(...)`** — 애너테이션 플래그로 등록할 설정 클래스를 결정. 메서드 보안 구성의 진입 분기.
- **각 `XxxAuthorizationMethodInterceptor`의 `Pointcut`** — 어떤 메서드(어떤 애너테이션)를 가로챌지 정의. `AuthorizationManagerBeforeMethodInterceptor.preAuthorize()` 등이 표준 포인트컷을 제공.
- **`PreAuthorizeAuthorizationManager.check(...)`** — 실제 표현식 평가. 인가 결정 로직은 `core`의 책임([../core/README.md](../core/README.md)).

## 설계 포인트 / 확장점

- **셀렉터 기반 조건부 구성**: 켠 기능만 인터셉터로 뜬다. 웹 보안의 `ImportSelector`(02장)와 같은 패턴.
- **AOP 어드바이저 = (Pointcut + AuthorizationManager)**: 인가 규칙이 `AuthorizationManager`로 통일되어, 웹(`AuthorizationFilter`)과 메서드(`*MethodInterceptor`)가 같은 추상화를 공유한다.
- **지연 인터셉터(Deferring)**: 인프라 빈 등록 순서와 협력자 주입 순서의 충돌을 우아하게 해결.
- **확장점**: `MethodSecurityExpressionHandler`/`AuthorizationManager`/`RoleHierarchy` 빈으로 표현식·역할 계층을 커스터마이즈. `mode = AdviceMode.ASPECTJ`로 프록시 대신 위빙 선택. 리액티브 환경은 `@EnableReactiveMethodSecurity`(별도 `ReactiveMethodSecuritySelector`/`ReactiveMethodSecurityConfiguration`).

## 정리

`@EnableMethodSecurity`는 `MethodSecuritySelector`를 통해, 켜진 기능(prePost/secured/jsr250)에 해당하는 설정 클래스만 import한다. `PrePostMethodSecurityConfiguration`은 `@PreAuthorize` 등 각 애너테이션에 대응하는 `MethodInterceptor`(어드바이저)를 등록하고, AOP 인프라가 그 포인트컷에 걸리는 빈을 프록시로 감싼다. 런타임에는 인터셉터가 메서드 호출 전후로 `AuthorizationManager`를 돌려 허용/거부를 결정한다. 마지막 장은 같은 의미를 다른 표면(Kotlin DSL, XML)으로 표현하는 방법을 본다.
