# 05. 인증 구성

## 무엇을 / 왜

앞 장들에서 로그인 필터는 `http.getSharedObject(AuthenticationManager.class)`로 인증 매니저를 받아 갔다. 그 `AuthenticationManager`는 누가, 언제 만드는가? 이 장은 `config` 모듈의 **인증(authentication) 측 빌더와 설정 클래스**를 본다. 핵심은 두 개의 인증 매니저가 있다는 사실이다.

- **전역 `AuthenticationManager`** — `AuthenticationConfiguration`이 만들고, `UserDetailsService`/`PasswordEncoder` 빈이나 `AuthenticationProvider` 빈을 자동으로 엮는다. 모든 `HttpSecurity`의 부모로 쓰인다.
- **체인 로컬 `AuthenticationManager`** — 각 `HttpSecurity`가 자기 `AuthenticationManagerBuilder`로 만드는 것. 부모로 전역 매니저를 둔다(`ProviderManager`의 parent).

## 핵심 타입

```
   @EnableGlobalAuthentication  ──@Import──▶ AuthenticationConfiguration
                                                  │ getAuthenticationManager()
                                                  ▼
                                       AuthenticationManagerBuilder (전역)
                                                  │ performBuild()
                                                  ▼
                                            ProviderManager (전역)
                                                  ▲ parent
                                                  │
   HttpSecurity 의 AuthenticationManagerBuilder (체인 로컬)
                                                  │ performBuild()
                                                  ▼
                                       ProviderManager (체인 로컬)


   GlobalAuthenticationConfigurerAdapter           (전역 빌더용 Configurer)
        ├── InitializeUserDetailsBeanManagerConfigurer       (UserDetailsService 빈 → DaoProvider)
        ├── InitializeAuthenticationProviderBeanManagerConfigurer (AuthenticationProvider 빈 등록)
        └── EnableGlobalAuthenticationAutowiredConfigurer    (@EnableGlobalAuthentication 빈 eager 초기화)
```

- 소스: `config/src/main/java/org/springframework/security/config/annotation/authentication/configuration/AuthenticationConfiguration.java`
- 소스: `.../authentication/builders/AuthenticationManagerBuilder.java`
- 소스: `.../authentication/configuration/InitializeUserDetailsBeanManagerConfigurer.java`
- 소스: `.../authentication/configuration/GlobalAuthenticationConfigurerAdapter.java`

### AuthenticationManagerBuilder

`AuthenticationManagerBuilder`는 [01장](01-빌더와-Configurer-기반구조.md)의 `AbstractConfiguredSecurityBuilder`를 상속해 `ProviderManager`를 만든다.

```java
public class AuthenticationManagerBuilder
        extends AbstractConfiguredSecurityBuilder<AuthenticationManager, AuthenticationManagerBuilder>
        implements ProviderManagerBuilder<AuthenticationManagerBuilder> {

    private AuthenticationManager parentAuthenticationManager;
    private List<AuthenticationProvider> authenticationProviders = new ArrayList<>();

    public AuthenticationManagerBuilder authenticationProvider(AuthenticationProvider p) {
        this.authenticationProviders.add(p);
        return this;
    }

    @Override
    protected ProviderManager performBuild() {
        if (!isConfigured()) {   // provider도 parent도 없으면
            this.logger.debug("No authenticationProviders and no parentAuthenticationManager defined. Returning null.");
            return null;
        }
        ProviderManager providerManager = new ProviderManager(this.authenticationProviders, this.parentAuthenticationManager);
        ...
        return providerManager;
    }
}
```

`inMemoryAuthentication()`/`jdbcAuthentication()`/`userDetailsService(...)`는 각각 그에 맞는 Configurer(`InMemoryUserDetailsManagerConfigurer`, `DaoAuthenticationConfigurer` 등)를 `apply`한다. 결과적으로 빌드 시 `DaoAuthenticationProvider` 같은 provider가 `authenticationProviders`에 채워지고, `performBuild()`가 그것들과 parent로 `ProviderManager`를 만든다.

### 전역 AuthenticationManager 의 lazy 조립

`AuthenticationConfiguration.getAuthenticationManager()`가 전역 매니저를 조립한다. 재귀 방지 로직이 핵심이다.

```java
public AuthenticationManager getAuthenticationManager() {
    if (this.authenticationManagerInitialized) return this.authenticationManager;
    AuthenticationManagerBuilder authBuilder = this.applicationContext.getBean(AuthenticationManagerBuilder.class);
    if (this.buildingAuthenticationManager.getAndSet(true))
        return new AuthenticationManagerDelegator(authBuilder);    // 이미 빌드 중 → 지연 위임자
    for (GlobalAuthenticationConfigurerAdapter config : this.globalAuthConfigurers)
        authBuilder.apply(config);                                  // 전역 Configurer 적용
    this.authenticationManager = authBuilder.build();
    if (this.authenticationManager == null)
        this.authenticationManager = getAuthenticationManagerBean();  // 사용자 정의 빈 fallback
    this.authenticationManagerInitialized = true;
    return this.authenticationManager;
}
```

- **`globalAuthConfigurers`** 는 `@Autowired`로 주입된 `GlobalAuthenticationConfigurerAdapter` 빈 목록이다. 여기에 `InitializeUserDetailsBeanManagerConfigurer`와 `InitializeAuthenticationProviderBeanManagerConfigurer`가 들어 있다(둘 다 `AuthenticationConfiguration`이 `@Bean`으로 노출).
- **`AuthenticationManagerDelegator`** — 인증 매니저를 빌드하는 도중에 다시 `getAuthenticationManager()`가 불리는 무한 재귀를 막는 지연 위임자. 첫 `authenticate()` 호출 때 실제 매니저를 한 번만 해석한다.

## 동작 흐름: UserDetailsService 빈 하나로 로그인이 되는 이유

가장 흔한 시나리오 — 사용자가 `UserDetailsService` 빈(또는 `InMemoryUserDetailsManager`)만 정의한 경우, 어떻게 그게 로그인에 연결되는가?

```
[기동]
  AuthenticationConfiguration.getAuthenticationManager()
    │
    ├─ globalAuthConfigurers 적용
    │    InitializeUserDetailsBeanManagerConfigurer.init(authBuilder)
    │      └ configure 단계에서 컨텍스트의 UserDetailsService 빈을 찾아
    │        DaoAuthenticationProvider 생성 (+ PasswordEncoder 빈, UserDetailsPasswordService 빈)
    │        authBuilder.authenticationProvider(daoProvider)
    │
    └─ authBuilder.build() → ProviderManager(providers=[DaoAuthenticationProvider])  = 전역 매니저

[HttpSecurity 빌드 — 02장]
  HttpSecurityConfiguration.httpSecurity()
    authenticationBuilder.parentAuthenticationManager( 전역 매니저 )

[HttpSecurity.build() — 03장 beforeConfigure]
  체인 로컬 AuthenticationManagerBuilder.build()
    → 로컬 provider 없으면 parent(전역)만 가진 ProviderManager
    → 공유객체 AuthenticationManager 로 등록

[로그인 요청 — 04장 configure 가 끼운 필터]
  UsernamePasswordAuthenticationFilter.setAuthenticationManager(공유 매니저)
    │ authenticate(token)
    ▼
  ProviderManager(로컬) → parent ProviderManager(전역) → DaoAuthenticationProvider
    → UserDetailsService.loadUserByUsername(...) → PasswordEncoder.matches(...)
```

`InitializeUserDetailsBeanManagerConfigurer`의 내부 `InitializeUserDetailsManagerConfigurer.configure()`가 그 연결을 만든다. 컨텍스트에 `UserDetailsService` 빈이 정확히 하나 있고 로컬 인증이 비어 있을 때만, `DaoAuthenticationProvider`를 구성해 등록한다(여러 개거나 사용자가 직접 provider를 줬으면 건드리지 않는다). 즉 **"빈 하나 던지면 알아서 엮인다"는 마법의 정체는 전역 Configurer가 빈을 스캔해 provider로 변환**하는 것이다.

마찬가지로 `InitializeAuthenticationProviderBeanManagerConfigurer`는 컨텍스트의 `AuthenticationProvider` 빈을 전역 빌더에 자동 등록한다.

## 핵심 메서드

- **`AuthenticationConfiguration.getAuthenticationManager()`** — 전역 매니저의 단일 조립 지점. lazy + 재귀 방지.
- **`AuthenticationManagerBuilder.parentAuthenticationManager(am)`** — 부모 설정. `ProviderManager`인 경우 `eraseCredentials` 설정을 부모에서 물려받는다.
- **`AuthenticationManagerBuilder.performBuild()`** — provider/parent가 하나도 없으면 `null` 반환(설정이 비었다는 신호). 그래서 `HttpSecurity.beforeConfigure()`는 `null`을 받으면 공유 매니저를 세팅하지 않는다.
- **`HttpSecurity.userDetailsService(...)` / `authenticationProvider(...)`** — 체인 로컬 빌더에 위임([03장](03-HttpSecurity-와-WebSecurity.md)).

## 설계 포인트 / 확장점

- **2단 계층(부모-자식) 인증 매니저**: 전역에서 공통 인증을, 체인별로 특수 인증을 더할 수 있다. `ProviderManager`의 parent 위임이 이를 구현.
- **빈 → provider 자동 변환**: `GlobalAuthenticationConfigurerAdapter` 계열이 컨텍스트를 스캔해 `UserDetailsService`/`AuthenticationProvider` 빈을 인증 체인에 끼운다. 설정 코드 없이 빈만으로 동작하는 비결.
- **lazy + Delegator**: 인증 매니저가 자기 자신을 만드는 과정에서의 순환을 안전하게 끊는다.
- **확장점**: `AuthenticationProvider` 빈을 정의하면 자동 참여. `GlobalAuthenticationConfigurerAdapter`를 직접 구현해 전역 인증 빌더를 후킹할 수도 있다. 체인별로 `http.authenticationProvider(...)`로 로컬 provider 추가.

## 정리

인증 구성은 "전역 매니저 하나 + 체인 로컬 매니저들"의 부모-자식 구조다. `AuthenticationConfiguration`이 전역 `AuthenticationManagerBuilder`에 `GlobalAuthenticationConfigurerAdapter`들을 적용해 컨텍스트의 `UserDetailsService`/`AuthenticationProvider` 빈을 provider로 엮고, `ProviderManager`를 만든다. 각 `HttpSecurity`는 그것을 부모로 둔 로컬 `ProviderManager`를 만들어 공유 객체로 노출하고, 로그인 필터가 그걸 받아 인증한다. 다음 장은 웹이 아닌 **메서드 레벨** 보안 구성을 본다.
