# 05. 데이터 암호화 · 키 생성 · 인코딩

## 무엇을 / 왜

비밀번호는 단방향 해시로 충분하다 — 다시 원문으로 돌릴 필요가 없으니까. 하지만 OAuth 토큰, 설정 파일의 DB 비밀번호, 리멤버-미 쿠키처럼 **나중에 복호화해야 하는** 데이터도 있다. 이런 양방향 암호화를 담당하는 게 `encrypt` 패키지다. 그리고 암호화에 필요한 솔트·IV·세션 키 같은 무작위 바이트는 `keygen`이, 바이트와 문자열을 안전하게 오가는 변환은 `codec`/`util`이 책임진다. 이 챕터는 이 세 보조 패키지를 묶어 본다.

## 핵심 타입

```
   encrypt 패키지
   ┌──────────────────────────────────────────────────────────┐
   │ «interface» BytesEncryptor   encrypt(byte[]) / decrypt(byte[]) │
   │ «interface» TextEncryptor    encrypt(String) / decrypt(String) │
   │                                                              │
   │ Encryptors (팩토리)                                          │
   │   standard / stronger → BytesEncryptor (AES-CBC / AES-GCM)   │
   │   text / delux        → TextEncryptor  (Hex 인코딩 래퍼)      │
   │   noOpText            → 무암호화 (테스트용)                    │
   │                                                              │
   │ 구현: AesBytesEncryptor, BouncyCastleAes(Cbc|Gcm)…,          │
   │       Rsa(Raw|Secret)Encryptor(비대칭), KeyStoreKeyFactory   │
   └──────────────────────────────────────────────────────────┘

   keygen 패키지                       codec / util 패키지
   ┌────────────────────────────┐     ┌──────────────────────────┐
   │ «interface» BytesKeyGenerator│     │ Hex   : byte[] ↔ 16진문자 │
   │ «interface» StringKeyGenerator│    │ Utf8  : byte[] ↔ UTF-8    │
   │ KeyGenerators (팩토리)        │     │ EncodingUtils:           │
   │   secureRandom / shared / string│   │   concatenate / subArray │
   └────────────────────────────┘     └──────────────────────────┘
```

소스: `crypto/src/main/java/org/springframework/security/crypto/{encrypt,keygen,codec,util}/`

## 동작 흐름 1 — Encryptors 팩토리와 AES

`Encryptors`는 직접 생성자를 노출하지 않고 의미 있는 이름의 정적 메서드만 제공한다.

```java
public static BytesEncryptor stronger(CharSequence password, CharSequence salt) {
    return new AesBytesEncryptor(password.toString(), salt,
            KeyGenerators.secureRandom(16), CipherAlgorithm.GCM);
}
public static BytesEncryptor standard(CharSequence password, CharSequence salt) {
    return new AesBytesEncryptor(password.toString(), salt, KeyGenerators.secureRandom(16));
}                                                          // 기본 CipherAlgorithm.CBC
```

- `stronger`: **AES-256-GCM**. GCM은 인증 암호화(authenticated encryption)라 변조를 탐지한다 — JavaDoc도 이쪽을 권장한다.
- `standard`: **AES-256-CBC**. 인증이 없어 데이터 무결성을 보장하지 않는다고 명시돼 있다.
- `delux`/`text`: 각각 `stronger`/`standard`를 `HexEncodingTextEncryptor`로 감싸 `String`을 다루게 한 `TextEncryptor`.
- `noOpText`: 평문을 그대로 반환하는 테스트용.

`AesBytesEncryptor`는 password와 hex 솔트로 **PBKDF2(`PBKDF2WithHmacSHA1`, 1024회)로 256비트 AES 키를 유도**한다.

```java
this(CipherUtils.newSecretKey("PBKDF2WithHmacSHA1",
        new PBEKeySpec(password.toCharArray(), Hex.decode(salt), 1024, 256)), ivGenerator, alg);
```

암호화는 매번 새 IV를 만들어 결과 앞에 붙인다 — 같은 평문이라도 매번 다른 암호문이 나오게 하는 핵심이다.

```java
public byte[] encrypt(byte[] bytes) {
    synchronized (this.encryptor) {
        byte[] iv = this.ivGenerator.generateKey();           // 16바이트 랜덤 IV
        CipherUtils.initCipher(this.encryptor, ENCRYPT_MODE, this.secretKey, alg.getParameterSpec(iv));
        byte[] encrypted = CipherUtils.doFinal(this.encryptor, bytes);
        return (ivGenerator != NULL_IV_GENERATOR)
                ? EncodingUtils.concatenate(iv, encrypted)    // [IV][암호문]
                : encrypted;
    }
}
```

```
encrypt(plain)
   │  IV = random(16)            ← 메시지마다 유일 → 같은 평문도 다른 암호문
   │  cipher.init(ENCRYPT, key, spec(IV))   spec: GCM→GCMParameterSpec(128), CBC→IvParameterSpec
   │  ct = cipher.doFinal(plain)
   └─ return IV || ct           decrypt 는 앞에서 IV 를 떼어 역순 수행
```

`CipherAlgorithm` enum이 CBC와 GCM의 차이(`AES/CBC/PKCS5Padding` vs `AES/GCM/NoPadding`, IvParameterSpec vs GCMParameterSpec, 기본 IV 생성기)를 캡슐화한다. `Cipher`는 스레드 안전하지 않으므로 `encryptor`/`decryptor` 인스턴스에 `synchronized`를 건다.

BouncyCastle 변형(`BouncyCastleAesGcmBytesEncryptor` 등)과 비대칭 `RsaSecretEncryptor`/`RsaRawEncryptor`(공개키 암호화/개인키 복호화), 키스토어에서 키를 읽는 `KeyStoreKeyFactory`도 같은 `BytesEncryptor`/`TextEncryptor` 계약을 구현해 서로 갈아끼울 수 있다.

## 동작 흐름 2 — KeyGenerators

`keygen` 패키지는 "무작위 바이트 N개"라는 단순하지만 어디서나 필요한 일을 담당한다. 두 인터페이스는 각각 `byte[]`와 `String`을 만든다.

```java
public static BytesKeyGenerator secureRandom(int keyLength) {  // SecureRandom 기반
    return new SecureRandomBytesKeyGenerator(keyLength);
}
public static BytesKeyGenerator shared(int keyLength) {        // 한 번 생성해 고정 반환
    return new SharedKeyGenerator(secureRandom(keyLength).generateKey());
}
public static StringKeyGenerator string() {                   // 8바이트 → hex 문자열
    return new HexEncodingStringKeyGenerator(secureRandom());
}
```

`secureRandom()`은 호출마다 새 키(솔트·IV용), `shared(n)`은 생성 시 한 번 만든 키를 매번 똑같이 반환(세션 전역 키용)한다는 의미 차이가 중요하다. `SecureRandomBytesKeyGenerator`는 플랫폼 기본 `SecureRandom`을 그대로 쓴다(특정 provider 지정 없음). `Base64StringKeyGenerator`는 기본 32바이트 키를 Base64 문자열로 만들어 주는데, OAuth2의 `state`/`nonce` 같은 토큰 생성에 쓰인다.

이 패키지가 [01~02장](02-단방향-해시-인코더.md)의 인코더들에 솔트를 공급한다는 점을 기억하자 — `AbstractPasswordEncoder`, `Argon2PasswordEncoder`, `Pbkdf2PasswordEncoder` 모두 내부에 `KeyGenerators.secureRandom(...)`을 들고 있다.

## 동작 흐름 3 — codec / util (바이트 ↔ 문자열)

해시·암호문은 모두 `byte[]`인데, DB 컬럼이나 쿠키에는 문자열로 담아야 한다. `codec` 패키지가 이 변환을 책임진다.

```java
// Hex: 바이트 한 개 → 16진 두 글자
public static char[] encode(byte[] bytes) {
    char[] result = new char[2 * bytes.length];
    for (byte aByte : bytes) {
        result[j++] = HEX[(0xF0 & aByte) >>> 4];   // 상위 4비트
        result[j++] = HEX[(0x0F & aByte)];          // 하위 4비트
    }
    return result;
}
```

`Hex.decode`는 역변환하며 길이가 홀수이거나 16진이 아닌 문자가 섞이면 `IllegalArgumentException`을 던진다. `Utf8`은 `String`↔UTF-8 바이트 변환을, `util.EncodingUtils`는 `concatenate(arrays...)`(여러 바이트 배열 이어붙이기)와 `subArray(array, begin, end)`(부분 추출)를 제공한다. 이 두 유틸이 "솔트 + 해시"를 합치고, 검증 시 다시 솔트만 떼어 내는 모든 인코더의 공통 일꾼이다. JavaDoc이 `Hex`를 "내부용(For internal use only)"이라 밝히듯, 이 패키지들은 상위 인코더/암호기를 떠받치는 저수준 도구다.

## 설계 포인트 / 확장점

- **의도가 드러나는 팩토리**: `Encryptors.stronger`/`standard`, `KeyGenerators.shared`/`secureRandom`처럼 생성자 대신 의미 있는 이름의 정적 메서드로 "무엇을 위한 것인지"를 코드에 새긴다. 잘못된 모드 선택을 줄인다.
- **IV는 메시지마다, 솔트는 키 유도용**: AES 암호화는 매 호출 새 IV를 붙여 동일 평문의 암호문이 달라지게 하고(의미 보안), PBKDF2 키 유도에는 별도 솔트를 쓴다. 두 역할을 혼동하지 않는 게 핵심.
- **인증 암호화 권장**: `standard`(CBC)는 변조 탐지가 없다고 명시하고 `stronger`(GCM)를 권한다. 새 코드는 GCM을 골라야 한다.
- **계약 기반 교체성**: AES·BouncyCastle·RSA 구현이 모두 `BytesEncryptor`/`TextEncryptor`를 구현하므로, 키스토어 기반이나 비대칭 방식으로 바꿔도 호출부는 변하지 않는다.

## 정리

`encrypt` 패키지는 복호화가 필요한 데이터를 위한 양방향 암호화를 `BytesEncryptor`/`TextEncryptor` 계약으로 제공하고, `Encryptors` 팩토리가 AES-CBC(`standard`)·AES-GCM(`stronger`)을 의도가 드러나는 이름으로 노출한다. `AesBytesEncryptor`는 PBKDF2로 키를 유도하고 메시지마다 새 IV를 붙인다. `keygen`은 이 모든 곳(과 비밀번호 인코더들)에 무작위 솔트·IV·키를 공급하고, `codec`/`util`은 바이트와 문자열을 안전하게 오가게 하는 저수준 일꾼이다. 이 세 보조 패키지가 모여 crypto 모듈 전체의 토대를 이룬다.
