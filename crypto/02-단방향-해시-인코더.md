# 02. 단방향 해시 인코더 (BCrypt · Argon2 · SCrypt · PBKDF2)

## 무엇을 / 왜

비밀번호 저장에 일반 해시(MD5, SHA-256)를 쓰면 안 되는 이유는 그것들이 **너무 빠르기** 때문이다. 빠르다는 건 공격자가 초당 수십억 개의 후보를 시도할 수 있다는 뜻이다. 그래서 비밀번호 전용 해시는 일부러 느리게, 그리고 하드웨어가 발전하면 더 느리게 조절할 수 있게(=적응형) 설계된다. 이 챕터는 Spring Security가 권장하는 네 가지 적응형 단방향 함수 — BCrypt, Argon2, SCrypt, PBKDF2 — 의 구현을 본다.

## 핵심 타입

```
   AbstractValidatingPasswordEncoder
        │
        ├── BCryptPasswordEncoder    ──▶ bcrypt.BCrypt (자체 구현)    | strength(log rounds)
        ├── Argon2PasswordEncoder    ──▶ BouncyCastle Argon2BytesGenerator | memory/iterations/parallelism
        ├── SCryptPasswordEncoder    ──▶ BouncyCastle SCrypt          | cpuCost/memoryCost/parallelism
        ├── Pbkdf2PasswordEncoder    ──▶ JCA SecretKeyFactory(PBKDF2) | iterations/algorithm
        │
        └── (레거시, @Deprecated)
            ├── MessageDigestPasswordEncoder  (MD5/SHA, {salt}digest)
            ├── StandardPasswordEncoder        (SHA-256)
            ├── Md4PasswordEncoder / LdapShaPasswordEncoder
            └── NoOpPasswordEncoder            (평문 그대로)
```

저장 출력 형식이 알고리즘마다 다르다는 점이 핵심이다.

```
BCrypt : $2a$10$dXJ3SW6G7P50lGmMkkmwe.20cQQubK3.HZWzG3YB1tlRy.fqvM/BG
         └┬┘└┬┘└──────────────┬──────────────────────────────────┘
         버전 strength      salt(22자) + hash, 모두 한 문자열에 포함
Argon2 : $argon2id$v=19$m=16384,t=2,p=1$<salt-b64>$<hash-b64>   (파라미터가 문자열에 박힘)
SCrypt : $e0801$<salt-b64>$<hash-b64>                            (N,r,p가 16진 헤더에)
PBKDF2 : <salt + hash> 를 통째로 Hex 또는 Base64 (파라미터는 외부 설정에 의존)
```

BCrypt·Argon2·SCrypt는 **자기 파라미터를 해시 문자열 안에 담는다.** 그래서 검증할 때 그 값을 읽어 같은 비용으로 재해싱할 수 있고, `upgradeEncoding`도 가능하다. PBKDF2는 파라미터를 담지 않아 인코더 설정에 의존한다.

## 동작 흐름 1 — BCrypt 인코딩과 검증

`BCryptPasswordEncoder`는 strength(기본 10, log rounds = 2^10회 키 확장)와 버전($2a/$2b/$2y)을 받는다. 인코딩은 자체 구현한 `BCrypt` 클래스에 위임한다.

```java
protected String encodeNonNullPassword(String rawPassword) {
    String salt = getSalt();                          // BCrypt.gensalt(version, strength, random)
    return BCrypt.hashpw(rawPassword.toString(), salt);
}
```

`getSalt()`는 `SecureRandom`으로 솔트를 만들어 `$2a$10$...` 형태의 솔트 문자열을 구성하고, `hashpw`가 그 솔트로 Blowfish 기반 키 확장을 strength번 돌려 최종 해시를 만든다. 솔트가 결과에 그대로 들어가므로 별도 저장이 필요 없다.

검증은 형식 먼저 확인하고 알고리즘에 위임한다.

```java
protected boolean matchesNonNull(String rawPassword, String encodedPassword) {
    if (!this.BCRYPT_PATTERN.matcher(encodedPassword).matches()) {
        this.logger.warn("Encoded password does not look like BCrypt");
        return false;                                 // 형식 불일치는 조용히 false
    }
    return BCrypt.checkpw(rawPassword.toString(), encodedPassword);
}
```

```
matches("secret", "$2a$10$dXJ3…")
   │
   ├─ 정규식 \A\$2(a|y|b)?\$(\d\d)\$[./0-9A-Za-z]{53} 매칭?  ── 아니오 ─▶ false
   └─ 예
        └─ BCrypt.checkpw: 저장값에서 salt·strength 추출
              → 같은 salt/strength 로 "secret" 재해싱 → 상수시간 비교
```

### BCrypt의 upgradeEncoding

```java
protected boolean upgradeEncodingNonNull(String encodedPassword) {
    Matcher matcher = this.BCRYPT_PATTERN.matcher(encodedPassword);
    ...
    int strength = Integer.parseInt(matcher.group(2));   // 저장된 log rounds
    return strength < this.strength;                      // 현재 설정보다 약하면 true
}
```

저장 해시에 박힌 strength를 읽어 현재 인코더 설정과 비교한다. 예전에 strength 8로 저장된 비밀번호를 strength 12 인코더로 검증하면 `true` → 업그레이드 대상이 된다.

## 동작 흐름 2 — Argon2 (메모리-하드 함수)

Argon2는 2015 패스워드 해싱 경연(PHC) 우승작으로, CPU뿐 아니라 **메모리 사용량**까지 비용으로 거는 점이 BCrypt와 다르다. `Argon2PasswordEncoder`는 BouncyCastle의 `Argon2BytesGenerator`에 위임한다.

```java
Argon2Parameters params = new Argon2Parameters.Builder(Argon2Parameters.ARGON2_id)
        .withSalt(salt).withParallelism(this.parallelism)
        .withMemoryAsKB(this.memory).withIterations(this.iterations).build();
Argon2BytesGenerator generator = new Argon2BytesGenerator();
generator.init(params);
generator.generateBytes(rawPassword.toString().toCharArray(), hash);
return Argon2EncodingUtils.encode(hash, params);   // $argon2id$... 문자열로 직렬화
```

기본값(`defaultsForSpringSecurity_v5_8`)은 솔트 16바이트, 해시 32바이트, parallelism 1, memory `1<<14`(16MB), iterations 2다. 검증은 저장 문자열을 `Argon2EncodingUtils.decode`로 파싱해 거기 담긴 파라미터로 재해싱한 뒤 직접 작성한 상수시간 루프로 비교한다.

```java
private static boolean constantTimeArrayEquals(byte[] expected, byte[] actual) {
    if (expected.length != actual.length) return false;
    int result = 0;
    for (int i = 0; i < expected.length; i++) result |= expected[i] ^ actual[i];
    return result == 0;     // XOR 누적: 어디서 틀렸는지 시간에 새어나가지 않음
}
```

`upgradeEncoding`은 저장된 `memory`나 `iterations`가 현재 설정보다 작으면 `true`를 반환한다.

## 동작 흐름 3 — PBKDF2 (JCA 표준)

`Pbkdf2PasswordEncoder`는 JDK 표준 `SecretKeyFactory`만으로 동작해 추가 의존이 필요 없다 — FIPS 환경에서 선호된다. 솔트와 (선택적) secret을 평문에 붙여 PBKDF2를 `iterations`번 돌린다.

```java
PBEKeySpec spec = new PBEKeySpec(rawPassword.toCharArray(),
        EncodingUtils.concatenate(salt, this.secret), this.iterations, this.hashWidth);
SecretKeyFactory skf = SecretKeyFactory.getInstance(this.algorithm);  // PBKDF2WithHmacSHA256 등
return EncodingUtils.concatenate(salt, skf.generateSecret(spec).getEncoded());  // salt || hash
```

기본값(`defaultsForSpringSecurity_v5_8`)은 솔트 16바이트, **310,000회 반복**, SHA-256, 256비트다. 출력은 `salt||hash`를 통째로 Hex(기본) 또는 Base64로 인코딩한 것이라, 검증 시 앞 16바이트를 솔트로 떼어 같은 절차를 반복한 뒤 `MessageDigest.isEqual`로 비교한다. 주석에 남은 흥미로운 역사: 과거 기본이 `algorithm=SHA-1`인데 `hashWidth=256`인 모순 설정이었고(gh-10506), 하위 호환을 위해 `overrideHashWidth` 플래그로 옛 생성자만 그 값을 존중하도록 분리했다.

## 레거시 인코더 — 왜 남겨 뒀나

`MessageDigestPasswordEncoder`(MD5/SHA), `StandardPasswordEncoder`(SHA-256), `Md4PasswordEncoder`, `LdapShaPasswordEncoder`, `NoOpPasswordEncoder`는 모두 `@Deprecated`다. 이들은 일반 다이제스트라 너무 빨라 안전하지 않다. 그럼에도 **삭제하지 않은** 이유는 분명하다 — 이미 이 형식으로 저장된 옛 비밀번호를 `DelegatingPasswordEncoder`가 검증하고, 로그인 순간 BCrypt로 마이그레이션할 수 있어야 하기 때문이다. JavaDoc도 "제거 계획은 없으며, 단지 레거시임을 알리려 deprecated 표시했다"고 못 박는다.

`MessageDigestPasswordEncoder`의 저장 형식은 `{salt}digest(password+{salt})`로, 솔트를 중괄호에 담아 평문 앞에 섞는다. `matchesNonNull`은 `PasswordEncoderUtils.equals`(상수시간)로 비교한다. `NoOpPasswordEncoder`는 평문을 그대로 반환하고 `equals`로 비교 — 테스트·데모 전용이다.

## 설계 포인트 / 확장점

- **적응형 비용**: 네 알고리즘 모두 "비용 파라미터"를 노출한다(strength / memory·iterations / cpuCost / iterations). 하드웨어가 빨라지면 이 숫자만 올려 새 비밀번호를 더 강하게 만든다.
- **파라미터 자가 기술**: BCrypt/Argon2/SCrypt는 비용을 해시 문자열에 담아, 같은 인코더로 옛 해시도 검증되고 `upgradeEncoding`도 자동 판정된다.
- **`defaultsForSpringSecurity_vX_Y` 정적 팩토리**: 보안 권장값이 버전마다 바뀌므로, 각 인코더는 버전별 기본 설정을 정적 메서드로 제공한다. `PasswordEncoderFactories`가 이를 `{argon2@SpringSecurity_v5_8}` 같은 id로 함께 등록한다([03장](03-Delegating과-비밀번호-업그레이드.md)).
- **상수시간 비교 일관성**: 모든 검증은 `MessageDigest.isEqual` 또는 XOR 누적 루프를 쓴다. 새 인코더를 만들 때도 반드시 따라야 할 규칙이다.

## 정리

BCrypt·Argon2·SCrypt·PBKDF2는 "일부러 느린, 비용 조절 가능한 단방향 함수"라는 같은 철학을 각자 다른 자원(시간/메모리)으로 구현한다. BCrypt는 자체 구현, Argon2·SCrypt는 BouncyCastle, PBKDF2는 JDK 표준에 위임한다. 셋(BCrypt/Argon2/SCrypt)은 파라미터를 해시에 담아 검증과 업그레이드를 자급하고, 레거시 다이제스트 인코더들은 마이그레이션 경로를 위해 deprecated 상태로 보존된다. 이 인코더들을 하나로 묶어 라우팅하고 업그레이드하는 장치가 다음 장의 `DelegatingPasswordEncoder`다.
