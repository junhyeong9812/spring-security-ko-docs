# 05. 메타데이터와 싱글 로그아웃(SLO)

## 무엇을 / 왜

로그인 왕복(§03·§04) 외에 SP가 갖춰야 할 두 가지 부수 기능이 있다.

1. **SP 메타데이터 노출** — IdP 관리자가 우리 SP를 등록하려면 우리의 entityId·ACS·인증서를 알아야 한다. 이걸 손으로 알려주는 대신 표준 `<EntityDescriptor>` XML로 게시한다.
2. **싱글 로그아웃(SLO)** — 한 곳에서 로그아웃하면 SP·IdP 세션을 함께 끊는다. SP가 시작(RP-initiated)할 수도, IdP가 시작(AP-initiated)할 수도 있어 양방향을 모두 처리해야 한다.

## 메타데이터 노출: `Saml2MetadataFilter`

`saml2/.../web/Saml2MetadataFilter.java`

기본 URL `/saml2/service-provider-metadata/{registrationId}` 를 가로채, 등록본을 메타데이터 XML로 변환해 파일로 내려준다.

```
브라우저/IdP ── GET /saml2/service-provider-metadata/{id} ──▶ Saml2MetadataFilter
   metadataResolver.resolve(request):
     requestMatcher 로 registrationId 추출
     → relyingPartyRegistrationResolver.resolve(req, id)
     → Saml2MetadataResolver.resolve(registration)  ← OpenSaml5MetadataResolver가 XML 생성
   응답: Content-Type: application/samlmetadata+xml
         Content-Disposition: attachment; filename="saml-{id}-metadata.xml"
```

```java
protected void doFilterInternal(req, res, chain) {
    Saml2MetadataResponse metadata;
    try { metadata = this.metadataResolver.resolve(req); }
    catch (Saml2Exception ex) { res.setStatus(SC_UNAUTHORIZED); return; }  // 등록 못 찾음
    if (metadata == null) { chain.doFilter(req, res); return; }            // 이 URL 아님
    writeMetadataToResponse(res, metadata);
}
```

`Saml2MetadataResolver`(인터페이스)는 `RelyingPartyRegistration → String(XML)` 변환이고, 실제 구현 `OpenSaml5MetadataResolver`(opensaml5Main)가 OpenSAML로 `<SPSSODescriptor>`(ACS, SLO 엔드포인트, 서명/암호화 인증서)를 빚는다. 매칭 URL·파일명은 setter로 바꿀 수 있으나, 커스텀 `Saml2MetadataResponseResolver` 를 직접 넣었다면 그쪽에서 설정해야 한다(필터가 `Assert.isInstanceOf` 로 강제).

## 로그아웃 1 — RP가 시작 (Relying Party Initiated)

사용자가 우리 앱에서 "로그아웃"을 누른 경우. 우리 세션을 끊은 뒤 IdP에게도 `<LogoutRequest>` 를 보내 IdP 세션을 끊게 한다. 담당자는 `LogoutSuccessHandler` 인 `Saml2RelyingPartyInitiatedLogoutSuccessHandler` 다.

`saml2/.../web/authentication/logout/Saml2RelyingPartyInitiatedLogoutSuccessHandler.java`

```
표준 LogoutFilter → 로컬 세션 무효화 → onLogoutSuccess():
   authentication == null → 401
   Saml2LogoutRequestResolver.resolve(request, authentication)
       └ 현재 Saml2Authentication(principal의 NameID, sessionIndex) + 등록본으로 <LogoutRequest> 생성·서명
   logoutRequestRepository.saveLogoutRequest(...)   ← 나중에 IdP의 LogoutResponse 대조용
   바인딩 분기:
     REDIRECT → 302 {IdP SLO}?SAMLRequest=...&RelayState=...&SigAlg=...&Signature=...
     POST     → 자동제출 폼
   ▼
브라우저 ── LogoutRequest ──▶ IdP   (IdP가 처리 후 LogoutResponse를 우리 SLO 엔드포인트로 돌려줌)
```

저장에는 `HttpSessionLogoutRequestRepository` 가 기본으로 쓰인다. IdP가 돌려주는 `<LogoutResponse>` 는 `Saml2LogoutResponseFilter` 가 받아 저장된 요청과 대조·검증하고 로그아웃을 마무리한다.

## 로그아웃 2 — IdP가 시작 (Asserting Party Initiated)

IdP(또는 다른 SP)에서 로그아웃이 일어나, IdP가 우리에게 `<LogoutRequest>` 를 보내는 경우. 담당자는 `Saml2LogoutRequestFilter` 다.

`saml2/.../web/authentication/logout/Saml2LogoutRequestFilter.java`

```
IdP ── POST/GET /logout/saml2/slo (SAMLRequest=...) ──▶ Saml2LogoutRequestFilter
   1. logoutRequestResolver.resolve(request, authentication)
        SAMLRequest 없으면 → chain 진행
        URL의 {registrationId} 또는 현재 Authentication에서 registrationId 도출
        등록본·요청을 묶은 Saml2LogoutRequestValidatorParameters 생성
   2. validateLogoutRequest():
        SLO 엔드포인트 설정됐나? (없으면 INVALID_DESTINATION)
        사용한 바인딩이 등록의 허용 바인딩에 포함되나? (아니면 INVALID_REQUEST)
        logoutRequestValidator.validate(params)   ← 서명·Issuer·Destination 검증
        실패 → 에러 LogoutResponse 또는 401
   3. handler.logout(request, response, authentication)   ← 실제 로컬 로그아웃(세션 무효화 등)
   4. logoutResponseResolver.resolve(...)  → <LogoutResponse> 생성
   5. 바인딩에 맞춰 IdP에게 LogoutResponse 전송 (redirect 또는 POST)
```

검증의 골자(필터 내부):

```java
if (registration.getSingleLogoutServiceLocation() == null)
    throw new Saml2AuthenticationException(new Saml2Error(INVALID_DESTINATION, ...));
Saml2MessageBinding binding = Saml2MessageBindingUtils.resolveBinding(request);
if (!registration.getSingleLogoutServiceBindings().contains(binding))
    throw new Saml2AuthenticationException(new Saml2Error(INVALID_REQUEST, "Logout request used invalid binding"));
Saml2LogoutValidatorResult result = this.logoutRequestValidator.validate(parameters);
if (result.hasErrors()) throw new Saml2AuthenticationException(new Saml2Error(INVALID_REQUEST, ...));
```

`registrationId` 를 못 구하면 현재 인증 객체에서 끌어온다 — `Saml2AssertionAuthentication` 이거나 principal이 `Saml2AuthenticatedPrincipal` 이면 그 `getRelyingPartyRegistrationId()` 를 쓴다. §04에서 principal에 registrationId를 심어둔 이유가 여기서 드러난다.

## 핵심 타입 지도

```
provider.service.authentication.logout/   (검증 로직 — 버전 중립 Base + 값 타입)
   ├── Saml2LogoutRequest / Saml2LogoutResponse        전송 단위(samlRequest/Response, binding, location)
   ├── Saml2LogoutRequestValidator  ← BaseOpenSamlLogoutRequestValidator ← OpenSaml5LogoutRequestValidator
   ├── Saml2LogoutResponseValidator ← BaseOpenSamlLogoutResponseValidator ← OpenSaml5LogoutResponseValidator
   ├── Saml2Logout(Request|Response)ValidatorParameters  검증 입력 묶음
   └── Saml2LogoutValidatorResult                        에러 누적 결과(§01 패턴)

provider.service.web.authentication.logout/  (서블릿 계층 — 필터·리졸버·핸들러)
   ├── Saml2LogoutRequestFilter                AP-initiated 수신
   ├── Saml2LogoutResponseFilter               RP-initiated의 응답 수신
   ├── Saml2RelyingPartyInitiatedLogoutSuccessHandler   RP-initiated 시작
   ├── Saml2LogoutRequestResolver  ← OpenSaml5LogoutRequestResolver     <LogoutRequest> 생성
   ├── Saml2LogoutResponseResolver ← OpenSaml5LogoutResponseResolver    <LogoutResponse> 생성
   ├── Saml2LogoutRequestValidatorParametersResolver ← OpenSaml5...     요청→검증파라미터
   └── Saml2LogoutRequestRepository / HttpSessionLogoutRequestRepository
```

로그인 계층과 똑같이 **값 타입·검증 로직은 `Base*`(src/main/java), OpenSAML 구체 구현은 `OpenSaml5*`(opensaml5Main)** 로 갈라져 있다.

## 설계 포인트 / 확장점

- **대칭 구조**: 로그아웃은 로그인을 거울처럼 닮았다. "리졸버가 XML 생성, 필터가 전송/수신, 밸리데이터가 서명·Issuer·Destination 검증, 리포지토리가 InResponseTo 대조용 저장". §03·§04를 이해했다면 로그아웃은 거의 그대로다.
- **양방향 SLO**: RP-initiated(`LogoutSuccessHandler`)와 AP-initiated(`Saml2LogoutRequestFilter`)를 별도 컴포넌트로 분리해, 같은 `/logout/saml2/slo` 계열 엔드포인트에서 방향에 맞게 처리한다.
- **바인딩 강제**: 들어온 로그아웃 요청의 바인딩이 등록에 명시된 허용 바인딩이 아니면 거부 — 다운그레이드/혼선 방지.
- **메타데이터 리졸버 교체**: `Saml2MetadataResponseResolver` 를 직접 구현해 멀티테넌트별 메타데이터, 추가 확장(Organization/ContactPerson) 등을 넣을 수 있다.

## 정리

`Saml2MetadataFilter` 는 SP 자신을 `<EntityDescriptor>` XML로 게시해 IdP 등록을 쉽게 한다. 싱글 로그아웃은 두 방향이다 — RP가 시작하면 `Saml2RelyingPartyInitiatedLogoutSuccessHandler` 가 `<LogoutRequest>` 를 IdP로 보내고, IdP가 시작하면 `Saml2LogoutRequestFilter` 가 받아 서명·바인딩·Destination을 검증한 뒤 로컬 로그아웃 후 `<LogoutResponse>` 로 답한다. 이 모든 컴포넌트가 로그인과 동일한 `Base*`/`OpenSaml5*` 분리와 누적 에러 검증 패턴을 따른다는 점이, 이 모듈 전체를 관통하는 일관성이다.
