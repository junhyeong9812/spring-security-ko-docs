# 04. 응답 검증과 인증 프로바이더

## 무엇을 / 왜

IdP가 사용자를 인증한 뒤 브라우저는 `<Response>`(서명된 `<Assertion>` 포함)를 들고 SP의 ACS(Assertion Consumer Service) URL `/login/saml2/sso/{registrationId}` 로 POST한다. 이 장은 그 응답을 받아 **디코딩 → 등록 매칭 → 서명검증 → 복호화 → assertion 검증 → `Authentication` 생성** 으로 흘려보내는 파이프라인을 다룬다. 이것이 SAML 로그인의 심장이며, 보안의 무게중심이 전부 여기 있다.

## 핵심 타입

```
요청 ──▶ Saml2WebSsoAuthenticationFilter (extends AbstractAuthenticationProcessingFilter)
            │ attemptAuthentication()
            │   ① authenticationConverter.convert(request)
            ▼
        Saml2AuthenticationTokenConverter  (AuthenticationConverter)
            └ BaseOpenSamlAuthenticationTokenConverter
                 → Saml2AuthenticationToken (검증 전: SAMLResponse + 확정 등록본 + 저장된 AuthnRequest)
            │   ② getAuthenticationManager().authenticate(token)
            ▼
        AuthenticationManager ──▶ OpenSaml5AuthenticationProvider
                                     └ (위임) BaseOpenSamlAuthenticationProvider.authenticate()
            │   검증 파이프라인 통과
            ▼
        Saml2Authentication / Saml2AssertionAuthentication  (검증 후, isAuthenticated=true)
            └ principal: Saml2AuthenticatedPrincipal (NameID + attributes + sessionIndexes)
```

## 1단계 — 요청을 토큰으로: `Saml2AuthenticationTokenConverter`

`saml2/.../web/BaseOpenSamlAuthenticationTokenConverter.java`

이 컨버터의 일은 "이 요청이 SAML 응답인가? 그렇다면 **어느 등록(RelyingPartyRegistration)** 에 속하는가?"를 판정해 검증 전 토큰 `Saml2AuthenticationToken` 을 만드는 것이다. 세 가지 방법을 순서대로 시도한다.

```java
public Saml2AuthenticationToken convert(HttpServletRequest request) {
    String serialized = request.getParameter(SAML_RESPONSE);     // SAMLResponse 없으면 대상 아님
    if (serialized == null) return null;
    if (!requestMatcher.matcher(request).isMatch()) return null;  // /login/saml2/sso(/{id})

    Saml2AuthenticationToken token = tokenByAuthenticationRequest(request); // ① 저장된 AuthnRequest로
    if (token == null) token = tokenByRegistrationId(request, result);       // ② URL의 {registrationId}로
    if (token == null) token = tokenByEntityId(request);                     // ③ Response의 Issuer로
    return token;
}
```

- **① by AuthenticationRequest**: §03에서 세션에 저장해 둔 요청을 꺼내 그 `registrationId` 로 등록을 찾는다. SP가 시작한 정상 로그인의 경로.
- **② by registrationId**: URL 경로 변수에 `registrationId` 가 있으면 그것으로 조회.
- **③ by entityId**: 위 둘이 실패하면(IdP-initiated 처럼 URL에 식별자가 없을 때) 응답을 디코딩해 `<Issuer>`(=AP entityId)로 `findUniqueByAssertingPartyEntityId` 를 호출.

디코딩은 메서드/바인딩에 맞춰 다르게 한다.

```java
boolean isGet = HttpMethod.GET.matches(request.getMethod());
Saml2Utils.withEncoded(encoded).requireBase64(true).inflate(isGet).decode();
// POST 바인딩(=폼) → base64만, GET/Redirect 바인딩 → base64 후 inflate(deflate 해제)
```

토큰을 만들 때도 등록본의 `entityId`/`acsLocation` 플레이스홀더를 현재 요청 기준으로 치환(`mutate()`)해 넣는다. 그래서 `Saml2AuthenticationToken` 은 항상 **확정된** 등록본과, 디코딩된 응답 XML 문자열, 그리고 (있다면) 저장된 AuthnRequest를 함께 들고 있다.

```java
// Saml2AuthenticationToken: 검증 전 입력. 항상 isAuthenticated()=false, principal=null
public boolean isAuthenticated() { return false; }
```

## 2단계 — 진입 필터: `Saml2WebSsoAuthenticationFilter`

`saml2/.../web/authentication/Saml2WebSsoAuthenticationFilter.java`

표준 폼 로그인 필터와 같은 골격(`AbstractAuthenticationProcessingFilter`)을 따른다. 기본 처리 URL은 `/login/saml2/sso/{registrationId}`.

```java
public Authentication attemptAuthentication(HttpServletRequest request, HttpServletResponse response) {
    Authentication authentication = this.authenticationConverter.convert(request); // 1단계 토큰
    if (authentication == null) {
        if (continueChainWhenNoRelyingPartyRegistrationFound) return null;          // 6.5+
        throw new Saml2AuthenticationException(
            new Saml2Error(RELYING_PARTY_REGISTRATION_NOT_FOUND, "No relying party registration found"));
    }
    setDetails(request, authentication);
    this.authenticationRequestRepository.removeAuthenticationRequest(request, response); // 1회용 소비
    return getAuthenticationManager().authenticate(authentication);                 // → Provider
}
```

세션 고정 보호로 `ChangeSessionIdAuthenticationStrategy` 를 쓰고, 저장된 AuthnRequest는 여기서 제거(소비)된다. 실제 검증은 `AuthenticationManager` → `OpenSaml5AuthenticationProvider` 에 맡긴다.

## 3단계 — 검증 파이프라인: `BaseOpenSamlAuthenticationProvider`

`saml2/.../authentication/BaseOpenSamlAuthenticationProvider.java`

`OpenSaml5AuthenticationProvider`(public)는 이 `Base` 클래스에 위임한다. `authenticate()` 의 뼈대:

```java
public Authentication authenticate(Authentication authentication) {
    Saml2AuthenticationToken token = (Saml2AuthenticationToken) authentication;
    Response response = parseResponse(token.getSaml2Response());      // XML → OpenSAML Response
    process(token, response);                                          // ★ 검증
    AbstractAuthenticationToken result =
        this.responseAuthenticationConverter.convert(new ResponseToken(response, token)); // → Saml2Authentication
    result.setDetails(authentication.getDetails());
    return result;
}
```

핵심은 `process()` 다. 검증을 **단계별 `Converter`/`Consumer` 의 조립** 으로 구성하고, 각 단계가 `Saml2Error` 를 누적한다.

```
process(token, response):
 ┌─────────────────────────────────────────────────────────────────┐
 │ 1. responseSignatureValidator                                      │
 │      Response.isSigned() 이면 AP verificationCredentials 로 서명검증  │
 │      → 실패 시 즉시 보고하고 중단 (서명 깨졌으면 더 볼 필요 없음)         │
 ├─────────────────────────────────────────────────────────────────┤
 │ 2. 복호화 분기                                                       │
 │      Response 서명됨 → responseElementsDecrypter (EncryptedAssertion 복호화) │
 │      서명 안 됨 + EncryptedAssertion 있음 → INVALID_SIGNATURE 에러     │
 ├─────────────────────────────────────────────────────────────────┤
 │ 3. responseValidator  (validateResponseAfterAssertions=false 기본)  │
 │      StatusCode == SUCCESS ?                                        │
 │      InResponseTo == 저장된 AuthnRequest.id ?                        │
 │      Destination == acsLocation ?                                   │
 │      Issuer == AP entityId ?                                        │
 │      assertion 최소 1개 존재 ?                                       │
 ├─────────────────────────────────────────────────────────────────┤
 │ 4. for each Assertion:                                              │
 │      assertionSignatureValidator (assertion.isSigned() 이면 검증)     │
 │      서명 OK & (Response 또는 assertion 서명됨) → assertion 복호화      │
 │      assertionValidator (SAML20AssertionValidator: 조건/주체/시간)    │
 ├─────────────────────────────────────────────────────────────────┤
 │ 5. Response·assertion 둘 다 미서명이면 INVALID_SIGNATURE             │
 │ 6. 첫 assertion에 NameID 없으면 SUBJECT_NOT_FOUND                    │
 └─────────────────────────────────────────────────────────────────┘
 → reportErrors(): 에러가 하나라도 있으면 첫 에러로 Saml2AuthenticationException throw
```

### 왜 "Response 또는 모든 assertion 중 하나는 서명돼야" 하나

```java
if (!responseSigned && !allAssertionsSigned) {
    errors.add(new Saml2Error(INVALID_SIGNATURE,
        "Either the response or one of the assertions is unsigned. ..."));
}
```

SAML에서 신뢰의 근거는 IdP의 디지털 서명이다. Response 전체가 서명됐거나, 각 assertion이 서명됐어야 위변조를 막을 수 있다. 둘 다 없으면 신뢰할 수 없으므로 거부한다. 또한 Response가 서명됐다면 그 안의 assertion 서명은 생략 가능하다(이미 전체가 보호되므로).

### assertion 검증의 실제 — `SAML20AssertionValidator`

`createDefaultAssertionValidator()` 는 OpenSAML의 `SAML20AssertionValidator` 에 조건/주체 검증기를 끼워 만든다. 검증 컨텍스트에는 **누가 무엇과 일치해야 하는지** 를 등록본에서 끌어와 채운다.

```java
params.put(COND_VALID_AUDIENCES,  singleton(registration.getEntityId()));                 // <Audience> = 우리
params.put(SC_VALID_RECIPIENTS,   singleton(registration.getAssertionConsumerServiceLocation())); // Recipient = ACS
params.put(VALID_ISSUERS,         singleton(assertingPartyEntityId));                      // Issuer = IdP
params.put(CLOCK_SKEW,            Duration.ofMinutes(5));                                  // 시간 허용오차
if (assertionContainsInResponseTo(assertion))
    params.put(SC_VALID_IN_RESPONSE_TO, getAuthnRequestId(token.getAuthenticationRequest()));
```

기본 동작 두 가지가 의도적으로 완화돼 있다(JavaDoc/주석 근거): `OneTimeUse` 조건은 "VALID"로 통과시키고(앱이 직접 재생방지하라), `BearerSubjectConfirmation` 의 주소(Address) 검증도 통과시킨다(gh-7514, 프록시/NAT 환경 때문). 즉 **replay 방지와 클라이언트 IP 검증은 애플리케이션 책임**이다.

### Response → `Authentication` 변환

검증을 통과하면 `responseAuthenticationConverter` 가 첫 assertion에서 사용자 정보를 추출한다.

```java
Assertion assertion = firstElement(response.getAssertions());
String username = assertion.getSubject().getNameID().getValue();
Map<String,List<Object>> attributes = getAssertionAttributes(assertion);   // AttributeStatement → Map
List<String> sessionIndexes = getSessionIndexes(assertion);                // SLO에 필요
DefaultSaml2AuthenticatedPrincipal principal =
    new DefaultSaml2AuthenticatedPrincipal(username, attributes, sessionIndexes);
principal.setRelyingPartyRegistrationId(registrationId);
return new Saml2Authentication(principal, token.getSaml2Response(),
    AuthorityUtils.createAuthorityList("ROLE_USER"));
```

기본 권한은 `ROLE_USER` 하나. assertion의 속성(부서, 그룹 등)을 권한으로 매핑하려면 `setResponseAuthenticationConverter(...)` 로 이 변환기를 교체한다. 결과 `Saml2Authentication`(7.0+에서는 assertion 접근자를 품은 하위형 `Saml2AssertionAuthentication`)이 `SecurityContext` 에 저장되어 로그인이 완성된다.

## 핵심 메서드 / 확장점

`OpenSaml5AuthenticationProvider` 는 파이프라인의 각 단계를 교체하는 setter들을 노출한다.

- `setResponseValidator(...)` — Status/Issuer/Destination/InResponseTo 등 Response 레벨 규칙 추가·교체. `createDefaultResponseValidator()` 에 자기 검증을 `concat` 하는 식으로 보강 가능.
- `setAssertionValidator(...)` — clock skew 조정, OneTimeUse(재생방지) 직접 구현, 커스텀 Condition 추가.
- `setResponseAuthenticationConverter(...)` — assertion 속성 → 권한/principal 매핑.
- `setResponseElementsDecrypter / setAssertionElementsDecrypter(...)` — 복호화 키·전략 교체.

`supports(Saml2AuthenticationToken.class)` 이므로 다른 인증과 한 `AuthenticationManager` 에 공존할 수 있다.

## 정리

응답 수신은 세 계단을 거친다. `Saml2AuthenticationTokenConverter` 가 요청을 어느 등록에 속하는지 판정해 검증 전 `Saml2AuthenticationToken` 으로 만들고, `Saml2WebSsoAuthenticationFilter` 가 그것을 `AuthenticationManager` 에 넘기며, `OpenSaml5AuthenticationProvider`(→ `BaseOpenSamlAuthenticationProvider`)가 서명검증·복호화·assertion 검증을 누적 에러 방식으로 수행한다. 통과하면 NameID·속성·세션인덱스를 담은 `Saml2Authentication` 이 만들어진다. 검증의 각 단계가 교체 가능한 `Converter` 라는 점이 이 설계의 확장성이다. 마지막 장에서는 SP 메타데이터 노출과 로그아웃을 본다.
