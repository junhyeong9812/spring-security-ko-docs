# 04. 인증 흐름 (Authentication)

> 소스: `webauthn/src/main/java/org/springframework/security/web/webauthn/authentication/`

## 무엇을 / 왜

인증(authentication) 의식은 등록해 둔 패스키로 **로그인**하는 과정이다. 이 패키지는 그 진입점들 — 인증 옵션 발급, assertion 검증 필터, 그리고 표준 Spring Security 인증 파이프라인에 끼워 넣는 `AuthenticationProvider`와 토큰들 — 을 제공한다. 등록과 달리 이 흐름의 끝에서는 `SecurityContext`에 담길 진짜 `Authentication`이 만들어진다.

설계의 묘는, WebAuthn이라는 특수한 메커니즘을 Spring Security의 일반 인증 골격(`AbstractAuthenticationProcessingFilter` → `AuthenticationManager` → `AuthenticationProvider`)에 그대로 얹었다는 점이다. 폼 로그인과 같은 틀을 쓰되 자격증명만 비밀번호 대신 서명으로 바꾼 것이다.

## 핵심 타입

```
authentication 패키지
 ├ PublicKeyCredentialRequestOptionsFilter    POST /webauthn/authenticate/options
 │     └ RequestOptions(challenge) 생성 → 세션 저장 → JSON 응답
 ├ WebAuthnAuthenticationFilter               POST /login/webauthn
 │     │ (extends AbstractAuthenticationProcessingFilter)
 │     └ 본문 파싱 + 세션 챌린지 → WebAuthnAuthenticationRequestToken → AuthenticationManager
 ├ WebAuthnAuthenticationProvider             토큰 검증 → UserDetails → WebAuthnAuthentication
 ├ WebAuthnAuthenticationRequestToken         (검증 전) 입력 토큰
 ├ WebAuthnAuthentication                     (검증 후) 인증된 결과 토큰
 └ PublicKeyCredentialRequestOptionsRepository (+HttpSession 구현)
```

토큰이 두 개인 이유가 핵심이다. **검증 전(`...RequestToken`)** 과 **검증 후(`WebAuthnAuthentication`)** 를 다른 타입으로 분리해, Provider가 전자를 받아 후자를 돌려주는 표준 패턴을 따른다.

## 동작 흐름


### ① 인증 옵션 발급 — PublicKeyCredentialRequestOptionsFilter

```java
protected void doFilterInternal(HttpServletRequest request, ...) {
    if (!this.matcher.matches(request)) { filterChain.doFilter(...); return; }  // POST /webauthn/authenticate/options

    SecurityContext context = this.securityContextHolderStrategy.getContext();
    var optionsRequest = new ImmutablePublicKeyCredentialRequestOptionsRequest(context.getAuthentication());
    PublicKeyCredentialRequestOptions credentialRequestOptions =
            this.rpOptions.createCredentialRequestOptions(optionsRequest);
    this.requestOptionsRepository.save(request, response, credentialRequestOptions); // 세션 보관
    this.converter.write(credentialRequestOptions, MediaType.APPLICATION_JSON, ...);
}
```

등록 옵션 필터와 달리 **인증 옵션은 비로그인 상태에서도 발급**된다(아직 로그인 전이니까). 현재 인증 객체를 넘기긴 하지만, 비인증이면 `createCredentialRequestOptions`가 `allowCredentials`를 빈 목록으로 만든다(05장). 발급한 옵션은 역시 세션에 저장해 둘째 요청과 챌린지를 대조한다.

### ② assertion 제출 — WebAuthnAuthenticationFilter

이 필터는 `AbstractAuthenticationProcessingFilter`를 상속하므로, 성공/실패 핸들러·`SecurityContextRepository` 같은 표준 인프라를 그대로 쓴다. 생성자에서 기본 경로·핸들러를 박아 둔다.

```java
public WebAuthnAuthenticationFilter() {
    super(pathPattern(HttpMethod.POST, "/login/webauthn"));
    setSecurityContextRepository(new HttpSessionSecurityContextRepository());
    setAuthenticationFailureHandler(new AuthenticationEntryPointFailureHandler(
            new HttpStatusEntryPoint(HttpStatus.UNAUTHORIZED)));        // 실패 → 401
    setAuthenticationSuccessHandler(new HttpMessageConverterAuthenticationSuccessHandler());
}

public Authentication attemptAuthentication(HttpServletRequest request, ...) {
    // 1) 본문을 PublicKeyCredential<AuthenticatorAssertionResponse>로 역직렬화
    var publicKeyCredential = (PublicKeyCredential<AuthenticatorAssertionResponse>)
            this.converter.read(resolvableType, httpRequest, null);
    // 2) 세션에서 첫 요청의 RequestOptions(challenge) 로드
    PublicKeyCredentialRequestOptions requestOptions = this.requestOptionsRepository.load(request);
    if (requestOptions == null) throw new BadCredentialsException("... No PublicKeyCredentialRequestOptions found.");
    this.requestOptionsRepository.save(request, response, null);   // ★ 챌린지 1회용 소모
    // 3) 입력 토큰으로 감싸 AuthenticationManager에 위임
    var authenticationRequest = new RelyingPartyAuthenticationRequest(requestOptions, publicKeyCredential);
    var token = new WebAuthnAuthenticationRequestToken(authenticationRequest);
    return getAuthenticationManager().authenticate(token);
}
```

폼 로그인의 `attemptAuthentication`이 username/password를 `UsernamePasswordAuthenticationToken`으로 감싸는 것과 똑같은 구조다. 다만 자격증명이 "클라이언트가 보낸 assertion + 세션의 챌린지"로 바뀌었을 뿐이다. 여기서도 검증 직전 `save(..., null)`로 챌린지를 소모한다.

### ③ 서명 검증과 권한 부여 — WebAuthnAuthenticationProvider

`AuthenticationManager`는 토큰 타입(`supports`)이 맞는 이 Provider에게 검증을 넘긴다. Provider가 하는 일은 명료하다.

```java
public Authentication authenticate(Authentication authentication) {
    WebAuthnAuthenticationRequestToken webAuthnRequest = (WebAuthnAuthenticationRequestToken) authentication;
    try {
        // (1) 실제 WebAuthn 서명 검증 → 성공 시 사용자 식별자 반환
        PublicKeyCredentialUserEntity userEntity =
                this.relyingPartyOperations.authenticate(webAuthnRequest.getWebAuthnRequest());
        // (2) username으로 표준 UserDetailsService에서 권한 로딩
        String username = userEntity.getName();
        UserDetails userDetails = this.userDetailsService.loadUserByUsername(username);
        Collection<GrantedAuthority> authorities = new HashSet<>(userDetails.getAuthorities());
        // (3) "이 인증은 WebAuthn 요소로 이뤄졌다"는 표식 권한 추가
        authorities.add(FactorGrantedAuthority.fromAuthority(FactorGrantedAuthority.WEBAUTHN_AUTHORITY));
        return new WebAuthnAuthentication(userEntity, authorities);
    } catch (RuntimeException ex) {
        throw new BadCredentialsException(ex.getMessage(), ex);
    }
}
```

세 단계가 들어있다. (1) **암호 검증은 `relyingPartyOperations.authenticate(...)`에 위임** — 여기서 webauthn4j가 저장된 공개키로 서명을 검증하고 서명 카운터를 갱신한다(05장). (2) 검증으로 얻은 username을 **표준 `UserDetailsService`** 에 넘겨 그 사용자의 권한(roles)을 로딩한다. 즉 "패스키로 누구인지 확인"과 "그 사람의 권한이 무엇인지"는 분리되어 있고, 후자는 기존 사용자 저장소를 재사용한다. (3) `FactorGrantedAuthority.WEBAUTHN_AUTHORITY`를 더해, 이 인증이 WebAuthn 요소로 달성됐음을 기록한다(다중요소 정책에서 활용).

### 두 토큰 — 입력과 결과

```java
// 검증 전: 미인증. 자격증명(assertion)과 principal(userHandle)을 실어 나르는 운반체
class WebAuthnAuthenticationRequestToken extends AbstractAuthenticationToken {
    public Object getCredentials() { return this.webAuthnRequest.getPublicKey(); }
    public Object getPrincipal()   { return ...getResponse().getUserHandle(); }
    public void setAuthenticated(boolean a) { Assert.isTrue(!a, "Cannot set ... trusted"); ... } // 신뢰 승격 금지
}

// 검증 후: 인증됨. principal이 PublicKeyCredentialUserEntity
class WebAuthnAuthentication extends AbstractAuthenticationToken {
    public PublicKeyCredentialUserEntity getPrincipal() { return this.principal; }
    public Object getCredentials() { return null; }   // 자격증명은 더 들고 있지 않음
    public String getName() { return this.principal.getName(); }
}
```

둘 다 `setAuthenticated(true)`를 막아(요청 토큰), 오직 Provider만이 `WebAuthnAuthentication` 생성자(`super.setAuthenticated(true)`)를 통해 신뢰된 토큰을 만들 수 있게 한다. 결과 토큰은 자격증명을 버리고(`getCredentials()==null`) principal로 WebAuthn 사용자 식별자를 들고 있다.

## 설계 포인트 / 확장점

- **표준 인증 골격 재사용**: `AbstractAuthenticationProcessingFilter` + `AuthenticationProvider` + 두 토큰 패턴을 그대로 따르므로, 성공/실패 핸들러·세션 고정 보호·`SecurityContextRepository`가 공짜로 붙는다.
- **식별과 권한의 분리**: WebAuthn은 "누구인지"만 책임지고, "무슨 권한인지"는 기존 `UserDetailsService`가 책임진다. 비밀번호 사용자 저장소를 그대로 재사용 가능.
- **요소 표식**: `WEBAUTHN_AUTHORITY`로 인증 수단을 권한에 기록 → MFA/단계적 인증 정책의 근거.
- **확장점**: `setRequestMatcher`(옵션 경로), `setRequestOptionsRepository`(챌린지 보관 전략), `setConverter`(직렬화), 그리고 `WebAuthnAuthenticationFilter`의 성공/실패 핸들러는 상위 클래스 setter로 교체.

## 정리

인증은 "옵션 발급(① `RequestOptionsFilter`) → 브라우저가 인증기로 챌린지 서명 → assertion 제출(② `WebAuthnAuthenticationFilter`)"의 두 POST다. 둘째 요청에서 필터는 본문과 세션 챌린지를 묶어 `WebAuthnAuthenticationRequestToken`을 만들고, `WebAuthnAuthenticationProvider`가 (1) webauthn4j로 서명을 검증해 사용자를 식별하고 (2) `UserDetailsService`로 권한을 로딩해 (3) `WebAuthnAuthentication`을 발급한다. 폼 로그인과 같은 골격 위에 자격증명만 서명으로 바꾼 셈이며, 실제 암호 검증의 속살은 [05-relyingparty-연산과-저장소.md](05-relyingparty-연산과-저장소.md)에서 본다.
