# 01. 개념과 전체 구조

## 무엇을 / 왜

이 챕터는 나머지 챕터를 읽기 위한 공통 지도다. WebAuthn이 무엇이고, 서버(Relying Party)가 떠맡는 책임이 무엇이며, 그 책임이 이 모듈의 어느 부품에 어떻게 흩어져 있는지를 먼저 잡는다. 개별 클래스는 다음 챕터들에서 깊게 다룬다.

### WebAuthn을 한 문장으로

비밀번호는 "사용자와 서버가 같은 비밀을 공유"하는 모델이다. WebAuthn은 이를 뒤집어, 인증기(authenticator) 안에서 **공개키/개인키 쌍**을 만든다. 개인키는 인증기 밖으로 절대 나오지 않고, 서버에는 **공개키만** 등록된다. 로그인할 때 서버는 일회용 난수 **챌린지**를 보내고, 인증기는 개인키로 그것에 서명해 돌려준다. 서버는 등록해 둔 공개키로 서명만 검증한다. "공유 비밀"이 없으니 유출·재사용·피싱이 구조적으로 불가능하다. 사용자에게 이 전체가 "패스키(passkey)"로 보인다.

## 두 개의 의식(Ceremony)

WebAuthn에는 딱 두 가지 흐름이 있다. 각 흐름은 모두 "서버가 옵션(특히 챌린지)을 발급한다 → 브라우저 JS가 인증기를 호출한다 → 인증기 응답을 서버가 검증한다"는 2-요청 왕복이다.

```
 [등록 Ceremony — Registration]              [인증 Ceremony — Authentication]
                                            
 ① POST /webauthn/register/options          ① POST /webauthn/authenticate/options
    └ CreationOptions(challenge) 발급           └ RequestOptions(challenge) 발급
            │                                          │
            ▼ 브라우저                                  ▼ 브라우저
 navigator.credentials.create(options)      navigator.credentials.get(options)
            │  (인증기가 키쌍 생성 + 서명)              │  (인증기가 개인키로 서명)
            ▼                                          ▼
 ② POST /webauthn/register                  ② POST /login/webauthn
    └ AttestationResponse 검증·저장             └ AssertionResponse 서명 검증
       → CredentialRecord 저장                    → Authentication 발급(로그인 완료)
```

핵심은 두 흐름 모두 **상태(챌린지)를 첫 요청과 둘째 요청 사이에 보관**해야 한다는 점이다. 그 보관소가 `*OptionsRepository`이며 기본 구현은 HttpSession이다. 둘째 요청에서 인증기가 서명한 챌린지가, 첫 요청에서 발급해 세션에 저장한 챌린지와 일치해야만 검증을 통과한다(재전송 공격 방지).

## Relying Party의 4대 연산

서버가 해야 할 일은 결국 위 다이어그램의 4개 박스이고, 이것이 그대로 `WebAuthnRelyingPartyOperations` 인터페이스의 네 메서드다(`management` 패키지).

```
WebAuthnRelyingPartyOperations
 ├ createPublicKeyCredentialCreationOptions(req)  → 등록 옵션  (등록 ①)
 ├ registerCredential(req)                        → 등록 검증·저장 (등록 ②)
 ├ createCredentialRequestOptions(req)            → 인증 옵션  (인증 ①)
 └ authenticate(req)                              → assertion 검증 (인증 ②)
```

필터들은 HTTP 파싱/저장소/응답쓰기만 담당하고, 실제 "WebAuthn 규칙"은 모두 이 인터페이스 뒤로 위임한다. 유일한 구현 `Webauthn4JRelyingPartyOperations`는 다시 암호 검증을 webauthn4j 라이브러리에 위임한다. 즉 책임이 3겹으로 분리되어 있다.

```
HTTP 필터 (registration/authentication)
   │  "요청 파싱 + 세션 챌린지 보관 + 응답 직렬화"
   ▼
WebAuthnRelyingPartyOperations (management)
   │  "WebAuthn 절차 + 저장소 조회/갱신"
   ▼
webauthn4j WebAuthnManager
      "서명/attestation의 실제 암호 검증"
```

## 모듈 전체 조감도

요청 관점에서 본 부품 배치는 다음과 같다. 위쪽이 HTTP 경계, 아래로 갈수록 도메인·영속화다.

```
                      브라우저 / 인증기(authenticator)
                                  │  JSON (WebAuthn 포맷)
 ┌────────────────────────────────┼─────────────────────────────────────┐
 │ SecurityFilterChain            │                                      │
 │                                ▼                                      │
 │  registration 패키지                  authentication 패키지            │
 │  ┌──────────────────────────┐         ┌──────────────────────────┐    │
 │  │ CreationOptionsFilter     │        │ RequestOptionsFilter      │    │
 │  │   POST .../register/options│       │   POST .../authenticate/.. │    │
 │  │ RegistrationFilter        │        │ WebAuthnAuthenticationFilter│   │
 │  │   POST .../register        │       │   POST /login/webauthn     │    │
 │  │   DELETE .../register/{id} │       └─────────────┬────────────┘    │
 │  │ RegistrationPageFilter    │                     │ token            │
 │  └────────────┬─────────────┘                      ▼                  │
 │               │                       WebAuthnAuthenticationProvider   │
 │               │                                   │                    │
 │   *OptionsRepository (HttpSession: 챌린지 보관)    │                    │
 └───────────────┼───────────────────────────────────┼────────────────────┘
                 ▼                                   ▼
            WebAuthnRelyingPartyOperations  ◀── UserDetailsService(권한 로딩)
                 │  (Webauthn4JRelyingPartyOperations)
       ┌─────────┴───────────┐                │
       ▼                     ▼                ▼
 UserCredentialRepository  PublicKeyCredentialUserEntityRepository
   (CredentialRecord 저장)    (사용자 식별자 저장)
   Map / Jdbc 구현            Map / Jdbc 구현
```

여기에 가로지르는 두 개의 횡단 관심사가 있다.

- **api 패키지**: 위 모든 박스가 주고받는 값 객체(데이터 모델). WebAuthn 스펙의 딕셔너리와 거의 1:1로 대응한다. → [02-api-도메인-모델.md](02-api-도메인-모델.md)
- **jackson 패키지**: 그 값 객체들을 WebAuthn이 요구하는 정확한 JSON 형태(예: byte[]는 Base64URL 문자열)로 직렬화/역직렬화. → [06-직렬화-jackson.md](06-직렬화-jackson.md)

## 설계 포인트

- **인터페이스로 절차를, 라이브러리로 암호를 분리**: 필터는 webauthn4j를 전혀 모른다. `WebAuthnRelyingPartyOperations`만 의존하므로, 검증 엔진을 교체하거나 테스트에서 목으로 대체하기 쉽다.
- **저장소 추상화**: 사용자(`PublicKeyCredentialUserEntityRepository`)와 자격증명(`UserCredentialRepository`)을 별도 인터페이스로 두고, 메모리(`Map*`)·JDBC(`Jdbc*`) 구현을 함께 제공한다.
- **상태 보관의 명시적 추상화**: 두 의식 사이의 챌린지 보관을 `*OptionsRepository`로 빼내어, 세션 대신 쿠키·Redis 등으로 갈아끼울 수 있게 했다.
- **DSL 분리**: 이 모듈은 필터/Provider/저장소 "부품"만 제공하고, 이들을 시큐리티 체인에 꽂는 조립은 `config` 모듈의 `WebAuthnConfigurer`가 한다.

## 정리

WebAuthn은 "서버가 챌린지를 주고 인증기가 개인키로 서명하면 서버가 공개키로 검증"하는 모델이며, 서버 측 일은 등록·인증 두 의식의 4개 연산으로 요약된다. 이 모듈은 그 4개 연산을 `WebAuthnRelyingPartyOperations`로 정의하고, HTTP 필터(registration/authentication) → 연산 인터페이스(management) → webauthn4j(암호 검증) → 저장소(영속화)로 책임을 층층이 나눠 구현한다. 이어지는 챕터들은 이 지도의 각 영역을 따라간다.
