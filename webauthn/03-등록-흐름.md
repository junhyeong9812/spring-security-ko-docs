# 03. 등록 흐름 (Registration)

> 소스: `webauthn/src/main/java/org/springframework/security/web/webauthn/registration/`

## 무엇을 / 왜

등록(registration) 의식은 **이미 로그인한 사용자**가 자기 계정에 새 패스키를 추가하는 과정이다. 이 패키지는 그 의식의 HTTP 진입점들 — 옵션 발급, 자격증명 등록/삭제, 그리고 기본 등록 UI — 을 제공한다. 실제 검증·저장은 `management`의 `WebAuthnRelyingPartyOperations`에 위임하고(05장), 이 패키지는 "HTTP 파싱 + 세션 챌린지 보관 + 응답 직렬화"만 담당한다.

왜 로그인 상태가 전제일까? 새 공개키를 누구의 계정에 묶을지 알아야 하기 때문이다. 그래서 옵션 발급 필터는 인증된 사용자만 통과시킨다.

## 핵심 타입

```
registration 패키지
 ├ PublicKeyCredentialCreationOptionsFilter   POST /webauthn/register/options
 │     └ 인증 사용자 검사 → CreationOptions 생성 → 세션 저장 → JSON 응답
 ├ WebAuthnRegistrationFilter                 POST /webauthn/register        (등록)
 │     │                                       DELETE /webauthn/register/{id} (삭제)
 │     └ 세션의 options + 클라이언트 응답 → registerCredential() → 저장
 ├ PublicKeyCredentialCreationOptionsRepository  (의식 사이 챌린지 보관)
 │     └ HttpSessionPublicKeyCredentialCreationOptionsRepository (기본)
 └ DefaultWebAuthnRegistrationPageGeneratingFilter  GET /webauthn/register (기본 HTML UI)
```

## 동작 흐름

등록은 두 번의 POST로 이루어진다. 중간에 브라우저가 인증기를 호출한다.

```
 사용자(로그인됨)        브라우저 JS            서버 필터                 management/저장소
      │                    │                      │                         │
      │ "패스키 추가" 클릭   │                      │                         │
      │───────────────────▶│                      │                         │
      │                    │ ① POST /register/options                       │
      │                    │─────────────────────▶│ CreationOptionsFilter   │
      │                    │                      │ authz: 인증됨?           │
      │                    │                      │─ createPublicKeyCredentialCreationOptions ─▶│
      │                    │                      │◀─ CreationOptions(challenge) ───────────────│
      │                    │                      │ repo.save(세션에 options 저장)            │
      │                    │◀── JSON(options) ────│                         │
      │                    │                      │                         │
      │                    │ navigator.credentials.create(options)          │
      │                    │  (인증기: 키쌍 생성 + attestation 서명)         │
      │                    │                      │                         │
      │                    │ ② POST /register {publicKey:{credential, label}}│
      │                    │─────────────────────▶│ RegistrationFilter      │
      │                    │                      │ repo.load(세션 options)  │
      │                    │                      │ repo.save(null) ← 챌린지 1회용 소모        │
      │                    │                      │─ registerCredential(options, publicKey) ──▶│
      │                    │                      │   (webauthn4j 검증 + 저장) │
      │                    │◀── JSON {success:true}│◀─ CredentialRecord ─────│
      │                    │                      │                         │
```

### ① 옵션 발급 — PublicKeyCredentialCreationOptionsFilter

```java
protected void doFilterInternal(HttpServletRequest request, ...) {
    if (!this.matcher.matches(request)) { filterChain.doFilter(...); return; }   // POST /webauthn/register/options

    Supplier<Authentication> authentication = () -> context.get().getAuthentication();
    AuthorizationResult result = this.authorization.authorize(authentication, request); // 기본: authenticated()
    if (result == null || !result.isGranted()) {
        response.setStatus(SC_BAD_REQUEST);   // 비로그인은 여기서 400
        return;
    }
    PublicKeyCredentialCreationOptions options = this.rpOperations.createPublicKeyCredentialCreationOptions(
            new ImmutablePublicKeyCredentialCreationOptionsRequest(authentication.get()));
    this.repository.save(request, response, options);     // 세션에 보관 (챌린지 포함)
    this.converter.write(options, MediaType.APPLICATION_JSON, new ServletServerHttpResponse(response));
}
```

핵심은 세 단계다. (1) `AuthenticatedAuthorizationManager.authenticated()`로 로그인 여부를 검사하고, (2) 인증 객체를 넘겨 옵션을 생성하며, (3) `repository.save(...)`로 옵션(=챌린지)을 세션에 저장한 뒤 JSON으로 내려보낸다. 저장이 없으면 둘째 요청에서 챌린지를 대조할 수 없다.

### ② 등록 검증 — WebAuthnRegistrationFilter

이 필터는 두 가지 동작을 매처로 갈라 처리한다(`doFilterInternal`). `POST /webauthn/register`는 등록, `DELETE /webauthn/register/{id}`는 삭제다.

```java
private void registerCredential(HttpServletRequest request, HttpServletResponse response) {
    WebAuthnRegistrationRequest registrationRequest = readRegistrationRequest(request); // 본문 파싱
    if (registrationRequest == null) { response.setStatus(BAD_REQUEST); return; }

    PublicKeyCredentialCreationOptions options = this.creationOptionsRepository.load(request); // 세션에서 챌린지
    if (options == null) { response.setStatus(BAD_REQUEST); return; }
    this.creationOptionsRepository.save(request, response, null);   // ★ 챌린지 1회용 소모(즉시 삭제)

    CredentialRecord credentialRecord = this.rpOptions.registerCredential(
            new ImmutableRelyingPartyRegistrationRequest(options, registrationRequest.getPublicKey()));
    // ... SuccessfulUserRegistrationResponse를 JSON으로 write
}
```

여기서 두 가지가 묶인다. **첫 요청에서 발급한 options(challenge)** 와 **둘째 요청에서 클라이언트가 보낸 publicKey** 를 함께 `registerCredential(...)`에 넘긴다. 검증은 05장에서 보듯 webauthn4j가 "클라이언트가 서명한 챌린지 == 발급했던 챌린지", "origin/rpId 일치"를 확인한다. 그리고 검증 직전 `save(..., null)`로 세션의 옵션을 즉시 비워 **챌린지를 1회용**으로 만든다 — 같은 챌린지의 재사용·재전송을 막는다.

### 자격증명 삭제 — defense in depth

`DELETE /webauthn/register/{id}`는 사용자가 자기 패스키를 지우는 경로다. 흥미로운 점은 인가 정책이다.

```java
private void removeCredential(HttpServletRequest request, ..., @Nullable String id) {
    Bytes credentialId = Bytes.fromBase64(id);
    AuthorizationResult result = this.deleteCredentialAuthorizationManager.authorize(authentication, credentialId);
    if (result != null && !result.isGranted()) { response.setStatus(FORBIDDEN); return; }
    this.userCredentials.delete(credentialId);
    response.setStatus(NO_CONTENT);
}
```

`deleteCredentialAuthorizationManager`의 **기본값이 `SingleResultAuthorizationManager.denyAll()`** 이다. 즉 별도로 인가 매니저를 주입하지 않으면 모든 삭제가 거부된다. 운영에서는 보통 `CredentialRecordOwnerAuthorizationManager`(05장)를 꽂아 "현재 사용자가 그 자격증명의 소유자일 때만 삭제"를 강제한다. 자격증명 id 자체가 사실상 추측 불가능하지만, 혹시 노출되더라도 남의 패스키를 못 지우게 하는 심층 방어다.

### 챌린지 보관소 — PublicKeyCredentialCreationOptionsRepository

두 의식 요청 사이의 상태 보관을 담당하는 추상화다. 기본 구현은 세션에 옵션 객체를 통째로 담는다.

```java
public void save(HttpServletRequest request, ..., @Nullable PublicKeyCredentialCreationOptions options) {
    request.getSession().setAttribute(this.attrName, options);   // options=null이면 사실상 삭제
}
public @Nullable PublicKeyCredentialCreationOptions load(HttpServletRequest request) {
    HttpSession session = request.getSession(false);
    return (session == null) ? null : (PublicKeyCredentialCreationOptions) session.getAttribute(this.attrName);
}
```

인터페이스로 빠져 있으므로 쿠키 기반·분산 캐시 기반 구현으로 교체할 수 있다. 옵션 객체가 `Serializable`인 이유이기도 하다.

### 기본 등록 UI — DefaultWebAuthnRegistrationPageGeneratingFilter

`GET /webauthn/register`에 대해 패스키를 등록/관리하는 기본 HTML 페이지를 렌더링한다. 현재 사용자의 자격증명 목록을 표로 보여주고, CSRF 토큰을 메타로 심어 위 JS 흐름이 동작하도록 돕는다. 애플리케이션이 자체 UI를 제공하면 비활성화하면 된다.

## 설계 포인트 / 확장점

- **책임 분리**: 필터는 검증 로직을 모른다. `WebAuthnRelyingPartyOperations`만 의존하므로 테스트·교체가 쉽다.
- **챌린지 1회용 강제**: 등록 검증 직전 `save(..., null)`로 세션 옵션을 비우는 패턴이 재전송을 막는 핵심이다.
- **삭제는 기본 deny**: 보안 민감 동작인 삭제를 기본적으로 막아두고, 명시적으로 소유권 인가를 주입하게 한 안전한 기본값.
- **확장점**: `setRegisterCredentialMatcher`/`setRemoveCredentialMatcher`(경로 변경), `setCreationOptionsRepository`(보관 전략 교체), `setConverter`(직렬화 교체), `setDeleteCredentialAuthorizationManager`(삭제 인가 정책).

## 정리

등록은 "옵션 발급(① `CreationOptionsFilter`) → 브라우저가 인증기로 키쌍 생성 → 등록 검증(② `RegistrationFilter`)"의 두 POST로 이루어진다. 옵션 발급은 로그인 사용자만 허용하고 챌린지를 세션에 저장하며, 등록 검증은 그 챌린지를 꺼내 `registerCredential()`에 넘긴 뒤 즉시 세션에서 비워 1회용으로 만든다. 삭제 경로는 기본 deny + 소유권 인가로 보호된다. 실제 암호 검증과 저장은 다음 흐름(인증)과 함께 [05-relyingparty-연산과-저장소.md](05-relyingparty-연산과-저장소.md)에서 본다.
