# 01. 아키텍처 개요 — Bearer Token 인증의 큰 그림

## 무엇을 / 왜

이 챕터는 모듈의 "지도"다. 리소스 서버가 토큰 하나로 어떻게 인증을 끝내는지, 그 과정에 어떤 타입들이 어떤 순서로 끼어드는지를 먼저 머릿속에 그려두면, 이후 JWT/Opaque 각 챕터가 같은 골격의 변주임을 알 수 있다.

리소스 서버 인증의 본질은 단순하다. **요청에서 토큰 문자열을 뽑고(추출) → 그 문자열이 진짜인지 확인하고(검증) → 결과를 `Authentication` 으로 만들어 `SecurityContext` 에 넣는다.** 어려운 점은 "검증" 방식이 둘(JWT 로컬 검증 / Opaque introspection)이라는 것뿐인데, 이 모듈은 그 둘을 **표준 `AuthenticationProvider` 인터페이스 뒤로 숨겨** 필터가 어느 쪽인지 몰라도 되게 만들었다.

## 핵심 타입

```
요청에서 토큰을 꺼내는 쪽            토큰을 검증하는 쪽
───────────────────────            ─────────────────────────────
BearerTokenAuthenticationFilter    AuthenticationManager
  └ AuthenticationConverter          └ AuthenticationProvider
      └ BearerTokenResolver               ├ JwtAuthenticationProvider
                                          ├ OpaqueTokenAuthenticationProvider
                                          └ DPoPAuthenticationProvider

토큰을 표현하는 타입
─────────────────────────────────────────────────────
BearerTokenAuthenticationToken   ← 인증 "요청" (검증 전, 토큰 문자열만)
        │ authenticate()
        ▼
JwtAuthenticationToken           ← JWT 경로의 인증 "결과"
BearerTokenAuthentication        ← Opaque 경로의 인증 "결과"
  (둘 다 AbstractOAuth2TokenAuthenticationToken 상속)
```

핵심은 입력과 출력 타입이 다르다는 점이다.

- **입력** `BearerTokenAuthenticationToken`: 아직 검증되지 않은 토큰 문자열만 담은 미인증 토큰. `isAuthenticated()==false`.
- **출력** `JwtAuthenticationToken` / `BearerTokenAuthentication`: 검증을 통과한 결과. principal(=토큰 속성), 권한(scope 기반), 만료/발급 정보를 담고 `isAuthenticated()==true`. 둘 다 `AbstractOAuth2TokenAuthenticationToken<T>` 를 상속해 `getToken()`/`getTokenAttributes()` 라는 공통 표면을 갖는다.

## 동작 흐름 — 요청 한 건이 통과하는 전체 경로

JWT를 예로 든 표준 시나리오다. Opaque도 Provider만 바뀔 뿐 골격은 같다.

```
HTTP 요청
  Authorization: Bearer eyJhbGci...   (JWS)
        │
        ▼
┌─────────────────────────────────────────────────────────────┐
│ BearerTokenAuthenticationFilter.doFilterInternal()           │
│                                                               │
│ 1) authenticationConverter.convert(request)                   │
│      → BearerTokenResolver 로 헤더에서 "eyJ..." 추출           │
│      → BearerTokenAuthenticationToken(token) 생성 (미인증)     │
│      · 토큰 없으면 → filterChain 그냥 통과 (익명으로 진행)     │
│      · 토큰 형식 오류 → EntryPoint.commence() 로 401          │
│                                                               │
│ 2) authenticationManagerResolver.resolve(request)             │
│      → AuthenticationManager 선택 (단일 or iss별 멀티테넌트)   │
│                                                               │
│ 3) authenticationManager.authenticate(token)                  │
│      ────────────────────────────────────────┐               │
│                                               ▼               │
│                            JwtAuthenticationProvider          │
│                              · jwtDecoder.decode(token)       │
│                                  → 서명·exp·iss 검증 (oauth2-jose)│
│                              · JwtAuthenticationConverter      │
│                                  → scope → SCOPE_* 권한        │
│                              → JwtAuthenticationToken 반환     │
│      ◀────────────────────────────────────────┘               │
│                                                               │
│ 4) DPoP 다운그레이드 차단 검사 (cnf.jkt 있으면 거부)          │
│ 5) SecurityContext 에 결과 저장 → securityContextRepository    │
│ 6) filterChain.doFilter()  → 컨트롤러 진입                    │
│                                                               │
│ 실패(AuthenticationException) 시:                             │
│   clearContext() → authenticationFailureHandler               │
│     → EntryPoint.commence() → 401 + WWW-Authenticate          │
└─────────────────────────────────────────────────────────────┘
```

이후 컨트롤러 진입 전 `AuthorizationFilter` 가 `SCOPE_*` 권한을 보고 인가를 결정한다. 권한이 모자라면 `BearerTokenAccessDeniedHandler` 가 403 `insufficient_scope` 를 돌려준다(05장).

## 핵심 메서드

### `BearerTokenAuthenticationFilter.doFilterInternal()`

이 모듈의 심장이다. 전체 골격이 위 흐름 그대로다. 주목할 설계 결정 세 가지.

- **토큰이 없으면 통과**(`authenticationRequest == null` → `filterChain.doFilter`). 즉 이 필터는 "토큰이 있으면 인증을 시도"할 뿐, 인증을 강제하지 않는다. 보호 여부는 뒤단 `AuthorizationFilter` 의 책임이다.
- **변환 단계 실패와 인증 단계 실패를 구분**한다. 토큰 추출 실패(`OAuth2AuthenticationException`)는 곧장 `EntryPoint` 로, 인증 실패(`AuthenticationException`)는 `clearContext()` 후 `authenticationFailureHandler` 로 보낸다.
- 성공 시 `SecurityContext` 를 만들어 `securityContextRepository.saveContext()` 로 저장한다. 기본 저장소는 `RequestAttributeSecurityContextRepository` — 세션이 아니라 요청 스코프에만 담는다. 무상태(stateless) 리소스 서버에 맞는 선택이다.

### `AuthenticationManagerResolver` — 단일 vs 멀티테넌트

필터는 `AuthenticationManager` 를 직접 들지 않고 `AuthenticationManagerResolver<HttpServletRequest>` 를 든다. 생성자에 매니저 하나를 넘기면 `(authentication) -> authenticationManager` 람다로 감싸 항상 같은 매니저를 돌려주고, 여러 인가 서버를 지원하려면 `JwtIssuerAuthenticationManagerResolver`(03장)를 넘긴다. 덕분에 단일/멀티테넌트가 같은 필터 코드로 처리된다.

## 설계 포인트 / 확장점

- **전략 패턴의 3단 분리**: "추출(`AuthenticationConverter`/`BearerTokenResolver`) → 선택(`AuthenticationManagerResolver`) → 검증(`AuthenticationProvider`)". 각 단계가 독립 인터페이스라 한 단계만 교체할 수 있다. 예: 헤더 대신 쿠키에서 토큰을 읽고 싶으면 `BearerTokenResolver` 만 바꾸면 된다.
- **7.0의 생성자 변화**: 과거엔 필터에 `BearerTokenResolver` 를 세터로 주입했지만, 7.0부터 생성자에 `AuthenticationConverter` 를 받는 형태가 표준이 됐다(`setBearerTokenResolver`/`setAuthenticationDetailsSource` 는 `@Deprecated`). 토큰 추출 책임이 `BearerTokenAuthenticationConverter` 로 옮겨진 것이다(02장).
- **Provider 교체로 검증 방식 선택**: JWT를 쓸지 introspection을 쓸지는 `AuthenticationManager` 에 어떤 `AuthenticationProvider` 를 꽂느냐로 결정된다. 보통 config 모듈의 `oauth2ResourceServer` DSL이 이 조립을 대신한다.

## 정리

리소스 서버 인증은 "추출 → 매니저 선택 → 검증 → 컨텍스트 저장"의 4박자이며, `BearerTokenAuthenticationFilter` 가 이 박자를 지휘한다. 검증 방식(JWT/Opaque/DPoP)은 모두 `AuthenticationProvider` 라는 같은 구멍에 꽂히는 부품이라, 필터는 어떤 방식인지 몰라도 된다. 다음 장부터 각 부품을 차례로 연다.
