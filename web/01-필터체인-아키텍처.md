# 01 · 필터 체인 아키텍처 — 요청이 보안에 진입하는 길

## 무엇을 / 왜

서블릿 컨테이너(Tomcat 등)는 "필터(`jakarta.servlet.Filter`)"라는 단 하나의 확장점만 제공한다. Spring Security는 수십 개의 보안 작업을 이 하나의 확장점에 욱여넣어야 한다. 해법은 **하나의 진입 필터 `FilterChainProxy`** 를 컨테이너에 등록하고, 그 안에서 Spring이 관리하는 보안 필터들을 직접 순서대로 호출하는 것이다.

이 장은 요청이 컨테이너에서 보안 세계로 들어오는 경계, 즉 `FilterChainProxy`가 어떻게 체인을 고르고, 요청을 검역(firewall)하고, 필터들을 실행하는지를 다룬다.

## 핵심 타입

```
┌──────────────────────┐     사용     ┌──────────────────────────┐
│ DelegatingFilterProxy│ ───────────▶ │ FilterChainProxy         │  (이 모듈)
│ (spring-web)         │              │  - List<SecurityFilterChain>
└──────────────────────┘              │  - HttpFirewall          │
   서블릿 web.xml/등록부              │  - FilterChainDecorator  │
                                      └──────────────────────────┘
                                                 │ 보유
                                ┌────────────────┴────────────────┐
                                ▼                                  ▼
                  ┌──────────────────────────┐      ┌──────────────────────────┐
                  │ SecurityFilterChain       │      │ DefaultSecurityFilterChain│
                  │  matches(request): bool   │◀─────│  RequestMatcher + Filter[]│
                  │  getFilters(): List<Filter>│      └──────────────────────────┘
                  └──────────────────────────┘
```

- **`FilterChainProxy`** (`web/.../FilterChainProxy.java`) — 컨테이너 필터 체인에 끼워지는 단 하나의 보안 필터. `GenericFilterBean`을 상속하므로 IoC 컨테이너가 라이프사이클을 관리한다.
- **`SecurityFilterChain`** — `(matches, getFilters)` 두 메서드뿐인 인터페이스. "이 요청에 적용되는가?"와 "적용된다면 어떤 필터들인가?"를 답한다.
- **`DefaultSecurityFilterChain`** — 표준 구현. `RequestMatcher` 하나와 `List<Filter>` 하나를 들고, `matches()`를 matcher에 위임한다.
- **`HttpFirewall` / `StrictHttpFirewall`** (`firewall/`) — 들어오는 요청을 검증·정규화하고, 위험하면 `RequestRejectedException`을 던진다.
- **`FilterChainProxy.FilterChainDecorator`** — 선택된 필터 목록을 실제 실행 가능한 `FilterChain`으로 감싸는 전략. 기본 구현 `VirtualFilterChainDecorator`가 `VirtualFilterChain`을 만든다.

## 동작 흐름 — 요청 한 건의 여정

`FilterChainProxy.doFilter()`(`web/.../FilterChainProxy.java:186`)와 `doFilterInternal()`(:213)이 핵심이다.

```
요청 ──▶ FilterChainProxy.doFilter
         │
         │ ① FILTER_APPLIED 속성 확인 (중첩 호출 방지)
         │    - 최초 진입이면 try/finally 로 감싸고 끝에서
         │      SecurityContextHolder.clearContext() + 속성 제거
         ▼
        doFilterInternal
         │ ② firewall.getFirewalledRequest/Response 로 래핑
         │    (경로 정규화, 위험 문자/인코딩 차단)
         │ ③ getFilters(req): filterChains 를 위에서부터 순회,
         │    chain.matches(req) 가 처음 true 인 체인의 필터 반환
         │      → 더 아래 체인은 보지 않는다 (first-match-wins)
         ▼
        ┌─ 매칭 필터 없음 ──▶ firewallRequest.reset() 후
        │                     원래 서블릿 체인으로 그냥 통과
        │
        └─ 매칭 필터 있음 ──▶ filterChainDecorator.decorate(reset, filters)
                              = VirtualFilterChain 생성
                                  │
                                  ▼ doFilter 반복 호출
                       Filter[0] → Filter[1] → ... → Filter[n]
                                  │ (각 필터가 chain.doFilter 호출 시 다음으로 진행)
                                  ▼
                       size 도달 → "Secured" 로그 → firewallRequest.reset()
                                  → 원래 컨테이너 체인(chain.doFilter) 으로 복귀
```

핵심 설계 포인트 세 가지:

1. **체인은 하나만 선택된다.** `getFilters()`(:245)는 매칭되는 **첫 번째** 체인의 필터를 반환하고 즉시 끝난다. 그래서 더 구체적인 매처를 가진 체인을 리스트 위쪽에 둬야 한다. 한 요청이 여러 체인을 거치는 일은 없다.
2. **필터가 빈 체인 = "이 경로는 보안 제외"** 가 아니라 "보안 필터 없이 그냥 통과"다. 정적 리소스 경로 등에 쓰인다.
3. **`clearContext()` 보장.** 최초 진입 필터만 `finally`에서 `SecurityContextHolder`를 비운다. 스레드 풀에서 재사용되는 스레드에 이전 요청의 인증이 남지 않도록 하는 안전장치다.

### VirtualFilterChain — 필터를 순서대로 끌고 가는 장치

`VirtualFilterChain`(`FilterChainProxy.java:357`)은 `FilterChain` 인터페이스를 구현하지만, 표준 컨테이너 체인이 아니라 **보안 필터 리스트를 인덱스로 순회하는 작은 상태기계**다.

```java
public void doFilter(ServletRequest request, ServletResponse response) {
    if (this.currentPosition == this.size) {
        this.originalChain.doFilter(request, response); // 보안 필터 소진 → 원래 체인 복귀
        return;
    }
    this.currentPosition++;
    Filter nextFilter = this.additionalFilters.get(this.currentPosition - 1);
    nextFilter.doFilter(request, response, this); // 다음 보안 필터에 자기 자신을 넘김
}
```

각 보안 필터는 자기 일을 마치고 `chain.doFilter(req, res)`를 부르는데, 그 `chain`이 바로 이 `VirtualFilterChain`이다. 즉 **필터가 체인을 한 칸 전진시키는 주체**다. 어떤 필터가 `chain.doFilter`를 호출하지 않으면(예: `AuthorizationFilter`가 거부하거나, 로그인 필터가 응답을 직접 마무리하면) 그 아래 필터들은 실행되지 않고 요청 처리가 거기서 끝난다.

## 핵심 메서드

- **`doFilter` (:186)** — `FILTER_APPLIED` 속성으로 최초 진입을 식별하고, 최초 진입에서만 `try/finally`로 컨텍스트 정리를 책임진다. 또한 실행 중 `RequestRejectedException`이 올라오면 `throwableAnalyzer`로 원인 사슬에서 찾아내 `RequestRejectedHandler`(기본 `HttpStatusRequestRejectedHandler` → 400)에 위임한다.
- **`getFilters(HttpServletRequest)` (:245)** — first-match-wins로 체인을 고른다. 이 한 메서드가 "URL → 적용 필터 집합" 매핑의 전부다.
- **`StrictHttpFirewall.getFirewalledRequest`** — `//`·역슬래시·인코딩된 슬래시·세미콜론(matrix variable)·제어문자 등을 가진 요청을 기본 거부한다. RequestMatcher들이 보는 경로 값을 일관되게 만들어, "매처는 통과시키고 컨트롤러는 다르게 해석하는" 류의 우회를 막는다.

## 설계 포인트 / 확장점

- **`setFilterChainDecorator`** — 6.0부터 도입. 필터 실행을 가로채는 확장점이다. `ObservationFilterChainDecorator`를 끼우면 Micrometer Observation으로 각 필터 실행을 관측할 수 있다.
- **`setFirewall` / `setRequestRejectedHandler`** — 기본 `StrictHttpFirewall`이 너무 엄격해 정상 요청을 막는 경우 완화하거나, 거부 시 응답(로그/상태코드)을 바꾼다.
- **`FilterChainValidator`** — `afterPropertiesSet()`에서 호출되는 검증 훅. config 모듈이 필터 순서·중복을 기동 시점에 검사하는 데 쓴다.
- 실무에서 `FilterChainProxy`를 직접 만드는 일은 드물다. config의 `HttpSecurity` DSL이 `DefaultSecurityFilterChain`을 만들고 그것들을 모아 `FilterChainProxy` 빈으로 등록한다. 이 모듈은 **그 부품과 실행 엔진**을 제공한다.

## 정리

`FilterChainProxy`는 서블릿 컨테이너의 단일 필터 확장점과 Spring Security의 다중 보안 관심사 사이를 잇는 다리다. 요청이 오면 firewall로 검역하고, first-match로 단 하나의 `SecurityFilterChain`을 고른 뒤, `VirtualFilterChain`으로 그 안의 보안 필터들을 순서대로 끌고 간다. 그리고 진입 스레드에서만 `SecurityContextHolder`를 끝에서 비워 스레드 재사용 안전성을 보장한다. 나머지 모든 장은 이 체인 위에 놓이는 개별 필터들의 이야기다.
