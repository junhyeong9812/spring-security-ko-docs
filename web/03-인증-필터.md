# 03 · 인증 필터 — 자격 증명을 Authentication 으로 바꾸는 자리

## 무엇을 / 왜

인증이란 "요청에 담긴 자격 증명(폼 파라미터, Basic 헤더, 토큰)을 검증해 누구인지 확정하는" 과정이다. `web` 모듈의 인증 필터들은 **요청에서 자격 증명을 추출하는 부분**만 담당하고, 실제 검증은 `core`의 `AuthenticationManager`에 위임한다. 이 분업이 핵심이다: 필터는 "어떻게 자격 증명을 읽는가(HTTP 프로토콜)"를 알고, `AuthenticationManager`는 "어떻게 검증하는가(도메인)"를 안다.

이 장은 폼 로그인의 본체인 `AbstractAuthenticationProcessingFilter`를 중심으로, HTTP Basic, remember-me, anonymous까지 인증 계열 필터 전체를 조망한다.

## 핵심 타입

```
AbstractAuthenticationProcessingFilter  (추상 골격)
   │ attemptAuthentication() 만 하위에서 구현
   ├── UsernamePasswordAuthenticationFilter  ← 폼 로그인 (POST username/password)
   ├── AuthenticationFilter                  ← Converter 기반 범용 인증
   └── (oauth2/saml2 등 외부 모듈의 필터들)

BasicAuthenticationFilter   ← OncePerRequestFilter, HTTP Basic 헤더 처리
AnonymousAuthenticationFilter ← 인증 없으면 "익명" Authentication 부여
RememberMeAuthenticationFilter ← 쿠키 기반 자동 로그인

협력자:
  AuthenticationConverter        request → Authentication(미인증 토큰)
  AuthenticationSuccessHandler   성공 후 리다이렉트/응답
  AuthenticationFailureHandler   실패 후 응답
  SessionAuthenticationStrategy  인증 직후 세션 처리 (5장)
  SecurityContextRepository      성공한 컨텍스트 저장 (2장)
  RememberMeServices             자동 로그인 토큰 발급/소비
```

## 동작 흐름 — 폼 로그인 한 번

`AbstractAuthenticationProcessingFilter.doFilter()`(`authentication/AbstractAuthenticationProcessingFilter.java:242`)가 전체 시나리오의 골격이다.

```
POST /login (username=alice&password=secret)
   │ requiresAuthentication(req)?  ← requiresAuthenticationRequestMatcher 로 판단
   │   아니면 → chain.doFilter (그냥 통과)
   ▼ 맞으면
  attemptAuthentication(req, res)        ← 하위 클래스 구현
   │  UsernamePasswordAuthenticationFilter:
   │    obtainUsername/Password → UsernamePasswordAuthenticationToken.unauthenticated
   │    → authenticationManager.authenticate(token)
   ▼
  ┌── 결과 == null ──▶ continueChainWhenNoAuthenticationResult 면 chain 진행, 아니면 즉시 반환
  │                    (멀티스텝 인증이 "아직 안 끝남"을 의미)
  │
  ├── AuthenticationException ──▶ unsuccessfulAuthentication
  │      clearContext + rememberMeServices.loginFail
  │      + failureHandler.onAuthenticationFailure (기본 401/실패 URL)
  │
  └── Authentication 반환 (성공)
        │ sessionStrategy.onAuthentication(result, req, res)  ← 세션 고정 보호 등
        ▼
       successfulAuthentication
        │ 1) context = createEmptyContext(); context.setAuthentication(result)
        │ 2) holder.setContext(context)
        │ 3) securityContextRepository.saveContext(context, req, res)  ← 2장의 명시 저장
        │ 4) rememberMeServices.loginSuccess(...)
        │ 5) eventPublisher.publish(InteractiveAuthenticationSuccessEvent)
        │ 6) successHandler.onAuthenticationSuccess(...)  ← 기본: 저장된 요청으로 리다이렉트
        ▼
       응답 완료 (체인 더 진행 안 함)
```

### attemptAuthentication — 프로토콜별로 다른 유일한 부분

상위 골격은 success/failure/세션/저장을 모두 공통화하고, **자격 증명을 어떻게 읽는가**만 하위에 위임한다. `UsernamePasswordAuthenticationFilter.attemptAuthentication()`(`authentication/UsernamePasswordAuthenticationFilter.java`)은 폼 파라미터를 읽어 미인증 토큰을 만든다.

```java
String username = obtainUsername(request); // 기본 파라미터명 "username"
String password = obtainPassword(request); // 기본 파라미터명 "password"
UsernamePasswordAuthenticationToken authRequest =
        UsernamePasswordAuthenticationToken.unauthenticated(username, password);
setDetails(request, authRequest);          // WebAuthenticationDetails (IP, 세션ID)
return this.getAuthenticationManager().authenticate(authRequest);
```

`postOnly`(기본 true)면 POST가 아닌 요청은 즉시 거부한다. 검증 자체는 전적으로 `AuthenticationManager`(→ `AuthenticationProvider` → `UserDetailsService`, `core` 모듈)의 몫이다.

### AuthenticationConverter — 골격을 일반화하다

상위 클래스의 기본 `attemptAuthentication`(:357)은 하위 구현 대신 **`AuthenticationConverter`** 를 쓰도록 되어 있다. `convert(request)`가 `null`이면 "이 요청엔 자격 증명 없음"으로 보고 체인을 계속 진행한다. `AuthenticationFilter`가 이 방식을 정면으로 채택해, "request → Authentication"을 함수형으로 갈아끼울 수 있게 한다(베어러 토큰, API 키 등).

## 다른 인증 필터들

- **`BasicAuthenticationFilter`** (`authentication/www/`) — `OncePerRequestFilter`. `Authorization: Basic <base64>` 헤더를 `BasicAuthenticationConverter`로 디코드해 인증한다. 성공하면 컨텍스트에 올리고 **체인을 계속 진행**한다(폼 로그인처럼 리다이렉트로 끝내지 않음 — Basic은 매 요청 인증이므로). 실패 시 `ignoreFailure`가 아니면 `AuthenticationEntryPoint`(보통 `BasicAuthenticationEntryPoint` → `WWW-Authenticate` 헤더)를 호출한다.
- **`AnonymousAuthenticationFilter`** — 체인 후반에서, 아직 컨텍스트에 인증이 없으면 `AnonymousAuthenticationToken`("anonymousUser", `ROLE_ANONYMOUS`)을 심는다. 이로써 이후 인가 단계가 "인증이 아예 null"과 "익명 사용자"를 구분할 수 있다(4장의 `ExceptionTranslationFilter`가 익명 여부로 분기).
- **`RememberMeAuthenticationFilter`** — 컨텍스트에 인증이 없을 때 `RememberMeServices.autoLogin(req, res)`를 호출해 쿠키로 자동 로그인을 시도한다. 성공하면 `RememberMeAuthenticationToken`을 컨텍스트에 올린다.

### RememberMe 서비스 계열

`RememberMeServices`(`authentication/RememberMeServices.java`)는 세 메서드 — `autoLogin`, `loginFail`, `loginSuccess` — 로 자동 로그인 모델을 정의한다. 인증 필터들은 `loginSuccess`/`loginFail`을 호출하고(쿠키 발급/폐기), `RememberMeAuthenticationFilter`는 `autoLogin`을 호출한다. 구현은 두 가지:

- **`TokenBasedRememberMeServices`** — 서명 토큰(`username:expiry:signature`)을 쿠키에 담는다. 서버 저장소 불필요.
- **`PersistentTokenBasedRememberMeServices`** — 시리즈/토큰을 서버(`PersistentTokenRepository`, JDBC/InMemory)에 저장하고 매번 토큰을 회전시켜 쿠키 탈취를 탐지한다(`CookieTheftException`).

## 핵심 메서드

- **`successfulAuthentication` (:392)** — 인증 성공의 "마무리 의식". 컨텍스트 생성 → holder 세팅 → **저장소에 명시 저장** → remember-me 통지 → 이벤트 발행 → success handler. 2장에서 본 "명시 저장"이 바로 여기서 일어난다.
- **`unsuccessfulAuthentication` (:419)** — 실패의 마무리. 컨텍스트를 비우고 remember-me에 실패를 알린 뒤 failure handler에 위임한다.
- **`SavedRequestAwareAuthenticationSuccessHandler`** — 기본 success handler. `ExceptionTranslationFilter`가 저장해둔 원래 요청(4장의 `RequestCache`)이 있으면 그 URL로, 없으면 `/`로 리다이렉트한다. "보호 자원 접근 → 로그인 → 원래 자원으로 복귀"의 사용자 경험을 완성한다.

## 설계 포인트 / 확장점

- **Template Method 패턴** — 상위가 흐름을, 하위가 `attemptAuthentication`만. 새 인증 프로토콜을 추가할 때 success/failure/세션/저장 로직을 재작성할 필요가 없다.
- **교체 가능한 협력자** — `setAuthenticationSuccessHandler`, `setAuthenticationFailureHandler`, `setSessionAuthenticationStrategy`, `setSecurityContextRepository`, `setRememberMeServices` 등 거의 모든 부분이 주입 가능하다. REST API라면 리다이렉트 대신 JSON을 반환하는 핸들러로 갈아끼운다.
- **MFA 지원(`setMfaEnabled`)** — `doFilter`(:259)에서 이미 인증된 같은 사용자에 대해 추가 인증이 들어오면 권한을 누적(merge)해, 1차/2차 인증의 권한을 합치는 다단계 인증을 지원한다.
- **`continueChainWhenNoAuthenticationResult`** — `AuthenticationConverter`를 세팅하면 자동으로 true가 되어, "이 요청엔 내 자격 증명이 없음"일 때 다음 필터로 자연스럽게 넘어간다(여러 인증 방식 공존).

## 정리

인증 필터들은 "HTTP에서 자격 증명을 읽어 미인증 `Authentication`을 만들고, `AuthenticationManager`에 검증을 위임한 뒤, 결과에 따라 컨텍스트를 채우고 저장·후처리한다"는 공통 골격을 공유한다. `AbstractAuthenticationProcessingFilter`가 이 골격을 Template Method로 고정하고, 프로토콜별 차이는 `attemptAuthentication`/`AuthenticationConverter`로만 흡수한다. Basic·anonymous·remember-me는 같은 무대 위에서 각자의 자격 증명 출처를 다룰 뿐이다.
