# 04 · 예외 변환과 인가 — 접근을 막고, 막힌 사람을 안내하기

## 무엇을 / 왜

인증된 사용자라도 모든 자원에 접근할 수는 없다. **인가(authorization)** 는 "이 주체가 이 자원에 접근해도 되는가"를 결정한다. 그리고 인증이 안 됐거나 권한이 부족할 때, 그것을 **HTTP 응답으로 번역**해 사용자를 로그인 페이지로 보내거나 403을 내려야 한다.

`web` 모듈은 이 둘을 두 필터로 분리한다:

- **`AuthorizationFilter`** — 실제 접근 결정. 거부되면 예외를 던진다(보안 강제).
- **`ExceptionTranslationFilter`** — 던져진 보안 예외를 잡아 응답으로 번역한다(UI 유지). 스스로는 어떤 보안 결정도 하지 않는다.

이 분리 덕분에 인가 로직(누가 통과하는가)과 응답 처리(거부를 어떻게 보여주는가)가 독립적으로 진화한다.

## 핵심 타입

```
ExceptionTranslationFilter  (try/catch 로 하위 필터 감쌈)
   ├── AuthenticationEntryPoint   미인증 → 인증 "개시" (로그인 리다이렉트, 401, WWW-Authenticate)
   ├── AccessDeniedHandler        인증됐으나 권한 부족 → 403
   ├── AuthenticationTrustResolver 익명/remember-me 여부 판별
   └── RequestCache               거부된 원래 요청 저장 (로그인 후 복귀용)

AuthorizationFilter  (체인 후반, 자원 직전)
   └── AuthorizationManager<HttpServletRequest>
         └── RequestMatcherDelegatingAuthorizationManager
               URL 패턴 → 개별 AuthorizationManager 매핑
               (AuthorityAuthorizationManager, AuthenticatedAuthorizationManager ...)
```

## 동작 흐름 — 거부에서 응답까지

두 필터의 상대적 위치가 중요하다. **`ExceptionTranslationFilter`가 `AuthorizationFilter`보다 앞**에 있어, 뒤에서 던진 예외를 잡을 수 있다.

```
요청 ──▶ ExceptionTranslationFilter.doFilter
          │ try { chain.doFilter(req,res) }   ← 여기 안에서 AuthorizationFilter 실행됨
          │                                       AuthorizationFilter:
          │                                         result = authorizationManager.authorize(auth, req)
          │                                         !granted → throw AuthorizationDeniedException
          │                                                    (AccessDeniedException 의 일종)
          │ catch (Exception ex) {
          │   throwableAnalyzer 로 원인 사슬에서
          │   AuthenticationException 또는 AccessDeniedException 추출
          │   ├─ 없음 → 그대로 rethrow (보안 예외 아님)
          │   └─ 있음 → handleSpringSecurityException
          │ }
          ▼
   ┌─ AuthenticationException ─▶ sendStartAuthentication
   │                              · 컨텍스트 비움
   │                              · requestCache.saveRequest(req,res)   ← 원래 요청 저장
   │                              · authenticationEntryPoint.commence(...)  ← 로그인으로 유도
   │
   └─ AccessDeniedException ─┬─ 익명 or remember-me 사용자?
                            │     예 → InsufficientAuthenticationException 로 감싸
                            │           sendStartAuthentication (로그인 시키자)
                            │     아니오(정식 인증됨) → accessDeniedHandler.handle (403)
```

### AuthorizationFilter — 실제 결정

`AuthorizationFilter.doFilter()`(`access/intercept/AuthorizationFilter.java:77`)의 본체는 짧고 명료하다.

```java
AuthorizationResult result = this.authorizationManager.authorize(this::getAuthentication, request);
this.eventPublisher.publishAuthorizationEvent(this::getAuthentication, request, result);
if (result != null && !result.isGranted()) {
    throw new AuthorizationDeniedException("Access Denied", result);
}
chain.doFilter(request, response);
```

- 인가 판단을 통째로 `AuthorizationManager`(core)에 위임한다. 필터는 "거부면 예외를 던진다"는 강제만 책임진다.
- `getAuthentication()`은 컨텍스트에서 인증을 꺼내되, **`null`이면 `AuthenticationCredentialsNotFoundException`** 을 던진다. 그래서 anonymous 필터가 익명 토큰이라도 심어두는 것이 의미를 갖는다.
- 인가 결과는 `AuthorizationEventPublisher`로 발행되어, 감사 로그/모니터링이 granted/denied를 관측할 수 있다.

### URL → 규칙 매핑: RequestMatcherDelegatingAuthorizationManager

웹 인가의 실체는 `RequestMatcherDelegatingAuthorizationManager`(`access/intercept/`)다. `authorizeHttpRequests { ... }` DSL이 만드는 매핑 테이블이 바로 이것이다.

```java
for (mapping : this.mappings) {              // 위에서부터 순서대로
    MatchResult matchResult = mapping.getRequestMatcher().matcher(request);
    if (matchResult.isMatch()) {
        manager = mapping.getEntry();
        return manager.authorize(authentication,
                new RequestAuthorizationContext(request, matchResult.getVariables()));
    }
}
return DENY;                                  // 어떤 매처에도 안 걸리면 거부
```

- **first-match-wins** — 처음 매칭된 매처의 `AuthorizationManager`만 적용한다. 그래서 `requestMatchers("/admin/**").hasRole("ADMIN")` 같은 구체 규칙을 `anyRequest()` 위에 둬야 한다.
- 매칭에서 추출된 경로 변수(`{id}` 등)는 `RequestAuthorizationContext`로 전달되어, `@PathVariable` 기반 인가식에 쓸 수 있다.
- 어떤 매처에도 걸리지 않으면 **기본 거부(`DENY`)** — "명시적으로 허용한 것만 통과"라는 안전한 기본값.

위임 대상 `AuthorizationManager`들은 core에 있다: `AuthorityAuthorizationManager`(역할/권한), `AuthenticatedAuthorizationManager`(인증 여부), `SingleResultAuthorizationManager.permitAll()/denyAll()` 등.

## 핵심 메서드

- **`ExceptionTranslationFilter.handleAccessDeniedException` (`access/ExceptionTranslationFilter.java:188`)** — 핵심 분기. `AuthenticationTrustResolver`로 현재 인증이 **익명/remember-me면** "아직 제대로 로그인 안 한 것"으로 보고 로그인을 개시한다. **정식 인증된 사용자**의 접근 거부만 진짜 403(`AccessDeniedHandler`)으로 처리한다. 즉 "권한이 부족한 게 아니라 애초에 로그인을 안 한 것"을 구분한다.
- **`sendStartAuthentication` (:214)** — 컨텍스트를 비우고(기존 인증 무효화), `requestCache.saveRequest`로 원래 요청을 저장한 뒤 `AuthenticationEntryPoint.commence`를 호출한다. 이 저장 덕분에 로그인 성공 후 3장의 `SavedRequestAwareAuthenticationSuccessHandler`가 원래 페이지로 복귀시킬 수 있다.
- **`AuthenticationEntryPoint.commence`** — 미인증 시 "인증을 시작"하는 방법을 추상화한다. 폼 로그인은 `LoginUrlAuthenticationEntryPoint`(로그인 페이지로 302), Basic은 `BasicAuthenticationEntryPoint`(401 + `WWW-Authenticate`), REST API는 `HttpStatusEntryPoint`(401만). `DelegatingAuthenticationEntryPoint`로 요청 유형(Accept 헤더 등)에 따라 분기할 수도 있다.

## RequestCache / SavedRequest — 인증 후 복귀

`savedrequest` 패키지는 "거부 당시의 요청"을 저장·복원한다.

- **`RequestCache`** — `saveRequest` / `getMatchingRequest`. 기본 구현 `HttpSessionRequestCache`는 세션에 `DefaultSavedRequest`(URL·헤더·파라미터 스냅샷)를 저장한다.
- **`RequestCacheAwareFilter`** — 로그인 성공 후 원래 URL로 돌아왔을 때, 저장된 요청과 매칭되면 원본 파라미터/헤더를 복원한 래핑 요청을 체인에 흘려보낸다. POST 폼 데이터까지 살릴 수 있다.

## 설계 포인트 / 확장점

- **결정과 표현의 분리** — `AuthorizationFilter`(결정) ↔ `ExceptionTranslationFilter`(표현). 인가 규칙을 바꿔도 응답 방식은 그대로고, 401/403 응답을 커스터마이즈해도 인가 로직은 그대로다.
- **`AuthorizationManager` 교체** — URL 인가 전체를 커스텀 `AuthorizationManager`로 대체하거나, 매핑 빌더에 커스텀 매니저를 추가할 수 있다. 메서드 보안과 동일한 `AuthorizationManager` 추상화를 공유한다.
- **`AccessDeniedHandler` 합성** — `DelegatingAccessDeniedHandler`, `RequestMatcherDelegatingAccessDeniedHandler` 등으로 예외 유형/경로별로 다른 403 처리를 조합한다(예: CSRF 실패는 별도 안내).
- **`observeOncePerRequest` / 디스패치 제어** — `AuthorizationFilter`는 기본적으로 ERROR·ASYNC 디스패치에도 적용된다(`filterErrorDispatch`, `filterAsyncDispatch`). 포워드/인클루드 시 재인가 여부를 조정할 수 있다.

## 정리

인가는 `AuthorizationFilter`가 `AuthorizationManager`(특히 URL→규칙을 first-match로 위임하는 `RequestMatcherDelegatingAuthorizationManager`)에 위임해 결정하고, 거부 시 예외를 던진다. 그 예외는 한 칸 앞의 `ExceptionTranslationFilter`가 잡아, 익명/remember-me면 `AuthenticationEntryPoint`로 로그인을 유도하고 정식 인증 사용자면 `AccessDeniedHandler`로 403을 내린다. 거부된 원래 요청은 `RequestCache`에 저장되어 로그인 후 복귀에 쓰인다. "보안 강제"와 "사용자 안내"를 두 필터로 나눈 것이 이 설계의 핵심이다.
