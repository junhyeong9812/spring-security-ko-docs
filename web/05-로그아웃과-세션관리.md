# 05 · 로그아웃과 세션 관리 — 세션 생애주기를 보안으로 다스리기

## 무엇을 / 왜

세션은 인증을 요청 간에 유지해주는 편리한 그릇이지만, 동시에 공격면이기도 하다. 로그인 직후 세션 식별자를 바꾸지 않으면 **세션 고정(session fixation)** 공격에 노출되고, 한 계정으로 무제한 동시 로그인을 허용하면 통제가 어렵다. 또 로그아웃은 단순히 "인증을 지운다"가 아니라 세션 무효화·쿠키 삭제·CSRF 토큰 정리 등 여러 정리 작업의 묶음이다.

이 장은 두 축을 다룬다: **로그아웃**(`authentication/logout`)과 **세션 관리**(`authentication/session` + `session`).

## 핵심 타입

```
LogoutFilter
   ├── LogoutHandler (실제 정리 작업들, 순서대로 실행)
   │     ├── SecurityContextLogoutHandler   컨텍스트 비우기 + 세션 무효화
   │     ├── CookieClearingLogoutHandler     쿠키 삭제
   │     ├── CsrfLogoutHandler               CSRF 토큰 제거
   │     └── HeaderWriterLogoutHandler / LogoutSuccessEventPublishing...
   └── LogoutSuccessHandler (로그아웃 후 어디로)

SessionManagementFilter            요청 중 새 인증 감지 → 세션 전략 적용
SessionAuthenticationStrategy      인증 시점의 세션 처리 (전략)
   ├── SessionFixationProtectionStrategy / ChangeSessionIdAuthenticationStrategy
   ├── ConcurrentSessionControlAuthenticationStrategy   동시 세션 제한
   ├── RegisterSessionAuthenticationStrategy            세션 레지스트리 등록
   └── CompositeSessionAuthenticationStrategy           위 전략들 묶음
ConcurrentSessionFilter            만료된 세션 감지 → 강제 로그아웃
```

## 동작 흐름 1 — 로그아웃

`LogoutFilter.doFilter()`(`authentication/logout/LogoutFilter.java:99`)는 단순한 디스패처다.

```
POST /logout
   │ requiresLogout(req)?  ← logoutRequestMatcher (기본 "/logout")
   │   아니면 → chain.doFilter (통과)
   ▼ 맞으면
  auth = holder.getContext().getAuthentication()
  handler.logout(req, res, auth)        ← CompositeLogoutHandler 가 등록된 핸들러를 순서대로
  │   · SecurityContextLogoutHandler: clearContext + session.invalidate
  │   · CookieClearingLogoutHandler: 쿠키 만료
  │   · CsrfLogoutHandler: CSRF 토큰 제거
  logoutSuccessHandler.onLogoutSuccess(req, res, auth)   ← 보통 로그인 페이지로 리다이렉트
  return  (체인 진행 안 함)
```

핵심은 **로그아웃이 "핸들러들의 리스트"** 라는 점이다. `CompositeLogoutHandler`가 등록된 `LogoutHandler`들을 순서대로 실행하므로, 정리해야 할 항목이 늘어도(예: remember-me 쿠키, 외부 SSO 통지) 핸들러를 추가하면 된다. `SecurityContextLogoutHandler`가 컨텍스트를 비우고 세션을 무효화하는 가장 핵심적인 정리를 맡는다.

## 동작 흐름 2 — 세션 고정 보호 (인증 시점)

세션 처리는 두 갈래로 일어난다. (a) **인증이 막 성공한 순간** `SessionAuthenticationStrategy`가, (b) **그 외 경로로 인증이 들어온 경우** `SessionManagementFilter`가 보강한다.

```
[a] 인증 필터 내부 (3장)
   AbstractAuthenticationProcessingFilter
     attemptAuthentication 성공
       └─▶ sessionStrategy.onAuthentication(auth, req, res)
              SessionFixationProtectionStrategy:
                기존 세션 속성을 복사 → 새 세션 생성(또는 session id 변경)
                → 공격자가 미리 심어둔 세션 ID 무효화
              ConcurrentSessionControlAuthenticationStrategy:
                이 사용자의 활성 세션 수 검사
                → 초과 시 예외(SessionAuthenticationException) 또는 가장 오래된 세션 만료
              RegisterSessionAuthenticationStrategy:
                SessionRegistry 에 새 세션 등록

[b] SessionManagementFilter (인증 필터를 안 거친 인증, 예: pre-auth/SSO)
   요청 끝 무렵, "요청 시작 땐 인증 없었는데 지금은 있다"를 감지
     securityContextRepository.containsContext(req) == false 이고
     현재 컨텍스트에 비익명 인증이 있으면
       └─▶ sessionAuthenticationStrategy.onAuthentication(...) 을 대신 호출
```

`SessionFixationProtectionStrategy`(추상 부모 `AbstractSessionFixationProtectionStrategy`)는 로그인 직후 **세션 식별자를 새로 발급**한다. 공격자가 피해자에게 자신이 아는 세션 ID를 미리 사용하게 만든 뒤 그 세션을 탈취하는 고정 공격을, 인증 순간 ID를 갈아치워 무력화한다. 서블릿 3.1+에서는 `ChangeSessionIdAuthenticationStrategy`(세션은 유지하고 ID만 변경)가 기본이다.

## 동작 흐름 3 — 동시 세션 만료 (이후 요청)

동시 세션 제어로 "가장 오래된 세션을 만료"시키기로 했다면, 그 만료된 세션의 주인이 다음 요청을 보낼 때 쫓아내야 한다. 그 일을 `ConcurrentSessionFilter`가 한다.

```
만료 표시된 세션의 사용자가 다음 요청
   │ ConcurrentSessionFilter
   │   SessionRegistry 에서 이 세션이 expired 인지 확인
   │   expired 면 → 로그아웃 핸들러 실행 + SessionInformationExpiredStrategy
   │                (기본: "다른 곳에서 로그인되어 만료됨" 안내 후 리다이렉트)
   │   아니면 → registry.refreshLastRequest(sessionId) 로 활동 시각 갱신 후 통과
```

## 핵심 메서드

- **`CompositeLogoutHandler.logout`** — 등록된 `LogoutHandler`들을 등록 순서대로 호출. 로그아웃을 확장 가능한 파이프라인으로 만든다.
- **`SecurityContextLogoutHandler.logout`** — `SecurityContextHolder` 비우기 + (옵션) `HttpSession.invalidate()`. 로그아웃의 의미적 핵심.
- **`SessionAuthenticationStrategy.onAuthentication`** — 인증 직후 단 한 번 불리는 훅. 세션 고정 보호·동시 세션 제어·레지스트리 등록이 모두 이 한 메서드의 구현으로 끼워진다. `CompositeSessionAuthenticationStrategy`가 여러 전략을 순서대로 실행한다.
- **`SessionManagementFilter.doFilter`** — `securityContextRepository.containsContext(request)`로 "요청 시작 시점에 저장된 인증이 있었는가"를 검사해, 인증 필터를 거치지 않은 인증(pre-auth, remember-me 등)에도 세션 전략이 적용되도록 보강한다.

## 설계 포인트 / 확장점

- **전략 패턴의 교과서** — 세션 처리 방식(`SessionAuthenticationStrategy`), 로그아웃 정리(`LogoutHandler`), 로그아웃 후 목적지(`LogoutSuccessHandler`), 만료 안내(`SessionInformationExpiredStrategy`)가 모두 인터페이스로 추상화되어 조합·교체된다.
- **`SessionLimit`** — 동시 세션 허용 수를 사용자별로 동적으로 결정하는 함수형 훅(고정 숫자 대신 권한에 따라 가변).
- **이벤트** — `HttpSessionEventPublisher`가 세션 생성/소멸을 Spring 이벤트로 발행해 `SessionRegistry`가 활성 세션을 추적한다. 동시 세션 제어가 정확히 동작하려면 이 퍼블리셔 등록이 필요하다.
- **REST/무상태** — 세션을 안 쓰는 경우 `SessionCreationPolicy.STATELESS`와 함께 이들 필터를 제거하거나 무력화한다(2장의 무상태 저장소와 짝).

## 정리

세션 보안은 세 시점에 작동한다: 인증 직후 `SessionAuthenticationStrategy`가 세션 ID를 갱신(고정 보호)하고 동시 세션 수를 통제하며, `SessionManagementFilter`가 인증 필터를 우회한 인증에도 같은 전략을 보강하고, 이후 요청에서 `ConcurrentSessionFilter`가 만료된 세션의 사용자를 쫓아낸다. 로그아웃은 `LogoutFilter`가 `LogoutHandler` 리스트를 순서대로 실행하는 확장 가능한 정리 파이프라인이다. 모든 변형 지점이 전략 인터페이스로 빠져 있어 정책을 코드 수정 없이 조립할 수 있다.
