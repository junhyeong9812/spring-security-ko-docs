# 02 · 보안 컨텍스트 영속화 — 인증을 요청 사이에 유지하기

## 무엇을 / 왜

HTTP는 무상태(stateless)다. 그러나 한 번 로그인한 사용자는 매 요청마다 다시 인증하길 원치 않는다. 그래서 인증 결과(`Authentication`)를 담은 `SecurityContext`를 **요청이 끝날 때 어딘가에 저장**하고, **다음 요청 시작 때 다시 꺼내** `SecurityContextHolder`(ThreadLocal)에 올려야 한다. 이 "저장-복원" 책임이 `web/context` 패키지다.

여기서 핵심은 두 역할의 분리다: 컨텍스트를 ThreadLocal에 올리고/내리는 **필터**(`SecurityContextHolderFilter`)와, 실제 저장 매체를 추상화한 **저장소**(`SecurityContextRepository`).

## 핵심 타입

```
SecurityContextHolderFilter (필터, 체인 맨 앞)
   │ loadDeferredContext(request)  ← "지연 로딩" Supplier 획득
   ▼
SecurityContextRepository  (인터페이스)
   ├── HttpSessionSecurityContextRepository   ← HttpSession 에 저장 (기본)
   ├── RequestAttributeSecurityContextRepository ← 요청 1건 동안만 (무상태)
   ├── DelegatingSecurityContextRepository    ← 여러 저장소에 동시 위임
   ├── NullSecurityContextRepository          ← 저장 안 함
   └── ...
```

- **`SecurityContextHolderFilter`** (`context/SecurityContextHolderFilter.java`) — 요청 시작 시 저장소에서 컨텍스트를 (지연) 로드해 `SecurityContextHolder`에 세팅하고, 요청 끝에 비운다. **저장은 직접 하지 않는다** — 이게 옛 `SecurityContextPersistenceFilter`와의 결정적 차이다.
- **`SecurityContextRepository`** — `loadDeferredContext(request)`, `saveContext(...)`, `containsContext(request)`. 저장 매체를 캡슐화한다.
- **`HttpSessionSecurityContextRepository`** — 기본 저장소. `HttpSession`의 `"SPRING_SECURITY_CONTEXT"` 키에 컨텍스트를 둔다.
- **`RequestAttributeSecurityContextRepository`** — 요청 속성에만 저장. 세션을 쓰지 않는 무상태(stateless) 시나리오에서 한 요청 내 필터 간 공유 용도.
- **`DeferredSecurityContext` / `SupplierDeferredSecurityContext`** — "실제로 필요해질 때까지" 로딩을 미루는 래퍼.

## 동작 흐름 — 로드는 지연되고, 저장은 명시적이다

```
요청 시작
  │ SecurityContextHolderFilter.doFilter
  │   deferred = repo.loadDeferredContext(request)   // 아직 세션 안 건드림!
  │   holder.setDeferredContext(deferred)            // Supplier 만 심어둠
  ▼
  체인 진행 (다른 필터/컨트롤러)
  │   누군가 SecurityContextHolder.getContext() 호출하는 순간
  │   → Supplier 가 비로소 repo 에서 실제 로드 (1회, 캐시됨)
  ▼
  인증 성공한 필터(예: UsernamePasswordAuthenticationFilter)가
  │   직접 repo.saveContext(context, req, res) 호출   ← 저장은 여기서
  ▼
요청 끝
  │ finally: holder.clearContext()  // ThreadLocal 비움
```

### 왜 "지연(deferred)"인가

`SecurityContextHolderFilter.doFilter()`(`SecurityContextHolderFilter.java:72`)를 보면 `loadDeferredContext`로 받은 `Supplier`를 `setDeferredContext`로 심을 뿐, 즉시 세션을 읽지 않는다.

```java
Supplier<SecurityContext> deferredContext = this.securityContextRepository.loadDeferredContext(request);
try {
    this.securityContextHolderStrategy.setDeferredContext(deferredContext);
    chain.doFilter(request, response);
}
finally {
    this.securityContextHolderStrategy.clearContext();
    request.removeAttribute(FILTER_APPLIED);
}
```

대부분의 요청(정적 리소스, 공개 API)은 인증을 꺼내볼 일이 없다. 즉시 로딩하면 그런 요청마다 불필요하게 `HttpSession`을 조회·생성하게 된다. 지연 로딩은 **`getContext()`가 실제 호출되는 순간에만** 세션을 건드리므로, 세션을 안 쓰는 요청은 세션을 만들지 않는다.

### 왜 저장이 "명시적"인가

옛 `SecurityContextPersistenceFilter`는 요청이 끝날 때 컨텍스트를 **자동으로** 세션에 다시 썼다. 반면 `SecurityContextHolderFilter`는 저장하지 않는다. 대신 **인증을 만든 주체**(폼 로그인 필터, Basic 필터 등)가 성공 시점에 직접 `securityContextRepository.saveContext(...)`를 호출한다(3장의 `successfulAuthentication` 참고).

이로써 "인증을 갱신한 메커니즘만 저장을 책임진다"는 명확한 규칙이 생기고, 매 요청 끝마다 변경 없는 컨텍스트를 다시 쓰는 낭비가 사라진다.

## 핵심 메서드

- **`SecurityContextRepository.loadDeferredContext` (인터페이스 default, `SecurityContextRepository.java:80`)** — 구버전 `loadContext(holder)`를 `SingletonSupplier`로 감싸 한 번만 실행되게 만든다. 신규 구현은 이 메서드를 직접 오버라이드해 응답 래핑 없이 효율적으로 로드한다.
- **`HttpSessionSecurityContextRepository.saveContext`** — 세션의 기존 컨텍스트와 비교해, 인증이 사라졌으면 세션 속성을 제거하고, 새 컨텍스트면 저장한다. 빈(익명) 컨텍스트는 저장하지 않아 불필요한 세션 생성을 피한다.
- **`HttpSessionSecurityContextRepository.containsContext`** — 세션에 컨텍스트 키가 있는지로 "이 요청에 저장된 인증이 있는가"를 답한다. `SessionManagementFilter` 등이 "요청 도중 새로 인증되었는지"를 판단할 때 쓴다.

## 설계 포인트 / 확장점

- **`DelegatingSecurityContextRepository`** — 예컨대 `HttpSessionSecurityContextRepository` + `RequestAttributeSecurityContextRepository`를 함께 등록해, 세션과 요청 속성 양쪽에서 로드/저장하도록 한다. config의 기본 조합이다.
- **무상태(stateless) API** — `SessionCreationPolicy.STATELESS`로 두면 `NullSecurityContextRepository` 또는 요청 속성 저장소를 써서 세션을 아예 만들지 않는다. JWT/OAuth2 리소스 서버처럼 매 요청 토큰으로 인증하는 경우에 맞다.
- **커스텀 저장소** — Redis 같은 외부 세션 저장소로 옮기고 싶으면 `SecurityContextRepository`를 구현해 끼운다. 필터들은 인터페이스만 알기 때문에 저장 매체 교체가 투명하다.

## 정리

`SecurityContextHolderFilter`는 보안 체인의 가장 앞에서 "이 요청의 인증을 ThreadLocal에 올렸다가 끝에 내리는" 일만 한다. 실제 저장 매체는 `SecurityContextRepository` 뒤에 숨고, 로드는 필요할 때까지 지연되며, 저장은 인증을 만든 필터가 명시적으로 책임진다. 이 "지연 로드 · 명시 저장" 모델 덕분에 인증이 필요 없는 요청은 세션을 건드리지 않고, 인증 갱신의 책임 소재도 분명해진다.
