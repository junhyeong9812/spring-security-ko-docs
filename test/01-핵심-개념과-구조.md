# 01. 핵심 개념과 구조 — 가짜 인증을 어디에, 어떻게 꽂는가

## 무엇을 / 왜

보안이 적용된 코드를 테스트하려면 결국 한 질문에 답해야 한다. **"내 코드가 `Authentication` 을 읽는 그 자리에, 어떻게 원하는 사용자를 미리 넣어 둘 것인가?"** 이 챕터는 그 "자리"에 해당하는 `TestSecurityContextHolder`, 그것을 채우는 메타 애너테이션 `WithSecurityContext`, 그리고 채우는 행위를 테스트 생명주기에 끼워 넣는 `TestExecutionListener` 의 자동 등록까지, 모듈의 뼈대를 먼저 잡는다.

## 핵심 타입

```
        ┌────────────────────────────────────────────────────────────┐
        │                    SecurityContextHolder                    │  (core 모듈)
        │            ThreadLocal<SecurityContext>  ← 운영 코드가 읽음    │
        └───────────────▲────────────────────────────▲───────────────┘
                        │ setContext / clearContext   │
        ┌───────────────┴───────────────┐            │ getContext() 위임
        │     TestSecurityContextHolder   │────────────┘
        │  ThreadLocal<SecurityContext>   │  ← 테스트가 "정답"을 보관하는 곳
        └───────────────▲────────────────┘
                        │ getContext/setContext/clearContext
        ┌───────────────┴────────────────────────────┐
        │   TestSecurityContextHolderStrategyAdapter   │  ← 위를 Strategy 로 감쌈
        │            implements SecurityContextHolderStrategy           │
        └─────────────────────────────────────────────┘
```

### TestSecurityContextHolder — 왜 별도의 홀더가 필요한가

`TestSecurityContextHolder`(`test/.../context/TestSecurityContextHolder.java`)는 자체 `ThreadLocal<SecurityContext>` 를 들고 있으면서, 값을 세팅할 때마다 **`SecurityContextHolder` 에도 같이 써 준다.**

```java
public static void setContext(SecurityContext context) {
    Assert.notNull(context, "Only non-null SecurityContext instances are permitted");
    contextHolder.set(context);          // 자기 ThreadLocal
    SecurityContextHolder.setContext(context);  // 운영용 ThreadLocal 에도 미러링
}
```

그런데 운영 `SecurityContextHolder` 가 이미 있는데 왜 굳이 별도 홀더를 둘까? 클래스 주석이 직접 답한다. **웹 요청에서는 `SecurityContextPersistenceFilter` 가 `SecurityContextRepository` 가 돌려준 값으로 `SecurityContextHolder` 를 덮어쓰고, 필터 체인 끝에서 비워 버린다.** 즉 `SecurityContextHolder` 하나만 믿으면 MockMvc 로 요청을 여러 번 보낼 때 "원래 어떤 사용자였는지"를 잃는다. `TestSecurityContextHolder` 는 그 "정답"을 필터와 무관하게 따로 보존하는 백업본이다.

읽기 동작도 영리하다. 자기 `ThreadLocal` 이 비어 있으면 `SecurityContextHolder.getContext()` 의 기본값으로 폴백한다.

```java
public static SecurityContext getContext() {
    SecurityContext ctx = contextHolder.get();
    if (ctx == null) {
        ctx = getDefaultContext();        // == SecurityContextHolder.getContext()
        contextHolder.set(ctx);
    }
    return ctx;
}
```

`clearContext()` 는 자기 ThreadLocal 과 `SecurityContextHolder` 를 **둘 다** 비운다. 테스트 간 누수를 막는 핵심이다.

### TestSecurityContextHolderStrategyAdapter — Strategy 로 감싸는 이유

`SecurityContextHolderStrategy` 는 core 모듈이 정의한 "보안 컨텍스트 저장소 전략" 인터페이스다. 모듈 곳곳(리스너, RequestPostProcessor)은 이 인터페이스에만 의존하도록 설계됐는데, 기본 구현으로 `TestSecurityContextHolder` 를 그대로 위임하는 어댑터를 둔다.

```java
public final class TestSecurityContextHolderStrategyAdapter implements SecurityContextHolderStrategy {
    public SecurityContext getContext()           { return TestSecurityContextHolder.getContext(); }
    public void setContext(SecurityContext c)      { TestSecurityContextHolder.setContext(c); }
    public void clearContext()                     { TestSecurityContextHolder.clearContext(); }
    public SecurityContext createEmptyContext()    { return SecurityContextHolder.createEmptyContext(); }
}
```

덕분에 애플리케이션 컨텍스트에 **커스텀 `SecurityContextHolderStrategy` 빈이 등록돼 있으면 그걸 쓰고**, 없으면 이 어댑터로 폴백하는 분기가 자연스럽게 가능해진다(02·03 챕터의 `securityContextHolderStrategyConverter` 참고).

### WithSecurityContext — 모든 @With* 의 메타 애너테이션

`WithSecurityContext`(`.../context/support/WithSecurityContext.java`)는 사용자가 직접 붙이는 애너테이션이 아니라, `@WithMockUser` 같은 애너테이션 **위에 다시 붙는 메타 애너테이션**이다. 두 가지 정보를 담는다.

```java
@Target(ElementType.ANNOTATION_TYPE)   // 다른 애너테이션에만 붙일 수 있음
public @interface WithSecurityContext {
    Class<? extends WithSecurityContextFactory<? extends Annotation>> factory();  // 변환 전략
    TestExecutionEvent setupBefore() default TestExecutionEvent.TEST_METHOD;       // 주입 타이밍
}
```

- `factory()` — 이 애너테이션을 어떤 `WithSecurityContextFactory` 로 `SecurityContext` 로 바꿀지.
- `setupBefore()` — 언제 심을지(`TEST_METHOD` vs `TEST_EXECUTION`, 02 챕터에서 상세).

예컨대 `@WithMockUser` 의 선언부는 다음과 같다.

```java
@WithSecurityContext(factory = WithMockUserSecurityContextFactory.class)
public @interface WithMockUser { ... }
```

### WithSecurityContextFactory — 변환 전략 인터페이스

```java
public interface WithSecurityContextFactory<A extends Annotation> {
    SecurityContext createSecurityContext(A annotation);
}
```

애너테이션 인스턴스 하나를 받아 `SecurityContext` 하나를 만들어 돌려주는 단일 책임 인터페이스다. 표준 구현 세 개(`WithMockUser/UserDetails/AnonymousUser` 용)가 있고, 사용자가 직접 구현해 자기만의 `@WithXxx` 를 만들 수도 있다(02 챕터 확장점).

## 동작 흐름: 리스너 자동 등록

이 모든 장치가 동작하려면 누군가 테스트 실행 직전에 애너테이션을 읽고 컨텍스트를 심어 줘야 한다. 그 "누군가"가 두 개의 `TestExecutionListener` 이고, Spring Framework 의 `spring.factories` 메커니즘으로 **자동 등록**된다.

`test/src/main/resources/META-INF/spring.factories`:

```
org.springframework.test.context.TestExecutionListener = \
    org.springframework.security.test.context.support.WithSecurityContextTestExecutionListener,\
    org.springframework.security.test.context.support.ReactorContextTestExecutionListener
```

```
   JUnit 실행
      │
      ▼
   SpringExtension / TestContextManager
      │   spring.factories 를 스캔해 TestExecutionListener 들을 수집
      ▼
   ┌─────────────────────────────────────────────┐
   │ 등록되는 보안 리스너 (getOrder 로 정렬)         │
   │  • WithSecurityContextTestExecutionListener   │  order 10000  → 컨텍스트를 심음
   │  • ReactorContextTestExecutionListener        │  order 11000  → Reactor Context 로 전파
   └─────────────────────────────────────────────┘
```

`getOrder()` 값(10000, 11000)이 핵심이다. `WithSecurityContextTestExecutionListener`(10000)가 **먼저** 실행되어 `TestSecurityContextHolder` 를 채우고, 그 다음 `ReactorContextTestExecutionListener`(11000)가 채워진 컨텍스트를 Reactor `Context` 로 옮긴다. 순서가 바뀌면 리액티브 전파가 빈 컨텍스트를 집어 가게 된다.

### 애플리케이션 컨텍스트가 필요 없을 때 — @SecurityTestExecutionListeners

`@SpringBootTest` 처럼 풀 컨텍스트를 띄우는 테스트는 위 자동 등록이 그대로 적용된다. 반면 `ApplicationContext` 자체가 필요 없는 가벼운 단위 테스트에서 `@WithMockUser` 만 쓰고 싶다면, 클래스에 `@SecurityTestExecutionListeners` 를 붙여 **보안 리스너 둘만** 켤 수 있다.

```java
@TestExecutionListeners(inheritListeners = false,
    listeners = { WithSecurityContextTestExecutionListener.class,
                  ReactorContextTestExecutionListener.class })
public @interface SecurityTestExecutionListeners { }
```

`inheritListeners = false` 로 다른 표준 리스너(의존성 주입, 트랜잭션 등)를 끄고 보안 리스너만 남긴다.

## 핵심 메서드 정리

- `TestSecurityContextHolder.setContext` / `getContext` / `clearContext` — 테스트가 보안 컨텍스트를 보관·조회·정리하는 단일 출입구. `SecurityContextHolder` 와 항상 동기화된다.
- `TestSecurityContextHolder.setAuthentication(Authentication)` — `Authentication` 하나만으로 빈 컨텍스트를 만들어 세팅하는 편의 메서드(since 5.1.1).
- `WithSecurityContextFactory.createSecurityContext(A)` — 애너테이션을 컨텍스트로 바꾸는 단 하나의 추상 메서드.

## 설계 포인트 / 확장점

- **메타 애너테이션 + 팩터리 분리**: "언제/어떤 권한으로"(애너테이션)와 "어떻게 컨텍스트를 만드나"(팩터리)를 분리해, 새 `@WithXxx` 를 추가할 때 리스너를 건드릴 필요가 없다.
- **Strategy 폴백**: 컨텍스트에 커스텀 `SecurityContextHolderStrategy` 빈이 있으면 그것을, 없으면 `TestSecurityContextHolderStrategyAdapter` 를 쓴다. 테스트 환경 유무에 따라 우아하게 분기한다.
- **AOT 힌트**: 네이티브 이미지에서도 팩터리를 리플렉션으로 생성할 수 있도록 `WithSecurityContextTestRuntimeHints` 가 `@With*` 의 `factory()` 클래스 생성자를 힌트로 등록한다.

## 정리

이 모듈의 뼈대는 세 조각이다. (1) 테스트가 "정답 사용자"를 보관하는 `TestSecurityContextHolder`, (2) 그 정답을 어떻게 만들지 선언하는 `WithSecurityContext` 메타 애너테이션 + `WithSecurityContextFactory`, (3) 테스트 실행 직전에 (1)을 (2)로 채우도록 `spring.factories` 로 자동 등록되는 두 개의 `TestExecutionListener`. 다음 챕터부터 이 조각들이 실제로 어떻게 맞물려 돌아가는지 살펴본다.
