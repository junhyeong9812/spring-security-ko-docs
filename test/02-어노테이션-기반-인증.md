# 02. 어노테이션 기반 인증 — @WithMockUser / @WithUserDetails / @WithAnonymousUser

## 무엇을 / 왜

서비스 계층의 메서드 보안(`@PreAuthorize("hasRole('ADMIN')")`)이나 인증을 전제로 한 비즈니스 로직을 테스트할 때는 HTTP 요청을 흉내 낼 필요조차 없다. **테스트 메서드가 실행되는 동안 `SecurityContextHolder` 에 원하는 사용자가 들어 있기만 하면 된다.** 이 챕터는 그 일을 선언적으로 해 주는 세 애너테이션과, 그것을 실제 컨텍스트로 바꿔 테스트 실행 직전에 심어 넣는 `WithSecurityContextTestExecutionListener` 의 동작을 추적한다.

## 핵심 타입

```
   @WithMockUser ──┐ factory = WithMockUserSecurityContextFactory
   @WithUserDetails─┤ factory = WithUserDetailsSecurityContextFactory   (모두 @WithSecurityContext 메타)
   @WithAnonymousUser┘ factory = WithAnonymousUserSecurityContextFactory
            │
            │  WithSecurityContextTestExecutionListener 가 읽음
            ▼
   WithSecurityContextFactory<A>.createSecurityContext(annotation)
            │
            ▼
   SecurityContext  ──▶  TestSecurityContextHolder.setContext(...)
```

### 세 애너테이션의 차이

| 애너테이션 | 만드는 Authentication | 사용자 출처 |
|-----------|----------------------|------------|
| `@WithMockUser` | `UsernamePasswordAuthenticationToken` | 애너테이션 속성으로 **즉석에서 생성**(존재하지 않아도 됨) |
| `@WithUserDetails` | `UsernamePasswordAuthenticationToken` | `UserDetailsService` 빈에서 **실제 조회** |
| `@WithAnonymousUser` | `AnonymousAuthenticationToken` | 익명 사용자 |

세 애너테이션 모두 `@Target({METHOD, TYPE})` 이라 **메서드 또는 클래스**에 붙일 수 있고, `@Inherited` 다. 메서드에 붙은 것이 클래스에 붙은 것보다 우선한다(뒤에서 설명).

### @WithMockUser — 즉석 사용자

```java
@WithSecurityContext(factory = WithMockUserSecurityContextFactory.class)
public @interface WithMockUser {
    String value()        default "user";       // username 단축 표기
    String username()     default "";           // 지정 시 value 보다 우선
    String[] roles()      default { "USER" };   // 자동으로 "ROLE_" 접두
    String[] authorities() default {};          // 지정 시 roles 무시, 접두 없음
    String password()     default "password";
    @AliasFor(annotation = WithSecurityContext.class)
    TestExecutionEvent setupBefore() default TestExecutionEvent.TEST_METHOD;
}
```

팩터리는 이 속성들을 그대로 권한으로 번역한다(`WithMockUserSecurityContextFactory`).

```java
String username = StringUtils.hasLength(withUser.username()) ? withUser.username() : withUser.value();
// authorities 가 있으면 그대로, 없으면 roles 각각에 "ROLE_" 접두
for (String authority : withUser.authorities())
    grantedAuthorities.add(new SimpleGrantedAuthority(authority));
if (grantedAuthorities.isEmpty())
    for (String role : withUser.roles()) {
        Assert.isTrue(!role.startsWith("ROLE_"), () -> "roles cannot start with ROLE_ Got " + role);
        grantedAuthorities.add(new SimpleGrantedAuthority("ROLE_" + role));
    }
User principal = new User(username, withUser.password(), true, true, true, true, grantedAuthorities);
Authentication authentication = UsernamePasswordAuthenticationToken.authenticated(
        principal, principal.getPassword(), principal.getAuthorities());
SecurityContext context = this.securityContextHolderStrategy.createEmptyContext();
context.setAuthentication(authentication);
return context;
```

두 가지 함정을 코드로 막아 둔 점에 주목하자. (1) `roles()` 값이 `ROLE_` 로 시작하면 즉시 예외 — 이중 접두를 방지한다. (2) `authorities()` 와 `roles()` 를 **둘 다** 비기본값으로 주면 `IllegalStateException` — 둘은 상호배타다.

### @WithUserDetails — 실제 UserDetailsService 조회

```java
@WithSecurityContext(factory = WithUserDetailsSecurityContextFactory.class)
public @interface WithUserDetails {
    String value() default "user";                         // 조회할 username
    String userDetailsServiceBeanName() default "";        // 특정 빈 이름(없으면 타입으로)
    @AliasFor(annotation = WithSecurityContext.class)
    TestExecutionEvent setupBefore() default TestExecutionEvent.TEST_METHOD;
}
```

팩터리는 `BeanFactory` 에서 `UserDetailsService` 를 찾아 `loadUserByUsername(value)` 를 호출한다. 빈 이름이 주어지면 이름으로, 아니면 타입으로 찾는다.

```java
UserDetailsService userDetailsService = findUserDetailsService(beanName);
UserDetails principal = userDetailsService.loadUserByUsername(withUser.value());
Authentication authentication = UsernamePasswordAuthenticationToken.authenticated(
        principal, principal.getPassword(), principal.getAuthorities());
```

여기엔 리액티브 배려도 있다. Reactor 가 클래스패스에 있으면 먼저 `ReactiveUserDetailsService` 빈을 찾고, 있으면 `findByUsername(...).block()` 으로 블로킹 어댑팅한다(`ReactiveUserDetailsServiceAdapter`). 없으면 일반 `UserDetailsService` 로 폴백한다. 즉 **WebFlux 앱에서도 `@WithUserDetails` 가 동작**한다.

> `@WithMockUser` 와 달리 `@WithUserDetails` 는 `BeanFactory` 가 필요하므로 `ApplicationContext` 가 있는 테스트에서만 의미가 있다.

### @WithAnonymousUser — 익명 override

이미 클래스 레벨에 `@WithMockUser` 를 깔아 두고 특정 메서드만 익명으로 돌리고 싶을 때 쓴다. 팩터리는 고정된 `AnonymousAuthenticationToken` 을 만든다.

```java
List<GrantedAuthority> authorities = AuthorityUtils.createAuthorityList("ROLE_ANONYMOUS");
Authentication authentication = new AnonymousAuthenticationToken("key", "anonymous", authorities);
```

## 동작 흐름: "@WithMockUser 가 붙으면 테스트 실행 전 무슨 일이 일어나는가"

핵심 무대는 `WithSecurityContextTestExecutionListener.beforeTestMethod(...)` 다(`.../context/support/WithSecurityContextTestExecutionListener.java`).

```
 JUnit: 테스트 메서드 호출 직전
   │
   ▼
 WithSecurityContextTestExecutionListener.beforeTestMethod(testContext)
   │
   ├─ 1) createTestSecurityContext(테스트 "메서드", ctx)
   │       AnnotatedElementUtils.findMergedAnnotation(method, WithSecurityContext.class)
   │       → 메서드에서 못 찾으면 null
   │
   ├─ 2) (메서드에 없으면) createTestSecurityContext(테스트 "클래스", ctx)
   │       TestContextAnnotationUtils 로 클래스/부모/메타 탐색
   │
   │   둘 다 못 찾으면 → return (아무 컨텍스트도 안 심음)
   │
   ├─ 3) WithSecurityContext.factory() 로 팩터리 인스턴스 생성
   │       createBean(factoryClass)  ← @Autowired 주입 가능
   │       실패하면 BeanUtils.instantiateClass(...) 로 폴백
   │
   ├─ 4) 팩터리의 제네릭 타입 A(예: WithMockUser)를 알아내
   │       그 A 애너테이션을 다시 찾아 (findAnnotation)
   │       Supplier<SecurityContext> = () -> factory.createSecurityContext(annotation)
   │
   └─ 5) setupBefore 분기
         ├ TEST_METHOD     → 지금 즉시 strategy.setContext(supplier.get())
         └ TEST_EXECUTION  → testContext 에 supplier 만 저장해 두고
                              나중에 beforeTestExecution() 에서 set
```

### setupBefore: TEST_METHOD vs TEST_EXECUTION

`TestExecutionEvent` 는 컨텍스트를 **언제** 심을지 고른다.

```java
public enum TestExecutionEvent {
    TEST_METHOD,     // beforeTestMethod 시점 (기본)
    TEST_EXECUTION   // beforeTestExecution 시점 (더 늦음)
}
```

왜 둘로 나뉠까? `@WithUserDetails` 처럼 팩터리가 빈(예: 트랜잭션으로 준비되는 DB 데이터)에 의존하면, `beforeTestMethod` 시점에는 아직 다른 리스너(예: SQL 셋업, 트랜잭션 시작)가 돌지 않았을 수 있다. `TEST_EXECUTION` 으로 미루면 그 셋업 이후에 컨텍스트를 만들 수 있다. 코드상 `TEST_METHOD` 면 즉시 `setContext`, `TEST_EXECUTION` 이면 `Supplier` 를 `SECURITY_CONTEXT_ATTR_NAME` 속성에 보관했다가 `beforeTestExecution` 에서 꺼내 실행한다.

```java
@Override public void beforeTestExecution(TestContext testContext) {
    Supplier<SecurityContext> supplier =
        (Supplier<SecurityContext>) testContext.removeAttribute(SECURITY_CONTEXT_ATTR_NAME);
    if (supplier != null)
        this.securityContextHolderStrategyConverter.convert(testContext).setContext(supplier.get());
}
```

### 어떤 SecurityContextHolderStrategy 에 심는가

리스너는 `securityContextHolderStrategyConverter` 로 전략을 고른다. 애플리케이션 컨텍스트가 없거나 `SecurityContextHolderStrategy` 빈이 없으면 **`TestSecurityContextHolderStrategyAdapter`**(즉 `TestSecurityContextHolder`)를, 있으면 그 빈을 쓴다.

```java
Converter<TestContext, SecurityContextHolderStrategy> securityContextHolderStrategyConverter = (testContext) -> {
    if (!testContext.hasApplicationContext()) return DEFAULT_SECURITY_CONTEXT_HOLDER_STRATEGY;
    ApplicationContext context = testContext.getApplicationContext();
    if (context.getBeanNamesForType(SecurityContextHolderStrategy.class).length == 0)
        return DEFAULT_SECURITY_CONTEXT_HOLDER_STRATEGY;
    return context.getBean(SecurityContextHolderStrategy.class);
};
```

### 테스트가 끝나면 — 정리

```java
@Override public void afterTestMethod(TestContext testContext) {
    this.securityContextHolderStrategyConverter.convert(testContext).clearContext();
}
```

`clearContext()` 는 `TestSecurityContextHolder` 와 `SecurityContextHolder` 를 모두 비운다. 덕분에 다음 테스트로 사용자가 새지 않는다.

## 핵심 메서드

- `beforeTestMethod(TestContext)` — 메서드 → 클래스 순으로 `@WithSecurityContext` 를 찾아 팩터리를 돌려 컨텍스트를 만들고(또는 미루고) 심는다. **메서드 우선**이 여기서 결정된다.
- `createFactory(...)` — `createBean` 으로 팩터리를 생성해 `@Autowired`(예: `BeanFactory`, `SecurityContextHolderStrategy`)를 주입한다. 컨텍스트가 없으면 `BeanUtils.instantiateClass` 로 폴백.
- `findAnnotation(...)` — 팩터리의 제네릭 타입 `A` 에 해당하는 실제 애너테이션 인스턴스를 (메타 애너테이션을 뚫고) 찾아낸다. 커스텀 합성 애너테이션을 지원하는 핵심.

## 설계 포인트 / 확장점

- **나만의 `@WithXxx` 만들기**: `WithSecurityContextFactory<MyAnno>` 를 구현하고, 새 애너테이션에 `@WithSecurityContext(factory = MyFactory.class)` 를 메타로 붙이면 끝이다. 리스너는 제네릭 타입을 보고 알아서 연결한다. 팩터리에는 `@Autowired` 로 빈을 주입받을 수 있다.
- **메서드 > 클래스 override**: 클래스에 기본 사용자를 깔고 메서드에서 `@WithAnonymousUser` 등으로 덮어쓰는 패턴이 그대로 동작한다.
- **`@AliasFor` 로 setupBefore 노출**: 각 애너테이션이 `setupBefore` 를 메타 애너테이션의 동명 속성에 별칭으로 연결해, 사용자가 타이밍을 직접 조정할 수 있게 한다.
- **AOT**: `WithSecurityContextTestRuntimeHints` 가 테스트 클래스/메서드의 `@WithSecurityContext.factory()` 생성자를 네이티브 리플렉션 힌트로 등록한다.

## 정리

세 애너테이션은 모두 `@WithSecurityContext(factory = ...)` 메타로 환원되며, `WithSecurityContextTestExecutionListener` 가 테스트 실행 직전(`TEST_METHOD`) 또는 직전 직전(`TEST_EXECUTION`)에 팩터리를 돌려 `SecurityContext` 를 만들어 `TestSecurityContextHolder` 에 심고, 끝나면 비운다. `@WithMockUser` 는 즉석 사용자, `@WithUserDetails` 는 실제 `UserDetailsService` 조회, `@WithAnonymousUser` 는 익명 — 차이는 오직 팩터리 한 줄에 있다.
