# 05. 웹 · 채널 · 메시징 보안

## 무엇을 / 왜

02장의 가로채기 엔진을 메서드가 아닌 **HTTP 요청**, **HTTP/HTTPS 채널**, **WebSocket 메시지**에 적용한 부품들을 모은다. 보안 객체 타입과 "실행"의 의미만 다를 뿐, 메타데이터 조회 → 투표 결정이라는 골격은 그대로다. 패키지로는 `web/access/intercept`, `web/access/expression`, `web/access/channel`, `messaging/access/**`.

## 핵심 타입 — 세 영역의 보안 객체

```
영역        보안 객체          인터셉터/필터                   메타데이터 소스
─────────  ────────────────  ─────────────────────────────  ──────────────────────────────
HTTP 인가   FilterInvocation  FilterSecurityInterceptor       FilterInvocationSecurityMetadataSource
                              (implements Filter)              → DefaultFilterInvocation…
                                                               → ExpressionBased…(SpEL)
HTTP 채널   FilterInvocation  ChannelProcessingFilter          FilterInvocationSecurityMetadataSource
                              (+ ChannelDecisionManager)       (REQUIRES_SECURE_CHANNEL 등)
메시지      Message<?>        ChannelSecurityInterceptor       MessageSecurityMetadataSource
                              (implements ChannelInterceptor)  → DefaultMessage… / Expression…
```

## 동작 흐름 — HTTP 요청 인가 (FilterSecurityInterceptor)

`FilterSecurityInterceptor`는 서블릿 `Filter`로서 필터 체인 끝부분에 위치하며, 요청을 `FilterInvocation`으로 감싸 02장 엔진에 넘긴다.

```
요청 ──▶ … SecurityFilterChain …
            ▼
   FilterSecurityInterceptor.doFilter(req, res, chain)
     fi = new FilterInvocation(req, res, chain)
        ▼ invoke(fi)
     token = super.beforeInvocation(fi)     ← 02장: 메타데이터 + 인가
        │   metadataSource.getAttributes(fi) → [WebExpressionConfigAttribute(...)]
        │   accessDecisionManager.decide(...) → WebExpressionVoter 투표
        ▼
     chain.doFilter(req, res)               ← 인가 통과 시 다음 필터/리소스로
        ▼ finally
     super.finallyInvocation(token)
     super.afterInvocation(token, null)
```

`invoke()`(소스 105~124행)는 `observeOncePerRequest` 플래그로 "요청당 한 번"을 제어한다. 켜져 있고 이미 처리된 요청이면 인가를 건너뛰고 체인만 진행시킨다(JSP forward 시 중복 인가 방지).

### 표현식 매칭 — WebExpressionVoter

`access("hasRole('ADMIN') and hasIpAddress('10.0.0.0/16')")` 같은 SpEL 권한은 `WebExpressionConfigAttribute`로 저장되고, `WebExpressionVoter`가 투표한다(`web/access/expression/WebExpressionVoter.java` 48~68행).

```java
WebExpressionConfigAttribute attr = findConfigAttribute(attributes); // instanceof로 자기 소관 확인
if (attr == null) return ACCESS_ABSTAIN;
EvaluationContext ctx = attr.postProcess(
        expressionHandler.createEvaluationContext(authentication, filterInvocation), filterInvocation);
boolean granted = ExpressionUtils.evaluateAsBoolean(attr.getAuthorizeExpression(), ctx);
return granted ? ACCESS_GRANTED : ACCESS_DENIED;
```

평가 컨텍스트(루트 객체)는 `DefaultWebSecurityExpressionHandler`가 만든다. 이 핸들러는 `FilterInvocationExpressionRoot`를 생성해 `hasRole`/`hasIpAddress`/`permitAll` 같은 웹 전용 함수와 `request` 변수를 노출하고, `AuthorizationManagerFactory`·`PermissionEvaluator`·역할 접두사를 주입한다(`DefaultWebSecurityExpressionHandler.java` 43~55행). 즉 표현식의 `hasRole(...)`은 내부적으로 현대 `AuthorizationManager`로 위임된다.

## 동작 흐름 — HTTP 채널 보안 (ChannelProcessingFilter)

"이 URL은 반드시 HTTPS로" 같은 전송 채널 요구를 강제한다. 인가가 아니라 **채널(평문/암호화) 결정**이라 별도 매니저(`ChannelDecisionManager`)를 쓴다.

```
ChannelProcessingFilter.doFilter(req, res, chain)
   fi = new FilterInvocation(req, res, chain)
   attributes = metadataSource.getAttributes(fi)   // 예: REQUIRES_SECURE_CHANNEL
   if (attributes != null)
       channelDecisionManager.decide(fi, attributes)
          └─ ChannelProcessor들을 순회 (Secure/Insecure ChannelProcessor)
               SecureChannelProcessor: 요청이 HTTPS 아니면
                  → ChannelEntryPoint(RetryWithHttpsEntryPoint)로 redirect
       if (response.isCommitted()) return;   // 리다이렉트했으면 체인 중단
   chain.doFilter(req, res)
```

`SecureChannelProcessor`/`InsecureChannelProcessor`가 각각 `REQUIRES_SECURE_CHANNEL`/`REQUIRES_INSECURE_CHANNEL` 속성을 보고, 현재 요청 채널이 요구와 다르면 `RetryWithHttpsEntryPoint`/`RetryWithHttpEntryPoint`로 같은 URL을 적절한 스킴으로 리다이렉트한다. 응답이 커밋되면 필터 체인은 거기서 멈춘다. 현대에는 `HttpsRedirectFilter`가 이 역할을 단순화해 대체한다.

## 동작 흐름 — 메시지 보안 (ChannelSecurityInterceptor)

WebSocket/STOMP 메시지를 보호한다. 보안 객체는 `Message<?>`이고, 인터셉터는 Spring Messaging의 `ChannelInterceptor`로 동작한다. 02장 엔진의 before/after를 메시징 콜백에 나눠 배치하는 점이 특징이다(`messaging/access/intercept/ChannelSecurityInterceptor.java`).

```
preSend(msg, channel)           → token = beforeInvocation(msg)  [인가]
                                   token을 ThreadLocal에 보관
   (메시지가 채널로 전송됨 = "실행")
postSend(msg, channel, sent)    → afterInvocation(token, null)
afterSendCompletion(...)        → finallyInvocation(token)        [컨텍스트 정리]
```

메서드/필터 인터셉터가 한 메서드 안에서 before/실행/after를 감싸는 것과 달리, 메시징은 send 라이프사이클 콜백이 분리돼 있어 `InterceptorStatusToken`을 `ThreadLocal`로 넘긴다. 메타데이터는 목적지 패턴별 권한을 담은 `MessageSecurityMetadataSource`(SpEL이면 `MessageExpressionVoter`가 투표)가 제공한다.

## 핵심 메서드 / 부품

- **`DefaultFilterInvocationSecurityMetadataSource`** — URL 패턴(RequestMatcher) → `ConfigAttribute` 매핑. 요청 URL을 매처로 찾아 해당 권한 목록을 반환.
- **`ExpressionBasedFilterInvocationSecurityMetadataSource`** — 위의 SpEL 버전. 패턴별 표현식을 `WebExpressionConfigAttribute`로 미리 파싱해 보관.
- **`DefaultWebInvocationPrivilegeEvaluator`** — "이 사용자가 이 URL에 접근 가능한가?"를 **실제 요청 없이** 사전 평가한다(예: 메뉴/링크 노출 여부 판단). 내부적으로 `FilterSecurityInterceptor`의 메타데이터+매니저를 재사용. 메서드판 대응물은 `intercept/MethodInvocationPrivilegeEvaluator`.

## 설계 포인트 / 확장점

- **동일 엔진, 다른 보안 객체**: HTTP 인가와 메시지 보안은 모두 `AbstractSecurityInterceptor`를 상속한다. 차이는 `getSecureObjectClass()`(FilterInvocation vs Message)와 "실행" 지점뿐이다. 채널 보안만 인가가 아닌 전송 결정이라 별도 매니저를 쓴다.
- **표현식 핸들러 위임**: 웹 표현식의 `hasRole` 등은 결국 `AuthorizationManagerFactory`를 통해 현대 인가 엔진으로 내려간다 — 레거시 표면 아래에서 신구가 이미 맞물려 있다.
- **현대 대체**: HTTP 인가는 `AuthorizationFilter`+`WebExpressionAuthorizationManager`, 채널은 `HttpsRedirectFilter`, 메시지는 `AuthorizationChannelInterceptor`로 각각 대체됐다.

## 정리

웹·채널·메시징 보안은 02장 엔진을 서로 다른 보안 객체에 끼운 어댑터들이다. `FilterSecurityInterceptor`는 `FilterInvocation`을 인가하고 `WebExpressionVoter`로 SpEL을 평가하며, `ChannelProcessingFilter`는 HTTPS 같은 전송 채널을 `ChannelDecisionManager`로 강제하고, `ChannelSecurityInterceptor`는 `Message`를 send 라이프사이클 콜백에 맞춰 인가한다. 모두 deprecated이며 각각 `AuthorizationFilter`·`HttpsRedirectFilter`·`AuthorizationChannelInterceptor`로 이어진다.
