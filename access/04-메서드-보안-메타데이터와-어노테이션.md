# 04. 메서드 보안 — 메타데이터 소스와 어노테이션

## 무엇을 / 왜

`@Secured`, `@RolesAllowed`(JSR-250), `@PreAuthorize`/`@PostAuthorize`/`@PreFilter`/`@PostFilter` — 이 어노테이션들이 어떻게 `ConfigAttribute`로 바뀌고, 메서드 호출 시 어떻게 인가에 쓰이는지를 다룬다. 02장의 `MethodSecurityInterceptor`가 엔진이라면, 이 챕터는 그 엔진에 "메서드용 메타데이터"를 공급하는 부품들(`method`, `annotation`, `prepost`, `expression/method` 패키지)을 본다.

## 핵심 타입 — 메서드 메타데이터의 계층

```
MethodSecurityMetadataSource  (extends SecurityMetadataSource)
   getAttributes(Method, targetClass) : ConfigAttribute[]
        △
   AbstractMethodSecurityMetadataSource (캐시 없는 골격)
        △
   AbstractFallbackMethodSecurityMetadataSource
     │  메서드→선언 클래스→인터페이스 순으로 어노테이션 탐색(fallback)
     ├─ SecuredAnnotationSecurityMetadataSource     @Secured
     ├─ Jsr250MethodSecurityMetadataSource          @RolesAllowed/@PermitAll/@DenyAll
     └─ PrePostAnnotationSecurityMetadataSource     @PreAuthorize 등 (expression)

   DelegatingMethodSecurityMetadataSource
     여러 MethodSecurityMetadataSource를 묶어 순차 위임 (결과 캐시)

   MapBasedMethodSecurityMetadataSource
     어노테이션 대신 메서드 패턴→속성 매핑(XML 시절)
```

`MethodInvocation`이 보안 객체이고, 메타데이터 소스는 호출되는 `Method`와 대상 클래스로부터 `ConfigAttribute` 목록을 만든다. 어노테이션 방식 소스들은 모두 `AbstractFallbackMethodSecurityMetadataSource`를 상속해, **구현 클래스의 메서드 → 인터페이스의 메서드 → 클래스 레벨 어노테이션** 순으로 권한 선언을 찾아 내려가는 fallback 탐색을 공유한다.

## 동작 흐름 — @Secured 가 ConfigAttribute가 되기까지

```
@Secured("ROLE_ADMIN")
void deleteUser() { ... }

  부팅/첫 호출 시
   SecuredAnnotationSecurityMetadataSource.findAttributes(method, targetClass)
     └─ AnnotationUtils.findAnnotation(method, Secured.class)
          └─ SecuredAnnotationMetadataExtractor.extractAttributes(secured)
               for token in secured.value():  → new SecurityConfig(token)
     결과: [ SecurityConfig("ROLE_ADMIN") ]

  호출 시 (02장 엔진)
   MethodSecurityInterceptor.beforeInvocation(mi)
     metadataSource.getAttributes(mi) → [ SecurityConfig("ROLE_ADMIN") ]
     accessDecisionManager.decide(...) → RoleVoter가 투표 (03장)
```

`@Secured`의 문자열들은 `SecurityConfig`(문자열형 `ConfigAttribute`)로 직행한다(`SecuredAnnotationSecurityMetadataSource.java` 88~99행). 즉 `@Secured`는 `RoleVoter`/`AuthenticatedVoter`가 이해하는 단순 권한 토큰만 표현할 수 있다.

JSR-250(`@RolesAllowed`, `@PermitAll`, `@DenyAll`)은 `Jsr250MethodSecurityMetadataSource`가 처리하며, `@RolesAllowed("ADMIN")`은 역할 접두사를 붙여 `Jsr250SecurityConfig`로, `@DenyAll`/`@PermitAll`은 거부/허용 의사 속성으로 변환된다. 전용 `Jsr250Voter`가 이를 집계한다.

## 동작 흐름 — @PreAuthorize / @PostAuthorize (표현식)

표현식 기반은 단순 문자열로 표현이 안 되므로(`getAttribute()`가 null), 전용 속성·전용 advice·전용 voter의 3단으로 처리된다.

```
@PreFilter("filterObject.owner == authentication.name")
@PreAuthorize("hasRole('ADMIN')")
List<Doc> list(List<Doc> docs)

  메타데이터:
   PrePostAnnotationSecurityMetadataSource
     └─ PrePostInvocationAttributeFactory (ExpressionBased…AttributeFactory)
          → PreInvocationExpressionAttribute(filterExpr, authorizeExpr, filterTarget)

  결정(before):
   PreInvocationAuthorizationAdviceVoter.vote(auth, mi, attrs)
     preAttr 있으면 → preAdvice.before(auth, mi, preAttr) ? GRANTED : DENIED
        │
        ▼ ExpressionBasedPreInvocationAdvice.before(...)
          ctx = expressionHandler.createEvaluationContext(auth, mi)
          preFilter 있으면 → expressionHandler.filter(target, preFilter, ctx)  ← 인자 컬렉션 필터링
          return evaluateAsBoolean(preAuthorize, ctx)                          ← 인가 boolean

  실행 후(after):
   PostInvocationAdviceProvider (AfterInvocationProvider)
     → ExpressionBasedPostInvocationAdvice.after(auth, mi, attr, returnedObject)
        @PostAuthorize 평가 / @PostFilter 로 반환 컬렉션 필터링
```

핵심은 `@PreFilter`/`@PostFilter`가 **인가 boolean이 아니라 컬렉션을 깎는다**는 점이다. `ExpressionBasedPreInvocationAdvice.before()`(소스 47~58행)는 `preAuthorize`를 boolean으로 평가하기 전에, `preFilter`가 있으면 `expressionHandler.filter()`로 메서드 인자(컬렉션/배열)에서 표현식을 만족하지 못하는 원소를 제거한다. `@PostFilter`는 같은 일을 반환값에 대해 `AfterInvocationProvider` 단계(02장 afterInvocation, 06장)에서 수행한다.

표현식 평가의 루트 객체(`hasRole`, `authentication`, `filterObject`, `principal` 등을 노출)는 `MethodSecurityExpressionHandler`가 만든다 — 이 핸들러와 `SecurityExpressionRoot`는 `core` 모듈에 있으며, access는 그것을 사용만 한다.

## 동작 흐름 — 리액티브 메서드 보안

`Mono`/`Flux`/Kotlin 코루틴을 반환하는 메서드는 블로킹 `SecurityContextHolder`를 쓸 수 없으므로 별도 인터셉터가 있다. `PrePostAdviceReactiveMethodInterceptor.invoke()`(소스 93~146행)는 반환 타입이 리액티브임을 검증한 뒤, `ReactiveSecurityContextHolder`에서 `Authentication`을 비동기로 꺼내 `preAdvice.before(...)`를 `filter`로 적용하고, 통과 못 하면 `AccessDeniedException`을 흘려보낸다. `@PostAuthorize`/`@PostFilter`는 `proceed()` 결과 스트림에 `map`으로 `postAdvice.after(...)`를 이어 붙인다.

```
ReactiveSecurityContextHolder.getContext()
   .map(getAuthentication).defaultIfEmpty(anonymous)
   .filter(auth -> preAdvice.before(auth, invocation, preAttr))   ← 통과 못하면 빈 스트림
   .switchIfEmpty(error(AccessDeniedException))
   .flatMap(auth -> proceed(invocation).map(r -> postAdvice.after(auth, ..., r)))
```

## 핵심 메서드 / 부품

- **`AbstractFallbackMethodSecurityMetadataSource.findAttributes(method, targetClass)`** — 구현체→인터페이스→클래스 순 어노테이션 탐색. 인터페이스에 단 권한도 구현체 호출에 적용되게 한다.
- **`DelegatingMethodSecurityMetadataSource`** — `@Secured`와 `@PreAuthorize`를 동시에 쓰는 구성에서, 여러 소스를 순서대로 물어보고 결과를 메서드별로 캐시한다.
- **`MethodSecurityInterceptor` + `MethodSecurityMetadataSourceAdvisor`** — 인터셉터(`MethodInterceptor`)를 실제 빈에 적용하는 AOP `Advisor`. 포인트컷이 "보안 어노테이션이 붙은 메서드"를 골라 인터셉터를 끼운다. AspectJ 변형은 `aspectj/AspectJMethodSecurityInterceptor`.
- **`P` (`method/P.java`)** — 표현식에서 메서드 파라미터를 이름으로 참조할 수 있게 하는 `@P("id")` 어노테이션 지원.

## 설계 포인트 / 확장점

- **메타데이터 ↔ 결정의 이중 표현**: 단순 권한(`@Secured`)은 `SecurityConfig`+`RoleVoter`로, 표현식 권한(`@PreAuthorize`)은 전용 Attribute+Advice+Voter로. 같은 엔진 위에서 두 세계가 공존한다.
- **before/after 분리**: `@Pre*`는 voter(인가 전), `@Post*`는 `AfterInvocationProvider`(실행 후). 02장 엔진의 before/after 두 슬롯에 정확히 대응한다.
- **현대 대체**: `AuthorizationManagerBeforeMethodInterceptor`/`AuthorizationManagerAfterMethodInterceptor`(블로킹)와 `...Reactive...`(리액티브)가 이 전체를 대체한다. `SecurityAnnotationScanner`가 메타데이터 소스 계층을, `AuthorizationManager`가 voter/advice를 흡수했다.

## 정리

메서드 보안은 "어노테이션 → `ConfigAttribute` → 투표/advice 평가"의 파이프라인이다. `@Secured`/JSR-250은 `SecurityConfig`류로 단순 변환돼 `RoleVoter`가 처리하고, `@Pre/Post*`는 전용 Expression 속성과 `PreInvocationAuthorizationAdviceVoter`·`PostInvocationAdviceProvider`로 SpEL을 평가하며 필터링까지 수행한다. 리액티브 반환 타입은 `PrePostAdviceReactiveMethodInterceptor`가 `ReactiveSecurityContextHolder` 위에서 같은 advice를 비동기로 적용한다.
