# 03. 인가 결정 흐름

## 무엇을 / 왜

이 챕터는 ACL 모듈의 심장인 두 갈래 판정 로직을 다룬다.

1. **읽기 인가** — "이 Sid들이 이 권한을 갖는가?" → `AclImpl.isGranted()` → `DefaultPermissionGrantingStrategy`. ACE 스캔, 거부 우선, 부모 상속의 규칙이 여기 있다.
2. **변경 인가** — "이 사용자가 ACL 자체를 바꿀 자격이 있는가?" → `AclAuthorizationStrategyImpl`. 소유자·관리 권위·`ADMINISTRATION` 권한을 검사한다.

두 가지를 한 챕터에 묶는 이유는, ACL 객체(`AclImpl`)가 **읽기 판정은 PermissionGrantingStrategy 에 위임**하고 **변경 시점엔 AclAuthorizationStrategy 로 자가 검문**하는, 대칭적인 두 전략을 동시에 들고 있기 때문이다.

## 핵심 타입

```
   AclImpl  (Acl + MutableAcl + Auditable + Ownership 일체 구현)
   │
   ├─ isGranted(perms, sids, adminMode)   ── 읽기 판정
   │        │  isSidLoaded 검사 → UnloadedSidException
   │        ▼
   │   PermissionGrantingStrategy.isGranted(this, perms, sids, adminMode)
   │        = DefaultPermissionGrantingStrategy  ★핵심 알고리즘★
   │
   └─ insertAce/updateAce/deleteAce/setOwner/setParent ── 변경
            │  맨 앞에서 항상:
            ▼
       AclAuthorizationStrategy.securityCheck(this, changeType)
            = AclAuthorizationStrategyImpl
            (소유자? 관리권위 보유? ADMINISTRATION 권한 보유?)
```

## 동작 흐름 1 — 읽기 판정 (DefaultPermissionGrantingStrategy)

### 진입: AclImpl.isGranted (`AclImpl.java`)

```java
public boolean isGranted(List<Permission> permission, List<Sid> sids, boolean administrativeMode) {
    Assert.notEmpty(permission, "Permissions required");
    Assert.notEmpty(sids, "SIDs required");
    if (!this.isSidLoaded(sids)) {
        throw new UnloadedSidException("ACL was not loaded for one or more SID");
    }
    return this.permissionGrantingStrategy.isGranted(this, permission, sids, administrativeMode);
}
```

먼저 요청된 Sid들이 이 ACL에 실제로 로딩됐는지 확인한다. 부분 로딩된 ACL에 로딩 안 된 Sid로 물으면 잘못된 false가 나올 수 있으므로, 차라리 `UnloadedSidException` 으로 단호히 중단한다. 통과하면 전략에 위임한다.

### 핵심 알고리즘 (`DefaultPermissionGrantingStrategy.java`)

이 메서드의 **인자 순서가 결과를 좌우한다**. 권한 목록과 Sid 목록을 중첩 루프로 도는데, 바깥이 권한, 안이 Sid다.

```java
for (Permission p : permission) {          // 1) 권한 순서대로
    for (Sid sid : sids) {                  // 2) Sid 순서대로
        for (AccessControlEntry ace : aces) {  // 3) ACE를 저장 순서대로 스캔
            if (isGranted(ace, p) && ace.getSid().equals(sid)) {
                // 권한 마스크 + Sid 가 정확히 일치하는 첫 ACE 발견
                if (ace.isGranting()) {
                    return true;            // 허용 ACE → 즉시 승인
                }
                // 거부 ACE → 이 (권한) 은 이 Sid에 대해 100% 거부
                if (firstRejection == null) firstRejection = ace;
                scanNextSid = false;
                break;                      // ACE 루프 탈출
            }
        }
        if (!scanNextSid) break;            // Sid 루프도 탈출 → 다음 권한으로
    }
}
```

규칙을 풀어 쓰면:

- **첫 매칭이 이긴다**: `(권한 마스크 == ACE 마스크) && (Sid 일치)` 인 첫 ACE를 찾으면, 그 ACE의 grant/deny가 최종 결정이다. 그래서 ACE의 저장 순서(`ace_order`)가 의미를 갖는다.
- **허용이면 즉시 true** 반환.
- **거부면** 그 (권한,Sid) 조합은 끝났다고 보고 `firstRejection` 에 기록한 뒤, 안쪽 Sid 루프를 깨고 **다음 권한**으로 넘어간다. 즉 "다른 권한으로라도 통과할 여지"는 남겨두되, 같은 권한에 대한 다른 Sid는 더 보지 않는다.

루프가 끝난 뒤의 후처리:

```java
if (firstRejection != null) {
    this.auditLogger.logIfNeeded(false, firstRejection);
    return false;                          // 명시적 거부가 있었다 → false
}
// 매칭이 하나도 없었다 → 부모에게 위임
if (acl.isEntriesInheriting() && acl.getParentAcl() != null) {
    return acl.getParentAcl().isGranted(permission, sids, false);
}
throw new NotFoundException("Unable to locate a matching ACE for passed permissions and SIDs");
```

결정 트리:

```
 매칭 ACE 스캔
   ├─ 허용 ACE 발견                          → true   (+감사 로그 success)
   ├─ 거부 ACE만 발견(firstRejection != null) → false  (+감사 로그 failure)
   └─ 매칭 전혀 없음
        ├─ isEntriesInheriting && parent != null → 부모 ACL.isGranted() 재귀
        └─ 부모 없음 / 상속 끊김                 → NotFoundException ★
```

마지막의 `NotFoundException` 이 중요하다. "권한 없음"과 "ACL 정보 자체가 없음"을 구분하는 신호다. `AclPermissionEvaluator` 는 이 예외를 잡아 `false` 로 변환한다([05장](05-Spring-Security-통합.md)).

### 권한 매칭 — 정확 일치 (그리고 오버라이드 훅)

```java
protected boolean isGranted(AccessControlEntry ace, Permission p) {
    return ace.getPermission().getMask() == p.getMask();   // 기본: 마스크 정확 일치
}
```

기본은 마스크 **완전 일치**다. Javadoc은 비트 단위(AND) 비교로 바꾸고 싶으면 이 메서드를 오버라이드하라고 안내한다 — 예컨대 "WRITE 권한 ACE가 READ도 함축한다" 같은 정책을 넣을 때.

```java
// 오버라이드 예시 (Javadoc 인용)
if (ace.isGranting() && p.getMask() != 0) {
    return (ace.getPermission().getMask() & p.getMask()) != 0;  // 비트가 겹치면 매칭
} else {
    return ace.getPermission().getMask() == p.getMask();
}
```

### 상속의 의미 — getParentAcl 과 isEntriesInheriting

ACL은 트리를 이룬다(폴더→문서). 두 조건이 **모두** 참일 때만 부모의 ACE가 자식 판정으로 흘러내린다.

- `getParentAcl() != null` — 네비게이션 링크가 존재.
- `isEntriesInheriting() == true` — 그 링크를 따라 ACE 상속도 한다.

링크는 있지만 상속은 끄는 것이 가능하다(독립 권한을 갖되 트리 구조만 유지). 부모 위임 호출 시 `administrativeMode` 를 `false` 로 강제해, 부모에서의 매칭은 정상 감사 로깅을 타게 한다.

## 동작 흐름 2 — 변경 인가 (AclAuthorizationStrategyImpl)

`AclImpl` 의 모든 변경 메서드는 첫 줄에서 자가 검문한다.

```java
public void insertAce(...) {
    this.aclAuthorizationStrategy.securityCheck(this, AclAuthorizationStrategy.CHANGE_GENERAL);
    ...
}
public void setOwner(Sid newOwner) {
    this.aclAuthorizationStrategy.securityCheck(this, AclAuthorizationStrategy.CHANGE_OWNERSHIP);
    ...
}
public void updateAuditing(...) {
    this.aclAuthorizationStrategy.securityCheck(this, AclAuthorizationStrategy.CHANGE_AUDITING);
    ...
}
```

변경 종류는 세 가지로 구분된다: `CHANGE_OWNERSHIP`(소유권), `CHANGE_AUDITING`(감사 플래그), `CHANGE_GENERAL`(ACE 추가/수정/삭제, 부모/상속 변경).

### securityCheck 의 판정 순서 (`AclAuthorizationStrategyImpl.java`)

```
 0. 인증 안 됨 → AccessDeniedException ("Authenticated principal required")

 1. 현재 사용자가 ACL 소유자인가?
      && (changeType == GENERAL || OWNERSHIP)        → 통과
      (소유자는 자기 객체의 일반 변경·소유권 이전 가능. 단 감사 변경은 불가)

 2. 소유자가 GrantedAuthoritySid이고, 사용자가 그 권위를 보유하는가? → 통과

 3. changeType에 해당하는 시스템 전역 권위 보유?
      (생성자 주입: gaTakeOwnership / gaModifyAuditing / gaGeneralChanges)
                                                      → 통과

 4. 이 ACL의 ACE 상으로 ADMINISTRATION 권한을 갖는가?
      acl.isGranted([BasePermission.ADMINISTRATION], sids, false)  → 통과

 그 무엇도 아니면 → AccessDeniedException
```

```java
// 1단계 — 소유권에 의한 허가
Sid currentUser = createCurrentUser(authentication);  // PrincipalSid(auth)
Sid owner = acl.getOwner();
if (owner != null && currentUser.equals(owner)
        && (changeType == CHANGE_GENERAL || changeType == CHANGE_OWNERSHIP)) {
    return;
}
...
// 4단계 — ACL 내 ADMINISTRATION 권한에 의한 허가
List<Sid> sids = this.sidRetrievalStrategy.getSids(authentication);
if (acl.isGranted(Arrays.asList(BasePermission.ADMINISTRATION), sids, false)) {
    return;
}
throw new AccessDeniedException("Principal does not have required ACL permissions ...");
```

이 4단계 위계가 ACL 보안 모델의 핵심이다. **역할(ROLE_ADMIN)이 없어도, 그 객체에 대한 `ADMINISTRATION` ACE만 있으면 그 객체의 ACL을 관리할 수 있다**(4단계). 생성자에는 3개(또는 공통 1개)의 `GrantedAuthority` 를 주입한다 — 인덱스 0=소유권 변경, 1=감사 변경, 2=일반 변경 권위.

`RoleHierarchy`(기본 `NullRoleHierarchy`)를 주입하면, 사용자의 권위를 계층 확장해 검사한다(6.4부터). `SecurityContextHolderStrategy` 도 교체 가능(5.8부터).

## 핵심 메서드 요약

| 메서드 | 위치 | 하는 일 |
|--------|------|---------|
| `AclImpl.isGranted` | AclImpl | Sid 로딩 검증 후 PermissionGrantingStrategy 위임 |
| `DefaultPermissionGrantingStrategy.isGranted(acl,...)` | strategy | ACE 스캔·거부 우선·부모 상속 재귀, 미발견 시 NotFoundException |
| `DefaultPermissionGrantingStrategy.isGranted(ace,p)` | strategy | 권한 매칭(기본 마스크 정확 일치, 오버라이드 훅) |
| `AclAuthorizationStrategyImpl.securityCheck` | strategy | 변경 권한 4단계 검사(소유자/권위/ADMIN) |

## 설계 포인트 / 확장점

- **전략 패턴 두 축**: 읽기 판정(`PermissionGrantingStrategy`)과 변경 인가(`AclAuthorizationStrategy`)가 각각 SPI로 분리돼 독립 교체된다.
- **순서 의존성**: ACE 저장 순서(`ace_order`)와 권한/Sid 인자 순서가 결과에 영향. 거부 ACE를 앞에 두면 더 일찍 차단된다.
- **"못 찾음"과 "거부"의 구분**: 매칭 ACE가 하나도 없으면 `NotFoundException`(정보 부재), 거부 ACE를 만나면 `false`(명시적 거부). 호출자가 다르게 다룰 수 있다.
- **권한 매칭 오버라이드**: `isGranted(ace, p)` 를 비트 AND로 바꿔 "상위 권한이 하위를 함축" 정책을 구현.

## 정리

읽기 인가는 `AclImpl.isGranted` → `DefaultPermissionGrantingStrategy` 로 흐르며, **권한×Sid×ACE를 순서대로 스캔해 첫 매칭의 grant/deny가 이기고, 매칭이 없으면 부모로 상속하며, 끝내 없으면 NotFoundException**을 던진다. 변경 인가는 `AclAuthorizationStrategyImpl.securityCheck` 가 **소유자 → 소유자권위 → 시스템 권위 → ADMINISTRATION ACE** 의 4단계로 검사한다. 두 전략 모두 교체 가능한 SPI다.
