# 01. 핵심 개념과 도메인 모델

## 무엇을 / 왜

이 챕터는 `org.springframework.security.acls.model` 패키지를 다룬다. 이 패키지는 **구현이 거의 없는 순수 계약(인터페이스·예외)** 만 모여 있다. ACL 서브시스템 전체가 공유하는 "유비쿼터스 언어"를 먼저 머릿속에 넣어야, 이후의 마스크 연산·판정 알고리즘·JDBC 영속화가 모두 같은 단어 위에서 돌아간다는 것이 보인다.

핵심 질문은 이것이다. **"누가, 무엇에 대해, 어떤 권한을, 허용/거부"** 를 어떻게 객체로 표현할 것인가? ACL은 이 네 가지를 각각 별도 추상화로 떼어낸다.

## 핵심 타입 — 4요소 + 컨테이너

```
 AccessControlEntry (ACE)  ── ACL을 구성하는 "한 줄"
 ┌──────────────────────────────────────────────────┐
 │  getSid()        : Sid          (누가)            │
 │  getPermission() : Permission   (어떤 권한 = 마스크)│
 │  isGranting()    : boolean      (허용 true/거부 false)│
 │  getAcl()        : Acl          (소속 ACL 역참조)  │
 │  getId()         : Serializable (DB PK, 미저장 시 null)│
 └──────────────────────────────────────────────────┘

 Acl  ── 하나의 ObjectIdentity 에 대한 ACE 목록 + 상속 정보
 ┌──────────────────────────────────────────────────┐
 │  getObjectIdentity() : ObjectIdentity (무엇에)     │
 │  getEntries()        : List<AccessControlEntry>    │
 │  getOwner()          : Sid                         │
 │  getParentAcl()      : Acl   (상속 트리의 부모)     │
 │  isEntriesInheriting(): boolean (부모 ACE 상속 여부) │
 │  isGranted(perms, sids, adminMode) : boolean ★판정★ │
 │  isSidLoaded(sids)   : boolean (부분 로딩 여부)     │
 └──────────────────────────────────────────────────┘
```

### Sid — 보안 주체의 추상화 (`Sid.java`)

`Sid`(Security IDentity)는 "권한의 대상이 되는 보안 주체"를 가리키는 **마커성 인터페이스**다. 선언하는 메서드는 `equals()` 와 `hashCode()` 둘뿐이다. 왜 이렇게 비어 있을까? ACL은 실제 `UserDetails` 나 `GrantedAuthority` 객체 전체를 저장하지 않고, **그 추상화(이름 문자열)** 만 저장한 뒤 비교만 하면 되기 때문이다.

구현은 `domain` 패키지에 둘 있다.

- `PrincipalSid` — `Authentication.getName()`(예: `"alice"`)을 들고 다니며 사용자 개인을 표현.
- `GrantedAuthoritySid` — 권위 문자열(예: `"ROLE_ADMIN"`)을 들고 다니며 역할/그룹을 표현.

```java
// PrincipalSid.equals — 같은 타입 + 같은 principal 문자열이면 동일 주체
return ((PrincipalSid) object).getPrincipal().equals(this.getPrincipal());
```

이 단순 문자열 비교가 ACL 판정의 기반이다. 즉 ACE에 저장된 `Sid` 와, 현재 로그인 사용자에게서 뽑아낸 `Sid` 목록을 `equals()` 로 맞춰본다.

### ObjectIdentity — 도메인 객체 인스턴스의 키 (`ObjectIdentity.java`)

ACL은 도메인 객체 자체를 참조하지 않는다. 대신 `(type, identifier)` 쌍으로 **간접 참조**한다.

- `getType()` — 보통 도메인 클래스의 FQCN (예: `com.example.BlogPost`).
- `getIdentifier()` — 인스턴스의 PK (예: `42L`). 비즈니스 의미 없는 합성 키(시퀀스 PK) 권장.

`equals()`/`hashCode()` 가 인터페이스에 명시적으로 선언된 점이 중요하다. ObjectIdentity는 캐시 키와 DB 조회 키로 쓰이므로 **참조 동일성이 아니라 값 동일성**으로 비교돼야 한다. 표준 구현 `ObjectIdentityImpl` 은 한 발 더 나아가, 식별자가 `Number` 면 `longValue()` 로 비교한다 — `Integer(42)` 와 `Long(42)` 를 같은 객체로 취급해 타입 불일치로 인한 캐시 미스를 막는다.

```java
// ObjectIdentityImpl.equals 발췌
if (this.identifier instanceof Number && other.identifier instanceof Number) {
    if (((Number) this.identifier).longValue() != ((Number) other.identifier).longValue()) {
        return false;
    }
}
```

`ObjectIdentityImpl(Object)` 생성자는 리플렉션으로 도메인 객체의 `getId()` 를 호출해 식별자를 뽑는다. 그래서 도메인 객체에 `getId()` 가 없으면 `IdentityUnavailableException` 이 난다.

### Permission — 32비트 마스크 권한 (`Permission.java`)

`Permission` 은 `getMask()`(int 비트마스크)와 `getPattern()`(사람이 읽는 32자 비트 문자열) 두 메서드만 갖는다. READ=1, WRITE=2, DELETE=8 처럼 각 권한이 비트 하나를 차지한다. 자세한 비트 연산은 [02장](02-권한-마스크와-Permission.md)에서 다룬다.

### Acl / MutableAcl — ACL 본체 (`Acl.java`, `MutableAcl.java`)

`Acl` 은 한 `ObjectIdentity` 에 묶인 ACE 목록이자, **유일하게 인가 결정을 내려도 되는 곳**이다. Javadoc이 단호하게 못 박는다.

> `getEntries()` 로 받은 목록을 직접 보고 인가 판단을 하지 말 것 — 반드시 `isGranted(...)` 를 쓸 것.

이유는 상속·순서·거부 우선 같은 규칙이 `isGranted` 안에 캡슐화돼 있기 때문이다(→ [03장](03-인가-결정-흐름.md)).

`Acl` 은 읽기 전용 계약이고, 변경은 `MutableAcl` 이 분리해서 제공한다.

```java
public interface MutableAcl extends Acl {
    void insertAce(int atIndexLocation, Permission permission, Sid sid, boolean granting);
    void updateAce(int aceIndex, Permission permission);
    void deleteAce(int aceIndex);
    void setOwner(Sid newOwner);
    void setParent(Acl newParent);
    void setEntriesInheriting(boolean entriesInheriting);
    Serializable getId();
}
```

여기에 두 개의 부가 확장 인터페이스가 더 있다.

- `OwnershipAcl` — `setOwner()` 를 별도 권한 검사 대상으로 분리.
- `AuditableAcl` — `updateAuditing(aceIndex, success, failure)` 로 ACE별 감사 플래그 토글.

표준 구현 `AclImpl` 은 이 네 인터페이스(`Acl, MutableAcl, AuditableAcl, OwnershipAcl`)를 한 클래스에서 모두 구현한다.

### 서비스 계약 — AclService / MutableAclService

```
 AclService                         (읽기 전용)
 ├─ readAclById(oid)                단건
 ├─ readAclById(oid, sids)          단건, 관심 Sid만
 ├─ readAclsById(oids)              일괄
 ├─ readAclsById(oids, sids)        일괄, 관심 Sid만  ← 성능 최적화 경로
 └─ findChildren(parentOid)         자식 ObjectIdentity 탐색

 MutableAclService extends AclService  (쓰기)
 ├─ createAcl(oid)        : MutableAcl   빈 ACL 생성
 ├─ updateAcl(acl)        : MutableAcl   ACE 통째로 교체
 └─ deleteAcl(oid, deleteChildren)       삭제(+자식 캐스케이드)
```

`readAclsById(objects, sids)` 에 `sids` 를 넘기는 오버로드가 따로 있는 이유는, 구현체가 "이 사용자가 가진 Sid에 해당하는 ACE만" 로딩해 성능을 끌어올릴 여지를 주기 위해서다. (단, 표준 `BasicLookupStrategy` 는 캐시 일관성을 위해 이 sids를 실제로는 무시하고 전부 로딩한다 — [04장](04-JDBC-영속화와-조회-전략.md).)

### 예외로 표현되는 상태

- `NotFoundException` — ACL/ACE를 찾을 수 없거나, 매칭되는 ACE가 없어 판정 불가. **"못 찾음 = 거부"** 의 신호로 상위에서 잡아 false 처리하는 경우가 많다.
- `UnloadedSidException` — ACL이 일부 Sid만 로딩됐는데, 로딩되지 않은 Sid로 판정을 요청했을 때.
- `AlreadyExistsException` / `ChildrenExistException` — 생성·삭제 시의 무결성 위반.

## 동작 흐름 — "객체 → ACL → 판정" 의 큰 그림

아래는 4요소가 실제 인가 한 번에서 어떻게 조립되는지의 개요다. (구체 코드는 03·05장)

```
 도메인객체(post #42), Authentication(alice/ROLE_USER), 요청권한 'WRITE'
        │
        │ ObjectIdentityRetrievalStrategy.getObjectIdentity(post)
        ▼
   ObjectIdentity(type=BlogPost, id=42)
        │                                  SidRetrievalStrategy.getSids(auth)
        │                                         ▼
        │                              [PrincipalSid(alice), GrantedAuthoritySid(ROLE_USER)]
        ▼
   AclService.readAclById(oid, sids)  ──▶  Acl (ACE 목록 + 부모)
        │
        ▼
   acl.isGranted([WRITE], [alice, ROLE_USER], false)  ──▶  true / false
```

## 설계 포인트

- **계약과 구현의 철저한 분리**: `model` 은 인터페이스만, `domain` 은 표준 구현, `jdbc` 는 영속화. 덕분에 `Permission`, `Sid`, `PermissionGrantingStrategy`, `AclService` 어느 것이든 독립적으로 교체할 수 있다.
- **간접 참조(indirection)**: 도메인 객체↔ObjectIdentity, 실제 주체↔Sid로 한 겹 추상화를 두어, ACL이 애플리케이션 도메인 코드와 결합되지 않는다.
- **읽기/쓰기 인터페이스 분리**(`Acl` vs `MutableAcl`): 인가 판정 경로에서는 불변 뷰만 노출해 의도치 않은 변경을 막는다.

## 정리

`model` 패키지는 ACL의 문법책이다. **Sid(누가) × ObjectIdentity(무엇에) × Permission(어떤 권한) × granting(허용/거부)** 가 `AccessControlEntry` 한 줄을 이루고, 그 줄들이 모여 한 객체의 `Acl` 이 된다. 인가 결정은 오직 `Acl.isGranted()` 를 통해서만 내려야 하며, ACL의 조회·생성·삭제는 `AclService`/`MutableAclService` 계약으로 추상화돼 있다. 다음 장부터 이 단어들의 실제 구현을 파고든다.
