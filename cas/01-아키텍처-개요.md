# 01. 아키텍처 개요 — CAS SSO의 전체 그림

## 무엇을 / 왜

CAS는 "중앙 인증 서버 한 곳에서 로그인하면, 여기에 연결된 모든 애플리케이션에서 다시 로그인하지 않아도 되는" SSO 프로토콜이다. 이 모듈을 코드로 따라가기 전에, 먼저 **누가 누구에게 무엇을 보내는지** 큰 그림을 잡아야 한다. CAS에는 세 개의 행위자가 등장한다.

- **사용자 브라우저(User Agent)** — 리다이렉트를 따라다니는 주체.
- **CAS 서버** — 실제 로그인 화면을 띄우고, 로그인 성공 시 **티켓(ticket)**을 발급하며, 나중에 그 티켓을 검증해준다.
- **서비스(Service)** — Spring Security로 보호되는 우리 애플리케이션. CAS 용어로 각 애플리케이션을 "service"라 부르고, 그 콜백 URL이 `ServiceProperties.getService()`다.

핵심 개념은 **티켓**이다. 티켓은 의미를 알 수 없는 불투명한 문자열(예: `ST-1-abcdef...`)이며, 발급받은 서비스는 그것을 직접 해석하지 못한다. 대신 CAS 서버에 "이 티켓 유효해?"라고 되물어 검증하고, 그 응답인 `Assertion`(누가 누구인지 + 속성)을 받는다. 비밀번호는 오직 사용자와 CAS 서버 사이에서만 오간다.

## 핵심 타입 한눈에

이 모듈은 "보내기" 쪽과 "받아 검증하기" 쪽으로 역할이 나뉜다.

```
설정값 보관      ServiceProperties      ← service URL, ticket/service 파라미터 이름, sendRenew 등

[보내기]        CasAuthenticationEntryPoint   ← 미인증 접근 시 CAS 로그인 페이지로 리다이렉트

[받아 검증]     CasAuthenticationFilter       ← 콜백 URL의 ticket 파라미터를 가로채
                      │                          CasServiceTicketAuthenticationToken 생성
                      ▼
                AuthenticationManager
                      │
                      ▼
                CasAuthenticationProvider     ← TicketValidator로 CAS 서버에 티켓 검증
                      │                          Assertion → UserDetails → 권한
                      ▼
                CasAuthenticationToken        ← 인증 완료 토큰(SecurityContext에 저장)

[보조]          StatelessTicketCache          ← 세션 없는 호출자용 티켓 캐시
                ServiceAuthenticationDetails  ← 프록시 티켓용 동적 service URL
                CasGatewayAuthenticationRedirectFilter ← 로그인 강제 없이 SSO 세션만 확인
```

`ServiceProperties`는 이 모든 컴포넌트가 공유하는 설정 보따리다. `cas/src/main/java/org/springframework/security/cas/ServiceProperties.java`를 보면 다음 값을 들고 있다.

- `service` — 우리 애플리케이션의 콜백 URL. 예: `https://www.mycompany.com/application/login/cas`. EntryPoint가 CAS에 알려주는 "돌아올 주소"이자, Provider가 티켓 검증 시 CAS에 넘기는 service 값이다. 그래서 **이 값과 필터가 처리하는 URL이 일치해야 한다.**
- `artifactParameter` — 티켓이 실려 오는 쿼리 파라미터 이름. 기본 `"ticket"`.
- `serviceParameter` — service URL을 실어 보낼 파라미터 이름. 기본 `"service"`.
- `sendRenew` — `true`면 SSO 세션이 있어도 CAS가 재인증을 강제한다. 고보안 앱에서 사용. 기본 `false`(자동 SSO).
- `authenticateAllArtifacts` — 프록시 티켓 처리를 켜는 스위치(→ 04장).

`SamlServiceProperties`는 이 클래스를 상속해 파라미터 기본값만 CAS의 SAML 모드용(`SAMLart`/`TARGET`)으로 바꾼 변형이다.

## 동작 흐름 — "로그인 한 바퀴"

처음 보호된 자원에 접근하는 사용자의 전체 왕복을 따라가 보자. 번호가 한 번의 SSO 로그인에서 일어나는 일의 순서다.

```
  ┌────────┐                ┌──────────────────────┐              ┌────────────┐
  │ 브라우저 │                │  우리 앱(Spring Sec)   │              │  CAS 서버   │
  └───┬────┘                └──────────┬───────────┘              └─────┬──────┘
      │ ①  GET /secure (티켓 없음)         │                              │
      │ ─────────────────────────────▶ │                              │
      │                                │ ExceptionTranslationFilter   │
      │                                │  → AccessDenied(미인증)         │
      │                                │  → CasAuthenticationEntryPoint│
      │ ②  302  Location:              │      .commence()              │
      │     {casLogin}?service={콜백URL} │                              │
      │ ◀───────────────────────────── │                              │
      │ ③  GET {casLogin}?service=...                                  │
      │ ─────────────────────────────────────────────────────────▶ │
      │ ④  로그인 폼 표시 → 사용자 ID/PW 입력 → CAS가 인증            │
      │ ◀ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ │
      │ ⑤  302  Location: {콜백URL}?ticket=ST-xxxx                     │
      │ ◀───────────────────────────────────────────────────────── │
      │ ⑥  GET /login/cas?ticket=ST-xxxx │                              │
      │ ─────────────────────────────▶ │ CasAuthenticationFilter      │
      │                                │  .attemptAuthentication()    │
      │                                │   → ticket 추출, 토큰 생성     │
      │                                │   → AuthenticationManager     │
      │                                │      → CasAuthentication      │
      │                                │         Provider              │
      │                                │ ⑦ TicketValidator.validate(  │
      │                                │     ticket, service) ───────▶│
      │                                │ ◀──────── Assertion ──────────│
      │                                │ ⑧ Assertion→UserDetails→권한  │
      │                                │   CasAuthenticationToken 저장  │
      │ ⑨  302 → 원래 요청 /secure       │                              │
      │ ◀───────────────────────────── │                              │
      │ ⑩  GET /secure  (이제 인증됨)     │                              │
      │ ─────────────────────────────▶ │ → 자원 응답                    │
```

- ①~②: 인증되지 않은 접근을 `ExceptionTranslationFilter`가 잡고, CAS용 `AuthenticationEntryPoint`가 사용자를 CAS 로그인 페이지로 보낸다. 이때 `service` 파라미터로 우리 콜백 URL을 함께 알려준다. (→ 02장)
- ③~⑤: 로그인은 전적으로 CAS 서버의 몫이다. 성공하면 CAS가 콜백 URL에 `ticket`을 붙여 브라우저를 되돌려보낸다.
- ⑥~⑧: 돌아온 티켓을 `CasAuthenticationFilter`가 가로채 `CasAuthenticationProvider`에게 넘기고, Provider가 CAS 서버에 티켓을 검증해 인증을 완성한다. (→ 03장)
- ⑨~⑩: 인증이 끝나면 원래 가려던 자원으로 다시 리다이렉트되고, 이번엔 `SecurityContext`에 인증이 있으므로 통과한다.

핵심은 **로그인 자체(③~⑤)는 이 모듈이 하지 않는다**는 점이다. 이 모듈은 "CAS로 보내는 일(②)"과 "돌아온 티켓을 검증하는 일(⑥~⑧)"만 책임진다.

## 설계 포인트

- **책임 분리**: `EntryPoint`(시작) / `Filter`(수신) / `Provider`(검증)의 3분할은 Spring Security의 표준 인증 구조를 그대로 따른다. 덕분에 CAS도 폼 로그인·OAuth2와 동일한 `AuthenticationManager` 파이프라인에 자연스럽게 끼워진다.
- **얇은 통합 계층**: 프로토콜의 실제 작업(서비스 URL 조립, 티켓 검증 HTTP 호출, 리다이렉트 URL 생성)은 모두 `org.apereo.cas.client`에 위임한다. 이 모듈은 그 결과를 Spring Security 타입(`Authentication`, `UserDetails`)으로 번역할 뿐이다.
- **설정 단일 출처**: `ServiceProperties` 하나를 EntryPoint·Filter·Provider·DetailsSource가 공유하므로, service URL이나 파라미터 이름이 어긋날 여지를 줄인다.

## 정리

CAS 모듈은 "중앙 CAS 서버가 발급한 티켓"을 Spring Security의 인증으로 바꾸는 통합 계층이다. 미인증 사용자를 CAS로 보내는 `CasAuthenticationEntryPoint`, 돌아온 티켓을 받는 `CasAuthenticationFilter`, 티켓을 CAS에 검증해 인증을 완성하는 `CasAuthenticationProvider`가 세 기둥이며, 이들이 공유하는 `ServiceProperties`가 service URL과 파라미터 이름을 일관되게 묶는다. 다음 장부터 각 기둥을 코드 레벨로 파고든다.
