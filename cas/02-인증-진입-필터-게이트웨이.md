# 02. 인증 진입 · 수신 필터 · 게이트웨이

## 무엇을 / 왜

01장의 "로그인 한 바퀴"에서 우리 애플리케이션이 직접 하는 일은 두 가지였다. **CAS로 내보내기**(②)와 **돌아온 티켓 받기**(⑥). 이 장은 그 두 일을 담당하는 웹 계층 컴포넌트, 즉 `CasAuthenticationEntryPoint`와 `CasAuthenticationFilter`를 코드로 본다. 마지막으로, 로그인 화면을 강제로 띄우지 않고 "이미 CAS에 로그인되어 있나?"만 조용히 확인하는 gateway 모드(`CasGatewayAuthenticationRedirectFilter`)를 다룬다.

## 핵심 타입

```
web/
├── CasAuthenticationEntryPoint              implements AuthenticationEntryPoint
│     └─ commence(): CAS 로그인 URL로 302 리다이렉트
│
├── CasAuthenticationFilter                  extends AbstractAuthenticationProcessingFilter
│     ├─ requiresAuthentication(): 이 요청을 처리할지(서비스/프록시/리셉터) 판단
│     ├─ attemptAuthentication(): ticket 추출 → 토큰 생성 → AuthenticationManager
│     └─ successfulAuthentication(): SecurityContext 저장 또는 (프록시면) 체인 계속
│
├── CasGatewayAuthenticationRedirectFilter   extends GenericFilterBean
│     └─ doFilter(): gateway=true를 붙여 CAS로 조용히 리다이렉트
└── CasGatewayResolverRequestMatcher         implements RequestMatcher
      └─ matches(): 한 번만 gateway 시도하도록(무한루프 방지)
```

## CasAuthenticationEntryPoint — CAS로 내보내기

`AuthenticationEntryPoint`는 "인증이 필요한데 안 되어 있을 때 무엇을 할지"를 정하는 표준 훅이다. `ExceptionTranslationFilter`가 `AccessDeniedException`(미인증)을 잡으면 이 `commence()`를 호출한다. CAS 버전은 사용자를 CAS 로그인 페이지로 리다이렉트한다.

`cas/src/main/java/org/springframework/security/cas/web/CasAuthenticationEntryPoint.java`:

```java
public final void commence(HttpServletRequest servletRequest, HttpServletResponse response,
        AuthenticationException authenticationException) throws IOException {
    String urlEncodedService = createServiceUrl(servletRequest, response);
    String redirectUrl = createRedirectUrl(urlEncodedService);
    preCommence(servletRequest, response);
    this.redirectStrategy.sendRedirect(servletRequest, response, redirectUrl);
}
```

두 단계로 나뉜다.

- `createServiceUrl(...)` — `WebUtils.constructServiceUrl(...)`(CAS 클라이언트)을 호출해 우리 콜백 URL(`serviceProperties.getService()`)을 인코딩한다. 즉 "로그인 끝나면 여기로 돌려보내줘"의 주소다. `encodeServiceUrlWithSessionId`가 기본 `true`인데, 이는 구버전 CAS(2.x)가 세션 식별자를 자동으로 제거하지 못하던 호환성 때문에 남아 있는 옵션이다.
- `createRedirectUrl(...)` — `CommonUtils.constructRedirectUrl(loginUrl, serviceParameter, serviceUrl, sendRenew, false)`로 최종 리다이렉트 URL을 만든다. 결과는 `{loginUrl}?service={콜백URL}` 형태이며, `sendRenew`가 `true`면 `renew` 파라미터가 붙어 CAS가 재로그인을 강제한다.

`preCommence()`는 리다이렉트 직전 후킹용 빈 템플릿 메서드로, 서브클래스가 추가 전처리를 넣을 수 있게 열어 둔 확장점이다.

## CasAuthenticationFilter — 돌아온 티켓 받기

이 필터는 `AbstractAuthenticationProcessingFilter`를 상속한다. 즉 폼 로그인 필터와 같은 골격(특정 URL을 감시하다가 매칭되면 `attemptAuthentication()` 실행)을 공유한다. 생성자에서 기본 처리 URL을 `/login/cas`로 잡는다.

```java
public CasAuthenticationFilter() {
    super("/login/cas");
    RequestMatcher processUri = pathPattern("/login/cas");
    setRequiresAuthenticationRequestMatcher(processUri);
    ...
}
```

### 이 요청을 처리할 것인가 — requiresAuthentication

표준 필터는 URL이 매칭되는지만 본다. 하지만 CAS 필터는 세 종류의 요청을 처리할 수 있어 판단이 더 복잡하다.

```java
protected boolean requiresAuthentication(HttpServletRequest request, HttpServletResponse response) {
    final boolean serviceTicketRequest = serviceTicketRequest(request, response);
    final boolean result = serviceTicketRequest          // ① /login/cas 로 온 서비스 티켓
            || proxyReceptorRequest(request)             // ② 프록시 리셉터(PGT 수신) URL
            || (proxyTicketRequest(serviceTicketRequest, request)); // ③ 아무 URL의 프록시 티켓
    return result;
}
```

- ① **서비스 티켓**: URL이 `/login/cas`에 매칭(`super.requiresAuthentication`)되면 평범한 SSO 로그인 콜백이다.
- ② **프록시 리셉터**: CAS 서버가 PGT(Proxy Granting Ticket)를 우리에게 전달하기 위해 호출하는 URL. (→ 04장)
- ③ **프록시 티켓**: `authenticateAllArtifacts`가 켜져 있고, 아직 인증되지 않은 요청에 티켓이 실려 있으면 어떤 URL이든 처리. (→ 04장)

### 인증 시도 — attemptAuthentication

```java
public Authentication attemptAuthentication(HttpServletRequest request, HttpServletResponse response)
        throws AuthenticationException, IOException {
    if (proxyReceptorRequest(request)) {                       // (A) 프록시 리셉터면
        WebUtils.readAndRespondToProxyReceptorRequest(request, response, this.proxyGrantingTicketStorage);
        return null;                                           //     PGT 저장하고 종료
    }
    String serviceTicket = obtainArtifact(request);            // (B) ticket 파라미터 추출
    if (!StringUtils.hasText(serviceTicket)) {
        // ... (C) gateway 실패 처리: 저장된 요청으로 되돌려보냄
        serviceTicket = "";
    }
    boolean serviceTicketRequest = serviceTicketRequest(request, response);
    CasServiceTicketAuthenticationToken authRequest = serviceTicketRequest
            ? CasServiceTicketAuthenticationToken.stateful(serviceTicket)   // (D) 상태형
            : CasServiceTicketAuthenticationToken.stateless(serviceTicket); //     무상태형
    authRequest.setDetails(this.authenticationDetailsSource.buildDetails(request));
    return this.getAuthenticationManager().authenticate(authRequest);       // (E) 검증 위임
}
```

- (A) 프록시 리셉터 요청이면 CAS 클라이언트에게 PGT 수신/응답을 맡기고 `null`을 반환한다. `null`은 "요청을 내가 소비했으니 인증 흐름을 더 진행하지 말라"는 신호다.
- (B)(D) `obtainArtifact()`가 `ticket` 파라미터에서 티켓 문자열을 꺼낸다. 그리고 이 요청이 서비스 티켓이면 `stateful(...)`, 프록시 티켓이면 `stateless(...)` 팩토리로 `CasServiceTicketAuthenticationToken`을 만든다. 이 토큰의 principal은 실제 사용자명이 아니라 `_cas_stateful_`/`_cas_stateless_` 같은 **식별 상수**이고, credentials가 티켓 문자열이다. (토큰 상세는 03장)
- (C) 티켓이 없는데 세션에 gateway 시도 흔적(`CAS_GATEWAY_AUTHENTICATION_ATTR`)이 있으면, "CAS에 로그인 안 되어 있었구나"로 판단하고 `RequestCache`에 저장해둔 원래 요청으로 되돌려보낸다. 이것이 gateway 모드의 "실패(=비로그인)" 경로다.
- (E) 만든 토큰을 `AuthenticationManager`에 넘긴다. 실제 티켓 검증은 그 뒤의 `CasAuthenticationProvider`가 한다.

### 인증 성공 후 — successfulAuthentication

```java
protected void successfulAuthentication(HttpServletRequest request, HttpServletResponse response,
        FilterChain chain, Authentication authResult) throws IOException, ServletException {
    boolean continueFilterChain = proxyTicketRequest(serviceTicketRequest(request, response), request);
    if (!continueFilterChain) {                       // 서비스 티켓이면
        super.successfulAuthentication(request, response, chain, authResult);
        return;                                       //   표준 성공 처리(저장+리다이렉트)
    }
    // 프록시 티켓이면: SecurityContext에 담되 체인을 계속 진행
    SecurityContext context = this.securityContextHolderStrategy.createEmptyContext();
    context.setAuthentication(authResult);
    this.securityContextHolderStrategy.setContext(context);
    this.securityContextRepository.saveContext(context, request, response);
    ...
    chain.doFilter(request, response);
}
```

서비스 티켓(일반 SSO 로그인)이면 부모의 표준 성공 처리를 그대로 쓴다 — 즉 성공 핸들러를 통해 원래 가려던 페이지로 리다이렉트한다. 반면 **프록시 티켓**은 사용자가 페이지를 "보러 온" 게 아니라 다른 서비스가 우리 API를 "호출"한 것이므로, 리다이렉트 없이 `SecurityContext`만 채우고 `chain.doFilter()`로 필터 체인을 계속 진행시켜 그 요청 자체를 처리하게 한다.

### 실패 핸들러 이중화

`setAuthenticationFailureHandler()`를 오버라이드해 내부 `CasAuthenticationFailureHandler`로 감싼 점도 눈여겨볼 만하다. 서비스 티켓 실패와 프록시 티켓 실패를 구분해 각각 다른 핸들러로 보낸다. 서비스 티켓 실패는 보통 로그인 실패 페이지로, 프록시 티켓 실패는 API 호출이므로 별도(`proxyFailureHandler`)로 처리하는 게 적절하기 때문이다.

## Gateway 모드 — 조용히 SSO만 확인하기

EntryPoint 방식은 "로그인 안 되어 있으면 무조건 로그인 화면을 보여준다." 그런데 "이미 CAS에 로그인되어 있으면 자동 로그인하고, 아니면 비로그인 상태 그대로 두고 싶은" 경우가 있다(예: 메인 페이지). 이를 위한 것이 `gateway` 파라미터이며, `CasGatewayAuthenticationRedirectFilter`가 담당한다.

`cas/src/main/java/org/springframework/security/cas/web/CasGatewayAuthenticationRedirectFilter.java`:

```java
public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) {
    if (!this.requestMatcher.matches(request)) {     // gateway 대상 아니면 통과
        chain.doFilter(request, response);
        return;
    }
    this.requestCache.saveRequest(request, response);          // 원래 요청 저장
    HttpSession session = request.getSession(true);
    session.setAttribute(CAS_GATEWAY_AUTHENTICATION_ATTR, true);// "gateway 시도함" 표시
    String urlEncodedService = WebUtils.constructServiceUrl(...);
    String redirectUrl = CommonUtils.constructRedirectUrl(this.casLoginUrl,
            ..., urlEncodedService, false, true);              // 마지막 true = gateway
    this.redirectStrategy.sendRedirect(request, response, redirectUrl);
}
```

`gateway=true`로 CAS에 보내면, CAS는 **로그인 화면을 띄우지 않고** 즉시 콜백 URL로 되돌려보낸다 — 이미 SSO 세션이 있으면 `ticket`을 붙여서, 없으면 티켓 없이. 클래스 주석에 나오듯 `gateway` 파라미터는 `renew`와 호환되지 않아, 이 필터는 `sendRenew`를 무시하고 항상 `false`로 보낸다.

티켓 없이 돌아온 경우는 앞서 본 `attemptAuthentication`의 (C) 경로에서 처리된다. 세션의 `CAS_GATEWAY_AUTHENTICATION_ATTR` 표시를 보고 "비로그인"으로 판정한 뒤 저장된 원래 요청으로 되돌린다.

### 무한루프 방지 — CasGatewayResolverRequestMatcher

gateway 리다이렉트가 매 요청마다 일어나면 CAS와 우리 앱 사이를 무한히 왕복하게 된다. `CasGatewayResolverRequestMatcher`가 이를 막는다.

```java
public boolean matches(HttpServletRequest request) {
    boolean wasGatewayed = this.gatewayStorage.hasGatewayedAlready(request, this.serviceProperties.getService());
    if (!wasGatewayed) {
        this.gatewayStorage.storeGatewayInformation(request, this.serviceProperties.getService());
        return true;     // 처음이면: 표시 남기고 리다이렉트 허용
    }
    return false;        // 이미 시도했으면: 다시 보내지 않음
}
```

CAS 클라이언트의 `GatewayResolver`(기본 `DefaultGatewayResolverImpl`)에 "이 세션은 이미 gateway 시도했음"을 기록해, 두 번째부터는 매칭하지 않는다.

## 정리

`CasAuthenticationEntryPoint`는 미인증 사용자를 CAS 로그인 URL(`{loginUrl}?service=...`)로 보내는 출구이고, `CasAuthenticationFilter`는 돌아온 `ticket`을 받아 `CasServiceTicketAuthenticationToken`으로 감싼 뒤 `AuthenticationManager`에 검증을 위임하는 입구다. 이 필터는 서비스 티켓·프록시 티켓·프록시 리셉터의 세 요청을 구분 처리하며, 프록시 티켓은 리다이렉트 없이 체인을 계속 진행시킨다. gateway 필터는 로그인 강제 없이 SSO 세션 유무만 확인하는 변형 진입 방식이다. 실제 티켓 검증은 다음 장의 `CasAuthenticationProvider`가 맡는다.
