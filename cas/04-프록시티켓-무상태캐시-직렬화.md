# 04. 프록시 티켓 · 무상태 캐시 · 직렬화

## 무엇을 / 왜

앞 장들은 "사용자가 브라우저로 우리 앱에 로그인하는" 가장 흔한 시나리오였다. 이 장은 그보다 까다로운 세 가지 운영 주제를 다룬다.

- **프록시 티켓(Proxy Ticket)** — 사용자가 서비스 A에 로그인했는데, A가 사용자 신원으로 백엔드 서비스 B를 호출해야 할 때. A는 사용자의 비밀번호를 모르지만, CAS로부터 "B를 호출해도 되는" 프록시 티켓을 받아 쓴다.
- **무상태(stateless) 티켓 캐시** — 세션을 못 쓰는 호출자(원격 클라이언트 등)가 같은 티켓을 반복 제시할 때, 매번 CAS에 검증하지 않도록 결과를 캐싱한다.
- **직렬화** — `CasAuthenticationToken`을 세션 저장소나 분산 캐시에 (역)직렬화하기 위한 Jackson 지원.

## 프록시 티켓의 큰 그림

```
 사용자 ── 로그인 ──▶ 서비스 A (우리 앱, proxyCallbackUrl 등록)
                         │
            ┌────────────┴─────────────┐
            │ ① CAS가 A에게 PGT 전달     │   PGT = Proxy Granting Ticket
            │    (proxyReceptorUrl로)    │
            ▼                            │
   ProxyGrantingTicketStorage           │  PGT를 저장해 둠
            │                            │
            │ ② A가 B를 호출할 때         │
            │    PGT로 CAS에서 PT 발급    │   PT = Proxy Ticket
            ▼                            │
        서비스 B (역시 우리 cas 모듈)      │
            │ ③ PT가 임의 URL에 ticket=   │
            │    파라미터로 도착           │
            ▼                            │
   CasAuthenticationFilter              │  authenticateAllArtifacts=true 이면
     proxyTicketRequest() == true ──────┘  → stateless 토큰으로 검증
```

두 개의 티켓 종류가 등장한다. **PGT**(Proxy Granting Ticket)는 "A가 앞으로 프록시 티켓을 받아갈 수 있는 권리"이고, **PT**(Proxy Ticket)는 "A가 B를 한 번 호출하기 위한 일회용 티켓"이다.

### PGT 수신 — proxyReceptorUrl

A가 PGT를 받으려면, `CasAuthenticationFilter`에 `proxyReceptorUrl`(또는 `proxyReceptorMatcher`)과 `proxyGrantingTicketStorage`를 설정해야 한다. 02장에서 본 `attemptAuthentication`의 첫 분기가 이것이다.

```java
if (proxyReceptorRequest(request)) {
    WebUtils.readAndRespondToProxyReceptorRequest(request, response, this.proxyGrantingTicketStorage);
    return null;
}
```

```java
private boolean proxyReceptorConfigured() {
    return this.proxyGrantingTicketStorage != null && this.proxyReceptorMatcher != null;
}
```

CAS 서버가 이 URL을 호출해 PGT를 전달하면, CAS 클라이언트(`WebUtils`)가 그것을 받아 `ProxyGrantingTicketStorage`에 저장한다. 이 스토리지를 `TicketValidator`와 공유하면, A는 저장된 PGT로 필요할 때 PT를 발급받을 수 있다. `null` 반환은 "이 요청은 PGT 수신용이었고 인증 흐름이 아니다"라는 신호다.

### PT 수신 — authenticateAllArtifacts

B 입장에서 들어오는 프록시 티켓은 `/login/cas` 같은 고정 URL이 아니라 **임의의 API URL**에 `ticket` 파라미터로 실려 온다. 그래서 `requiresAuthentication`이 이를 별도로 잡는다.

```java
private boolean proxyTicketRequest(boolean serviceTicketRequest, HttpServletRequest request) {
    if (serviceTicketRequest) {
        return false;
    }
    boolean result = this.authenticateAllArtifacts        // ServiceProperties로 켠 스위치
            && obtainArtifact(request) != null            // ticket 파라미터 존재
            && !authenticated();                          // 아직 인증 안 된 요청
    return result;
}
```

세 조건이 모두 참이어야 프록시 티켓으로 처리한다. `authenticateAllArtifacts`는 `ServiceProperties.setAuthenticateAllArtifacts(true)`로 켜고, 이 값은 `CasAuthenticationFilter.setServiceProperties()`에서 필터로 복사된다. 이미 인증된 요청(`authenticated()`)은 건드리지 않아, 정상 로그인된 사용자의 후속 요청을 방해하지 않는다.

### 동적 service URL — ServiceAuthenticationDetails

프록시 티켓의 service URL은 고정값(`ServiceProperties.getService()`)이 아니라 **티켓이 도착한 실제 요청 URL**이어야 한다. CAS는 PT를 발급할 때 특정 service에 묶어 발급하므로, 검증 시 그 URL을 정확히 되돌려줘야 한다. 이를 위해 03장에서 본 Provider의 `getServiceUrl()`이 details를 본다.

```java
private String getServiceUrl(Authentication authentication) {
    if (authentication.getDetails() instanceof ServiceAuthenticationDetails) {
        return ((ServiceAuthenticationDetails) authentication.getDetails()).getServiceUrl();
    }
    // 아니면 serviceProperties.getService() (고정)
}
```

details에 `ServiceAuthenticationDetails`를 심는 것이 `ServiceAuthenticationDetailsSource`다. `CasAuthenticationFilter`의 `authenticationDetailsSource`로 이를 설정하면, `attemptAuthentication`의 `buildDetails(request)`가 동적 URL을 가진 details를 만든다.

`DefaultServiceAuthenticationDetails`는 현재 요청 URL에서 **artifact(ticket) 파라미터만 제거**한 URL을 service로 삼는다.

```java
DefaultServiceAuthenticationDetails(String casService, HttpServletRequest request, Pattern artifactPattern) {
    super(request);
    URL casServiceUrl = new URL(casService);
    int port = getServicePort(casServiceUrl);
    final String query = getQueryString(request, artifactPattern);   // ticket=... 제거
    this.serviceUrl = UrlUtils.buildFullRequestUrl(casServiceUrl.getProtocol(),
            casServiceUrl.getHost(), port, request.getRequestURI(), query);
}
```

`artifactPattern`은 `&?ticket=[^&]*` 형태의 정규식으로, 쿼리스트링에서 티켓 파라미터를 지운다. 프로토콜/호스트/포트는 설정된 `casService`에서 가져오되 경로와 나머지 쿼리는 실제 요청에서 취해, "티켓을 뺀 원래 호출 URL"을 재구성한다. 이 값이 CAS가 PT 발급 때 묶었던 service와 일치해야 검증이 통과한다.

## 무상태 티켓 캐시 — StatelessTicketCache

`StatelessTicketCache`는 세션을 쓸 수 없는 호출자를 위한 장치다. 인터페이스 주석(`cas/src/main/java/.../authentication/StatelessTicketCache.java`)이 배경을 잘 설명한다. 티켓은 일회용이라 한 번 검증하면 다시 못 쓰는데, 상태형 호출자는 검증 결과를 `HttpSession`에 보관하면 된다. 하지만 원격 프로토콜 같은 무상태 호출자는 세션이 없어, 같은 티켓을 매 호출마다 제시한다. 이때 결과를 캐싱하지 않으면 CAS에 매번 새 티켓을 받아야 해 비싸다.

그래서 03장에서 본 Provider의 흐름은 무상태 토큰일 때만 캐시를 거친다.

```java
if (stateless) {
    result = this.statelessTicketCache.getByTicketId(ticket);   // 캐시 조회
}
if (result == null) {
    result = this.authenticateNow(authentication);              // 미스면 실제 검증
}
if (stateless) {
    this.statelessTicketCache.putTicketInCache(result);         // 캐시에 저장
}
```

구현은 셋이다.

- **`NullStatelessTicketCache`** — 기본값. 모든 메서드가 no-op이고 조회는 항상 `null`. 무상태 기능을 쓰지 않는 앱이 불필요한 캐시 의존성을 갖지 않도록 한 "캐싱 안 함" 구현이다.
- **`SpringCacheBasedTicketCache`** — Spring의 `Cache` 추상화 위에 동작. 티켓 문자열(`token.getCredentials()`)을 키로 `CasAuthenticationToken`을 저장한다. 실제 캐시 제공자(Caffeine, Redis 등)는 Spring Cache 설정에 따른다.

```java
public void putTicketInCache(CasAuthenticationToken token) {
    String key = token.getCredentials().toString();   // 티켓 ID가 캐시 키
    this.cache.put(key, token);
}
```

## 직렬화 — jackson / jackson2 패키지

`CasAuthenticationToken`을 세션 저장소(분산 세션 등)나 무상태 캐시(Redis)에 넣으려면 (역)직렬화가 필요하다. 문제는 토큰이 CAS 클라이언트의 `Assertion`/`AttributePrincipal` 구현(`AssertionImpl`, `AttributePrincipalImpl`)을 품고 있어, Spring Security 외부 타입이라는 점이다. 이를 위해 Jackson Mixin과 Module을 제공한다.

```java
public void setupModule(SetupContext context) {
    SecurityJackson2Modules.enableDefaultTyping(context.getOwner());
    context.setMixInAnnotations(AssertionImpl.class, AssertionImplMixin.class);
    context.setMixInAnnotations(AttributePrincipalImpl.class, AttributePrincipalImplMixin.class);
    context.setMixInAnnotations(CasAuthenticationToken.class, CasAuthenticationTokenMixin.class);
}
```

Mixin은 이들 외부 타입에 Jackson 애너테이션을 "주입"해, 생성자/필드 매핑과 타입 정보 포함(default typing)을 가능하게 한다. 두 패키지로 나뉜 이유는 Jackson 버전 전환 때문이다.

- **`jackson` (`CasJacksonModule`)** — Jackson 3 기반. 현재 권장.
- **`jackson2` (`CasJackson2Module`)** — Jackson 2 기반. 클래스에 `@Deprecated(forRemoval = true)`가 붙어 있으며 주석에 "7.0부터 `jackson` 패키지로 대체"라고 명시된다.

실무에서는 직접 등록하기보다 `SecurityJackson2Modules.getModules(ClassLoader)`로 클래스패스의 모든 보안 모듈을 한꺼번에 등록하는 방식이 권장된다.

## 정리

프록시 티켓은 "한 서비스가 사용자 신원으로 다른 서비스를 호출"하는 시나리오를 위한 것으로, 우리 앱은 `proxyReceptorUrl`로 PGT를 받아 `ProxyGrantingTicketStorage`에 저장하고, 들어오는 PT는 `authenticateAllArtifacts`를 켜 임의 URL에서도 처리한다. 이때 service URL은 고정값이 아니라 `ServiceAuthenticationDetails`로 만든 동적 URL(티켓 파라미터를 뺀 실제 요청 URL)을 쓴다. 무상태 호출자는 `StatelessTicketCache`로 일회용 티켓의 검증 결과를 재사용하고, 이렇게 만들어진 `CasAuthenticationToken`은 Jackson Mixin/Module을 통해 외부 저장소에 (역)직렬화된다.
