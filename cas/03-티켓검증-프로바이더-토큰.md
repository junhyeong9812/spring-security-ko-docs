# 03. 티켓 검증 — Provider · 토큰 3종 · UserDetailsService

## 무엇을 / 왜

02장에서 `CasAuthenticationFilter`는 티켓을 `CasServiceTicketAuthenticationToken`에 담아 `AuthenticationManager`에 넘겼다. 그 토큰을 실제로 **검증**해 인증을 완성하는 것이 `CasAuthenticationProvider`다. 이 장은 (1) 검증 전·검증 후를 나타내는 토큰 3종, (2) Provider가 티켓을 CAS 서버에 검증하고 `Assertion`을 사용자로 번역하는 과정, (3) 그 번역을 담당하는 `UserDetailsService`를 본다.

## 핵심 타입 — 토큰 3종

CAS 인증은 세 단계를 거치며 각 단계의 상태를 서로 다른 토큰이 표현한다. 모두 `AbstractAuthenticationToken`을 상속한다.

```
검증 전           CasServiceTicketAuthenticationToken   (authenticated = false)
  │                 principal = "_cas_stateful_" 또는 "_cas_stateless_"  (식별 상수)
  │                 credentials = 티켓 문자열
  │
  │  Provider 내부에서 UserDetailsService를 부르기 위한 임시 토큰
  ├──────────────▶ CasAssertionAuthenticationToken
  │                 principal = assertion.getPrincipal().getName()
  │                 credentials = ticket(보통 "")
  │                 assertion = CAS 검증 결과
  ▼
검증 후           CasAuthenticationToken                (authenticated = true)
                    principal = UserDetails
                    credentials = 티켓 문자열
                    userDetails + assertion + keyHash
```

### CasServiceTicketAuthenticationToken — 검증 전 요청

`cas/src/main/java/org/springframework/security/cas/authentication/CasServiceTicketAuthenticationToken.java`. 이 토큰의 특이점은 principal이 사용자가 아니라 **상수**라는 것이다.

```java
static final String CAS_STATELESS_IDENTIFIER = "_cas_stateless_";
static final String CAS_STATEFUL_IDENTIFIER  = "_cas_stateful_";

public static CasServiceTicketAuthenticationToken stateful(Object credentials) {
    return new CasServiceTicketAuthenticationToken(CAS_STATEFUL_IDENTIFIER, credentials);
}
public static CasServiceTicketAuthenticationToken stateless(Object credentials) {
    return new CasServiceTicketAuthenticationToken(CAS_STATELESS_IDENTIFIER, credentials);
}
public boolean isStateless() {
    return CAS_STATELESS_IDENTIFIER.equals(this.identifier);
}
```

아직 누가 로그인했는지 모르므로 principal에 사용자명을 넣을 수 없다. 대신 "이건 CAS 서비스 티켓 요청이고 상태형/무상태형이다"라는 메타 정보만 담는다. `setAuthenticated(true)`를 호출하면 예외를 던져, 외부에서 임의로 신뢰 토큰을 만들지 못하게 막는다.

### CasAuthenticationToken — 검증 후 완료

`CasAuthenticationToken`은 검증이 끝난 인증을 표현한다. principal에 진짜 `UserDetails`가 들어가고, CAS가 돌려준 `Assertion`과 `keyHash`를 함께 보관한다.

```java
public CasAuthenticationToken(String key, Object principal, Object credentials,
        Collection<? extends GrantedAuthority> authorities, UserDetails userDetails, Assertion assertion) {
    this(extractKeyHash(key), principal, credentials, authorities, userDetails, assertion);
}
private static Integer extractKeyHash(String key) {
    Assert.hasLength(key, "key cannot be null or empty");
    return key.hashCode();
}
```

`keyHash`가 핵심 장치다. Provider는 설정된 `key` 문자열의 해시를 토큰에 박아둔다. 나중에 같은 토큰이 다시 인증 흐름을 탈 때(예: 세션에서 복원), Provider는 "이 키 해시가 내가 만든 것과 같은가"만 확인하고 티켓을 재검증하지 않는다. 티켓은 일회용이라 재검증할 수 없기 때문이다.

### CasAssertionAuthenticationToken — UserDetailsService 호출용

`Assertion`을 `UserDetailsService`에 넘기기 위한 얇은 임시 토큰이다. principal은 `assertion.getPrincipal().getName()`이고, `getAssertion()`으로 원본 `Assertion`을 노출한다.

## 동작 흐름 — Provider의 티켓 검증

`cas/src/main/java/org/springframework/security/cas/authentication/CasAuthenticationProvider.java`의 `authenticate()`가 전체를 지휘한다.

```
authenticate(authentication)
  │
  ├─ supports() 아니면 → null (다른 Provider에게)
  │
  ├─ 이미 CasAuthenticationToken 이면 (재인증)
  │     └─ keyHash 비교 → 맞으면 그대로 반환, 틀리면 BadCredentials
  │
  ├─ credentials(티켓) 없으면 → BadCredentials("noServiceTicket")
  │
  ├─ stateless && 캐시에 있으면 → 캐시된 CasAuthenticationToken 반환
  │
  └─ 없으면 authenticateNow(authentication):
        ① ticketValidator.validate(ticket, serviceUrl) → Assertion   (CAS 서버 호출)
        ② loadUserByAssertion(assertion) → UserDetails               (UserDetailsService)
        ③ userDetailsChecker.check(userDetails)                       (계정 상태 확인)
        ④ 권한 매핑 + FactorGrantedAuthority(CAS) 추가
        ⑤ new CasAuthenticationToken(key, userDetails, ticket, authorities, ...)
```

```java
public Authentication authenticate(Authentication authentication) throws AuthenticationException {
    if (!supports(authentication.getClass())) {
        return null;
    }
    if (authentication instanceof CasAuthenticationToken) {        // 재인증 경로
        if (this.key.hashCode() != ((CasAuthenticationToken) authentication).getKeyHash()) {
            throw new BadCredentialsException(...);                //   내가 만든 게 아니면 거부
        }
        return authentication;                                     //   재검증 없이 통과
    }
    if ((authentication.getCredentials() == null) || "".equals(authentication.getCredentials())) {
        throw new BadCredentialsException(...);                    // 티켓 없음
    }
    boolean stateless = (authentication instanceof CasServiceTicketAuthenticationToken token
            && token.isStateless());
    CasAuthenticationToken result = null;
    if (stateless) {                                               // 무상태: 캐시 먼저
        result = this.statelessTicketCache.getByTicketId(authentication.getCredentials().toString());
    }
    if (result == null) {
        result = this.authenticateNow(authentication);             // 실제 검증
        result.setDetails(authentication.getDetails());
    }
    if (stateless) {
        this.statelessTicketCache.putTicketInCache(result);        // 무상태: 캐시에 저장
    }
    return result;
}
```

### 핵심 메서드 — authenticateNow

```java
private CasAuthenticationToken authenticateNow(Authentication authentication) throws AuthenticationException {
    try {
        Object credentials = authentication.getCredentials();
        Assertion assertion = this.ticketValidator.validate(credentials.toString(), getServiceUrl(authentication));
        UserDetails userDetails = loadUserByAssertion(assertion);
        this.userDetailsChecker.check(userDetails);
        Collection<GrantedAuthority> authorities = new ArrayList<>(
                this.authoritiesMapper.mapAuthorities(userDetails.getAuthorities()));
        authorities.add(FactorGrantedAuthority.fromAuthority(AUTHORITY));      // CAS 인증 표식
        return new CasAuthenticationToken(this.key, userDetails, credentials, authorities, userDetails, assertion);
    }
    catch (TicketValidationException ex) {
        throw new BadCredentialsException(ex.getMessage(), ex);
    }
}
```

- **티켓 검증**: `ticketValidator.validate(ticket, serviceUrl)`가 CAS 서버에 HTTP로 티켓을 검증하고 `Assertion`을 받는다. `TicketValidator`는 CAS 클라이언트 라이브러리의 인터페이스이며, 프로토콜별 구현(`Cas20ServiceTicketValidator`, `Cas20ProxyTicketValidator`, SAML 등)을 주입한다. **검증의 실제 작업은 전적으로 이 객체의 몫**이고, Provider는 그 결과를 번역할 뿐이다.
- **serviceUrl 결정** — `getServiceUrl()`: details가 `ServiceAuthenticationDetails`면 거기서 동적 URL을 꺼내고(프록시 티켓 시나리오, → 04장), 아니면 `serviceProperties.getService()`(고정 URL)를 쓴다. 티켓 검증 시 service 값이 발급 때와 일치해야 CAS가 검증을 통과시킨다.
- **권한 부여**: `UserDetails`의 권한을 `GrantedAuthoritiesMapper`로 매핑한 뒤, `FactorGrantedAuthority.CAS_AUTHORITY`를 추가한다. 이 표식은 "이 인증은 CAS 요소로 이뤄졌다"는 표시로, 다요소(multi-factor) 인가 판단에 쓰일 수 있다.
- **예외 변환**: `TicketValidationException`(CAS 클라이언트 예외)을 Spring Security의 `BadCredentialsException`으로 바꿔, 표준 인증 실패 흐름에 합류시킨다.

### supports — 세 토큰을 받는다

```java
public boolean supports(Class<?> authentication) {
    return CasServiceTicketAuthenticationToken.class.isAssignableFrom(authentication)
        || CasAuthenticationToken.class.isAssignableFrom(authentication)
        || CasAssertionAuthenticationToken.class.isAssignableFrom(authentication);
}
```

검증 전 토큰(`CasServiceTicket...`)뿐 아니라 검증 후 토큰(`CasAuthenticationToken`, 재인증용)까지 받는다.

## Assertion → UserDetails — userdetails 패키지

검증 결과 `Assertion`을 Spring Security의 `UserDetails`로 바꾸는 일은 Provider 밖으로 위임된다.

```java
protected UserDetails loadUserByAssertion(Assertion assertion) {
    CasAssertionAuthenticationToken token = new CasAssertionAuthenticationToken(assertion, "");
    return this.authenticationUserDetailsService.loadUserDetails(token);
}
```

`authenticationUserDetailsService`는 `AuthenticationUserDetailsService<CasAssertionAuthenticationToken>` 타입이다. 두 가지 사용법이 있다.

- **`setUserDetailsService(UserDetailsService)`** — 평범한 `UserDetailsService`를 `UserDetailsByNameServiceWrapper`로 감싸 사용. 이 경우 Assertion의 principal 이름으로 로컬 DB/디렉토리에서 사용자를 조회한다.
- **`setAuthenticationUserDetailsService(...)`** — Assertion 자체에서 사용자를 구성. 대표 구현이 `userdetails` 패키지에 있다.

### AbstractCasAssertionUserDetailsService 와 속성 기반 권한

`AbstractCasAssertionUserDetailsService`는 토큰에서 `Assertion`을 꺼내 추상 메서드로 위임하는 템플릿이다.

```java
public final UserDetails loadUserDetails(CasAssertionAuthenticationToken token) {
    return loadUserDetails(token.getAssertion());     // Assertion만 넘김
}
protected abstract UserDetails loadUserDetails(Assertion assertion);
```

대표 구현 `GrantedAuthorityFromAssertionAttributesUserDetailsService`는 **로컬 DB 조회 없이** CAS가 Assertion에 실어 보낸 속성에서 권한을 직접 만든다.

```java
protected UserDetails loadUserDetails(Assertion assertion) {
    List<GrantedAuthority> grantedAuthorities = new ArrayList<>();
    for (String attribute : this.attributes) {
        Object value = assertion.getPrincipal().getAttributes().get(attribute);
        if (value != null) {
            if (value instanceof List) {
                for (Object o : (List<?>) value) grantedAuthorities.add(createSimpleGrantedAuthority(o));
            } else {
                grantedAuthorities.add(createSimpleGrantedAuthority(value));
            }
        }
    }
    return new User(assertion.getPrincipal().getName(), NON_EXISTENT_PASSWORD_VALUE, ...,  grantedAuthorities);
}
```

지정한 속성 이름들(예: `"role"`, `"memberOf"`)의 값을 모두 `GrantedAuthority`로 변환한다. 비밀번호 자리에는 `"NO_PASSWORD"` 더미를 넣는데, CAS가 인증을 책임지므로 로컬에 비밀번호를 둘 필요가 없기 때문이다. `convertToUpperCase`(기본 `true`)로 권한 문자열을 대문자화한다.

## 설계 포인트 / 확장점

- **검증 위임**: `TicketValidator`를 주입식으로 두어 CAS 프로토콜 버전(CAS 1.0/2.0/3.0, SAML, 프록시)을 갈아 끼울 수 있다. Provider 코드는 어떤 프로토콜이든 동일하다.
- **번역 위임**: Assertion→UserDetails를 `AuthenticationUserDetailsService`로 분리해, "로컬 DB로 사용자 조회" vs "Assertion 속성만으로 사용자 구성" 두 전략을 자유롭게 선택하게 한다. `loadUserByAssertion`은 `protected`라 서브클래싱으로 완전히 재정의할 수도 있다.
- **계정 상태 점검**: `userDetailsChecker`(기본 `AccountStatusUserDetailsChecker`)로 잠김/만료 등을 확인한다. 6.4부터 교체 가능.
- **key 무결성**: `keyHash`로 "이 토큰을 내가 만들었는가"를 검증해, 재인증 시 일회용 티켓을 다시 검증하지 않으면서도 위조 토큰을 거른다.

## 정리

`CasAuthenticationProvider`는 `CasServiceTicketAuthenticationToken`(티켓을 담은 검증 전 토큰)을 받아 `TicketValidator`로 CAS 서버에 티켓을 검증하고, 돌려받은 `Assertion`을 `UserDetailsService`로 `UserDetails`+권한으로 번역한 뒤 `CasAuthenticationToken`(검증 후 토큰)을 만든다. 토큰 3종은 인증의 "전·검증중·후" 상태를 분리해 표현하며, `keyHash`는 재인증 시 일회용 티켓을 재검증하지 않게 해주는 안전장치다. 무상태 호출자와 프록시 티켓은 다음 장에서 이어진다.
