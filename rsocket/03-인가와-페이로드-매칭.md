# 03. 인가와 페이로드 매칭 (Authorization & Matching)

## 무엇을 / 왜

인증으로 "누구인지"가 확정되면, 인가는 "이 사람이 이 요청을 해도 되는가"를 판단한다. HTTP에서는 URL 패턴으로 보호 범위를 나누지만, RSocket에는 URL이 없다. 대신 **라우트(route)** — 메시지의 목적지 매핑 경로 — 로 요청을 구분한다. 이 챕터는 라우트 등으로 요청을 가려내는 `PayloadExchangeMatcher`와, 매칭된 요청에 인가 규칙을 적용하는 `AuthorizationPayloadInterceptor`를 다룬다.
(`authorization/`, `util/matcher/`)

## 핵심 타입

```
util/matcher/
┌───────────────────────────────────────────────────────────────┐
│ PayloadExchangeMatcher        요청 선별 (HTTP RequestMatcher 대응)│
│   Mono<MatchResult> matches(PayloadExchange)                    │
│   MatchResult: isMatch() + getVariables() (경로 변수)            │
│        △ 구현                                                    │
│   ├─ RoutePayloadExchangeMatcher   라우트 패턴 매칭 + 변수 추출    │
│   └─ PayloadExchangeMatchers       setup()/anyRequest()/anyExchange()│
│ PayloadExchangeMatcherEntry<T>   (matcher, entry=T) 쌍           │
│ PayloadExchangeAuthorizationContext  exchange + 경로 변수         │
└───────────────────────────────────────────────────────────────┘

authorization/
┌───────────────────────────────────────────────────────────────┐
│ AuthorizationPayloadInterceptor  (PayloadInterceptor, Ordered)  │
│   Context의 인증을 꺼내 authorizationManager.verify              │
├───────────────────────────────────────────────────────────────┤
│ PayloadExchangeMatcherReactiveAuthorizationManager              │
│   (ReactiveAuthorizationManager<PayloadExchange>)               │
│   매처→하위 매니저 매핑 테이블을 순서대로 평가                      │
└───────────────────────────────────────────────────────────────┘
```

## 동작 흐름: 인가 인터셉터

`AuthorizationPayloadInterceptor.intercept()`는 짧지만 흐름이 분명하다.

```java
public Mono<Void> intercept(PayloadExchange exchange, PayloadInterceptorChain chain) {
    return ReactiveSecurityContextHolder.getContext()
        .mapNotNull(SecurityContext::getAuthentication)
        .switchIfEmpty(Mono.error(() -> new AuthenticationCredentialsNotFoundException(
            "An Authentication (possibly AnonymousAuthenticationToken) is required.")))
        .as((authentication) -> this.authorizationManager.verify(authentication, exchange))
        .then(chain.next(exchange));
}
```

```
intercept(exchange, chain)
      │
   1) ReactiveSecurityContextHolder.getContext()
      │   .getAuthentication()  — 02챕터에서 심은 인증을 읽음
      │
      ├─ 인증 없음 ──▶ AuthenticationCredentialsNotFoundException
      │                (익명 인터셉터가 없거나 누락된 경우)
      │
      └─ 인증 있음(익명 포함)
            │
         2) authorizationManager.verify(authentication, exchange)
            │   인가 결정. 거부 시 AccessDeniedException으로 onError
            │
            └─ 허용
                 │
              3) chain.next(exchange)   — 다음 인터셉터/실제 처리로 진행
```

`verify()`는 인가 매니저의 `authorize()`를 호출해 결과가 거부면 예외를 던지는 디폴트 메서드다. 거부되면 체인은 더 진행하지 않고 에러가 위로 전파되어 클라이언트는 처리 실패를 받는다.

## 라우트 매칭: PayloadExchangeMatcher

인가 규칙은 "어떤 요청에 어떤 규칙"이라는 매핑이다. 그 "어떤 요청"을 정하는 게 매처다.

### RoutePayloadExchangeMatcher

```java
public Mono<MatchResult> matches(PayloadExchange exchange) {
    Map<String, Object> metadata = this.metadataExtractor.extract(
        exchange.getPayload(), exchange.getMetadataMimeType());
    return Optional.ofNullable((String) metadata.get(MetadataExtractor.ROUTE_KEY))
        .map(this.routeMatcher::parseRoute)
        .map((route) -> this.routeMatcher.matchAndExtract(this.pattern, route))
        .map(MatchResult::match)
        .orElse(MatchResult.notMatch());
}
```

```
PayloadExchange
   │ MetadataExtractor.extract → 메타데이터에서 ROUTE_KEY(목적지 경로) 추출
   ▼
"users.{id}.delete" 같은 라우트 문자열
   │ RouteMatcher.parseRoute → Route 객체
   │ routeMatcher.matchAndExtract(pattern, route)
   │   pattern "users.{id}.delete" 와 대조
   ├─ 매칭 + 변수 {id=42} ──▶ MatchResult.match({id:42})
   └─ 라우트 없음/불일치   ──▶ MatchResult.notMatch()
```

라우트가 패턴과 맞으면 경로 변수(`{id}` → `42`)까지 추출해 `MatchResult`에 담는다. 이 변수들은 `PayloadExchangeAuthorizationContext`로 전달되어, 표현식 기반 인가(`#id == ...`)에서 쓰일 수 있다. `spring-messaging`의 `MetadataExtractor`/`RouteMatcher`에 의존하는 이유가 여기에 있다.

### PayloadExchangeMatchers (정적 팩토리)

```java
setup()       → exchange.getType() == SETUP 이면 매치
anyRequest()  → exchange.getType().isRequest() 이면 매치 (SETUP·PAYLOAD 제외)
anyExchange() → 항상 매치
```

`anyRequest()`가 `isRequest()` 플래그를 쓰므로(01챕터), SETUP 프레임과 채널 후속 PAYLOAD는 "요청"에서 빠진다. 즉 `anyRequest().authenticated()` 같은 규칙은 SETUP 자체에는 적용되지 않는다.

## 매핑 테이블: PayloadExchangeMatcherReactiveAuthorizationManager

여러 (매처 → 인가 매니저) 매핑을 순서대로 평가하는 합성 인가 매니저다. HTTP의 `authorizeHttpRequests {}` 매핑 테이블에 해당한다.

```java
public Mono<AuthorizationResult> authorize(Mono<Authentication> authentication, PayloadExchange exchange) {
    return Flux.fromIterable(this.mappings)
        .concatMap((mapping) -> mapping.getMatcher().matches(exchange)
            .filter(MatchResult::isMatch)
            .mapNotNull(MatchResult::getVariables)
            .flatMap((variables) -> mapping.getEntry()
                .authorize(authentication, new PayloadExchangeAuthorizationContext(exchange, variables))))
        .next()                                              // 첫 매칭만 사용
        .switchIfEmpty(Mono.fromCallable(() -> new AuthorizationDecision(false)));  // 매칭 없으면 거부
}
```

```
mappings (순서 있는 리스트)
   [ (matcher1 → manager1),
     (matcher2 → manager2),
     ... ]
        │  concatMap — 위에서부터 순서대로
        ▼
   matcher.matches(exchange)?
     ├─ 매치 ──▶ 해당 manager.authorize(auth, ctx(exchange, 변수)) 평가
     │            → 그 결과를 .next()로 채택하고 중단
     └─ 불일치 ──▶ 다음 매핑으로
        │
   어떤 매핑도 매치 안 됨
        ▼
   AuthorizationDecision(false)  — 기본 거부(deny by default)
```

핵심은 **첫 번째로 매칭된 매핑의 결정만 사용**(`concatMap` + `next()`)하고, 어디에도 매칭되지 않으면 **거부**한다는 점이다. 그래서 규칙 순서가 중요하고(좁은 규칙을 위에), 빠뜨린 라우트는 자동으로 막힌다. 빌더(`builder().add(entry).build()`)로 매핑을 누적해 구성하며, 이 매핑은 보통 `config` 모듈의 `RSocketSecurity` DSL이 채운다.

각 매핑의 하위 매니저는 `ReactiveAuthorizationManager<PayloadExchangeAuthorizationContext>` 타입이라, 코어의 `AuthorityReactiveAuthorizationManager`(역할 기반), `AuthenticatedReactiveAuthorizationManager`(인증 필요) 등을 그대로 재사용한다.

## 전체 인증→인가 시퀀스 (한 요청 기준)

```
요청 payload
   │
   ▼ PayloadInterceptorRSocket → ContextPayloadInterceptorChain
   │
   ├─ AuthenticationPayloadInterceptor
   │     (SETUP에서 이미 인증됨 → Context에 인증 존재)
   │
   ├─ AnonymousPayloadInterceptor
   │     인증 없으면 익명 토큰 채움 → Context 항상 인증 보유
   │
   └─ AuthorizationPayloadInterceptor
         Context 인증 꺼냄
         → PayloadExchangeMatcherReactiveAuthorizationManager
              라우트 매처로 규칙 선택 → 하위 매니저 verify
         ├─ 허용 ──▶ chain.next → 실제 메시지 처리
         └─ 거부 ──▶ AccessDeniedException 전파
```

## 설계 포인트 / 확장점

- **코어 계약 재사용**: `ReactiveAuthorizationManager<PayloadExchange>`를 그대로 구현하므로, 인가 판단 로직(역할/표현식/커스텀)을 HTTP·메서드 보안과 공유한다.
- **deny-by-default**: 매핑에 없는 라우트는 거부된다. 보안상 안전한 기본값.
- **경로 변수 노출**: `RoutePayloadExchangeMatcher`가 추출한 변수를 `PayloadExchangeAuthorizationContext`로 전달해, 표현식 인가에서 활용 가능.
- **매처 합성**: `setup()`으로 연결 수립 자체에 별도 규칙(예: SETUP 시 인증 강제)을, `anyRequest()`로 일반 요청에 규칙을 따로 줄 수 있다.

## 정리

인가는 Context에서 인증을 꺼내(없으면 즉시 에러) `ReactiveAuthorizationManager.verify`로 결정을 강제한다. 어떤 규칙을 적용할지는 `PayloadExchangeMatcher`가 라우트로 가려내고, `PayloadExchangeMatcherReactiveAuthorizationManager`가 매핑 테이블을 순서대로 평가해 첫 매칭의 결정을 쓰며, 매칭이 없으면 거부한다. RSocket의 라우트를 HTTP의 URL처럼 다루되, 인가 판단 자체는 코어의 리액티브 인가 계약을 그대로 빌려 쓰는 구조다.
