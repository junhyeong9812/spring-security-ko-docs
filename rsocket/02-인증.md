# 02. 인증 (Authentication)

## 무엇을 / 왜

RSocket에는 HTTP의 `Authorization` 헤더 같은 표준 헤더가 없다. 대신 자격증명을 **payload의 메타데이터**에 인코딩해 싣는다. 이 챕터는 그 메타데이터에서 자격증명을 꺼내 `Authentication`으로 바꾸고, `ReactiveAuthenticationManager`로 인증한 뒤, 결과를 `ReactiveSecurityContextHolder`(= Reactor Context)에 심는 인터셉터들을 다룬다. (`rsocket/src/main/java/org/springframework/security/rsocket/authentication/`)

인증은 보통 **SETUP 단계에서 한 번** 수행된다. 01챕터에서 봤듯 SETUP 체인이 만든 인증 Context가 연결 전체로 전파되기 때문이다.

## 핵심 타입

```
┌─────────────────────────────────────────────────────────────────┐
│ PayloadExchangeAuthenticationConverter   (변환 계약)               │
│   Mono<Authentication> convert(PayloadExchange)                   │
│        △                                                          │
│        │ 구현                                                      │
│   ┌────┴───────────────────────────────────────────────────┐    │
│   │ AuthenticationPayloadExchangeConverter (표준, since 5.3) │    │
│   │   composite metadata → Simple/Bearer 인증 추출           │    │
│   ├──────────────────────────────────────────────────────────┤   │
│   │ BasicAuthenticationPayloadExchangeConverter (@Deprecated)│    │
│   │ BearerPayloadExchangeConverter             (@Deprecated)│    │
│   └──────────────────────────────────────────────────────────┘   │
├─────────────────────────────────────────────────────────────────┤
│ AuthenticationPayloadInterceptor   (PayloadInterceptor, Ordered)  │
│   converter로 변환 → authenticationManager.authenticate           │
│   → 성공 시 Context에 인증 저장 후 chain.next                       │
├─────────────────────────────────────────────────────────────────┤
│ AnonymousPayloadInterceptor        (PayloadInterceptor, Ordered)  │
│   Context에 인증이 없으면 AnonymousAuthenticationToken 주입         │
└─────────────────────────────────────────────────────────────────┘
```

`AuthenticationPayloadInterceptor`는 `Ordered`를 구현하므로 체인 안에서의 위치를 `order`로 조절한다. 인증은 인가보다 먼저 와야 하므로 더 낮은 order를 갖는다.

## 동작 흐름: 자격증명 → 인증 → Context

`AuthenticationPayloadInterceptor.intercept()`는 네 단계의 리액티브 파이프라인이다.

```java
public Mono<Void> intercept(PayloadExchange exchange, PayloadInterceptorChain chain) {
    return this.authenticationConverter.convert(exchange)                    // 1
        .switchIfEmpty(chain.next(exchange).then(Mono.empty()))              // 2
        .flatMap((a) -> this.authenticationManager.authenticate(a))         // 3
        .flatMap((a) -> onAuthenticationSuccess(chain.next(exchange), a));  // 4
}
```

```
PayloadExchange (예: SETUP)
      │
   1) converter.convert(exchange)
      │   payload 메타데이터에서 자격증명 추출 시도
      │
      ├─ 비어있음(자격증명 없음) ──▶ 2) switchIfEmpty:
      │                              chain.next(exchange)로 그냥 통과
      │                              (인증 없이 다음 인터셉터로)
      │
      └─ Authentication(미인증) 있음
             │
          3) authenticationManager.authenticate(a)
             │   ReactiveAuthenticationManager가 검증
             │   (UsernamePassword면 비밀번호 대조, Bearer면 토큰 검증)
             │
             ├─ 실패 ──▶ onError (AuthenticationException 전파)
             │
             └─ 성공: 인증된 Authentication
                    │
                 4) onAuthenticationSuccess(chain.next(exchange), auth)
                    │   chain.next(...)의 Context에
                    │   ReactiveSecurityContextHolder.withAuthentication(auth) 기록
                    ▼
              이후 체인/요청은 이 인증을 Context에서 읽음
```

2단계의 `switchIfEmpty`가 설계의 핵심이다. **자격증명이 없으면 인증을 건너뛰고 그냥 다음으로 넘긴다.** 인증을 강제하는 책임은 인가 인터셉터(03챕터)에 있다. 즉 이 인터셉터는 "자격증명이 있으면 인증해서 심어준다"까지만 하고, "인증이 꼭 있어야 하는가"는 판단하지 않는다.

4단계 `onAuthenticationSuccess`는 한 줄이다.

```java
private Mono<Void> onAuthenticationSuccess(Mono<Void> payload, Authentication authentication) {
    return payload.contextWrite(ReactiveSecurityContextHolder.withAuthentication(authentication));
}
```

`chain.next(exchange)`가 만든 다운스트림에 인증을 담은 Context를 `contextWrite`로 덧씌운다. 01챕터의 `ContextPayloadInterceptorChain`이 체인 끝에서 이 Context를 캡처해, SETUP의 경우 연결 전체로 운반한다.

## 자격증명 추출: AuthenticationPayloadExchangeConverter

표준 컨버터는 RSocket의 **Authentication Extension**(composite metadata 안의 `message/x.rsocket.authentication.v0`)을 파싱한다.

```
payload.metadata (composite metadata)
      │
   MetadataExtractor.extract(payload, COMPOSITE_METADATA_MIME_TYPE)
      │   "authentication" 키로 인증 바이트 추출
      ▼
   byte[] authenticationMetadata
      │
   AuthMetadataCodec.isWellKnownAuthType?  ── 아니오 ──▶ null (인증 없음)
      │ 예
   readWellKnownAuthType()
      ├─ SIMPLE  ──▶ readUsername/readPassword
      │              → UsernamePasswordAuthenticationToken.unauthenticated(u, p)
      └─ BEARER  ──▶ readBearerTokenAsCharArray
                     → BearerTokenAuthenticationToken(token)
```

- `SIMPLE`이면 username/password를 읽어 **미인증** `UsernamePasswordAuthenticationToken`을 만든다. 실제 검증은 뒤의 `ReactiveAuthenticationManager`가 한다.
- `BEARER`이면 토큰을 읽어 `BearerTokenAuthenticationToken`(oauth2-resource-server 모듈 타입)을 만든다. 그래서 이 모듈이 oauth2-resource-server에 의존한다.
- 둘 다 아니거나 자격증명이 아예 없으면 `null`을 반환하고, 이는 `Mono.empty()`가 되어 위 흐름의 `switchIfEmpty`로 빠진다.

`BasicAuthenticationPayloadExchangeConverter`와 `BearerPayloadExchangeConverter`는 표준화 이전의 레거시 MIME 타입(`message/x.rsocket.authentication.basic.v0` 등)을 쓰며 모두 `@Deprecated`다. 신규 코드는 표준 `AuthenticationPayloadExchangeConverter`를 쓴다. (단, `AuthenticationPayloadInterceptor`의 **기본** 컨버터는 하위 호환을 위해 여전히 `BasicAuthenticationPayloadExchangeConverter`이며, `setAuthenticationConverter`로 교체한다.)

## 익명 처리: AnonymousPayloadInterceptor

인증이 끝난 뒤(또는 자격증명이 없어 인증을 건너뛴 뒤), Context에 인증이 비어 있으면 익명 토큰을 채운다. HTTP의 `AnonymousAuthenticationFilter`와 같은 역할이다.

```java
public Mono<Void> intercept(PayloadExchange exchange, PayloadInterceptorChain chain) {
    return ReactiveSecurityContextHolder.getContext().switchIfEmpty(Mono.defer(() -> {
        AnonymousAuthenticationToken authentication =
            new AnonymousAuthenticationToken(this.key, this.principal, this.authorities);
        return chain.next(exchange)
            .contextWrite(ReactiveSecurityContextHolder.withAuthentication(authentication))
            .then(Mono.empty());
    })).flatMap((securityContext) -> chain.next(exchange));
}
```

```
ReactiveSecurityContextHolder.getContext()
      ├─ 비어있음 ──▶ AnonymousAuthenticationToken(key, "anonymousUser",
      │                                            [ROLE_ANONYMOUS]) 생성
      │              chain.next(exchange) + 익명 인증 Context 기록
      │
      └─ 인증 있음 ──▶ 그대로 chain.next(exchange) (덮어쓰지 않음)
```

이 덕분에 인가 인터셉터는 항상 어떤 `Authentication`(최소한 익명)이 Context에 있다고 가정할 수 있다. 익명도 엄연한 인증이므로, "인증 없음" 에러는 익명 인터셉터가 빠졌거나 그 앞에서 끊겼을 때만 발생한다.

## 설계 포인트 / 확장점

- **변환과 검증의 분리**: payload→`Authentication` 변환(`Converter`)과 실제 인증(`ReactiveAuthenticationManager`)이 분리돼, 자격증명 표현 방식이 바뀌어도 매니저는 그대로 재사용된다.
- **컨버터 교체**: `setAuthenticationConverter()`로 커스텀 메타데이터 포맷을 지원할 수 있다.
- **인증 강제는 인가의 몫**: 인증 인터셉터는 자격증명이 없으면 통과시킨다. 보호 여부는 인가 규칙으로 결정한다 — 관심사 분리의 좋은 예다.
- **Context 기반 전파**: 모든 인증 결과는 `ReactiveSecurityContextHolder.withAuthentication`으로 Reactor Context에 담겨, 메서드 보안(`@PreAuthorize`)이나 컨트롤러에서 `@AuthenticationPrincipal`로 읽힌다.

## 정리

인증 인터셉터는 payload 메타데이터의 자격증명을 컨버터로 꺼내 `Authentication`을 만들고, `ReactiveAuthenticationManager`로 검증한 뒤 결과를 Reactor Context에 심는다. 자격증명이 없으면 검증을 건너뛰고, 그 뒤 익명 인터셉터가 빈 Context를 익명 토큰으로 채운다. 이렇게 확정된 인증은 SETUP의 경우 연결 전체로, 요청의 경우 해당 요청 처리 전체로 전파되어 다음 단계인 인가의 입력이 된다.
