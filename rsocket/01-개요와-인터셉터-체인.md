# 01. 개요와 인터셉터 체인

## 무엇을 / 왜

RSocket은 하나의 연결 위에서 여러 종류의 상호작용이 흐르는 양방향 프로토콜이다. 보안을 걸려면 "연결이 열릴 때"와 "그 위로 요청이 오갈 때"를 모두 가로채야 한다. 이 챕터는 그 가로채기 지점인 `SocketAcceptorInterceptor` 어댑터와, 실제 보안 로직을 담는 `PayloadInterceptor` 체인이 어떻게 맞물리는지를 다룬다.

핵심 아이디어는 단순하다. **HTTP의 서블릿 필터 체인을 RSocket 세계로 옮긴 것**이다. 다만 반환 타입이 `Mono<Void>`인 리액티브 체인이고, 가로채는 대상이 HTTP 요청이 아니라 `PayloadExchange`라는 점이 다르다.

```
   서블릿(HTTP)                       RSocket
   ───────────                        ───────
   Filter                       →     PayloadInterceptor
   FilterChain                  →     PayloadInterceptorChain
   HttpServletRequest           →     PayloadExchange
   SecurityFilterChainProxy     →     PayloadInterceptorRSocket
```

## 핵심 타입

`api` 패키지는 프로토콜에 의존하지 않는 순수 계약 4개로 이루어진다. (`rsocket/src/main/java/org/springframework/security/rsocket/api/`)

```
┌──────────────────────────────────────────────────────────────┐
│ PayloadExchange   한 번의 상호작용을 추상화                       │
│   getType()           → PayloadExchangeType                    │
│   getPayload()        → io.rsocket.Payload (data + metadata)   │
│   getDataMimeType()   → 데이터 MIME 타입                         │
│   getMetadataMimeType()→ 메타데이터 MIME 타입                    │
├──────────────────────────────────────────────────────────────┤
│ PayloadExchangeType   상호작용 종류 enum                         │
│   SETUP(isRequest=false)        연결 수립 프레임                  │
│   FIRE_AND_FORGET(true)         단방향 호출                      │
│   REQUEST_RESPONSE(true)        단일 요청-응답                    │
│   REQUEST_STREAM(true)          스트림 요청                       │
│   REQUEST_CHANNEL(true)         양방향 채널                       │
│   PAYLOAD(false)                채널/스트림의 후속 데이터          │
│   METADATA_PUSH(true)           메타데이터 푸시                   │
├──────────────────────────────────────────────────────────────┤
│ PayloadInterceptor    보안 단위                                  │
│   Mono<Void> intercept(PayloadExchange, PayloadInterceptorChain)│
├──────────────────────────────────────────────────────────────┤
│ PayloadInterceptorChain   다음으로의 위임                         │
│   Mono<Void> next(PayloadExchange)                             │
└──────────────────────────────────────────────────────────────┘
```

`PayloadExchangeType`의 `isRequest` 플래그가 중요하다. SETUP과 PAYLOAD(채널/스트림의 두 번째 이후 데이터)는 `isRequest=false`이고, 나머지 실제 "요청"들은 `true`다. 인가 매처 `PayloadExchangeMatchers.anyRequest()`가 이 플래그로 SETUP을 인가 대상에서 제외한다(03챕터).

## 진입점: SocketAcceptorInterceptor 어댑터들

RSocket 서버는 연결마다 `SocketAcceptor.accept(setup, sendingSocket)`로 `RSocket`을 만들어 준다. RSocket 라이브러리는 이 acceptor를 감쌀 수 있는 `SocketAcceptorInterceptor` 훅을 제공한다. 이 모듈은 그 훅에 보안을 끼워 넣는다.

```
core/SecuritySocketAcceptorInterceptor   ──감쌈──▶  PayloadSocketAcceptorInterceptor
   (보안을 지연 적용하는 얇은 래퍼)                      (실제로 인터셉터들을 꽂는 본체)
```

- `PayloadSocketAcceptorInterceptor`는 `List<PayloadInterceptor>`를 들고 있다가, `apply(socketAcceptor)`가 호출되면 원본 acceptor를 `PayloadSocketAcceptor`로 감싼다. SETUP 프레임에 data/metadata MIME 타입이 없을 때 쓸 기본값(`defaultDataMimeType`, 기본 metadata는 `message/x.rsocket.composite-metadata.v0`)도 여기서 설정한다.
- `SecuritySocketAcceptorInterceptor`는 또 한 겹 위의 래퍼다. `config` 모듈이 보안 적용을 지연(lazy)시키기 위해 위임 대상 `SocketAcceptorInterceptor`를 감싸는 용도로 쓴다.

## 동작 흐름 1: 연결 수립(SETUP)

클라이언트가 연결을 열면 `PayloadSocketAcceptor.accept()`가 호출된다. 여기서 SETUP 프레임 자체에 대해 한 번 체인을 돌리고, 그 결과 Context를 연결 전체에 심는다.

```
클라이언트 connect()
      │
      ▼
PayloadSocketAcceptor.accept(setup, sendingSocket)
      │  1) setup에서 data/metadata MIME 타입 파싱(없으면 기본값)
      │
      │  2) intercept(setup, ...) — SETUP 타입으로 체인 실행
      │     ┌──────────────────────────────────────────┐
      │     │ ContextPayloadInterceptorChain            │
      │     │   DefaultPayloadExchange(type=SETUP, ...) │
      │     │   인터셉터들 순서대로 intercept() 호출       │
      │     │   → 인증 인터셉터가 자격증명 검사            │
      │     │   → 인증 결과를 Reactor Context에 저장       │
      │     └──────────────────────────────────────────┘
      │           │ 체인이 캡처한 Context(ctx) 반환
      │           ▼
      │  3) delegate.accept(setup, sendingSocket)  (실제 앱 acceptor)
      │           │ 만들어진 RSocket을
      │           ▼
      │  4) new PayloadInterceptorRSocket(socket, interceptors, ..., ctx)
      │     로 감싸고 .contextWrite(ctx)
      ▼
가로채는 RSocket 반환 (이후 모든 요청이 여기를 통과)
```

`accept()`의 핵심은 SETUP 단계에서 인증을 끝내고 그 인증을 담은 `ctx`를 만들어, 이후 만들어지는 모든 요청용 `RSocket`(`PayloadInterceptorRSocket`)에 미리 주입한다는 점이다. 덕분에 개별 요청은 다시 인증할 필요 없이 SETUP 때 확정된 인증을 이어받는다.

## 동작 흐름 2: 개별 요청

SETUP을 통과해 만들어진 `PayloadInterceptorRSocket`은 `RSocketProxy`를 상속한다. RSocket의 다섯 가지 상호작용 메서드를 모두 오버라이드해, 실제 위임(`source`) 전에 체인을 한 번 돌린다.

```
요청 메서드 호출 (예: requestResponse(payload))
      │
      ▼
PayloadInterceptorRSocket.requestResponse(payload)
      │  intercept(REQUEST_RESPONSE, payload)
      │     ┌──────────────────────────────────────────┐
      │     │ ContextPayloadInterceptorChain (새 인스턴스)│
      │     │   DefaultPayloadExchange(REQUEST_RESPONSE) │
      │     │   인증 인터셉터(이미 ctx에 인증 있음)         │
      │     │   인가 인터셉터 → 라우트 매칭 → verify        │
      │     └──────────────────────────────────────────┘
      │           │ 통과 시 Context 반환, 실패 시 onError
      │           ▼
      │  source.requestResponse(payload).contextWrite(context)
      ▼
실제 응답 Mono<Payload>
```

각 메서드는 `intercept(type, payload)`로 그 상호작용 종류에 맞는 `PayloadExchangeType`을 만들어 체인을 돌린 뒤, 통과하면 원본 `source`의 같은 메서드로 위임한다. `requestChannel`은 특별하다. `switchOnFirst`로 **첫 payload만** REQUEST_CHANNEL로 가로채고, 두 번째부터의 payload는 `PAYLOAD` 타입으로 따로 가로챈다(인덱스 0은 그대로 통과, 그 외만 `intercept`).

## 핵심 메서드

### ContextPayloadInterceptorChain.next()

체인은 불변 연결 리스트로 미리 조립된다. 생성자가 `List<PayloadInterceptor>`를 받아 뒤에서 앞으로 순회하며 `(currentInterceptor, next)` 노드들을 만든다(`init`). 실행은 다음과 같다.

```java
public Mono<Void> next(PayloadExchange exchange) {
    return Mono.defer(() ->
        shouldIntercept()
            ? this.currentInterceptor.intercept(exchange, this.next)   // 현재 인터셉터 실행
            : Mono.deferContextual(Mono::just).cast(Context.class)
                  .doOnNext((c) -> this.context = c).then());          // 끝: Context 캡처
}
```

- `shouldIntercept()`가 참이면 현재 인터셉터를 호출하고 `this.next`를 체인으로 넘긴다. 인터셉터가 `chain.next(exchange)`를 부르면 다음 노드로 진행한다.
- 마지막 빈 노드에 닿으면 인터셉터를 부르는 대신 **현재 Reactor Context를 멤버 변수에 캡처**한다. 인증 인터셉터가 `contextWrite`로 심어둔 인증이 이 시점에 Context에 들어있다.

### getContext()의 의미

```java
Context getContext() {
    if (this.next == null) return this.context;
    return this.next.getContext();
}
```

체인의 맨 끝 노드까지 따라가 거기서 캡처한 Context를 돌려준다. 이게 왜 필요한가? Reactor의 Context는 다운스트림(구독자 방향)으로만 흐르고 업스트림으로는 자동 전파되지 않는다. SETUP 체인 안에서 만들어진 인증 Context를, 체인 바깥의 `PayloadSocketAcceptor.accept()`가 받아 이후 RSocket에 다시 심어주려면, 이렇게 명시적으로 끄집어내야 한다. 클래스 주석이 밝히듯 이 체인은 **스레드 안전하지 않으며 매 실행마다 새 인스턴스를 만든다**(`intercept()`가 매번 `new ContextPayloadInterceptorChain(...)`).

## 설계 포인트 / 확장점

- **단일 추상화로 통일**: 인증·인가·익명 모두 `PayloadInterceptor` 하나로 표현된다. 새 보안 관심사(예: 타임아웃, 감사 로깅)도 같은 인터페이스로 끼워 넣을 수 있고, `Ordered`로 순서를 조절한다.
- **SETUP과 요청의 분리**: 비싼 인증은 연결당 한 번(SETUP), 가벼운 인가는 요청마다 수행하도록 자연히 나뉜다.
- **RSocketProxy 위임**: `PayloadInterceptorRSocket`이 모든 상호작용 메서드를 오버라이드하므로, 보안을 빠뜨리는 상호작용 종류가 없다.
- **Reactor Context를 인증 운반체로 사용**: `ThreadLocal` 대신 Context를 써서, 비동기 경계를 넘어도 인증이 따라간다(`ReactiveSecurityContextHolder`).

## 정리

`rsocket` 모듈의 골격은 "RSocket 진입점(`SocketAcceptorInterceptor`)을 가로채 `PayloadInterceptor` 체인을 끼우는" 어댑터다. 연결 수립(SETUP) 때 인증을 끝내 Context에 심고, 그 Context를 이후 만들어지는 요청용 RSocket에 미리 주입한다. 개별 요청은 `RSocketProxy`가 모든 상호작용 메서드마다 같은 체인을 다시 돌려 인가를 적용한다. 다음 챕터부터 이 체인에 꽂히는 인증·인가 인터셉터의 내부를 본다.
